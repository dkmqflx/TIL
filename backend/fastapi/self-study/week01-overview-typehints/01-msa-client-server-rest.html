웹 백엔드의 큰 그림
클라이언트–서버 · API · REST · MSA

코드를 짜기 전에, "내가 만들 서버가 전체 시스템의 어디에 서 있는지"를 먼저 그려봅니다. 면접에서 가장 먼저 물어보는 지도(map)입니다.

“

백엔드는 요청을 받아 → 데이터를 처리하고 → 응답을 돌려주는 식당의 주방이다. REST API는 그 주방에 주문을 넣는 표준화된 메뉴판이고, MSA는 거대한 식당 하나를 여러 전문 매장으로 쪼갠 푸드코트다.

  • 클라이언트–서버 모델을 한 문장으로 설명할 수 있다
  • API와 REST API가 무엇이고 왜 필요한지 안다
  • HTTP 메서드 · 상태 코드 · 자원(Resource)의 관계를 이해한다
  • 모놀리식과 MSA의 차이, 그 속에서 FastAPI의 자리를 안다

1클라이언트와 서버 — 누가 무엇을 하나

웹은 결국 요청(Request)과 응답(Response)의 끝없는 주고받음입니다. 요청하는 쪽이 Client, 응답하는 쪽이 Server입니다.

🍽️ 비유 — 레스토랑

손님(클라이언트)은 직접 주방에 들어가 요리하지 않습니다. 웨이터(API)에게 메뉴판의 이름으로 주문하고, 주방(서버)이 요리해서 접시(응답)를 돌려받죠. 손님은 주방이 가스레인지를 쓰는지 인덕션을 쓰는지 몰라도 됩니다 — 그게 바로 관심사의 분리입니다.

2API — 시스템 사이의 약속된 창구

API = Application Programming Interface. 풀어 쓰면 "프로그램끼리 대화하기 위해 미리 약속해 둔 창구"입니다. 손님이 "그 빨간 거 하나요"가 아니라 메뉴판의 정확한 이름으로 주문하듯, 클라이언트는 서버가 정해둔 규칙(URL, 메서드, 데이터 형식)대로 요청해야 합니다.

💡 프론트 경험과 연결

이미 fetch('/api/todos') 같은 코드를 써봤다면, 당신은 평생 API의 소비자(consumer)였습니다. 백엔드 개발자가 된다는 건 이제 그 메뉴판을 직접 설계하고 주방을 만드는 쪽으로 넘어간다는 뜻입니다.

3REST API — 가장 널리 쓰이는 메뉴판 양식

API를 설계하는 방식에는 여러 스타일이 있지만, 웹에서 사실상 표준은 REST(REpresentational State Transfer)입니다. 핵심 아이디어는 단 하나 —

세상의 모든 것을 자원(Resource)으로 보고, 각 자원에 주소(URL)를 주고, 행동은 HTTP 메서드로 표현한다.

예를 들어 "할 일(todo)"이라는 자원을 다룬다면:

하고 싶은 일HTTP 메서드URL(주소)성공 상태코드
할 일 전체 조회GET/todos200 OK
특정 할 일 1건 조회GET/todos/42200 OK
새 할 일 생성POST/todos201 Created
할 일 수정PATCH/todos/42200 OK
할 일 삭제DELETE/todos/42204 No Content
📮 비유 — 같은 주소, 다른 행동

주소 /todos/42는 "42번 할 일"이라는 집 주소 하나입니다. 그 집에 대해 GET은 "안을 들여다봐", PATCH는 "내용을 고쳐", DELETE는 "허물어"입니다. 주소(명사)와 행동(동사)을 분리하는 것이 REST의 우아함입니다.

상태 코드는 응답의 한 줄 요약입니다. 자세한 건 Week 2에서 다루지만, 큰 분류만 기억해 두세요:

범위뜻대표 예시
2xx성공200 OK · 201 Created · 204 No Content
4xx클라이언트 잘못400 Bad Request · 401 인증 안 됨 · 404 Not Found
5xx서버 잘못500 Internal Server Error

4모놀리식 vs MSA — 주방을 몇 개 둘 것인가

서비스가 커지면 "서버 하나에 모든 기능을 다 넣을 것인가, 아니면 기능별로 쪼갤 것인가"라는 질문에 부딪힙니다.

🏬 비유 — 만능 식당 vs 푸드코트

모놀리식은 한식·중식·디저트를 한 주방에서 다 만드는 만능 식당입니다. 처음엔 단순하고 빠르지만, 디저트 담당이 실수로 불을 내면 식당 전체가 멈춥니다. MSA는 각 요리를 전문 매장으로 분리한 푸드코트 — 중식당이 문을 닫아도 디저트 가게는 멀쩡히 장사합니다.

그래서 FastAPI는 어디에 서 있나

FastAPI는 가볍고(경량) · 빠르고(비동기) · 타입 기반으로 명세가 자동 생성되는 프레임워크라, MSA의 개별 서비스를 만들기에 이상적입니다. 머신러닝 모델 서빙 API로 인기가 많은 것도 같은 이유죠. 이 강의에서 당신은 결국 이 "매장 하나"를 처음부터 끝까지 만들게 됩니다.

NestJS ↔ FastAPI

5NestJS를 알면 이미 절반은 안다

당신은 이미 NestJS로 서버를 만들어 봤습니다. 그렇다면 "라우트 하나에 핸들러 함수를 연결한다"는 핵심 멘탈 모델을 이미 갖고 있는 겁니다. 표현만 다를 뿐입니다.

Nest NestJS — 데코레이터 + 클래스
todos.controller.ts
@Controller('todos')
export class TodosController {
  @Get()
  findAll() {
    return this.service.findAll();
  }
}

클래스 + 메서드 데코레이터로 라우트를 선언.

Fast FastAPI — 함수 + 데코레이터
main.py
@app.get("/todos")
def find_all():
    return service.find_all()

함수 하나 + 경로 데코레이터면 끝. 더 적은 보일러플레이트.

🧭 핵심 대응표

Nest @Controller → FastAPI APIRouter (Week 3) · Nest @Get/@Post → @app.get/@app.post · Nest DTO + class-validator → Pydantic 모델 (Week 3) · Nest Provider/DI → FastAPI Depends (Week 6). 이름표만 바꿔 끼우면 됩니다.

6⚠️ 자주 빠지는 함정

⚠️ 동사를 URL에 넣지 말 것

POST /createTodo, GET /getTodoById는 REST 위반입니다. URL은 명사(자원), 동작은 메서드가 원칙 — POST /todos, GET /todos/42처럼 쓰세요. RPC 스타일과 REST를 섞으면 면접관이 바로 알아챕니다.

⚠️ "MSA가 항상 정답"이라는 착각

MSA는 공짜가 아닙니다 — 네트워크 통신, 분산 트랜잭션, 운영 복잡도가 따라옵니다. 작은 서비스는 모놀리식으로 시작하는 게 보통 옳습니다. "왜 MSA를 쓰는가"만큼 "왜 안 쓰는가"도 말할 수 있어야 합니다.

7✅ 셀프 체크 — 면접관처럼 물어보기

Q 클라이언트와 서버를 한 문장으로 설명해 보세요.

요청을 보내고 응답을 받는 쪽이 클라이언트, 요청을 받아 처리하고 응답을 돌려주는 쪽이 서버입니다. 둘은 약속된 규칙(API)으로만 소통하며, 서로의 내부 구현을 몰라도 됩니다.

Q REST에서 PUT과 PATCH, POST의 차이는?

POST는 새 자원 생성(주소를 서버가 정함), PUT은 자원 전체 교체, PATCH는 자원의 일부만 수정입니다. 이 강의의 ToDo API는 부분 수정을 위해 주로 PATCH를 씁니다.

Q 작은 스타트업에 MSA를 권하겠습니까?

대개는 아니요. 초기에는 모놀리식이 개발·배포·디버깅이 단순해 속도가 빠릅니다. 트래픽·팀 규모가 커지고 특정 기능만 독립적으로 확장/배포해야 할 필요가 생길 때 MSA로 분리하는 것이 일반적인 진화 경로입니다.
