Python 타입 힌트
TypeScript를 안다면 거의 다 안다

FastAPI의 거의 모든 마법(검증·자동 문서·자동완성)은 타입 힌트에서 나옵니다. 다행히 당신은 이미 TypeScript로 타입을 써봤습니다. 그 직관을 그대로 옮겨봅시다.

“

TypeScript의 타입은 컴파일 후 사라지는 주석에 가깝다. 하지만 Python 타입 힌트는 실행 중에도 살아남아 FastAPI가 그것을 읽고 요청을 검증하고 문서를 만든다. 같은 문법, 더 강력한 쓰임.

  • 기본 타입·컬렉션·Optional·Union·Literal을 TS와 1:1로 매핑한다
  • 타입 힌트가 "왜 FastAPI에서 단순 주석이 아닌지" 이해한다
  • 함수 시그니처에 타입을 달고, 클래스로 구조화된 타입을 만든다

1왜 타입 힌트가 FastAPI의 심장인가

NestJS에서 DTO 클래스와 class-validator로 요청 본문을 검증했던 걸 떠올려 보세요. FastAPI는 그 역할을 타입 힌트 + Pydantic이 통째로 합니다. 함수 매개변수에 타입만 적으면:

  • 들어온 요청 데이터를 그 타입으로 자동 검증·변환
  • Swagger 문서를 자동 생성 (Week 1·03에서 직접 봅니다)
  • 에디터 자동완성·타입 체크가 작동
🏷️ 비유 — 택배 상자의 운송장

타입 힌트는 함수라는 택배 상자에 붙은 운송장입니다. "이 칸에는 정수(int)가 들어갑니다"라고 적어두면, FastAPI라는 택배 기사가 그 운송장을 읽고 내용물이 맞는지 검사합니다. TypeScript에서는 상자가 배송되기 전(컴파일) 운송장만 확인하고 떼버리지만, Python에서는 운송장이 배송 내내 붙어 있어 런타임에 검사가 가능합니다.

2기본 타입 — TS와 나란히 보기

TS TypeScript
types.ts
let name: string = "todo";
let count: number = 42;
let done: boolean = true;
Py Python
types.py
name: str = "todo"
count: int = 42
done: bool = True
TypeScriptPython메모
stringstr
numberint / floatPython은 정수와 실수를 구분!
booleanboolTrue/False 대문자

⚠️ number 하나가 int / float 둘로

TS의 number는 Python에서 int(정수)와 float(실수)로 나뉩니다. 가격·수량은 int, 평점·좌표는 float처럼 의미에 맞게 골라야 검증이 정확해집니다.

3컬렉션 — 배열과 객체

TS TypeScript
collections.ts
// 문자열 배열
let tags: string[];
// 키:문자열, 값:숫자
let scores: Record<string, number>;
Py Python 3.9+
collections.py
# 문자열 리스트
tags: list[str]
# 키:문자열, 값:정수
scores: dict[str, int]

TS의 T[]가 Python에선 list[T], Record<K,V>가 dict[K, V]입니다. 고정 길이 묶음은 tuple[int, str] (TS의 튜플 [number, string]과 동일).

4Optional — "있을 수도, 없을 수도"

TS에서 값이 없을 수 있음을 string | null 또는 ?로 표현했죠. Python도 똑같이 합니다.

TS TypeScript
optional.ts
let memo: string | null;
// 또는 선택적 프로퍼티
interface Todo {
  memo?: string;
}
Py Python
optional.py
# 3.10+ 권장 표기
memo: str | None = None

# 예전 표기 (동일 의미)
from typing import Optional
memo: Optional[str] = None
💡 Optional은 "필수 아님"이 아니다

흔한 오해: Optional[str]은 "값이 None일 수 있다"는 뜻이지, "이 필드를 생략해도 된다"는 뜻이 아닙니다. 생략 가능 여부는 기본값(= None)이 결정합니다. Week 3 Pydantic에서 이 둘의 차이가 중요해집니다.

5Union · Literal — 여러 타입, 정해진 값

TS TypeScript
union.ts
let id: number | string;
type Status =
  "todo" | "doing" | "done";
Py Python
union.py
id: int | str

from typing import Literal
Status = Literal["todo", "doing", "done"]

TS의 유니온 A | B는 Python에서도 A | B. TS의 문자열 리터럴 유니온은 Python의 Literal[...]로 대응됩니다 — 정해진 값만 허용하고 싶을 때 씁니다.

6함수와 구조화된 타입

함수 시그니처에 타입을 다는 법, 그리고 TS interface에 대응하는 구조화된 타입(미리보기)입니다.

TS TypeScript
shape.ts
function repeat(
  text: string, n: number
): string {
  return text.repeat(n);
}

interface Todo {
  id: number;
  title: string;
  done: boolean;
}
Py Python
shape.py
def repeat(
    text: str, n: int
) -> str:
    return text * n

from pydantic import BaseModel
class Todo(BaseModel):
    id: int
    title: str
    done: bool

반환 타입은 TS가 ): string, Python은 -> str로 적습니다. 그리고 TS의 interface는 FastAPI 세계에서 Pydantic BaseModel로 진화합니다 — 단순 모양 정의를 넘어 실제 검증까지 해주는 강화판이죠. Week 3에서 본격적으로 다룹니다.

7⚠️ 가장 중요한 함정 하나

⚠️ 순수 Python은 타입 힌트를 강제하지 않는다

이건 TS와 결정적으로 다른 점입니다. 아래 코드는 에러 없이 그냥 돌아갑니다.

gotcha.py
def add(a: int, b: int) -> int:
    return a + b

add("hello", "world")  # 💥 런타임 에러 안 남! → "helloworld"

타입 힌트는 그 자체로는 검사하지 않습니다. 검사를 해주는 건 ① 정적 분석 도구(mypy·에디터)와 ② FastAPI/Pydantic입니다. FastAPI가 특별한 이유가 바로 이것 — 런타임에 힌트를 읽어 실제로 검증해 주기 때문입니다.

8✅ 셀프 체크

Q TS 타입과 Python 타입 힌트의 가장 큰 차이는?

TS 타입은 컴파일 시점에만 존재하고 JS로 변환되며 사라집니다(타입 소거). Python 타입 힌트는 런타임에도 객체로 남아 있어 typing.get_type_hints() 등으로 읽을 수 있고, FastAPI/Pydantic이 이를 활용해 실제 검증을 수행합니다.

Q list[str]와 List[str]는 뭐가 다른가요?

의미는 같습니다. List는 typing에서 import해야 하는 예전 표기이고, Python 3.9부터는 내장 list[str]를 바로 쓸 수 있습니다. 새 코드는 소문자 내장 표기를 권장합니다.

Q memo: str | None과 memo: str | None = None의 차이는?

앞은 "None일 수 있는 필수 값", 뒤는 "기본값이 None이라 생략 가능한 값"입니다. 타입(None 허용 여부)과 필수성(기본값 존재 여부)은 별개입니다 — Pydantic 모델에서 자주 헷갈리는 지점입니다.
