요청의 모든 입구
Path · Query · Body · Header · Cookie · Form

클라이언트가 보내는 데이터는 요청의 여러 입구로 나뉘어 들어옵니다. 어떤 데이터를 어느 입구로 받을지 고르는 것이 API 설계의 첫 단추입니다.

“

택배 하나에도 주소(Path) · 포스트잇 메모(Query) · 상자 속 내용물(Body) · 운송 정보(Header) · 회원카드(Cookie)가 따로 있다. FastAPI는 함수 매개변수의 선언 방식만 보고 어느 입구인지 자동으로 알아챈다.

  • HTTP 요청의 구조와 6가지 데이터 입구를 구분한다
  • Path / Query / Body를 언제 쓰는지 판단 기준을 갖는다
  • 각 입구를 FastAPI에서 선언하는 법을 NestJS와 비교해 익힌다

1HTTP 요청의 해부도

먼저 "요청 하나가 실제로 어떻게 생겼는지"를 봅시다. curl -X POST가 만들어내는 날것의 HTTP 메시지입니다.

📦 비유 — 택배 상자

Path는 상자에 적힌 받는 주소(어떤 자원인가) · Query는 붙여둔 포스트잇 메모(어떻게 처리할까: 정렬·필터) · Body는 상자 속 실제 내용물(생성/수정할 데이터) · Header는 운송장의 메타 정보(형식·인증) · Cookie는 단골 회원카드입니다.

2Path 파라미터 — "어떤 자원인가"

URL 경로에 박혀 있는 값입니다. 특정 자원을 식별할 때 씁니다. 중괄호 {}로 선언하고, 같은 이름의 함수 매개변수로 받습니다. 타입 힌트를 달면 자동 검증됩니다.

Nest NestJS
todos.controller.ts
@Get(':id')
findOne(@Param('id') id: number) {
  return ...;
}
Fast FastAPI
main.py
@app.get("/todos/{id}")
def find_one(id: int):
    return ...

Path 파라미터는 항상 필수입니다 — URL의 일부이니 빠질 수 없죠. /todos/abc처럼 타입이 안 맞으면 422가 자동 응답됩니다.

3Query 파라미터 — "어떻게 처리할까"

URL ? 뒤의 key=value 쌍입니다. 주로 정렬·필터·페이지네이션처럼 "선택적 옵션"에 씁니다. 함수 매개변수가 경로에 없고 기본 타입이면 FastAPI가 자동으로 쿼리로 인식합니다.

main.py
# GET /todos?skip=0&limit=10&done=true
@app.get("/todos")
def list_todos(skip: int = 0, limit: int = 10, done: bool | None = None):
    # skip, limit → 기본값 있으니 선택적
    # done → None 기본값이라 생략 가능
    return {"skip": skip, "limit": limit, "done": done}
💡 필수 vs 선택은 "기본값"이 가른다

매개변수에 기본값이 있으면 선택적, 없으면 필수 쿼리가 됩니다. q: str → 필수, q: str = "x" → 선택. Path는 이와 무관하게 늘 필수라는 점과 대비됩니다.

4Request Body — "생성/수정할 데이터"

POST·PATCH로 보내는 JSON 본문입니다. FastAPI에서는 Pydantic 모델로 받습니다 (Week 3에서 깊게). 매개변수 타입이 Pydantic 모델이면 자동으로 "본문에서 읽어라"가 됩니다.

Nest NestJS — DTO
create-todo.dto.ts
class CreateTodoDto {
  title: string;
  done: boolean;
}
// controller
@Post()
create(@Body() dto: CreateTodoDto) {}
Fast FastAPI — Pydantic
main.py
class CreateTodo(BaseModel):
    title: str
    done: bool

@app.post("/todos")
def create(todo: CreateTodo):
    return todo

NestJS의 DTO + class-validator 조합을, FastAPI는 Pydantic 모델 하나로 선언과 검증을 동시에 해결합니다.

5Header · Cookie · Form — 나머지 입구

이들은 명시적으로 Header(), Cookie(), Form()을 써서 "이 매개변수는 이 입구에서 받아라"라고 알려줍니다.

main.py
from fastapi import Header, Cookie, Form

@app.get("/me")
def me(
    user_agent: str | None = Header(None),   # 헤더 User-Agent
    session: str | None = Cookie(None),      # 쿠키 session
):
    return {"ua": user_agent, "session": session}

@app.post("/login")
def login(username: str = Form(), password: str = Form()):
    # HTML form (application/x-www-form-urlencoded)
    return {"user": username}
⚠️ Header 이름 변환 규칙

Python 변수 user_agent는 HTTP 헤더 User-Agent에 자동 매핑됩니다 (언더스코어 → 하이픈, 대소문자 무시). 그리고 Form()을 쓰려면 uv add python-multipart가 필요합니다 — 안 깔면 에러가 납니다.

그럼 프론트에서는 어떻게 호출하나? (프론트 출신을 위한 보너스)

위 /me 엔드포인트는 헤더와 쿠키를 받습니다. 그런데 이 둘은 브라우저가 상당 부분 자동으로 처리해줘서, 프론트가 할 일은 케이스마다 다릅니다.

🪪 비유 — 입장할 때 자동으로 따라오는 것들

헤더(특히 User-Agent)와 쿠키는 클럽에 들어갈 때 몸에 이미 붙어 있는 신분증 같은 겁니다. 손님이 매번 "제 신분증이요!" 하고 꺼내지 않아도, 입구(브라우저)가 알아서 같이 들여보냅니다. 단, 다른 건물(다른 출처)로 갈 땐 "이 신분증도 같이 보내주세요"라고 명시해야 합니다.

① User-Agent 헤더 — 브라우저가 모든 요청에 자동으로 붙입니다. 프론트는 아무것도 안 해도 채워집니다.

client.js
// 이것만으로 충분 — User-Agent는 브라우저가 자동으로 넣어줌
fetch("http://127.0.0.1:8000/me");

// ❌ User-Agent는 '금지된 헤더'라 직접 덮어쓸 수 없다 (무시됨)
fetch("/me", { headers: { "User-Agent": "내맘대로" } });

만약 X-Client-Version 같은 커스텀 헤더였다면 그땐 직접 넣어야 합니다 (FastAPI: x_client_version: str | None = Header(None)):

client.js
fetch("/me", {
  headers: { "X-Client-Version": "1.2.3" },  // ← 직접 지정
});

② session 쿠키 — 브라우저가 자동으로 실어 보냅니다. 단 같은 출처냐 다른 출처냐가 갈림길입니다.

same 같은 출처
client.js
// 프론트·API가 같은 도메인:포트
fetch("/me");
// → 쿠키 자동 포함 ✅
cross 다른 출처
client.js
// 예: 프론트 :5173 ↔ API :8000
fetch("http://127.0.0.1:8000/me", {
  credentials: "include",  // ← 필수!
});

다른 출처일 때는 credentials: "include"(axios는 withCredentials: true)가 없으면 쿠키가 안 실립니다. 그리고 서버(FastAPI)의 CORS 설정도 쿠키를 허용해야 하는데, 이 부분은 Week 7 (CORS)에서 다룹니다.

받는 것프론트가 할 일
user_agent (User-Agent)아무것도 안 해도 됨 (브라우저 자동) · 직접 변경 불가
session 쿠키 — 같은 출처아무것도 안 해도 됨 (자동 전송)
session 쿠키 — 다른 출처credentials: "include" + 서버 CORS 허용
🔭 그 쿠키는 누가 만드나?

보통 프론트가 만드는 게 아니라, 로그인 시 서버가 Set-Cookie로 내려줍니다 (대개 HttpOnly라 JS로 못 읽어 더 안전). 이후 요청부터 브라우저가 자동으로 다시 실어 보내는 구조입니다. 이 세션/쿠키 인증 흐름 전체는 Week 8 (인증/인가)에서 직접 구현합니다.

인자 순서가 바뀌어도 될까?

결론부터: FastAPI 입장에선 순서가 전혀 상관없습니다. FastAPI는 인자의 위치가 아니라 이름 + Header()/Cookie()/타입 선언을 보고 어디서 값을 꺼낼지 정하기 때문입니다. user_agent와 session의 자리를 바꿔도 동작은 똑같습니다.

⚠️ 단, 파이썬 문법 규칙 하나 — "기본값 없는 인자가 먼저"

이건 FastAPI가 아니라 순수 파이썬 규칙입니다: 기본값이 있는 인자 뒤에 기본값 없는 인자가 올 수 없습니다. 위 예시는 Header(None)·Cookie(None) 둘 다 기본값이 있어 자유롭지만, 필수 인자와 섞이면 걸립니다.

main.py
# ❌ SyntaxError: non-default argument follows default argument
@app.get("/items/{item_id}")
def read(
    user_agent: str | None = Header(None),  # 기본값 있음
    item_id: int,                           # 기본값 없음 → 뒤에 못 옴!
): ...

# ✅ 필수 인자를 앞으로
@app.get("/items/{item_id}")
def read(
    item_id: int,                           # 필수 먼저
    user_agent: str | None = Header(None),  # 선택 나중
): ...
💡 추천 — Annotated를 쓰면 순서 고민이 사라진다

FastAPI 최신 권장 방식인 Annotated를 쓰면 Header()/Cookie() 메타데이터가 타입 쪽으로 들어가고 실제 기본값은 = None으로 분리됩니다. 그러면 위 문법 제약에서 자유로워지고, 특히 필수 쿼리/헤더를 받을 때 깔끔합니다.

잠깐 — 그래서 Annotated가 뭔데?

typing 모듈이 주는 도구로, 형태는 Annotated[진짜_타입, 부가정보...]입니다. 첫 번째는 진짜 타입(str·int…), 나머지는 그 타입에 붙이는 메타데이터입니다. 타입은 그대로 두고 "추가 메모"만 붙이는 거죠 — 타입 체커는 첫 번째 타입만 보고, FastAPI는 뒤의 메타데이터를 읽어 활용합니다.

🔖 비유 — 책에 붙인 포스트잇

책의 내용(타입)은 그대로입니다. 포스트잇(메타데이터)은 필요한 사람만 떼어 읽죠. Annotated[str, Header()]는 "이건 str인데, FastAPI 너는 이걸 헤더에서 꺼내"라는 포스트잇이 붙은 타입입니다.

📌 먼저 — 여기서 "출처"란?

출처 = 이 값을 요청(Request)의 어느 부분에서 꺼낼지를 뜻합니다. 이 강의 앞에서 본 요청의 입구들(Path · Query · Body · Header · Cookie) 중 어디서 값을 가져오느냐죠. 그래서 매개변수 하나에는 늘 두 정보가 필요합니다 — 타입(값의 모양, 예: str)과 출처(어디서 꺼낼지, 예: Query()). 같은 str이라도 Query()면 ?q=...에서, Header()면 헤더에서 값을 찾습니다.

사실 당신은 NestJS에서 이미 비슷한 패턴을 썼습니다 — 데코레이터로 파라미터에 "출처(어디서 꺼낼지)"를 붙였죠. Annotated는 정확히 같은 일을 합니다.

Nest NestJS — 데코레이터로 출처 표시
todos.controller.ts
// 타입(string)과 출처(@Query)가 분리
findAll(@Query('q') q: string) {}
Fast FastAPI — Annotated로 출처 표시
main.py
# 타입(str)과 출처(Query)가 분리
def find_all(q: Annotated[str, Query()]): ...

매개변수 하나에는 ① 타입 · ② 출처 · ③ 기본값 세 정보가 있는데, 옛 방식은 이걸 두 자리에 욱여넣고 Annotated는 세 자리에 나눠 담습니다:

main.py
# 옛 방식: '기본값 자리'(= 뒤)에 출처 정보를 욱여넣음
q: str = Query(default=None)
#  ↑타입        ↑'기본값 자리'인데 여기에 출처(Query)가 들어가 충돌·순서 제약

# Annotated: 세 정보가 각자 제자리에
q: Annotated[str, Query()] = None
#            ↑타입  ↑출처      ↑기본값  → 깔끔 + 재사용 + 테스트 쉬움

그래서 Annotated는 ① 순서 제약이 사라지고, ② 같은 Annotated[...]를 변수로 빼서 재사용할 수 있고, ③ 테스트에서 함수를 find_all("hello")처럼 직접 부를 때 기본값이 정상 동작합니다. Week 6의 Depends에서도 Annotated가 다시 등장합니다.

main.py
from typing import Annotated
from fastapi import Header, Cookie

@app.get("/me")
def me(
    user_agent: Annotated[str | None, Header()] = None,
    session: Annotated[str | None, Cookie()] = None,
):
    return {"ua": user_agent, "session": session}

옛 방식 q: str = Query(...)는 "기본값처럼" 취급돼 순서 제약에 걸리지만, q: Annotated[str, Query()]는 기본값이 없으니 어디 둬도 됩니다. 한 줄 요약: 순서는 의미를 바꾸지 않되, "필수 → 선택" 순으로만 적으면 되고, Annotated를 쓰면 그 고민조차 거의 사라집니다.

조회용 GET 요청에 본문(Body)을 싣는 건 피하세요 — 많은 프록시·캐시가 GET 본문을 무시하고, REST 관례에도 어긋납니다. 조회 조건은 Query로 보내는 게 정석입니다.

복습

7✅ 셀프 체크

Q FastAPI는 매개변수가 Path/Query/Body 중 무엇인지 어떻게 구분하나요?

① 경로 문자열 {}에 같은 이름이 있으면 Path, ② 없고 타입이 기본형(int·str 등)이면 Query, ③ 타입이 Pydantic 모델이면 Body로 해석합니다. Header/Cookie/Form은 명시적으로 해당 함수(Header() 등)로 선언합니다.

Q 검색어를 받는 /search는 Query, Body 중 무엇이 맞나요?

일반적으로 GET /search?q=...처럼 Query를 씁니다. 조회는 멱등·캐시 가능해야 하므로 GET + Query가 정석입니다. 다만 검색 조건이 매우 복잡하면(중첩 필터 등) 실무에서 예외적으로 POST + Body를 쓰기도 합니다.

Q limit: int = 10과 limit: int의 차이는?

앞은 기본값 10이 있어 생략 가능한 선택 쿼리, 뒤는 기본값이 없어 반드시 보내야 하는 필수 쿼리입니다. 빠뜨리면 422가 납니다.
