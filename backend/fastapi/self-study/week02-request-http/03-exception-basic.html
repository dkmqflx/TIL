예외 처리
HTTPException으로 정중하게 거절하기

좋은 API는 잘못된 요청에도 친절하고 일관된 에러를 돌려줍니다. 서버가 그냥 터져 500을 뱉는 것과, "42번 할 일은 없습니다(404)"라고 말해주는 것의 차이입니다.

“

처리할 수 없는 요청을 만나면 raise HTTPException(status_code, detail)로 즉시 멈추고 의미 있는 에러를 돌려준다. 그러면 FastAPI가 {"detail": ...} JSON으로 깔끔히 응답한다.

  • 왜 예외를 명시적으로 처리해야 하는지 이해한다
  • HTTPException으로 404·400 등을 발생시킨다
  • FastAPI의 기본 에러 응답 형태를 안다

1왜 예외 처리가 중요한가

조회한 자원이 없거나, 권한이 없거나, 비즈니스 규칙에 어긋나는 요청은 늘 들어옵니다. 이때 코드를 그냥 진행시키면 None에 접근하다 서버가 500을 뱉습니다. 클라이언트는 "왜 안 되는지" 알 수 없죠.

🍳 비유 — 재료가 떨어진 식당

주문한 메뉴의 재료가 없을 때, 좋은 식당은 "죄송합니다, 그 메뉴는 품절입니다"라고 정중히 안내합니다(404·400). 나쁜 식당은 아무 말 없이 주방에 불이 나죠(500). HTTPException은 "정중한 거절"을 코드로 표현하는 방법입니다.

2HTTPException — 멈추고 에러 돌려주기

가장 흔한 패턴: 자원을 못 찾으면 404를 발생시킵니다. raise 하는 순간 함수 실행이 멈추고 FastAPI가 에러 응답을 만들어 보냅니다.

main.py
from fastapi import FastAPI, HTTPException, status

app = FastAPI()
todos = {1: {"title": "장보기"}}

@app.get("/todos/{id}")
def read_todo(id: int):
    if id not in todos:
        raise HTTPException(
            status_code=status.HTTP_404_NOT_FOUND,
            detail=f"id={id} 할 일을 찾을 수 없습니다",
        )
    return todos[id]

존재하지 않는 /todos/99를 요청하면 이런 응답이 돌아갑니다:

response · 404
# HTTP/1.1 404 Not Found
{
  "detail": "id=99 할 일을 찾을 수 없습니다"
}
💡 return이 아니라 raise

HTTPException은 반환하는 게 아니라 던집니다(raise). 던지는 순간 그 아래 코드는 실행되지 않으니, NestJS의 throw new NotFoundException()처럼 "조기 종료" 흐름을 만들 수 있습니다.

3NestJS와 비교

Nest NestJS — 예외 클래스
todos.service.ts
import { NotFoundException }
  from '@nestjs/common';

if (!todo) {
  throw new NotFoundException(
    `id=${id} 없음`
  );
}

상태별 예외 클래스가 준비돼 있음.

Fast FastAPI — 하나로 통일
main.py
from fastapi import HTTPException

if id not in todos:
    raise HTTPException(
        404, f"id={id} 없음"
    )

HTTPException 하나에 코드만 바꿔 쓴다.

NestJS가 상태마다 클래스(BadRequestException, ForbiddenException …)를 두는 반면, FastAPI는 HTTPException 하나에 status_code를 바꿔 끼우는 단순한 방식입니다.

4헤더가 필요한 에러 — 401 예시

인증 실패(401)는 보통 WWW-Authenticate 헤더를 함께 요구합니다. headers 인자로 줄 수 있습니다.

main.py
raise HTTPException(
    status_code=status.HTTP_401_UNAUTHORIZED,
    detail="인증이 필요합니다",
    headers={"WWW-Authenticate": "Bearer"},  # Week 8 인증에서 재등장
)

5검증 에러(422)는 자동

타입·필수 검증 실패는 직접 raise 하지 않아도 FastAPI가 자동으로 422를 만들어줍니다. 어떤 필드가 왜 틀렸는지까지 상세히 담깁니다.

response · 422 (자동 생성)
{
  "detail": [
    {
      "type": "int_parsing",
      "loc": ["path", "id"],
      "msg": "Input should be a valid integer",
      "input": "abc"
    }
  ]
}
🔭 다음 단계 미리보기

지금은 라우트마다 raise를 쓰지만, 같은 에러 모양을 앱 전역에서 일관되게 만들고 싶을 때가 옵니다. 그건 Week 6의 커스텀 Exception Handler에서 다룹니다 — NestJS의 ExceptionFilter에 대응합니다.

⚠️ 에러 메시지에 민감 정보 금지

detail에 DB 쿼리문, 내부 경로, 스택 트레이스 같은 걸 노출하지 마세요. 공격자에게 힌트를 줍니다. 사용자에겐 "무엇이 잘못됐는지"만 알려주고, 자세한 원인은 서버 로그에 남기세요.

복습

6✅ 셀프 체크

Q HTTPException은 return인가요 raise인가요? 왜 중요한가요?

raise입니다. 던지면 그 즉시 함수 실행이 중단되고 이후 코드가 돌지 않습니다. 덕분에 "없으면 404 던지고 끝, 있으면 정상 처리"라는 명확한 조기 종료 흐름을 만들 수 있습니다.

Q 검증 실패(422)와 직접 던지는 404의 차이는?

422는 요청 데이터의 타입·필수 검증 실패로 FastAPI가 자동 생성합니다(함수 본문 실행 전). 404는 "형식은 맞지만 해당 자원이 없다"처럼 비즈니스 상황이라 개발자가 직접 raise HTTPException(404) 해야 합니다.

Q 기본 에러 응답의 JSON 구조는?

{"detail": ...}입니다. detail에는 문자열(HTTPException) 또는 검증 에러 배열(422)이 담깁니다. 전역에서 이 모양을 바꾸려면 Week 6의 커스텀 Exception Handler를 씁니다.
