Pydantic 모델 & 검증
FastAPI를 떠받치는 엔진

NestJS의 DTO + class-validator가 하던 일을, FastAPI에서는 Pydantic 모델 하나가 통째로 합니다. 선언과 검증이 한 곳에서 일어나죠.

“

Pydantic 모델은 요청 데이터의 입국 심사대다. 타입이 맞는지, 길이·범위 규칙을 지키는지 검사해서 통과시키거나(파싱) 돌려보낸다(422). 클래스에 타입만 적으면 검사기가 공짜로 생긴다.

  • BaseModel로 데이터 모델을 정의하고 자동 검증을 받는다
  • Field로 세부 제약(길이·범위·기본값)을 건다
  • 중첩 모델과 커스텀 validator를 작성한다
  • model_dump/model_validate로 직렬화·역직렬화한다

1Pydantic이란 — 타입 힌트로 하는 검증

Pydantic은 "타입 힌트를 읽어 실제로 데이터를 검증·변환"해주는 라이브러리입니다. Week 1에서 "타입 힌트는 런타임에 살아남는다"고 했죠? Pydantic이 바로 그 힌트를 런타임에 읽어 일하는 주인공입니다.

🛂 비유 — 공항 입국 심사대

해외에서 들어온 승객(요청 JSON)은 입국 심사대(Pydantic 모델)를 통과해야 합니다. 여권 형식이 맞는지(타입), 비자 기간이 유효한지(제약)를 검사해서, 통과하면 입국 도장(파싱된 객체)을 찍고, 문제가 있으면 정중히 돌려보냅니다(422). 심사대 덕분에 공항 안(=당신의 비즈니스 로직)은 늘 "검증된 데이터"만 돌아다닙니다.

2BaseModel 기초

BaseModel을 상속하고 필드에 타입 힌트를 적으면 끝입니다. NestJS DTO와 나란히 보죠.

Nest NestJS — DTO + 데코레이터
create-todo.dto.ts
import { IsString, IsBoolean }
  from 'class-validator';

class CreateTodoDto {
  @IsString()
  title: string;

  @IsBoolean()
  done: boolean;
}
Fast FastAPI — Pydantic
schemas.py
from pydantic import BaseModel

class CreateTodo(BaseModel):
    title: str
    done: bool = False

NestJS는 타입 위에 @IsString() 같은 데코레이터를 일일이 붙이지만, Pydantic은 타입 힌트 자체가 곧 검증 규칙입니다. title: str 한 줄이 "문자열이어야 함"을 이미 강제합니다.

3Field — 세부 제약 걸기

"문자열인데 1~50자", "0보다 큰 정수" 같은 세부 규칙은 Field로 겁니다. NestJS의 @MinLength(1), @Max(...)에 대응합니다.

schemas.py
from pydantic import BaseModel, Field

class CreateTodo(BaseModel):
    title: str = Field(min_length=1, max_length=50)
    priority: int = Field(default=1, ge=1, le=5)   # 1 이상 5 이하
    done: bool = False
Field 옵션의미NestJS 대응
min_length / max_length문자열 길이@MinLength / @MaxLength
gt / ge초과 / 이상@Min
lt / le미만 / 이하@Max
default기본값DTO 기본값
pattern정규식@Matches
💡 Field도 Annotated로 쓸 수 있다

Week 2에서 배운 Annotated 기억하시죠? Field도 title: Annotated[str, Field(min_length=1)]처럼 쓸 수 있고, 이게 모델 간 재사용에 유리한 최신 스타일입니다.

4중첩 모델 — 모델 안의 모델

실무 데이터는 평평하지 않습니다. "할 일에 작성자 정보가 들어 있다"처럼 객체 안에 객체가 옵니다. Pydantic 모델을 필드 타입으로 그냥 쓰면 됩니다.

schemas.py
class Author(BaseModel):
    name: str
    email: str

class Todo(BaseModel):
    title: str
    author: Author              # 중첩!
    tags: list[str] = []     # 리스트 필드

들어온 JSON의 author도 자동으로 Author 규칙으로 검증됩니다. 중첩 깊이에 상관없이 전부 검사되죠.

5커스텀 validator — 규칙을 직접 쓰기

"비밀번호는 숫자를 포함해야 한다" 같은 비즈니스 규칙은 @field_validator로 직접 작성합니다.

schemas.py
from pydantic import BaseModel, field_validator

class SignUp(BaseModel):
    username: str
    password: str

    @field_validator("password")
    @classmethod
    def check_password(cls, v: str) -> str:
        if not any(c.isdigit() for c in v):
            raise ValueError("비밀번호에 숫자가 1개 이상 필요합니다")
        return v

ValueError를 던지면 FastAPI가 자동으로 422 검증 에러로 바꿔줍니다. NestJS의 커스텀 @Validate 데코레이터와 같은 역할입니다.

6직렬화 — 모델 ↔ dict/JSON

메서드방향용도
model_validate(data)dict → 모델데이터를 검증하며 객체로
model_dump()모델 → dict저장·가공용 파이썬 dict
model_dump_json()모델 → JSON 문자열네트워크 전송용
📝 Pydantic v1 → v2 메서드 이름

예전 자료엔 .dict(), .parse_obj()가 나오는데, 이건 v1 표기입니다. v2에서는 .model_dump(), .model_validate()로 바뀌었습니다. FastAPI 최신 버전은 v2를 쓰니 새 이름을 익히세요.

⚠️ 가변 기본값(mutable default) 함정

리스트·딕셔너리 기본값은 tags: list[str] = []처럼 써도 Pydantic이 안전하게 처리하지만, 일반 파이썬 함수에서 def f(x=[])는 모든 호출이 같은 리스트를 공유하는 악명 높은 버그입니다. Pydantic 밖에서는 None 기본값 + 내부 초기화 패턴을 쓰세요.

복습

7✅ 셀프 체크

Q NestJS의 DTO+class-validator와 Pydantic의 차이를 한마디로?

NestJS는 타입(DTO)과 검증 규칙(class-validator 데코레이터)을 따로 선언하지만, Pydantic은 타입 힌트 자체가 검증 규칙이 되어 한 클래스에서 선언·검증이 동시에 일어납니다. 세부 제약만 Field로 보탭니다.

Q 커스텀 validator에서 검증 실패를 어떻게 알리나요?

@field_validator 함수 안에서 ValueError(또는 AssertionError)를 raise하면 됩니다. FastAPI가 이를 422 응답으로 자동 변환하고, 어떤 필드가 왜 실패했는지 detail에 담아줍니다.

Q .dict()가 동작 안 한다고 나옵니다. 왜죠?

Pydantic v2부터 .dict()는 deprecated이고 .model_dump()로 바뀌었습니다. 마찬가지로 .parse_obj() → .model_validate()입니다. 구버전 튜토리얼을 보고 있을 가능성이 큽니다.
