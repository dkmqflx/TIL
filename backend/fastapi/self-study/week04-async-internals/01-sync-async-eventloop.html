동기 · 비동기 · Event Loop
FastAPI가 빠른 진짜 이유

NestJS가 Node의 이벤트 루프 위에서 돌듯, FastAPI도 asyncio 이벤트 루프 위에서 돕니다. "스레드를 늘리지 않고도 동시에 많은 요청을 처리"하는 마법의 정체를 파헤칩니다.

“

비동기는 일을 더 빨리 하는 것이 아니라, 기다리는 시간을 놀지 않는 것이다. I/O(DB·네트워크)를 기다리는 동안 다른 요청을 처리한다. 그래서 I/O 바운드에서 빛나고, CPU 바운드엔 큰 효과가 없다.

  • 동기(blocking)와 비동기(non-blocking)의 차이를 설명한다
  • 동시성(concurrency)과 병렬성(parallelism)을 구분한다
  • 멀티프로세스·멀티스레드·비동기 세 방식의 장단점을 비교한다
  • coroutine·await·Event Loop가 협력하는 흐름을 그린다

1동기(blocking) vs 비동기(non-blocking)

동기 방식은 "한 작업이 완전히 끝날 때까지 다음 작업으로 못 넘어가는" 방식입니다. 비동기 방식은 "오래 걸리는 작업(특히 기다림)을 시작해두고, 끝나기를 기다리는 동안 다른 일을 먼저 하는" 방식이죠.

☕ 비유 — 카페 점원 한 명

동기 점원은 첫 손님의 라떼가 완성될 때까지 카운터에 서서 기다립니다. 에스프레소가 추출되는 30초 동안 뒷줄은 그냥 멈춰 있죠. 비동기 점원은 라떼 주문을 받아 머신에 걸어두고(=I/O 시작), 추출되는 동안 다음 손님 주문을 받습니다. 머신이 "삐—" 울리면(=I/O 완료) 그때 음료를 건네죠. 점원은 한 명(단일 스레드)인데도 줄이 훨씬 빨리 줄어듭니다. 비결은 "기다리는 시간에 다른 일을 하는 것"입니다.

여기서 핵심은 점원이 더 빨라진 게 아니라는 점입니다. 에스프레소 추출 시간 자체는 그대로입니다. 다만 그 대기 시간(I/O)을 놀리지 않고 다른 주문을 받는 데 썼을 뿐이죠. 비동기가 "일을 빨리 하는 게 아니라 기다림을 활용하는 것"이라는 말이 이 뜻입니다.

2동시성(concurrency) ≠ 병렬성(parallelism)

둘은 자주 헷갈리지만 전혀 다른 개념입니다. 한 줄로 구분하면 이렇습니다.

개념한 줄 정의카페 비유
동시성
(concurrency)		여러 작업을 번갈아 진행해 "동시에 진행 중"으로 보이게 함. 실제 일꾼은 한 명일 수 있음.점원 1명이 여러 주문을 왔다 갔다 처리
병렬성
(parallelism)		여러 작업을 물리적으로 같은 순간에 실행. 일꾼(코어)이 여러 명 필요.점원 4명이 각자 한 주문씩 동시에 처리

비동기(asyncio)는 동시성을 단일 스레드로 구현하는 기법입니다. 한 명이 빠르게 번갈아 일할 뿐, 실제로 두 코드가 같은 순간에 도는 건 아닙니다. 반면 병렬성은 코어가 여러 개 필요하고, 파이썬에선 멀티프로세스로 얻습니다(Lesson 3).

📝 파이썬의 GIL — 왜 스레드로는 병렬 CPU가 안 되나

CPython에는 GIL(Global Interpreter Lock)이 있어 한 순간에 파이썬 바이트코드를 실행하는 스레드는 하나뿐입니다. 그래서 멀티스레드를 써도 CPU 연산은 진짜 병렬이 되지 않죠. 다만 I/O 대기 중에는 GIL을 놓아주므로, I/O 작업의 동시 처리는 가능합니다. "CPU 병렬"이 필요하면 스레드가 아니라 프로세스를 늘려야 합니다.

세 방식은 배타적이지 않습니다. 실무 FastAPI 배포는 흔히 "비동기(I/O 동시성) + 멀티프로세스 워커(CPU 활용)"를 함께 씁니다. 이 조합은 Lesson 3에서 자세히 다룹니다.

4coroutine · await · Event Loop

이제 파이썬 비동기의 부품 세 개를 봅시다. NestJS의 async/await를 써봤다면 모양은 거의 똑같습니다.

부품역할
async def (coroutine)"중간에 멈췄다 재개할 수 있는 함수". 호출해도 바로 실행되지 않고 코루틴 객체를 만든다.
await"여기서 기다림이 생긴다. 끝날 때까지 이벤트 루프에 제어권을 양보한다"는 표시.
Event Loop준비된 코루틴을 골라 실행하고, await에서 양보하면 다른 코루틴으로 갈아타는 관제탑.
Nest Node.js — 익숙한 async/await
service.ts
async getUser(id: number) {
  // await에서 이벤트 루프에 양보
  const u = await db.find(id);
  return u;
}
Fast Python — 거의 동일한 모양
service.py
async def get_user(id: int):
    # await에서 이벤트 루프에 양보
    u = await db.find(id)
    return u

흐름은 이렇습니다. 코루틴이 await db.find(id)를 만나면 "DB 응답을 기다려야 하니 나는 잠깐 멈출게"라며 이벤트 루프에 제어권을 넘깁니다. 루프는 그 틈에 다른 요청의 코루틴을 실행하죠. DB 응답이 도착하면 루프가 원래 코루틴을 멈췄던 줄부터 재개합니다.

💡 await가 없는 async 함수는 의미가 거의 없다

이벤트 루프가 "다른 일을 할 틈"은 오직 await가 양보하는 순간에 생깁니다. async def로 선언만 하고 안에서 한 번도 await하지 않으면, 그 코루틴은 끝까지 루프를 독점합니다. 양보 지점(await)이 곧 동시성의 숨구멍입니다.

5비동기는 I/O 바운드에서 빛난다

마지막으로 가장 중요한 한 줄입니다. 비동기의 이득은 "기다림(I/O)"을 다른 일로 메우는 것에서 나옵니다.

작업 종류예시비동기 효과
I/O 바운드DB 쿼리, 외부 API 호출, 파일 읽기, 네트워크 대기매우 큼 — 대기 시간을 다른 요청 처리에 활용
CPU 바운드이미지 리사이즈, 암호화, 대규모 수치 계산거의 없음 — 양보할 "대기"가 없어 루프를 통째로 막음
⚠️ CPU 작업은 이벤트 루프를 통째로 멈춘다

무거운 계산(예: 큰 행렬 곱)을 async def 안에서 그냥 돌리면, 그 코드는 await 양보 없이 CPU를 붙잡습니다. 그 사이 모든 다른 요청이 멈춥니다. CPU 바운드는 비동기가 아니라 멀티프로세스(또는 워커 스레드 오프로딩)로 풀어야 합니다. 다음 Lesson에서 이 "블로킹 함정"을 정면으로 다룹니다.

복습

6✅ 셀프 체크

Q "비동기가 더 빠르다"는 말은 왜 부정확한가요?

개별 작업의 실행 시간 자체를 줄이는 게 아니기 때문입니다. 비동기는 I/O를 기다리는 동안 다른 작업을 처리해 전체 처리량(throughput)을 높입니다. 기다림이 없는 순수 CPU 작업은 비동기로도 빨라지지 않습니다.

Q 동시성과 병렬성의 차이를 한마디로?

동시성은 "여러 작업을 번갈아 진행해 동시에 진행 중처럼 보이게" 하는 것(일꾼 1명도 가능)이고, 병렬성은 "여러 작업을 같은 순간에 물리적으로 실행"하는 것(코어가 여러 개 필요)입니다. asyncio는 동시성, 멀티프로세스는 병렬성입니다.

Q 이벤트 루프는 코루틴을 언제 다른 것으로 갈아타나요?

코루틴이 await를 만나 제어권을 양보하는 순간입니다. 그 틈에 루프는 준비 큐의 다른 코루틴을 실행하고, 원래 코루틴의 I/O가 끝나면 멈췄던 지점부터 재개합니다. await가 없으면 양보가 일어나지 않아 동시성이 생기지 않습니다.
