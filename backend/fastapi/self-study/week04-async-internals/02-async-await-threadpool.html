async def vs def
그리고 블로킹의 함정

FastAPI는 async def와 일반 def 경로 함수를 서로 다른 방식으로 실행합니다. 이 차이를 모르면 서버 전체가 멈추는 가장 흔한 사고를 냅니다.

“

async def는 이벤트 루프에서, 일반 def는 자동으로 threadpool에서 돈다. 가장 큰 함정은 async def 안에서 블로킹 코드(time.sleep, 동기 DB, requests)를 부르는 것 — 이벤트 루프 전체가 멈춰 모든 요청이 느려진다.

  • FastAPI가 async def와 def를 어떻게 다르게 실행하는지 안다
  • async def 안 블로킹 호출이 왜 전체를 멈추는지 설명한다
  • 언제 async def, 언제 def를 쓸지 결정 가이드를 익힌다
  • 잘못된 예를 올바른 비동기·threadpool 패턴으로 고친다

1FastAPI의 두 갈래 실행 경로

FastAPI(정확히는 Starlette)는 경로 함수를 보고 두 가지 방식 중 하나로 실행합니다. 함수를 어떻게 선언했는지에 따라 자동으로 갈립니다.

선언실행 위치의미
async def이벤트 루프 (메인 스레드)코루틴으로 직접 실행. await로 양보하며 동시 처리.
def (일반 함수)threadpool (워커 스레드)이벤트 루프를 막지 않도록 별도 스레드로 자동 오프로딩.
🏢 비유 — 관제탑과 별관 작업실

이벤트 루프는 관제탑 한 명입니다. async def 작업은 관제탑이 직접 처리하되, 기다림이 생기면(await) 잠깐 다른 일을 봅니다. 그런데 누군가 관제탑에게 "여기서 5분만 가만히 서 계세요(블로킹)"라고 하면 관제탑이 멈춰 모든 항공기가 대기합니다. 그래서 FastAPI는 def로 선언된, 멈출 수 있는 동기 작업은 아예 별관 작업실(threadpool)로 보내 관제탑을 자유롭게 둡니다.

2⚠️ 가장 중요한 함정 — async 안의 블로킹

이 한 가지만 기억해도 절반은 성공입니다. async def 안에서 동기(블로킹) 코드를 직접 호출하면 이벤트 루프가 통째로 멈춥니다. 그 코루틴이 await 없이 시간을 잡아먹는 동안, 같은 워커가 처리하던 다른 모든 요청이 함께 멈춥니다.

대표적인 블로킹 범인들입니다.

블로킹 코드대신 써야 할 것
time.sleep(n)await asyncio.sleep(n)
requests.get(url) (동기 HTTP)await httpx.AsyncClient().get(url)
동기 DB 드라이버 (psycopg2 등)async 드라이버 (asyncpg, SQLAlchemy async)
무거운 CPU 연산run_in_threadpool 또는 멀티프로세스(Lesson 3)
bad.py — 절대 이렇게 쓰지 말 것
import time

@app.get("/bad")
async def bad():
    time.sleep(5)        # 😱 블로킹! 이벤트 루프가 5초간 정지
    return {"ok": True}  # 그 5초 동안 서버의 모든 요청이 멈춘다
good.py — 비동기로 양보
import asyncio

@app.get("/good")
async def good():
    await asyncio.sleep(5)  # ✅ 대기 동안 루프가 다른 요청 처리
    return {"ok": True}

둘 다 "5초 기다림"이지만 결과는 정반대입니다. time.sleep은 스레드를 붙잡고, asyncio.sleep은 이벤트 루프에 양보합니다. 양보 한 줄이 "한 요청만 5초"와 "모든 요청이 5초"를 가릅니다.

3그럼 동기 라이브러리만 있으면? — def로 두기

async 버전이 없는 라이브러리(예: 어떤 SDK, 동기 ORM)를 써야 할 때가 있습니다. 그땐 억지로 async def에 넣지 말고, 그냥 def로 선언하세요. FastAPI가 알아서 threadpool로 보내 이벤트 루프를 보호합니다.

Nest NestJS — 늘 async가 기본
cats.controller.ts
@Get()
async findAll() {
  return this.svc.findAll();
}

Node엔 동기 I/O가 드물어 거의 async로 통일.

Fast FastAPI — 선택이 의미를 가짐
main.py
# 동기 라이브러리 → def로!
@app.get("/report")
def report():
    return sync_orm.query()

def면 threadpool로 자동 오프로딩.

⚠️ threadpool은 무한하지 않다 (기본 ~40 스레드)

"동기는 def로 두면 안전"은 맞지만 무제한은 아닙니다. FastAPI의 threadpool(AnyIO 기반)은 기본적으로 약 40개 스레드로 제한됩니다. 동시 요청이 이 수를 넘기면 나머지는 큐에서 대기하죠. 느린 동기 작업이 폭주하면 풀이 고갈되어 결국 지연이 생깁니다. 트래픽이 높은 I/O라면 진짜 비동기 라이브러리 + async def가 정답입니다.

4결정 가이드 — async def? def?

실무에서 헷갈릴 때 이 표 하나로 정리됩니다.

상황선택이유
I/O인데 async 라이브러리가 있다 (httpx, asyncpg)async def + await이벤트 루프에서 최고 효율로 동시 처리
I/O인데 동기 라이브러리밖에 없다 (requests, psycopg2)defthreadpool이 루프를 막지 않게 오프로딩
가벼운 CPU 작업 (간단 계산)defthreadpool로 보내 루프 보호
무거운 CPU 작업 (대규모 연산)멀티프로세스 / 큐(Celery 등)GIL 때문에 스레드론 부족 → Lesson 3
💡 async 함수 안에서 동기 함수를 꼭 불러야 할 때

이미 async def 안인데 동기 함수를 한 번 호출해야 한다면, 직접 부르지 말고 run_in_threadpool로 감싸 워커 스레드로 보내세요. 이러면 그 동기 호출이 이벤트 루프를 막지 않습니다.

main.py
from fastapi.concurrency import run_in_threadpool

@app.get("/mixed")
async def mixed():
    # 동기 blocking 함수를 워커 스레드로 오프로딩
    result = await run_in_threadpool(sync_blocking_call, arg)
    return {"data": result}

5✅ 셀프 체크

Q async def 경로와 일반 def 경로는 어디서 실행되나요?

async def는 메인 이벤트 루프에서 코루틴으로 직접 실행됩니다. 일반 def는 FastAPI가 자동으로 threadpool(워커 스레드)로 보내 이벤트 루프를 막지 않게 합니다.

Q async def 안에 time.sleep(5)를 쓰면 무슨 일이 일어나죠?

time.sleep은 블로킹이라 await 양보 없이 이벤트 루프를 5초간 붙잡습니다. 그동안 같은 워커가 처리하던 모든 요청이 함께 멈춥니다. await asyncio.sleep(5)로 바꾸면 대기 중 루프가 다른 요청을 처리합니다.

Q "동기 라이브러리는 그냥 def로 두면 항상 안전"한가요?

대체로 안전하지만 무제한은 아닙니다. threadpool은 기본 약 40개 스레드로 제한되어, 느린 동기 작업이 폭주하면 풀이 고갈되고 나머지 요청이 큐에서 대기합니다. 트래픽이 큰 I/O는 async 라이브러리 + async def가 더 낫습니다.
