◆ Week 4 — Lesson 03

멀티프로세스 & 3층 구조
Uvicorn · Starlette · FastAPI

비동기는 한 코어 안의 동시성을 줍니다. 여러 코어를 쓰려면 멀티프로세스(워커)가 필요하죠. 그리고 그 위에 올라탄 3층 구조가 FastAPI 요청 한 건을 처리합니다.

“

비동기(I/O 동시성)와 멀티프로세스(CPU 병렬성)는 다른 층위의 해결책이다. 실무는 둘을 합친다. 그리고 요청 한 건은 Uvicorn(서버) → Starlette(프레임워크) → FastAPI(검증·문서) 3층을 통과한다. FastAPI는 Starlette를 상속한다.

  • 단일 프로세스가 왜 보통 코어 하나만 쓰는지 안다
  • 멀티프로세스 워커로 CPU를 활용하는 방법을 익힌다
  • Uvicorn·Starlette·FastAPI의 역할 계층을 그린다
  • ASGI와 WSGI의 차이를 한 줄로 설명한다

1단일 프로세스 = 코어 한 개

Lesson 1~2에서 본 비동기 서버는 한 프로세스 · 한 이벤트 루프입니다. GIL 때문에 이 프로세스는 사실상 CPU 코어 하나만 제대로 활용합니다. I/O 대기를 메우는 데는 충분하지만, 8코어 머신에서 코어 7개가 놀고 있다면 낭비죠.

📝 비동기로는 CPU 병렬이 안 되는 이유

이벤트 루프는 단일 스레드입니다. await로 갈아타도 같은 순간엔 코드 한 줄만 실행됩니다. 동시성은 주지만 병렬성은 주지 않죠. 여러 코어에서 진짜 동시에 계산하려면 인터프리터(=프로세스)를 여러 개 띄워야 합니다.

2멀티프로세스 워커 — 코어 활용하기

해법은 같은 앱을 여러 프로세스(워커)로 띄워 들어오는 요청을 나눠 처리하는 것입니다. 각 워커는 자신의 이벤트 루프를 가진 독립 프로세스라, 서로 다른 코어에서 진짜 병렬로 돕니다.

terminal
# 방법 1) Uvicorn이 직접 워커 4개 기동
uvicorn main:app --workers 4

# 방법 2) Gunicorn이 프로세스를 관리하고, 각 워커는 Uvicorn
gunicorn main:app -k uvicorn.workers.UvicornWorker --workers 4

두 축을 헷갈리지 마세요. 비동기는 "한 워커 안에서 I/O 동시 처리", 멀티프로세스는 "워커를 늘려 CPU 병렬 처리"입니다. 서로 다른 문제를 푸는 다른 층위의 도구이고, 실무에선 보통 함께 씁니다 (예: async def 앱 + 워커 4개).

해결책해결하는 문제단위
비동기 (async/await)I/O 대기 중 다른 요청 처리 (동시성)코루틴 / 이벤트 루프
멀티프로세스 (--workers)여러 코어 동시 계산 (병렬성)프로세스(워커)
⚠️ 워커는 상태를 공유하지 않는다

워커는 독립된 프로세스라 메모리를 공유하지 않습니다. 전역 변수에 캐시·세션·카운터를 담아두면 워커마다 값이 따로 놀아 버그가 됩니다. 공유 상태는 Redis·DB 같은 외부 저장소에 두세요. (NestJS를 PM2 cluster로 여러 인스턴스 띄울 때와 같은 주의입니다.)

33층 구조 — Uvicorn · Starlette · FastAPI

요청 한 건이 처리되기까지, 사실 세 개의 라이브러리가 층을 이뤄 협력합니다. 각자의 역할이 또렷하게 나뉘어 있죠.

층정체역할
UvicornASGI 서버소켓 수신, HTTP 파싱, 이벤트 루프 구동. 네트워크 ↔ 파이썬 객체 변환.
Starlette웹 마이크로프레임워크라우팅, 미들웨어, 요청/응답 객체, WebSocket, 백그라운드 태스크.
FastAPIStarlette 위 확장타입 기반 검증(Pydantic), 의존성 주입, 자동 OpenAPI 문서.

중요한 사실: FastAPI는 Starlette를 상속합니다. 즉 FastAPI 앱은 그 자체가 Starlette 앱이고, 미들웨어·WebSocket 같은 기능은 전부 Starlette에서 물려받은 것입니다.

fastapi/applications.py (개념)
from starlette.applications import Starlette

class FastAPI(Starlette):   # ← Starlette를 그대로 상속
    # 여기에 검증·의존성·자동 문서 기능을 얹는다
    ...

4ASGI vs WSGI

Uvicorn을 "ASGI 서버"라고 불렀습니다. Django·Flask가 쓰던 옛 규격은 WSGI인데, 한 줄 차이는 이렇습니다.

규격특징대표 서버
WSGI동기 전용. 요청 1건 = 워커 1개가 끝까지 점유. WebSocket 불가.Gunicorn(기본), uWSGI
ASGIasync/await · WebSocket · 장기 연결 지원. 비동기 동시성 가능.Uvicorn, Hypercorn

한 줄로: ASGI는 WSGI에 비동기와 WebSocket을 더한 차세대 규격입니다. FastAPI가 비동기·WebSocket을 쓸 수 있는 건 ASGI 위에 서 있기 때문이죠.

🏨 비유 — 호텔로 보는 전체 구조

호텔 한 채를 프로세스(워커)라 합시다. 더 많은 손님을 받으려면 여러 동(멀티프로세스)을 짓습니다. 각 동의 전화 교환원이 외부 전화(HTTP)를 받아 내부 언어로 연결하는데, 이게 Uvicorn입니다. 건물의 골조와 복도·엘리베이터—손님을 객실로 안내하는 길(라우팅·미들웨어)—가 Starlette입니다. 그리고 각 객실의 인테리어와 컨시어지 서비스(타입 검증·의존성·자동 문서)가 FastAPI죠. 손님(요청)은 교환원 → 복도 → 객실 순으로 안내받습니다.

💡 Week 4 정리 — 두 축으로 기억하기

① 동시성 축: async/await + 이벤트 루프로 한 워커 안 I/O를 겹쳐 처리. ② 병렬성 축: 멀티프로세스 워커로 여러 코어를 활용. 그리고 이 모든 게 Uvicorn → Starlette → FastAPI 3층 위에서 돌아갑니다. 두 축과 3층, 이 그림이 머리에 있으면 성능 튜닝의 큰 그림이 보입니다.

복습

5✅ 셀프 체크

Q 비동기를 쓰는데도 워커를 여러 개 띄우는 이유는?

비동기는 한 코어 안의 I/O 동시성만 줍니다. GIL 때문에 단일 프로세스는 CPU 코어 하나만 제대로 씁니다. 여러 코어를 활용해 CPU를 병렬로 쓰고 처리량을 높이려면 멀티프로세스 워커(--workers N)가 필요합니다. 둘은 다른 층위의 해결책이라 함께 씁니다.

Q Uvicorn, Starlette, FastAPI의 역할을 한마디씩 구분하면?

Uvicorn은 ASGI 서버(소켓·HTTP 파싱·이벤트 루프), Starlette는 웹 프레임워크(라우팅·미들웨어·WebSocket), FastAPI는 그 위 확장(타입 검증·의존성·자동 문서)입니다. FastAPI는 Starlette를 상속하므로 그 자체가 Starlette 앱이기도 합니다.

Q ASGI가 WSGI와 다른 핵심 한 가지는?

ASGI는 async/await와 WebSocket·장기 연결을 지원한다는 점입니다. WSGI는 동기 전용이라 비동기 동시성이나 WebSocket을 다룰 수 없습니다. FastAPI가 비동기로 동작하는 토대가 바로 ASGI(그 위의 Uvicorn)입니다.
