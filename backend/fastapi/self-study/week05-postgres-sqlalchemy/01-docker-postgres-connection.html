Docker로 PostgreSQL 띄우기
그리고 DB에 연결하기

로컬에 DB를 설치하지 않고도, docker-compose.yml 한 장으로 PostgreSQL을 켜고 끕니다. 그리고 SQLAlchemy Engine으로 그 DB에 처음 연결해 봅니다.

“

DB는 창고, 드라이버(psycopg)는 창고 언어 통역사, SQLAlchemy는 주문을 대신 처리하는 비서다. Docker는 그 창고를 컴퓨터에 직접 짓지 않고 컨테이너 한 채로 빌려 쓰는 것. docker compose up -d 한 줄이면 창고가 선다.

  • 왜 DB를 Docker로 띄우는지 — 격리·재현성의 이점을 안다
  • docker-compose.yml로 PostgreSQL 16을 띄우고 끈다
  • DATABASE_URL 형식과 드라이버(psycopg)의 역할을 이해한다
  • SQLAlchemy create_engine으로 Engine을 만들어 연결한다

1왜 DB를 Docker로 띄우는가

"PostgreSQL을 내 맥에 직접 설치"하는 방식은 버전이 꼬이고, 다른 프로젝트와 충돌하고, 팀원마다 환경이 달라집니다. Docker는 이 문제를 격리(isolation)와 재현성(reproducibility)으로 해결합니다.

📦 비유 — 집 마당에 창고를 짓지 말고, 컨테이너를 빌려라

DB를 직접 설치하는 건 마당에 창고를 콘크리트로 짓는 것과 같습니다. 한 번 지으면 옮기기 힘들고, 헐 때도 흙바닥이 더럽혀집니다. Docker 컨테이너는 레고처럼 가져다 놓는 조립식 창고입니다. up으로 세우고 down으로 흔적 없이 치웁니다. "내 컴퓨터에선 됐는데?"가 사라지는 이유 — 모두가 같은 컨테이너 설계도(이미지)를 받기 때문입니다.

방식설치충돌팀 공유
로컬 직접 설치OS마다 다름버전·포트 충돌 잦음"내 환경에선…" 문제
Docker 컨테이너이미지 pull 한 번컨테이너 안에 격리compose 파일 1장 공유

2docker-compose.yml 작성

컨테이너 한 채를 띄우는 설계도입니다. 이미지·계정·포트·데이터 보관 위치를 선언합니다.

docker-compose.yml
services:
  db:
    image: postgres:16          # PostgreSQL 16 공식 이미지
    environment:
      POSTGRES_USER: todo
      POSTGRES_PASSWORD: todo_pw
      POSTGRES_DB: todo_db
    ports:
      - "5432:5432"          # 호스트 5432 → 컨테이너 5432
    volumes:
      - pgdata:/var/lib/postgresql/data  # 데이터 영속화

volumes:
  pgdata:
💡 volume이 없으면 데이터가 증발한다

컨테이너는 휘발성입니다. down으로 지우면 안에 쌓인 데이터도 같이 사라지죠. volumes로 데이터 폴더를 컨테이너 바깥의 named volume(pgdata)에 연결해 두면, 컨테이너를 지웠다 다시 띄워도 데이터가 살아남습니다. 진짜로 다 지우려면 docker compose down -v(볼륨까지)를 씁니다.

3띄우고 끄기

같은 폴더에서 명령 한 줄이면 됩니다.

terminal
# 백그라운드(-d, detached)로 DB 컨테이너 시작
docker compose up -d

# 잘 떴는지 확인
docker compose ps

# 종료(데이터는 volume에 보존)
docker compose down

# 데이터까지 완전 삭제
docker compose down -v

4DATABASE_URL — 연결 주소의 문법

이제 코드가 이 창고로 찾아갈 주소가 필요합니다. SQLAlchemy는 하나의 URL 문자열로 "어떤 DB·어떤 드라이버·누가·어디로" 전부를 표현합니다.

DATABASE_URL 형식
postgresql+psycopg://todo:todo_pw@localhost:5432/todo_db
└────┬───┘ └──┬──┘  └─┬─┘ └──┬──┘ └───┬───┘ └─┬┘ └──┬──┘
   DB종류  드라이버  유저   비번     호스트  포트  DB이름

postgresql은 DB 종류, +psycopg는 그 DB와 실제로 대화할 드라이버를 지정합니다. 이 둘을 합쳐 "PostgreSQL에 psycopg로 말 걸겠다"는 뜻이 됩니다.

🧩 psycopg(v3)란 — 동기 드라이버

driver psycopg는 파이썬 ↔ PostgreSQL 사이를 잇는 드라이버(통역사)입니다. SQLAlchemy가 만든 SQL을 받아 실제 네트워크로 DB에 보내고 결과를 받아옵니다. 여기서는 동기(sync) 버전을 씁니다 — 요청이 끝날 때까지 기다리는 평범한 함수 호출 방식이죠. (Week 4의 async와 짝을 이루는 async 드라이버 asyncpg는 나중에 다룹니다.)

5설치 — uv add

드라이버와 SQLAlchemy를 의존성으로 추가합니다.

terminal
uv add sqlalchemy psycopg
⚠️ 연결이 안 되면 psycopg[binary]를 의심하라

psycopg만 설치하면 시스템에 libpq(PostgreSQL C 라이브러리)가 있어야 빌드/실행됩니다. 깨끗한 로컬에서 바로 돌리고 싶다면 미리 컴파일된 uv add "psycopg[binary]"가 무난합니다. 서버에서 직접 빌드하는 운영 환경에서는 psycopg(소스)를 권장합니다.

6create_engine — Engine 만들기

주소가 준비됐으니 create_engine으로 Engine을 만듭니다. Engine은 "연결 공장 + 커넥션 풀"이며, 앱 전체에서 딱 하나만 만들어 재사용합니다.

database.py
import os
from sqlalchemy import create_engine

DATABASE_URL = os.getenv(
    "DATABASE_URL",
    "postgresql+psycopg://todo:todo_pw@localhost:5432/todo_db",
)

# 앱당 하나. 내부에 Connection Pool을 들고 있다.
engine = create_engine(DATABASE_URL, pool_pre_ping=True)
🗂️ 비유 — 창고·통역사·비서

그림으로 정리하면 이렇습니다. DB(창고)에는 물건(데이터)이 쌓여 있고, 당신은 창고 언어를 모릅니다. psycopg(통역사)가 창고 직원과 대화해 주고, SQLAlchemy(비서)는 당신의 파이썬 주문("Todo 다 가져와")을 통역사가 알아들을 SQL로 바꿔 전달합니다. 당신은 비서에게만 말하면 됩니다.

7✅ 셀프 체크

Q DB를 직접 설치하지 않고 Docker로 띄우는 핵심 이점은?

격리와 재현성입니다. 컨테이너 안에 DB가 갇혀 있어 로컬 환경과 충돌하지 않고, compose 파일 한 장만 공유하면 팀 전체가 동일한 버전·설정의 DB를 똑같이 띄울 수 있습니다. up/down으로 흔적 없이 켜고 끕니다.

Q postgresql+psycopg://...에서 +psycopg는 무슨 뜻인가요?

실제로 PostgreSQL과 대화할 드라이버를 지정합니다. postgresql은 DB 종류, psycopg는 통역사 역할의 드라이버입니다. 여기서는 동기 버전을 쓰며, SQLAlchemy가 만든 SQL을 받아 네트워크로 DB에 보내고 결과를 받아옵니다.

Q Engine은 요청마다 새로 만들어야 하나요?

아니요. Engine은 내부에 Connection Pool을 들고 있는 무거운 객체라 앱당 하나만 만들어 모듈 전역에서 재사용합니다. 요청마다 새로 빌려 쓰는 것은 Engine이 아니라 그 풀에서 나오는 연결/세션입니다(Lesson 3).
