SQLAlchemy 2.0 ORM 모델링
클래스로 테이블을 그리기

파이썬 클래스 하나가 테이블 하나가 됩니다. NestJS의 TypeORM @Entity가 하던 일을, SQLAlchemy 2.0의 DeclarativeBase + Mapped[]로 합니다.

“

ORM은 객체 ↔ 테이블 번역기다. class Todo(Base)가 todos 테이블이 되고, Todo(...) 인스턴스 하나가 한 행(row)이 된다. SQL을 손으로 쓰지 않고 파이썬 객체로 데이터를 다룬다. 2.0 스타일은 Mapped[] 타입 힌트로 컬럼을 선언한다.

  • ORM이 객체와 테이블을 어떻게 매핑하는지 이해한다
  • SQLAlchemy 2.0 스타일(DeclarativeBase · Mapped · mapped_column)로 모델을 짠다
  • 타입·nullable·default·index 등 컬럼 옵션을 건다
  • create_all로 테이블을 생성(DDL)한다

1ORM이란 — 객체와 테이블의 번역기

DB는 행과 열로 데이터를 저장하지만, 우리 코드는 객체로 생각합니다. 이 둘 사이의 간극(임피던스 불일치)을 메우는 게 ORM(Object-Relational Mapping)입니다.

🔤 비유 — 양식 작성 대행

raw SQL은 모든 서류를 손으로 직접 작성하는 것입니다. 빠르지만 오타(SQL 문법 오류)가 나기 쉽고, 양식이 바뀌면 전부 고쳐야 하죠. ORM은 "이름·완료여부만 알려주세요" 하고 양식을 대신 채워주는 대행 창구입니다. 당신은 파이썬 객체만 다루고, ORM이 INSERT INTO todos ... 같은 SQL로 번역합니다.

SQL 문자열을 직접 작성파이썬 객체로 표현
DB 종류 바뀌면 SQL 수정대부분 코드 그대로 이식

22.0 스타일 — DeclarativeBase & Mapped

SQLAlchemy 2.0은 타입 힌트 기반으로 모델을 선언합니다. DeclarativeBase를 상속한 Base를 만들고, 컬럼은 Mapped[타입] + mapped_column(...)로 적습니다. 타입 힌트가 곧 컬럼 타입이 됩니다.

models.py
from datetime import datetime
from sqlalchemy import String, func
from sqlalchemy.orm import (
    DeclarativeBase, Mapped, mapped_column,
)

class Base(DeclarativeBase):
    pass                              # 모든 모델의 부모. metadata가 모인다.

class Todo(Base):
    __tablename__ = "todos"

    id: Mapped[int] = mapped_column(primary_key=True)
    title: Mapped[str] = mapped_column(String(100), index=True)
    done: Mapped[bool] = mapped_column(default=False)
    created_at: Mapped[datetime] = mapped_column(
        server_default=func.now(),
    )
💡 Mapped[int] vs Mapped[int | None]

2.0 스타일에서 nullable 여부는 타입 힌트가 결정합니다. Mapped[str]는 NOT NULL, Mapped[str | None]은 NULL 허용입니다. NestJS에서 @Column({ nullable: true })를 옵션으로 주던 것을, 여기서는 타입에 | None을 붙이는 것만으로 표현합니다.

3컬럼 옵션 — 타입 · nullable · default · index

옵션의미예
primary_key=True기본키(PK)id
Mapped[T] / Mapped[T | None]NOT NULL / NULL 허용title / 선택 필드
default=...파이썬 측 기본값done=False
server_default=func.now()DB가 채우는 기본값created_at
index=True조회용 인덱스title
String(100)VARCHAR 길이 제한title
🕐 created_at은 왜 server_default일까

파이썬 쪽 default=datetime.utcnow는 3.12+에서 deprecated이고, 앱 서버 시계에 의존합니다. server_default=func.now()는 DB 서버가 직접 시각을 찍어 정확하고 일관됩니다. 여러 인스턴스가 떠 있어도 기준 시계가 하나(DB)라 안전합니다.

4NestJS TypeORM과 나란히

같은 "할 일" 엔티티를 양쪽으로 그려 봅니다. 데코레이터(Nest) ↔ 타입 힌트(SQLAlchemy)의 대비가 핵심입니다.

Nest NestJS — TypeORM Entity
todo.entity.ts
@Entity('todos')
export class Todo {
  @PrimaryGeneratedColumn()
  id: number;

  @Column({ length: 100 })
  title: string;

  @Column({ default: false })
  done: boolean;

  @CreateDateColumn()
  createdAt: Date;
}
Fast FastAPI — SQLAlchemy 2.0
models.py
class Todo(Base):
    __tablename__ = "todos"

    id: Mapped[int] = \
        mapped_column(primary_key=True)
    title: Mapped[str] = \
        mapped_column(String(100))
    done: Mapped[bool] = \
        mapped_column(default=False)
    created_at: Mapped[datetime] = \
        mapped_column(server_default=func.now())

@PrimaryGeneratedColumn ↔ primary_key=True, @Column ↔ mapped_column. 데코레이터 대신 타입 힌트가 일합니다.

5create_all — 테이블 생성(DDL)

모델을 선언했다고 테이블이 생기진 않습니다. Base.metadata.create_all(engine)를 호출하면, 등록된 모든 모델을 보고 CREATE TABLE(DDL)을 실행합니다. 이미 있는 테이블은 건드리지 않습니다.

create_tables.py
from database import engine
from models import Base

# 모델 → CREATE TABLE 변환 후 실행. 있으면 건너뜀.
Base.metadata.create_all(engine)
⚠️ create_all은 "변경"을 못 한다

create_all은 없는 테이블을 만들 뿐, 이미 있는 테이블의 컬럼 추가·타입 변경은 하지 못합니다. 모델을 고쳐도 기존 테이블엔 반영되지 않죠. 그래서 데모·초기 개발에만 쓰고, 운영의 스키마 변경은 Week 10의 Alembic 마이그레이션으로 버전 관리합니다.

🧭 다음 단계 예고 — Alembic

실무에선 "컬럼 하나 추가"도 기록(마이그레이션)으로 남겨 팀·운영 DB에 순서대로 적용합니다. NestJS의 typeorm migration:generate에 대응하는 게 Alembic의 alembic revision --autogenerate입니다. 지금은 create_all로 빠르게 시작하고, 구조가 안정되면 Alembic으로 갈아탑니다.

6✅ 셀프 체크

Q SQLAlchemy 2.0 스타일에서 컬럼의 nullable 여부는 어떻게 정하나요?

타입 힌트로 정합니다. Mapped[str]는 NOT NULL, Mapped[str | None]은 NULL 허용입니다. 별도 옵션 없이 타입 자체가 nullable 여부를 선언합니다.

Q default와 server_default의 차이는?

default는 파이썬(앱) 쪽에서 값을 채우고, server_default는 DB 서버가 채웁니다. created_at처럼 정확한 시각·여러 인스턴스 일관성이 중요한 경우 server_default=func.now()로 DB가 찍게 하는 편이 안전합니다.

Q 모델 컬럼을 바꿨는데 create_all로 테이블이 안 바뀌어요.

create_all은 없는 테이블을 만들 뿐, 기존 테이블의 컬럼 변경은 하지 못합니다. 스키마 변경은 Alembic 마이그레이션(Week 10)으로 처리해야 합니다. 데모라면 테이블을 드롭 후 다시 만들 수도 있지만, 실데이터가 있으면 절대 그러면 안 됩니다.
