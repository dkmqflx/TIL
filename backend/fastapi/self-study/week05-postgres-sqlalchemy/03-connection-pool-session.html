Connection Pool · Session
그리고 안전한 쿼리

DB 연결은 비싼 자원입니다. 미리 만들어 돌려 쓰는 Connection Pool, 작업 단위인 Session, 그리고 SQL Injection을 막는 bind 변수까지 — DB를 실제로 다루는 마지막 조각들입니다.

“

DB 연결은 매번 새로 맺기엔 비싸다. 그래서 풀(pool)에 미리 만들어 두고 빌려 쓰고 반납한다 — 콜택시 대기열처럼. 작업은 Session(작업 단위)으로 묶고, 값은 문자열로 잇지 말고 bind 변수로 바인딩해 인젝션을 막는다.

  • Connection Pool이 왜 필요한지와 주요 파라미터를 안다
  • Session(Unit of Work)으로 작업을 묶고 commit/close한다
  • 2.0 스타일로 쿼리를 실행하고 결과를 fetch한다
  • bind 변수로 SQL Injection을 막는다

1Connection Pool — 연결을 재사용하기

DB 연결은 TCP 핸드셰이크·인증을 거치는 무거운 작업입니다. 요청마다 새로 맺고 끊으면 그 비용이 누적됩니다. 그래서 연결을 미리 몇 개 만들어 두고 빌려 쓰고 반납합니다.

🚕 비유 — 콜택시 대기열

손님(요청)이 올 때마다 새 택시를 사는 것은 미친 짓입니다. 대신 회사는 택시 몇 대를 대기열(pool)에 세워 둡니다. 손님이 오면 대기 중인 택시를 배차(빌림)하고, 목적지에 도착하면 택시는 다시 대기열로 복귀(반납)합니다. 손님이 몰리면 임시 차량을 더 부르고(max_overflow), 한가하면 줄입니다.

파라미터의미비유
pool_size평상시 유지하는 연결 수상시 대기 택시 대수
max_overflow몰릴 때 추가 허용 한도피크 때 임시 콜 한도
pool_recycle일정 초 지난 연결은 폐기·재생성오래된 차 교체 주기
pool_pre_ping빌려주기 전 "살아있니?" 점검배차 전 시동 확인
database.py
engine = create_engine(
    DATABASE_URL,
    pool_size=5,        # 상시 5개
    max_overflow=10,    # 피크 때 +10까지
    pool_recycle=1800,  # 30분 지난 연결 재생성
    pool_pre_ping=True, # 끊긴 연결 자동 폐기
)

2Session — 작업 단위(Unit of Work)

풀에서 빌린 연결 위에서 실제 작업을 묶는 게 Session입니다. 한 Session 안의 변경들은 commit 전까지 한 묶음(트랜잭션)으로 모였다가 함께 반영됩니다.

database.py
from sqlalchemy.orm import Session, sessionmaker

SessionLocal = sessionmaker(
    bind=engine, autoflush=False, expire_on_commit=False,
)

def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()      # 끝나면 연결을 풀로 반납
메서드역할
session.add(obj)새 객체를 세션에 등록(아직 DB 반영 X)
session.commit()모은 변경을 트랜잭션으로 확정
session.refresh(obj)DB가 채운 값(id·created_at)을 다시 읽어옴
session.close()세션 종료, 연결을 풀로 반납
💡 with 문 / get_db 의존성

세션은 반드시 닫아야 연결이 풀로 돌아갑니다. with Session(engine) as s: 컨텍스트나, FastAPI에선 위처럼 get_db 제너레이터를 Depends로 주입해 finally: close()를 보장합니다. NestJS에서 요청 스코프로 Repository/EntityManager를 받는 것과 같은 역할입니다.

3쿼리 실행 & fetch

2.0 스타일은 select()로 쿼리를 만들고, scalars()로 ORM 객체 리스트를 받습니다.

query.py
from sqlalchemy import select
from models import Todo

# 전체 조회 — Todo 객체 리스트
todos = db.scalars(select(Todo).order_by(Todo.id)).all()

# 동일하게 execute + scalars로도 가능
result = db.execute(select(Todo))
todos = result.scalars().all()
🔎 scalars()를 쓰는 이유 — Row vs 객체

execute()의 결과는 Row 객체(컬럼 묶음)입니다. select(Todo)처럼 엔티티 하나만 고를 땐 Row의 첫 칸이 곧 Todo 객체죠. scalars()는 그 첫 칸을 꺼내 Todo 객체 자체를 돌려줍니다. 그래서 todo.title처럼 바로 다룰 수 있습니다. 여러 컬럼을 골랐다면 scalars() 없이 Row로 받아 풀어 씁니다.

4bind 변수 — 값을 "끼워 넣지" 말고 "바인딩"하라

가장 중요한 안전 규칙입니다. 사용자 입력을 문자열로 직접 이어 붙여 SQL을 만들면 SQL Injection에 뚫립니다. 대신 값을 파라미터(bind 변수)로 따로 전달하면, DB가 그것을 "코드가 아닌 데이터"로 취급해 안전합니다.

⚠️ f-string으로 SQL을 만들지 마라

아래 왼쪽처럼 f"... WHERE title = '{q}'"로 SQL을 조립하면, q에 ' OR '1'='1 같은 값이 들어왔을 때 쿼리 의미가 통째로 바뀝니다. 비밀번호를 우회하거나 테이블을 통째로 노출시킬 수 있죠. 값은 절대 문자열로 잇지 말고 바인딩하세요.

위험 문자열 포매팅 (절대 금지)
danger.py
# ❌ 사용자 입력을 그대로 이어붙임
q = request_query
sql = text(
  f"SELECT * FROM todos "
  f"WHERE title = '{q}'"
)
db.execute(sql)
# q = "' OR '1'='1"  →  전체 노출!
안전 bind 변수 바인딩
safe.py
from sqlalchemy import select, text

# ✅ 값을 파라미터로 따로 전달
# 1) ORM은 기본이 바인딩
select(Todo).where(Todo.title == q)

# 2) raw SQL도 :param 으로
db.execute(
  text("SELECT * FROM todos "
       "WHERE title = :q"),
  {"q": q},   # 데이터로 취급
)
🛡️ ORM을 쓰면 기본이 안전

select(Todo).where(Todo.title == q)처럼 ORM 표현식을 쓰면 SQLAlchemy가 자동으로 값을 바인딩합니다. 즉, 평소처럼 ORM으로 짜기만 해도 인젝션이 막힙니다. 위험은 거의 항상 raw SQL을 f-string으로 직접 조립할 때 생깁니다.

5전체 흐름 한눈에

지금까지의 조각을 하나의 엔드포인트로 모으면 이렇습니다.

main.py
from fastapi import Depends, FastAPI
from sqlalchemy import select
from sqlalchemy.orm import Session
from database import get_db
from models import Todo

app = FastAPI()

@app.get("/todos")
def list_todos(db: Session = Depends(get_db)):
    return db.scalars(select(Todo).order_by(Todo.id)).all()

6✅ 셀프 체크

Q Connection Pool은 왜 필요한가요?

DB 연결은 핸드셰이크·인증을 거치는 비싼 작업이라, 요청마다 새로 맺으면 비용이 큽니다. 풀에 연결을 미리 만들어 두고 빌려 쓰고 반납하면, 콜택시 대기열처럼 재사용으로 빠르게 처리할 수 있습니다.

Q scalars()는 왜 쓰나요?

execute()는 컬럼 묶음인 Row를 돌려줍니다. select(Todo)처럼 엔티티 하나만 골랐다면 scalars()로 Row의 첫 칸(=Todo 객체)을 꺼내, 곧바로 todo.title처럼 객체로 다룰 수 있게 합니다.

Q SQL Injection을 막는 핵심 규칙은?

값을 문자열로 이어 붙이지 말고 bind 변수(파라미터)로 따로 전달하는 것입니다. ORM 표현식(where(Todo.title == q))은 자동으로 바인딩하므로 안전하고, raw SQL을 쓸 때도 :q 플레이스홀더 + 파라미터 dict를 씁니다. f-string으로 SQL을 조립하는 것이 가장 흔한 사고 원인입니다.
