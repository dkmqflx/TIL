REST API 명세 & CRUD
스키마 · 모델 · 라우터를 묶다

Week 3의 Pydantic, Week 5의 SQLAlchemy 모델, 그리고 라우터 — 흩어져 있던 부품을 하나로 합쳐 진짜로 DB에 저장되는 ToDo REST API를 완성합니다.

“

REST API는 "자원(resource)을 URL로, 행동을 HTTP 메서드로" 표현하는 약속이다. GET /todos(조회), POST /todos(생성), PATCH /todos/{id}(부분 수정), DELETE /todos/{id}(삭제). 여기에 Pydantic·SQLAlchemy를 끼우면 검증된 데이터가 DB까지 흐른다.

  • ToDo 자원을 REST 규칙에 맞게 메서드·경로·상태코드로 설계한다
  • 요청/응답용 Pydantic 스키마(Create/Update/Public)를 나눠 쓴다
  • DB 세션으로 조회·생성·수정·삭제(CRUD)를 구현한다
  • ORM 객체를 response_model로 자동 직렬화한다(from_attributes)
  • 없는 자원에 404를 일관되게 응답한다

1REST API 명세 — 먼저 표로 그린다

코드를 짜기 전에 "어떤 요청에 어떻게 응답할지"를 표로 정리합니다. 이게 곧 API의 계약서입니다. NestJS에서 컨트롤러 메서드를 설계하던 것과 같은 단계죠.

메서드경로설명성공 상태코드
GET/todos할 일 전체 목록 조회200 OK
GET/todos/{id}할 일 단건 조회200 OK (없으면 404)
POST/todos새 할 일 생성201 Created
PATCH/todos/{id}일부 필드만 수정200 OK (없으면 404)
DELETE/todos/{id}할 일 삭제204 No Content (없으면 404)
🍽️ 비유 — 식당의 주문 규칙

REST는 손님과 주방 사이의 주문 규칙입니다. "메뉴(자원)는 메뉴판 번호(URL)로 가리키고, 하고 싶은 행동(조회·추가·변경·취소)은 정해진 손짓(HTTP 메서드)으로 한다." 손님이 규칙대로만 주문하면, 어떤 주방이든 같은 방식으로 알아듣습니다. 상태코드는 주방이 돌려주는 "됐어요(201)/없어요(404)" 같은 표준 답변이고요.

2스키마 3종 — 입력과 출력을 분리한다

"받을 때 필요한 필드"와 "돌려줄 때 보여줄 필드"는 다릅니다. 그래서 Pydantic 모델을 용도별로 나눕니다. NestJS에서 CreateDto, UpdateDto, ResponseDto를 나누던 것과 똑같습니다.

스키마용도특징
TodoCreatePOST 요청 본문title 필수, id는 받지 않음
TodoUpdatePATCH 요청 본문모든 필드 Optional (보낸 것만 수정)
TodoPublic응답(response_model)id·created_at 포함, ORM에서 직렬화
schemas.py
from datetime import datetime
from pydantic import BaseModel, ConfigDict, Field

class TodoCreate(BaseModel):
    title: str = Field(min_length=1, max_length=100)
    done: bool = False

class TodoUpdate(BaseModel):     # PATCH: 보낸 필드만 수정
    title: str | None = Field(default=None, min_length=1, max_length=100)
    done: bool | None = None

class TodoPublic(BaseModel):
    model_config = ConfigDict(from_attributes=True)   # ← 핵심!
    id: int
    title: str
    done: bool
    created_at: datetime

3from_attributes — ORM 객체를 그대로 직렬화

DB에서 꺼낸 건 dict가 아니라 SQLAlchemy 객체(todo.id, todo.title처럼 속성으로 접근)입니다. 기본 Pydantic은 dict만 읽으려 하기 때문에, "속성으로도 읽어라"고 켜 주는 스위치가 ConfigDict(from_attributes=True)입니다.

📝 v1의 orm_mode가 이름만 바뀐 것

Pydantic v1에서 class Config: orm_mode = True로 쓰던 그 기능입니다. v2에서는 model_config = ConfigDict(from_attributes=True)로 이름이 바뀌었습니다. 이게 켜져 있으면 라우터에서 ORM 객체를 그대로 return해도 FastAPI가 TodoPublic 모양의 JSON으로 변환합니다.

4SQLAlchemy 모델과 DB 세션 (Week 5 복습)

Week 5에서 만든 Todo 테이블 모델과, 요청마다 세션을 열고 닫는 get_db입니다. 다음 레슨에서 이 get_db를 의존성으로 주입하게 됩니다.

models.py
from datetime import datetime
from sqlalchemy import func
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column

class Base(DeclarativeBase): ...

class Todo(Base):
    __tablename__ = "todos"
    id: Mapped[int] = mapped_column(primary_key=True)
    title: Mapped[str] = mapped_column(index=True)
    done: Mapped[bool] = mapped_column(default=False)
    created_at: Mapped[datetime] = mapped_column(server_default=func.now())

5CRUD 라우터 — 5개 엔드포인트

이제 명세표의 5줄을 실제 코드로 옮깁니다. (DB 접근은 다음 레슨에서 서비스 계층으로 빼지만, 여기서는 흐름을 한눈에 보기 위해 라우터에 직접 적습니다.)

routers/todos.py
from fastapi import APIRouter, Depends, HTTPException, status
from sqlalchemy import select
from sqlalchemy.orm import Session
from database import get_db
from models import Todo
from schemas import TodoCreate, TodoUpdate, TodoPublic

router = APIRouter(prefix="/todos", tags=["todos"])

@router.get("", response_model=list[TodoPublic])
def list_todos(db: Session = Depends(get_db)):
    return db.scalars(select(Todo).order_by(Todo.id)).all()

@router.get("/{todo_id}", response_model=TodoPublic)
def get_todo(todo_id: int, db: Session = Depends(get_db)):
    todo = db.get(Todo, todo_id)
    if todo is None:
        raise HTTPException(status.HTTP_404_NOT_FOUND, "Todo not found")
    return todo                       # ORM 객체를 그대로 반환 → 직렬화
routers/todos.py (이어서)
@router.post("", response_model=TodoPublic,
             status_code=status.HTTP_201_CREATED)   # 생성 → 201
def create_todo(payload: TodoCreate, db: Session = Depends(get_db)):
    todo = Todo(title=payload.title, done=payload.done)
    db.add(todo)
    db.commit()
    db.refresh(todo)              # DB가 채운 id·created_at 다시 읽기
    return todo

@router.patch("/{todo_id}", response_model=TodoPublic)
def update_todo(todo_id: int, payload: TodoUpdate,
                db: Session = Depends(get_db)):
    todo = db.get(Todo, todo_id)
    if todo is None:
        raise HTTPException(status.HTTP_404_NOT_FOUND, "Todo not found")
    # exclude_unset=True → 요청에 실제로 보낸 필드만 수정
    for field, value in payload.model_dump(exclude_unset=True).items():
        setattr(todo, field, value)
    db.commit(); db.refresh(todo)
    return todo

@router.delete("/{todo_id}",
              status_code=status.HTTP_204_NO_CONTENT)   # 삭제 → 204
def delete_todo(todo_id: int, db: Session = Depends(get_db)):
    todo = db.get(Todo, todo_id)
    if todo is None:
        raise HTTPException(status.HTTP_404_NOT_FOUND, "Todo not found")
    db.delete(todo); db.commit()       # 본문 없이 204 응답
💡 PATCH와 PUT은 다르다

PUT은 자원 전체를 통째로 교체(모든 필드 필수)하고, PATCH는 일부만 수정합니다. 그래서 TodoUpdate의 필드를 전부 Optional로 두고, model_dump(exclude_unset=True)로 "실제로 보낸 필드만" 골라 덮어씁니다. done만 보내면 title은 건드리지 않죠.

6전체 흐름 — 요청부터 응답까지

하나의 요청이 라우터를 거쳐 DB를 만나고 다시 JSON으로 돌아오는 길입니다.

⚠️ commit 뒤엔 refresh를 잊지 말 것

id와 created_at은 INSERT 시점에 DB가 채워 주는 값입니다. db.commit()만 하고 바로 객체를 반환하면 이 값들이 비어 있거나(만료 상태) 추가 쿼리가 필요할 수 있습니다. db.refresh(todo)로 DB가 채운 최신 값을 다시 읽어 와야 응답에 제대로 담깁니다.

7✅ 셀프 체크

Q 스키마를 Create/Update/Public 셋으로 나누는 이유는?

입력과 출력의 필드가 다르기 때문입니다. 생성 때는 id를 받지 않고, 수정 때는 모든 필드가 선택적이며, 응답에는 id·created_at이 들어갑니다. 하나의 모델로 다 처리하려 하면 "선택적이어야 할 곳이 필수가 되는" 모순이 생깁니다. NestJS의 Create/Update/Response DTO 분리와 같은 발상입니다.

Q 라우터에서 SQLAlchemy 객체를 그냥 return해도 되는 까닭은?

response_model=TodoPublic을 지정했고, TodoPublic에 ConfigDict(from_attributes=True)를 켜 뒀기 때문입니다. FastAPI가 ORM 객체의 속성을 읽어 TodoPublic 모양의 JSON으로 변환합니다. 직접 dict로 변환할 필요가 없습니다.

Q PATCH에서 model_dump(exclude_unset=True)를 쓰는 이유는?

요청 본문에 실제로 보낸 필드만 추려 수정하기 위해서입니다. exclude_unset 없이 dump하면 보내지 않은 필드도 기본값(예: None)으로 들어와 멀쩡한 값을 덮어써 버립니다. 부분 수정(PATCH)의 핵심 옵션입니다.
