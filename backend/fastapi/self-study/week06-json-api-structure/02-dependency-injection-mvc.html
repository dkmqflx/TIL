Dependency Injection
서비스 계층으로 리팩터링

FastAPI의 Depends는 NestJS의 DI 컨테이너에 대응합니다. 공통 의존성(DB 세션)을 주입받고, 라우터에 뭉쳐 있던 로직을 서비스 계층(MVC)으로 분리합니다.

“

DI는 "필요한 걸 직접 만들지 말고 받아 쓰라"는 원칙이다. FastAPI에서는 함수 인자에 Depends(get_db)를 적기만 하면 세션이 자동으로 주입된다. 그리고 라우터(컨트롤러)는 요청/응답만, 서비스는 비즈니스 로직만 — 이렇게 계층을 나누면 테스트와 유지보수가 쉬워진다.

  • Depends로 공통 의존성(DB 세션)을 함수에 주입한다
  • yield 의존성으로 세션을 열고 닫는 생명주기를 관리한다
  • Annotated[Session, Depends(get_db)] 스타일을 익힌다
  • 라우터·서비스·리포지토리로 계층을 분리한다(MVC)
  • 라우터에 뭉친 코드를 서비스 함수로 리팩터링한다

1Depends — 의존성을 주입받기

지난 레슨에서 모든 라우트가 db: Session = Depends(get_db)를 적었죠. 이게 바로 의존성 주입입니다. 함수가 세션을 직접 만들지 않고, "세션 주세요"라고 선언만 하면 FastAPI가 get_db를 호출해 결과를 넣어 줍니다.

👨‍🍳 비유 — 재료를 받아 쓰는 요리사

요리사(라우트 함수)가 요리할 때마다 직접 장을 보러 간다면? 시간도 낭비되고, 테스트하려면 매번 진짜 마트가 필요합니다. 대신 준비된 재료(DB 세션)를 받아 쓰면, 요리사는 요리에만 집중하고 — 테스트할 땐 가짜 재료(목 세션)를 건네 똑같이 검증할 수 있습니다. DI는 "재료 조달"과 "요리"를 떼어 놓아 결합도를 낮추고 테스트를 쉽게 만듭니다.

2yield 의존성 — 열고, 쓰고, 닫기

get_db는 단순 함수가 아니라 yield를 쓰는 생성기 의존성입니다. yield 위쪽은 요청 처리 전(셋업), 아래쪽은 응답 후 정리(teardown)로 실행됩니다. 세션을 열고 → 라우트에 넘기고 → 끝나면 반드시 닫는 패턴이죠.

database.py
from collections.abc import Generator
from sqlalchemy.orm import Session

def get_db() -> Generator[Session, None, None]:
    db = SessionLocal()
    try:
        yield db          # ← 여기서 라우트가 db를 받아 사용
    finally:
        db.close()        # 응답이 끝나면 항상 닫힘(예외가 나도)
📝 try/finally가 핵심

finally 안에서 close()를 하므로, 라우트에서 예외가 터져도 세션은 누수 없이 닫힙니다. 커넥션 풀(Week 5)로 세션을 빌려 쓰고 꼬박꼬박 반납하는 셈입니다. NestJS에서 요청 스코프 프로바이더가 요청 끝에 정리되는 것과 같은 생명주기입니다.

3Annotated 스타일 — 재사용 가능한 의존성

모든 라우트마다 db: Session = Depends(get_db)를 반복하면 지저분합니다. Annotated로 별칭 타입을 하나 만들어 두면 깔끔하게 재사용할 수 있습니다 — FastAPI가 권장하는 최신 스타일입니다.

routers/todos.py
from typing import Annotated
from fastapi import APIRouter, Depends
from sqlalchemy.orm import Session
from database import get_db

# 의존성 별칭 — 한 번 정의하고 모든 라우트에서 재사용
DbSession = Annotated[Session, Depends(get_db)]

router = APIRouter(prefix="/todos", tags=["todos"])

@router.get("/{todo_id}")
def get_todo(todo_id: int, db: DbSession):   # ← 깔끔!
    return services.get_todo(db, todo_id)

4NestJS Provider/DI와 나란히

두 프레임워크 모두 "의존성을 컨테이너가 주입"한다는 발상은 같습니다. 표현 방식만 다릅니다.

Nest NestJS — @Injectable + 생성자 주입
todos.controller.ts
@Injectable()
export class TodosService { ... }

@Controller('todos')
export class TodosController {
  // 생성자에서 주입받음
  constructor(
    private todos: TodosService,
  ) {}
}
Fast FastAPI — Depends
routers/todos.py
# 함수 인자에서 주입받음
@router.get("/{id}")
def get_todo(
    id: int,
    db: DbSession,   # = Depends(get_db)
):
    return services.get_todo(db, id)

생성자 주입 ↔ 함수 인자 주입. 클래스 없이 함수 단위로 더 가볍게.

5계층 분리(MVC) — 왜 나누나

지난 레슨의 라우터는 "요청 받기 + 비즈니스 판단 + DB 쿼리"를 한 함수에 다 했습니다. 작을 땐 괜찮지만, 로직이 커지면 라우터가 비대해지고 재사용·테스트가 어려워집니다. 그래서 책임을 계층으로 나눕니다.

계층역할NestJS 대응
라우터(컨트롤러)요청 받기 · 응답 보내기 · 검증Controller
서비스비즈니스 로직 (무엇을·어떻게)Service
리포지토리/모델DB 접근 (쿼리·저장)Repository / Entity

6리팩터링 — 라우터의 로직을 서비스로

이제 라우터에 박혀 있던 DB 코드를 services.py로 옮깁니다. 라우터는 "받아서 서비스에 넘기고 결과를 돌려주는" 얇은 껍데기가 됩니다.

services.py
from sqlalchemy import select
from sqlalchemy.orm import Session
from models import Todo
from schemas import TodoCreate, TodoUpdate
from exceptions import TodoNotFoundError   # 도메인 예외(HTTP 모름)

def get_todo(db: Session, todo_id: int) -> Todo:
    todo = db.get(Todo, todo_id)
    if todo is None:
        raise TodoNotFoundError(todo_id)   # HTTPException 아님!
    return todo

def create_todo(db: Session, data: TodoCreate) -> Todo:
    todo = Todo(title=data.title, done=data.done)
    db.add(todo); db.commit(); db.refresh(todo)
    return todo

def update_todo(db: Session, todo_id: int, data: TodoUpdate) -> Todo:
    todo = get_todo(db, todo_id)            # 없으면 예외 재사용
    for field, value in data.model_dump(exclude_unset=True).items():
        setattr(todo, field, value)
    db.commit(); db.refresh(todo)
    return todo
routers/todos.py (얇아진 라우터)
import services
from fastapi import APIRouter, status
from schemas import TodoCreate, TodoUpdate, TodoPublic

router = APIRouter(prefix="/todos", tags=["todos"])

@router.get("/{todo_id}", response_model=TodoPublic)
def get_todo(todo_id: int, db: DbSession):
    return services.get_todo(db, todo_id)   # 위임만 한다

@router.post("", response_model=TodoPublic,
             status_code=status.HTTP_201_CREATED)
def create_todo(payload: TodoCreate, db: DbSession):
    return services.create_todo(db, payload)
💡 서비스는 HTTP를 몰라야 한다

서비스 계층에서 HTTPException(상태코드, 헤더)을 던지면, 그 로직은 "웹 요청 안에서만" 쓸 수 있게 묶여 버립니다. 대신 TodoNotFoundError 같은 도메인 예외를 던지고, HTTP 변환은 다음 레슨의 전역 예외 핸들러에 맡기세요. 그래야 서비스를 배치 작업·CLI·테스트 어디서든 재사용할 수 있습니다.

⚠️ commit을 누가 책임지나

이 예제는 단순함을 위해 서비스 함수 안에서 commit()합니다. 트랜잭션이 여러 서비스 호출에 걸치는 복잡한 경우엔, 의존성/미들웨어에서 "요청 끝에 한 번 커밋"하도록 경계를 한 곳으로 모으는 패턴이 더 안전합니다. 일단은 "한 요청 = 한 작업 = 한 커밋"으로 익히면 됩니다.

복습

7✅ 셀프 체크

Q FastAPI의 Depends는 NestJS의 무엇에 대응하나요?

NestJS의 DI(의존성 주입)에 대응합니다. NestJS가 @Injectable 프로바이더를 생성자로 주입한다면, FastAPI는 Depends(...)로 함수 인자에 주입합니다. 둘 다 "필요한 의존성을 직접 만들지 않고 컨테이너가 넣어 준다"는 원리는 같고, FastAPI는 클래스 없이 함수 단위로 더 가볍습니다.

Q get_db가 일반 return이 아니라 yield를 쓰는 이유는?

요청 처리가 끝난 뒤 정리(세션 닫기)를 보장하기 위해서입니다. yield 위쪽에서 세션을 열고, 라우트에 넘긴 뒤, 응답이 끝나면 yield 아래쪽(finally: db.close())이 실행됩니다. 예외가 나도 세션이 누수되지 않습니다.

Q 라우터의 로직을 서비스로 빼면 뭐가 좋아지나요?

책임이 분리됩니다. 라우터는 요청/응답만 담당하고, 비즈니스 로직은 서비스에 모입니다. 그 결과 ① 같은 로직을 여러 라우트나 배치 작업에서 재사용할 수 있고, ② 서비스 함수를 HTTP 없이 단위 테스트할 수 있으며, ③ 라우터가 얇아져 읽기 쉬워집니다.
