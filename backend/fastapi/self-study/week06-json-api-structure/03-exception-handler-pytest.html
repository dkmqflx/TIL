“

좋은 API는 에러도 일관된 모양으로 돌려준다. @app.exception_handler로 도메인 예외를 한 곳에서 HTTP 응답으로 바꾸고, 설정은 pydantic-settings + .env로 코드 밖에 두며, pytest + TestClient로 "정말 동작하는가"를 자동으로 검증한다.

  • @app.exception_handler로 전역 예외 핸들러를 등록한다
  • 커스텀 예외를 일관된 에러 JSON으로 변환한다
  • pydantic-settings + .env로 환경변수를 안전하게 로드한다

1전역 예외 핸들러 — 에러를 한 곳에서

지난 레슨에서 서비스는 TodoNotFoundError 같은 도메인 예외를 던졌습니다. 이걸 받아 HTTP 응답으로 바꾸는 일을 라우트마다 반복하면 지저분하죠. @app.exception_handler로 앱 전체에서 한 번만 등록합니다.

🚑 비유 — 건물의 중앙 비상 대응실

층마다 사고가 날 때 각 사무실이 따로 119에 전화하면 대응이 제각각입니다. 대신 중앙 비상실을 두면, 어느 층에서 무슨 사고가 나든 같은 절차로 일관되게 처리됩니다. 전역 예외 핸들러가 바로 그 중앙 비상실 — 앱 어디서 어떤 예외가 올라오든 같은 모양의 에러 JSON으로 응답합니다.

exceptions.py
class TodoNotFoundError(Exception):
    def __init__(self, todo_id: int) -> None:
        self.todo_id = todo_id
        super().__init__(f"Todo {todo_id} not found")
main.py
from fastapi import FastAPI, Request, status
from fastapi.responses import JSONResponse
from exceptions import TodoNotFoundError

app = FastAPI()

@app.exception_handler(TodoNotFoundError)
async def todo_not_found_handler(request: Request, exc: TodoNotFoundError):
    return JSONResponse(
        status_code=status.HTTP_404_NOT_FOUND,
        content={
            "error": "todo_not_found",
            "message": f"id={exc.todo_id} 인 할 일을 찾을 수 없습니다.",
        },
    )

2NestJS ExceptionFilter와 나란히

발상이 거의 같습니다. "예외 타입 → 응답 변환"을 한 곳에 모아 두는 것이죠.

Nest NestJS — ExceptionFilter
todo.filter.ts
@Catch(TodoNotFoundError)
export class TodoFilter
  implements ExceptionFilter {
  catch(e, host) {
    const res = host
      .switchToHttp()
      .getResponse();
    res.status(404).json({ ... });
  }
}
Fast FastAPI — exception_handler
main.py
@app.exception_handler(
    TodoNotFoundError
)
async def handler(req, exc):
    return JSONResponse(
        status_code=404,
        content={ ... },
    )

@Catch(Type) ↔ @app.exception_handler(Type). 데코레이터 하나로 등록.

3설정 관리 — pydantic-settings + .env

DATABASE_URL 같은 값을 코드에 박아 두면 환경(로컬·운영)마다 코드를 고쳐야 하고, 비밀값이 노출됩니다. pydantic-settings의 BaseSettings는 환경변수와 .env 파일을 읽어 타입 검증까지 해 줍니다.

settings.py
from pydantic_settings import BaseSettings, SettingsConfigDict

class Settings(BaseSettings):
    database_url: str = "sqlite:///./test.db"   # 기본값
    app_name: str = "Todo API"

    model_config = SettingsConfigDict(env_file=".env")

settings = Settings()   # 어디서든 import 해서 settings.database_url
.env
# 환경별 값은 여기에. 코드에는 절대 하드코딩하지 않는다.
DATABASE_URL=postgresql+psycopg://user:secret@localhost:5432/app
APP_NAME=Todo API
⚠️ 비밀값을 코드/깃에 넣지 말 것

.env는 반드시 .gitignore에 등록하세요. DB 비밀번호·API 키·토큰이 깃 히스토리에 한 번이라도 올라가면 사실상 영구히 노출된 것으로 봐야 합니다. 저장소에는 값이 빠진 .env.example만 커밋해 "어떤 키가 필요한지"만 알려 주세요.

4pytest 입문 — 첫 자동 테스트

"브라우저로 직접 눌러 확인"은 매번 반복되고 실수도 잦습니다. pytest와 FastAPI의 TestClient로 "GET이 200을, POST가 201을 주는가"를 코드로 못 박아 둡니다.

terminal
# 개발 의존성으로 pytest 추가 (uv)
uv add --dev pytest httpx
uv run pytest -v
test_todos.py
from fastapi.testclient import TestClient
from main import app

client = TestClient(app)   # 서버를 띄우지 않고 앱을 직접 호출

def test_list_todos_returns_200():
    res = client.get("/todos")
    assert res.status_code == 200
    assert isinstance(res.json(), list)

def test_create_todo_returns_201():
    res = client.post("/todos", json={"title": "pytest 배우기"})
    assert res.status_code == 201
    body = res.json()
    assert body["title"] == "pytest 배우기"
    assert body["done"] is False
💡 TestClient는 진짜 서버가 필요 없다

TestClient는 uvicorn으로 서버를 띄우지 않고, 앱 객체를 메모리에서 직접 호출합니다. 그래서 빠르고, 포트 충돌도 없습니다. 내부적으로 httpx를 쓰기 때문에 pytest와 함께 httpx를 설치합니다. NestJS의 supertest로 e2e 테스트하던 경험과 비슷합니다.

여러 테스트가 공유하는 준비물(테스트 전용 DB, 깨끗한 세션, 인증된 클라이언트 등)은 conftest.py에 fixture로 모읍니다. 지금은 "이런 게 있다"만 알아 두세요. 테스트 전용 DB 분리·픽스처·DI 오버라이드는 Week 10에서 본격적으로 다룹니다.

5✅ 셀프 체크

Q 전역 예외 핸들러를 쓰면 라우트마다 try/except를 쓰는 것보다 뭐가 낫나요?

에러 처리를 한 곳에 모아 일관성을 보장합니다. 서비스는 도메인 예외만 던지고, HTTP 변환(상태코드·JSON 모양)은 핸들러가 전담하므로 중복이 사라지고, 응답 포맷을 바꿀 때도 한 곳만 고치면 됩니다. NestJS의 ExceptionFilter와 같은 이점입니다.

Q 설정을 BaseSettings로 관리하면 좋은 점은?

환경변수·.env 값을 읽어 타입 검증까지 해 주고, 코드 어디서든 settings.database_url로 일관되게 접근합니다. 비밀값을 코드 밖(.env)에 두어 노출을 막고, 환경(로컬/운영)마다 코드를 고치지 않고 값만 바꾸면 됩니다.

Q TestClient로 테스트할 때 서버를 따로 실행해야 하나요?

아니요. TestClient(app)는 ASGI 앱을 메모리에서 직접 호출하므로 uvicorn 실행이 필요 없습니다. 그래서 빠르고 포트 충돌도 없습니다. uv run pytest만으로 GET 200·POST 201 같은 동작을 자동 검증할 수 있습니다.
