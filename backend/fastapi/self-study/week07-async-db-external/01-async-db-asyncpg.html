비동기 DB (asyncpg)
이벤트 루프를 막지 마라

Week 4에서 "async 함수 안에서 동기 I/O를 호출하면 이벤트 루프가 멈춘다"고 배웠죠. DB 호출도 똑같은 I/O입니다. 이번엔 SQLAlchemy 2.0 + asyncpg로 DB까지 완전 비동기로 만듭니다.

“

비동기 서버에서 동기 DB 드라이버를 쓰면, DB가 응답을 주는 동안 이벤트 루프 전체가 멈춰 다른 요청을 못 받는다. asyncpg + create_async_engine + AsyncSession으로 갈아끼우고, 모든 쿼리 앞에 await를 붙이면 DB 대기 중에도 서버가 다른 일을 한다.

  • 왜 비동기 서버에서 동기 DB 호출이 위험한지 이벤트 루프 관점에서 이해한다
  • asyncpg 드라이버 + create_async_engine + async_sessionmaker로 엔진/세션을 구성한다
  • async 엔드포인트에서 async with 세션 + await execute(select(...))로 쿼리한다
  • Lifespan으로 startup/shutdown 리소스를 관리한다 (engine.dispose())

1왜 비동기 DB인가 — Week 4 함정의 연장

Week 4에서 핵심 규칙 하나를 배웠습니다. "async def 엔드포인트 안에서 동기(blocking) I/O를 부르면 이벤트 루프가 통째로 멈춘다". 그 함정의 가장 흔한 현장이 바로 데이터베이스입니다.

DB 쿼리는 본질적으로 네트워크 I/O입니다. 쿼리를 보내고 결과가 돌아올 때까지 수십 ms를 기다리죠. 이 "기다림"을 동기 드라이버(psycopg2 같은)로 하면, 그동안 이벤트 루프가 그 한 요청에 묶여 다른 요청을 하나도 처리하지 못합니다.

☕ 비유 — 카페 바리스타 한 명

이벤트 루프는 바리스타 한 명입니다. 동기 DB는 "에스프레소 머신이 추출하는 30초 동안 바리스타가 가만히 머신만 쳐다보는" 것과 같습니다. 그동안 줄 선 다른 손님(요청)은 아무도 주문을 못 합니다. 비동기 DB는 "추출 버튼만 누르고(await) 다음 손님 주문을 받으러 가는" 똑똑한 바리스타입니다. 추출이 끝나면(결과 도착) 그 커피를 마저 내어줍니다. 같은 바리스타 한 명인데 처리량이 몇 배가 됩니다.

2드라이버 교체 — postgresql+asyncpg://

비동기 DB의 출발점은 비동기 드라이버입니다. PostgreSQL에서는 asyncpg가 표준입니다. SQLAlchemy에게 "이 드라이버로 비동기 연결을 맺어라"라고 알려주는 건 연결 URL의 접두사입니다.

구분연결 URL엔진 생성
동기 (지금까지)postgresql+psycopg2://...create_engine
비동기 (이번 주)postgresql+asyncpg://...create_async_engine
⚠️ URL 접두사 하나만 틀려도 터진다

비동기 엔진(create_async_engine)에 동기 URL(postgresql:// 또는 +psycopg2)을 주면 InvalidRequestError가 납니다. 반드시 postgresql+asyncpg://로 시작해야 합니다. 드라이버(asyncpg)도 따로 설치돼 있어야 하고요(uv add asyncpg).

3엔진 & 세션 구성 — async_sessionmaker

비동기 버전은 이름 앞에 대부분 async가 붙거나 Async로 바뀐다고 기억하면 쉽습니다. 동기와 1:1로 대응됩니다.

동기 (SQLAlchemy 2.0)비동기 (이번 주)
create_enginecreate_async_engine
sessionmakerasync_sessionmaker
SessionAsyncSession
session.execute(...)await session.execute(...)
database.py
from sqlalchemy.ext.asyncio import (
    create_async_engine,
    async_sessionmaker,
    AsyncSession,
)

DATABASE_URL = "postgresql+asyncpg://user:pw@localhost:5432/app"

# 1) 비동기 엔진 — 커넥션 풀을 관리한다
engine = create_async_engine(DATABASE_URL, echo=True)

# 2) 세션 팩토리 — 요청마다 세션을 찍어낸다
AsyncSessionLocal = async_sessionmaker(
    engine,
    class_=AsyncSession,
    expire_on_commit=False,   # 커밋 후에도 객체 속성 접근 가능
)
💡 expire_on_commit=False 가 비동기에서 특히 중요

기본값(True)이면 커밋 직후 객체 속성에 접근할 때 SQLAlchemy가 자동으로 DB를 다시 읽으려 합니다. 비동기에서는 이 "숨은 재조회"가 await 없이 일어나 MissingGreenlet 에러를 냅니다. expire_on_commit=False로 꺼두는 게 비동기의 정석입니다.

4엔드포인트 — 동기 → 비동기 전환 (before / after)

Week 5·6에서 쓰던 동기 라우트를 비동기로 바꿔봅시다. 바뀌는 건 ① def → async def, ② with → async with, ③ execute 앞에 await. 이 세 가지뿐입니다.

Sync Before — 동기 (Week 5·6)
routers/items.py
@router.get("/items")
def list_items():
    with SessionLocal() as session:
        result = session.execute(
            select(Item)
        )
        return result.scalars().all()
Async After — 비동기 (Week 7)
routers/items.py
@router.get("/items")
async def list_items():
    async with AsyncSessionLocal() as session:
        result = await session.execute(
            select(Item)
        )
        return result.scalars().all()

result.scalars().all()은 이미 메모리에 받은 결과를 가공하는 거라 await가 없습니다. await는 DB와 통신하는 순간(execute, commit, flush)에만 붙는다고 기억하세요.

📝 쓰기는 await session.commit()

생성·수정·삭제는 session.add(obj) 뒤에 await session.commit()으로 마무리합니다. 새로 생성된 id 등을 바로 써야 하면 await session.refresh(obj)로 DB 값을 다시 채웁니다. 이 둘 모두 DB와 통신하므로 await가 필요합니다.

5Lifespan — startup/shutdown 리소스 관리

엔진은 앱 전체에서 단 하나여야 합니다(커넥션 풀을 공유해야 하니까). 그리고 서버가 꺼질 때는 풀을 깨끗이 정리해야 하죠. 이 "시작할 때 준비 / 끝날 때 정리"를 담는 곳이 Lifespan입니다.

main.py
from contextlib import asynccontextmanager
from fastapi import FastAPI

@asynccontextmanager
async def lifespan(app: FastAPI):
    # 🟢 startup — yield 이전
    print("서버 시작: 커넥션 풀 준비 완료")
    yield
    # 🔴 shutdown — yield 이후
    await engine.dispose()       # 풀의 모든 연결을 닫는다
    print("서버 종료: 커넥션 정리 완료")

app = FastAPI(lifespan=lifespan)

yield를 기준으로 위는 startup, 아래는 shutdown입니다. async with 컨텍스트 매니저와 똑같은 구조죠. 서버가 살아있는 동안은 yield 지점에 "멈춰" 있다가, 종료 신호가 오면 그 아래 정리 코드가 실행됩니다.

⚠️ @app.on_event("startup") 은 구식 — Lifespan을 써라

예전 튜토리얼에 나오는 @app.on_event("startup") / @app.on_event("shutdown")은 deprecated입니다. 지금은 위처럼 @asynccontextmanager + lifespan 하나로 시작·종료를 같은 함수에 묶는 방식이 권장됩니다. 시작에서 만든 변수를 종료에서 그대로 쓸 수 있어 더 안전하고요.

복습

6✅ 셀프 체크

Q async 엔드포인트에서 동기 DB 드라이버를 쓰면 뭐가 문제인가요?

동기 DB 호출은 결과가 올 때까지 스레드를 블로킹합니다. 그 스레드가 곧 이벤트 루프이므로, DB가 응답하는 수십 ms 동안 다른 모든 요청이 멈춥니다. 비동기의 장점(대기 중 다른 일 처리)이 통째로 사라지죠. 그래서 asyncpg 같은 비동기 드라이버 + await가 필요합니다.

Q 동기 코드를 비동기로 바꿀 때 await는 어디에 붙이나요?

DB와 실제로 통신하는 호출에만 붙입니다: await session.execute(...), await session.commit(), await session.refresh(...). 반면 result.scalars().all()처럼 이미 받은 결과를 가공하는 코드에는 await가 없습니다. 함수는 async def, 세션은 async with로 엽니다.

Q Lifespan은 왜 쓰고, 거기서 engine.dispose()는 왜 하나요?

Lifespan은 앱의 시작·종료 시점에 한 번씩 실행되는 코드 자리입니다. 엔진(커넥션 풀)은 앱 전체에서 하나만 두고 공유하는데, 서버가 꺼질 때 await engine.dispose()로 풀에 열린 연결들을 깔끔히 닫지 않으면 연결 누수가 생깁니다. @asynccontextmanager의 yield 이후가 종료 정리 자리입니다.
