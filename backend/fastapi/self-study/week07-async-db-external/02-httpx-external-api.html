외부 API 호출
이번엔 우리가 클라이언트다

백엔드는 혼자 일하지 않습니다. 결제·알림·다른 마이크로서비스를 HTTP로 호출하죠. 그 순간 우리 서버가 또 다른 서버의 클라이언트가 됩니다. 비동기 환경의 표준 도구 httpx.AsyncClient를 익힙니다.

“

비동기 엔드포인트에서 외부 API를 부를 땐 반드시 httpx.AsyncClient를 쓴다. requests는 동기라 이벤트 루프를 막는다. 클라이언트는 async with로 열고, await client.get(...)으로 호출하고, 타임아웃과 상태코드 검사를 빼먹지 않는다.

  • 백엔드가 다른 서비스를 호출하는 상황(MSA·결제·알림)을 이해한다
  • httpx.AsyncClient로 비동기 HTTP 요청을 보낸다
  • requests를 async 엔드포인트에서 쓰면 안 되는 이유를 안다
  • 타임아웃·상태코드 확인·예외 처리로 외부 호출을 안전하게 만든다

1백엔드가 클라이언트가 되는 순간

지금까지 우리 서버는 프론트엔드의 요청을 받기만 했습니다. 하지만 실무에서는 우리가 직접 다른 서버에 요청을 보내야 할 일이 많습니다.

  • 결제: 우리 API → 결제 게이트웨이(토스·스트라이프) 호출
  • 알림: 우리 API → 카카오·슬랙·SMS 발송 API 호출
  • MSA: 주문 서비스 → 재고 서비스 → 배송 서비스로 이어지는 호출
🔁 비유 — 식당의 식자재 주문

당신의 백엔드는 손님(프론트)에게 음식을 내는 식당입니다. 그런데 그 식당도 고기를 직접 키우진 않죠. 정육점(결제 API)·채소가게(알림 API)에 전화로 주문합니다. 이때 식당은 손님을 받는 가게인 동시에, 정육점에게는 주문하는 손님(클라이언트)입니다. 프론트가 우리에게 했던 일을, 이번엔 우리가 외부 API에게 합니다.

2httpx.AsyncClient — 기본 사용법

httpx는 requests와 거의 같은 API를 제공하면서, 비동기(async/await)를 정식 지원합니다. 핵심은 AsyncClient를 async with로 열고, 메서드 앞에 await를 붙이는 것입니다.

routers/weather.py
import httpx
from fastapi import APIRouter

router = APIRouter()

@router.get("/weather/{city}")
async def get_weather(city: str):
    url = f"https://api.example.com/weather?q={city}"
    async with httpx.AsyncClient() as client:
        r = await client.get(url)
    data = r.json()
    return {"city": city, "temp": data["temp"]}

async with로 열면 블록을 벗어날 때 연결이 자동으로 정리됩니다. POST·헤더·바디도 requests와 똑같습니다: await client.post(url, json={...}, headers={...}).

3⚠️ 함정 — requests는 async에서 금물

가장 흔한 실수입니다. 익숙하다고 requests를 async def 안에서 쓰면, Week 1의 그 함정이 그대로 재현됩니다.

Bad ❌ requests — 동기 (루프 차단)
bad.py
import requests

@router.get("/bad")
async def bad():
    # 🔒 응답 올 때까지 루프 정지!
    r = requests.get(url)
    return r.json()
Good ✅ httpx — 비동기 (양보)
good.py
import httpx

@router.get("/good")
async def good():
    async with httpx.AsyncClient() as c:
        r = await c.get(url)   # 대기 중 양보
    return r.json()

requests.get()은 await가 없는(걸 수도 없는) 동기 호출입니다. async def 안에 있어도 그 줄에서 이벤트 루프가 통째로 멈춰 외부 응답을 기다립니다. Lesson 01의 동기 DB와 똑같은 문제죠. 비동기 엔드포인트에서는 HTTP도 httpx입니다.

📝 정 requests를 써야 한다면

레거시 코드 등 불가피하게 동기 라이브러리를 호출해야 하면, await run_in_threadpool(requests.get, url)(Starlette 제공)로 별도 스레드에 떠넘겨 루프를 막지 않게 할 수 있습니다. 하지만 새 코드라면 그냥 httpx를 쓰는 게 훨씬 깔끔합니다.

4타임아웃 & 클라이언트 재사용

외부 호출에서 가장 무서운 건 상대가 응답을 안 주는 경우입니다. 타임아웃을 안 걸면 우리 요청 처리도 무한정 매달립니다. 또한 클라이언트를 매 요청마다 새로 만들면 TCP/TLS 핸드셰이크 비용이 반복되죠.

설정왜 필요한가
timeout=5.0외부가 느릴 때 무한 대기 방지 (초 단위)
클라이언트 재사용커넥션 풀·keep-alive로 핸드셰이크 비용 절약
Lifespan에 보관앱 전체에서 클라이언트 하나를 공유
💡 클라이언트는 Lifespan에 두고 재사용하라

Lesson 01의 DB 엔진과 똑같은 패턴입니다. lifespan 시작에서 httpx.AsyncClient(timeout=5.0)를 한 번 만들어 app.state.http 같은 곳에 보관하고, 종료에서 await app.state.http.aclose()로 닫습니다. 매 요청 async with로 새로 여는 것보다 커넥션을 재활용해 빠릅니다.

5에러 처리 — 상태코드 & 예외

외부 API는 4xx·5xx를 줄 수도, 아예 응답을 안 줄 수도 있습니다. 이 실패를 우리 API의 적절한 응답으로 번역해야 합니다. Week 6의 HTTPException과 자연스럽게 이어집니다.

routers/weather.py
import httpx
from fastapi import APIRouter, HTTPException

@router.get("/weather/{city}")
async def get_weather(city: str):
    try:
        async with httpx.AsyncClient(timeout=5.0) as c:
            r = await c.get(url)
        r.raise_for_status()        # 4xx·5xx 면 예외 발생
    except httpx.TimeoutException:
        raise HTTPException(504, "외부 API 응답 시간 초과")
    except httpx.HTTPStatusError as e:
        raise HTTPException(502, f"외부 API 오류: {e.response.status_code}")
    return r.json()
httpx 예외 / 메서드의미권장 응답
raise_for_status()4xx·5xx면 예외를 던짐—
TimeoutException타임아웃 초과504 Gateway Timeout
HTTPStatusError외부가 에러 상태코드 반환502 Bad Gateway
ConnectError연결 자체 실패503 Service Unavailable
⚠️ 외부 실패를 500으로 흘려보내지 마라

외부 API가 죽었는데 우리가 그냥 500 Internal Server Error를 내면, 프론트는 "우리 서버 버그"로 오해합니다. "외부 의존성 문제"라는 의미를 담은 502/503/504로 번역해 주는 게 디버깅과 모니터링에 훨씬 유리합니다.

복습

6✅ 셀프 체크

Q async 엔드포인트에서 requests를 쓰면 왜 안 되나요?

requests는 동기 라이브러리라 응답이 올 때까지 스레드를 블로킹합니다. 그 스레드가 이벤트 루프이므로, 외부 응답을 기다리는 동안 다른 모든 요청이 멈춥니다. 비동기 환경에서는 await를 걸 수 있는 httpx.AsyncClient를 써야 대기 중에 다른 요청을 처리할 수 있습니다.

Q 클라이언트를 매 요청마다 새로 만들지 말라는 이유는?

매번 새 AsyncClient를 만들면 요청마다 TCP 연결과 TLS 핸드셰이크를 새로 합니다. 클라이언트 하나를 Lifespan에 만들어 app.state에 보관하고 재사용하면, 커넥션 풀과 keep-alive로 같은 연결을 재활용해 지연과 비용이 줄어듭니다. DB 엔진을 하나만 두는 것과 같은 이유입니다.

Q 외부 API가 타임아웃 나면 클라이언트엔 몇 번 코드를 줘야 하나요?

504 Gateway Timeout이 적절합니다. 외부가 에러 상태코드를 주면 502 Bad Gateway, 연결 자체가 안 되면 503을 쓰는 게 관례입니다. 핵심은 "외부 의존성 문제"를 500으로 뭉뚱그리지 말고 5xx 중 의미 있는 코드로 번역하는 것입니다.
