CORS & 미들웨어
모든 요청을 가로채는 계층

프론트(:5173)에서 API(:8000)를 부르면 십중팔구 만나는 CORS 에러. 그 정체와 해결법, 그리고 모든 요청을 전·후로 감싸는 미들웨어를 배웁니다. Week 2에서 예고했던 쿠키/credentials 이야기도 여기서 매듭짓습니다.

“

CORS 에러는 서버 버그가 아니라 브라우저의 보안 정책이다. 다른 출처의 프론트가 호출하면 브라우저가 막는다. 서버가 CORSMiddleware로 "이 출처는 허용"이라고 응답 헤더에 명시하면 풀린다. 미들웨어는 모든 요청을 전·후로 가로채는 공통 계층이다.

  • CORS가 왜·언제 발생하는지 동일 출처 정책으로 설명한다
  • CORSMiddleware의 핵심 옵션(origins·credentials·methods·headers)을 설정한다
  • 프론트 CORS 에러 메시지를 읽고 디버깅한다
  • 커스텀 미들웨어로 모든 요청을 전·후 처리한다 (처리시간 헤더 예제)

1CORS란 — 출처가 다르면 브라우저가 막는다

출처(Origin)는 "스킴 + 호스트 + 포트"입니다. http://localhost:5173과 http://localhost:8000은 포트가 달라 다른 출처죠. 브라우저는 보안을 위해 "다른 출처로 보내는 JS 요청의 응답은, 서버가 명시적으로 허락하지 않으면 읽지 못하게" 막습니다. 이것이 동일 출처 정책(Same-Origin Policy)입니다.

🛂 비유 — 출입 허가 명단

API 서버는 출입 통제가 있는 건물입니다. 같은 회사 직원(동일 출처)은 그냥 들어가지만, 다른 건물(다른 출처)에서 온 방문자는 경비(브라우저)가 막습니다. 서버가 "이 방문자(http://localhost:5173)는 허용"이라고 명단(Access-Control-Allow-Origin 헤더)을 경비에게 보여줘야 통과됩니다. 명단에 없으면 응답이 와 있어도 경비가 손님에게 안 넘겨줍니다.

📝 CORS는 '브라우저'만의 규칙이다

중요한 포인트: CORS는 브라우저가 강제하는 규칙입니다. curl이나 다른 서버(Lesson 02처럼 우리 백엔드가 외부 API를 호출하는 경우)에서는 CORS 에러가 절대 안 납니다. 그래서 "Postman/curl은 되는데 브라우저만 안 돼요"가 전형적인 CORS 증상입니다.

2CORSMiddleware 설정

FastAPI는 CORSMiddleware를 기본 제공합니다. 허용할 출처·메서드·헤더·credentials를 등록하면, 미들웨어가 응답에 알맞은 CORS 헤더를 자동으로 붙여줍니다.

main.py
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware

app = FastAPI()

app.add_middleware(
    CORSMiddleware,
    allow_origins=["http://localhost:5173"],  # 정확한 출처 목록
    allow_credentials=True,                      # 쿠키·인증 허용
    allow_methods=["*"],                         # GET, POST, ...
    allow_headers=["*"],                         # Authorization 등
)
옵션의미
allow_origins허용할 프론트 출처 목록 (정확히 일치해야 함)
allow_credentials쿠키·Authorization 헤더 등 인증 정보 전송 허용
allow_methods허용 HTTP 메서드 (["*"]로 전체)
allow_headers허용 요청 헤더 (["*"]로 전체)
⚠️ credentials를 쓸 땐 allow_origins=["*"] 불가

Week 2에서 예고했던 쿠키/credentials 이야기가 여기 연결됩니다. allow_credentials=True(쿠키·인증 전송)와 allow_origins=["*"](모든 출처)는 동시에 쓸 수 없습니다. 보안상 브라우저가 거부합니다. 인증을 쓴다면 반드시 정확한 출처를 하나하나 적어야 합니다. 흔히 빠지는 함정이니 꼭 기억하세요.

3프론트 CORS 에러 읽는 법 & 디버깅

브라우저 콘솔의 그 빨간 메시지를 해독해 봅시다.

browser console
# 전형적인 CORS 에러
Access to fetch at 'http://localhost:8000/items'
from origin 'http://localhost:5173' has been blocked
by CORS policy: No 'Access-Control-Allow-Origin'
header is present on the requested resource.

해석: ":5173에서 :8000으로 보낸 요청의 응답에 허용 헤더가 없어서 막았다." 즉 서버가 그 출처를 허용 명단에 안 넣었다는 뜻입니다.

증상원인 / 체크
No 'Access-Control-Allow-Origin'allow_origins에 프론트 출처가 없음 → 정확히 추가
credentials 관련 에러allow_credentials=True인데 origins=["*"] → 정확한 출처로
OPTIONS만 보이고 본 요청 안 감Preflight 실패 → allow_methods/headers 확인
curl은 되는데 브라우저만 안 됨정상 — CORS는 브라우저 전용. 서버 헤더 설정 문제
💡 Preflight(사전 요청)란

인증 헤더나 PUT/DELETE 같은 요청은 브라우저가 본 요청 전에 OPTIONS 요청을 먼저 보내 "이런 요청 보내도 돼?"라고 묻습니다. 이게 Preflight입니다. CORSMiddleware가 이 OPTIONS에 자동 응답해 주므로, 직접 OPTIONS 핸들러를 만들 필요는 없습니다.

4미들웨어 — 모든 요청을 감싸는 계층

CORSMiddleware도 사실 미들웨어의 한 종류였습니다. 미들웨어는 라우트 함수에 도달하기 전과 응답이 나가기 후를, 모든 요청에 공통으로 가로채는 계층입니다. 로깅·인증·처리시간 측정 같은 공통 관심사를 한 곳에 모읍니다.

5커스텀 미들웨어 — 처리시간 헤더 추가

@app.middleware("http")로 직접 만들 수 있습니다. call_next(request)가 "다음 계층(결국 라우트)으로 넘기고 응답을 받아오는" 호출입니다. 그 전후로 우리 코드를 끼웁니다.

main.py
import time
from fastapi import Request

@app.middleware("http")
async def add_process_time(request: Request, call_next):
    start = time.perf_counter()      # 전처리
    response = await call_next(request)  # 라우트 실행
    elapsed = time.perf_counter() - start
    response.headers["X-Process-Time"] = f"{elapsed:.4f}"  # 후처리
    return response

이제 모든 응답에 X-Process-Time 헤더가 자동으로 붙습니다. 라우트를 한 줄도 안 건드리고요. 이게 미들웨어의 힘입니다 — 공통 관심사를 한 곳에서.

6NestJS와 비교 — Middleware / Interceptor

NestJS 경험자라면 익숙한 개념입니다. FastAPI의 미들웨어는 NestJS의 Middleware + Interceptor를 합친 느낌입니다.

Nest NestJS — Interceptor
timing.interceptor.ts
@Injectable()
class TimingInterceptor {
  intercept(ctx, next) {
    const s = Date.now();
    return next.handle().pipe(
      tap(() => log(Date.now() - s)),
    );
  }
}

RxJS 파이프로 전후를 감쌈.

Fast FastAPI — middleware
main.py
@app.middleware("http")
async def timing(req, call_next):
    s = time.perf_counter()
    res = await call_next(req)
    log(time.perf_counter() - s)
    return res

call_next 전후로 감쌈.

NestJS역할FastAPI 대응
Middleware요청 전처리 (라우트 도달 전)@app.middleware("http")
Interceptor응답 전·후 변형@app.middleware("http") (call_next 전후)
Guard접근 허용/차단의존성(Depends) 또는 미들웨어
📝 미들웨어 vs 의존성(Depends)

모든 요청에 공통(로깅·CORS·처리시간)이면 미들웨어가 맞습니다. 하지만 "이 라우트들만 인증 필요" 같은 선택적·세밀한 제어는 Week 8에서 배울 Depends(의존성 주입)가 더 적합합니다. 둘은 경쟁이 아니라 역할 분담입니다.

복습

7✅ 셀프 체크

Q 프론트(:5173)에서 API(:8000) 호출이 CORS로 막혔습니다. 어디를 고치나요?

서버 쪽입니다. CORSMiddleware의 allow_origins에 "http://localhost:5173"을 정확히 추가합니다. CORS는 서버가 응답 헤더로 출처를 허용해야 풀리는 문제라, 프론트 코드로는 해결되지 않습니다. (curl로는 잘 되는데 브라우저만 막힌다면 거의 확실히 이 경우입니다.)

Q allow_credentials=True와 allow_origins=["*"]를 같이 쓰면?

동작하지 않습니다. 보안상 브라우저는 "쿠키·인증을 모든 출처에 허용"을 금지합니다. 쿠키나 Authorization 헤더를 쓰는 인증 API라면 allow_origins에 정확한 출처를 하나하나 적어야 합니다.

Q 모든 응답에 처리시간 헤더를 넣으려면 어떻게 하나요?

@app.middleware("http")로 커스텀 미들웨어를 만들고, call_next(request) 호출 전후로 시간을 재서 response.headers["X-Process-Time"]에 넣은 뒤 그 response를 반환합니다. 라우트 함수는 전혀 수정할 필요가 없습니다 — 모든 요청에 공통 적용되니까요.
