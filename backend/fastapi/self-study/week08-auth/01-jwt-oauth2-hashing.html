JWT · OAuth2 · 해싱
“너 누구냐”를 증명하기

로그인은 결국 “이 요청을 누가 보냈는가”를 증명하는 일입니다. 패스워드를 안전하게 보관하고(해싱), 신원을 위조 불가능한 토큰(JWT)으로 발급하는 — 인증의 3대 부품을 익힙니다.

“

패스워드는 일방향으로 분쇄(해싱)해서 저장하고, 로그인에 성공하면 서버만 서명할 수 있는 토큰(JWT)을 발급한다. 이후 요청은 그 토큰을 Bearer로 들고 오고, 서버는 서명만 검증하면 “누구인지”를 알 수 있다.

  • 인증(Authentication)과 인가(Authorization)를 구분한다
  • 패스워드를 bcrypt로 해싱·검증하고, 평문 저장이 왜 금지인지 안다
  • JWT의 3-파트 구조와 서명으로 위변조를 막는 원리를 이해한다
  • OAuth2PasswordBearer + /token 엔드포인트로 토큰을 발급한다

1인증 vs 인가 — 헷갈리지 말 것

두 단어가 비슷해 보이지만 묻는 질문이 완전히 다릅니다. 인증이 먼저, 인가가 그 다음입니다.

구분묻는 질문예시
인증 (Authentication)“너 누구냐?”아이디·비밀번호 로그인, 토큰 검증
인가 (Authorization)“너 이거 할 권한 있냐?”관리자만 삭제 가능, 본인 글만 수정
🎫 비유 — 공연장 입장

인증은 입구에서 신분증을 확인하고 입장 팔찌를 채워주는 일입니다(너 누구냐). 인가는 그 팔찌 색깔을 보고 “일반석은 들어가지만 백스테이지는 안 됨”을 판정하는 일이죠(권한 있냐). 이번 레슨은 인증과 팔찌(JWT) 발급, 다음 레슨은 인가(권한 체크)를 다룹니다.

2패스워드 해싱 — 절대 평문 저장 금지

DB가 유출되는 사고는 언제든 일어납니다. 그때 패스워드가 평문으로 들어 있으면 모든 사용자 계정이 그 즉시 털립니다. 그래서 패스워드는 원본을 복구할 수 없는 형태로 변환해 저장합니다 — 이게 해싱입니다.

⚙️ 비유 — 일방향 분쇄기

해싱은 종이(평문 비밀번호)를 분쇄기에 넣는 것과 같습니다. 같은 종이를 넣으면 늘 같은 조각 패턴(해시)이 나오지만, 조각을 보고 원래 종이를 복원할 수는 없습니다. 로그인 때는 “들어온 비밀번호를 똑같이 분쇄해서, 저장된 조각과 일치하는가”만 비교합니다. 서버조차 원본 비밀번호를 모르는 게 정상입니다.

실무 표준은 bcrypt입니다. 파이썬에선 최신 pwdlib[bcrypt](FastAPI 공식 문서가 채택)나 전통의 passlib[bcrypt]로 씁니다. 핵심 동작은 두 가지 — hash(저장할 때)와 verify(로그인할 때)입니다.

security.py — pwdlib
from pwdlib import PasswordHash
from pwdlib.hashers.bcrypt import BcryptHasher

password_hash = PasswordHash((BcryptHasher(),))

def hash_password(raw: str) -> str:
    return password_hash.hash(raw)            # 가입 시 1회 — 이 결과만 DB에 저장

def verify_password(raw: str, hashed: str) -> bool:
    return password_hash.verify(raw, hashed)  # 로그인 시 — 일치 여부만 반환
📝 pwdlib vs passlib

둘 다 bcrypt를 감싸는 래퍼이고 API도 거의 같습니다. 다만 passlib(1.7.4)는 최신 bcrypt 패키지(5.x)와 충돌해 에러가 나는 경우가 있어, 새 프로젝트에선 pwdlib를 권장합니다. 이 스터디의 실행 예제(code/main.py)도 pwdlib로 통일했습니다.

⚠️ 평문·MD5·SHA256 단독 저장은 모두 금지

평문은 말할 것도 없고, md5(password)나 sha256(password) 같은 단순 해시도 금지입니다. 이런 해시는 너무 빨라서 공격자가 초당 수십억 번 시도(레인보우 테이블·무차별 대입)할 수 있습니다. bcrypt는 의도적으로 느리고(work factor), 매번 다른 솔트(salt)를 자동 포함해 같은 비밀번호라도 해시가 달라집니다. 비밀번호엔 반드시 bcrypt / argon2 같은 전용 알고리즘을 쓰세요.

⚠️ bcrypt의 72바이트 제한

bcrypt는 입력의 앞 72바이트까지만 다룹니다. 매우 긴 비밀번호(또는 멀티바이트 한글이 많은 경우) 73바이트 이후는 무시되거나, 최신 bcrypt 5.x에선 아예 오류(password cannot be longer than 72 bytes)가 납니다. 실무에선 입력 길이를 제한하거나(예: 64자), 72바이트 제한이 없는 argon2를 고려합니다. 데모 수준에선 신경 쓰지 않아도 되지만, “왜 긴 비밀번호가 안 먹지?”의 원인이 이것입니다.

3JWT — 위조 방지 입장 팔찌

로그인에 성공하면 서버는 JWT(JSON Web Token)를 발급합니다. JWT는 점(.)으로 구분된 세 조각을 base64로 인코딩한 한 줄 문자열입니다.

JWT 한 줄
eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiLrr7zsiJgiLCJleHAiOjE3MzB9.dQw4w9WgXcQ
└──── header ────┘ └────────── payload ──────────┘ └─ signature ─┘
파트내용비고
header서명 알고리즘 (예: HS256)base64 — 누구나 디코딩 가능
payload담은 정보(클레임): sub(누구), exp(만료) 등base64 — 암호화 아님!
signatureheader+payload를 시크릿 키로 서명서버만 만들 수 있음
🎗️ 비유 — 위조 방지 입장 팔찌

JWT는 공연장 입장 팔찌와 같습니다. 팔찌 안엔 “좌석 정보(payload)”가 적혀 있어 누구나 읽을 수 있지만, 특수 인장(signature)이 찍혀 있어 손님이 마음대로 좌석을 바꿔 쓰면 인장이 깨져 즉시 들킵니다. 인장 도장은 주최 측(서버)만 갖고 있으니까요. 서버는 매번 인장만 확인하면 되고, 손님 명단을 따로 들고 다닐 필요가 없습니다.

⚠️ payload는 암호화가 아니라 ‘서명’이다

JWT의 payload는 base64일 뿐 누구나 디코딩해 읽을 수 있습니다. 서명은 위변조를 막을 뿐 내용을 가리지 않습니다. 따라서 비밀번호·카드번호 같은 민감 정보를 payload에 넣지 마세요. 보통 sub(사용자 식별자)와 exp 정도만 담습니다.

발급은 jwt.encode, 검증은 jwt.decode입니다. 시크릿 키는 서버만 갖습니다.

security.py — PyJWT
import jwt
from datetime import datetime, timedelta, timezone

SECRET_KEY = "..."           # 프로덕션에선 환경변수/시크릿. 절대 커밋 금지!
ALGORITHM = "HS256"

def create_access_token(sub: str) -> str:
    payload = {
        "sub": sub,                                       # 누구인지
        "exp": datetime.now(timezone.utc) + timedelta(minutes=30),  # 만료
    }
    return jwt.encode(payload, SECRET_KEY, algorithm=ALGORITHM)

def decode_token(token: str) -> dict:
    # 서명이 깨졌거나 exp 가 지났으면 예외 발생
    return jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
💡 exp(만료)는 보안의 핵심

토큰이 유출돼도 피해를 줄이려면 수명을 짧게 둡니다(예: access 30분). jwt.decode는 exp가 지난 토큰을 자동으로 거부합니다(PyJWT는 ExpiredSignatureError). 더 오래 로그인을 유지하려면 별도의 refresh 토큰으로 재발급하는 패턴을 씁니다.

4OAuth2PasswordBearer + /token

FastAPI는 “비밀번호로 로그인해 토큰을 받는” 흐름(OAuth2 Password Flow)을 표준으로 지원합니다. 두 부품을 씁니다.

부품역할
OAuth2PasswordRequestForm/token이 받을 username/password 폼을 정의
OAuth2PasswordBearer보호된 요청에서 Authorization: Bearer ... 헤더의 토큰을 꺼내옴 + Swagger에 자물쇠 버튼 추가
auth.py — /token 엔드포인트
from fastapi import APIRouter, Depends, HTTPException
from fastapi.security import OAuth2PasswordRequestForm, OAuth2PasswordBearer

oauth2_scheme = OAuth2PasswordBearer(tokenUrl="token")  # 토큰 발급 경로 안내
router = APIRouter()

@router.post("/token")
def login(form: OAuth2PasswordRequestForm = Depends()):
    user = get_user(form.username)
    if not user or not verify_password(form.password, user.hashed):
        raise HTTPException(401, "아이디 또는 비밀번호가 틀렸습니다")
    token = create_access_token(sub=user.username)
    return {"access_token": token, "token_type": "bearer"}
📝 라이브러리 선택 — 한 가지로 통일

JWT 라이브러리는 PyJWT(import jwt)와 python-jose(from jose import jwt) 두 가지가 있습니다. 이 스터디는 PyJWT로 통일합니다 — 의존성이 가볍고 유지보수가 활발합니다. 예외 클래스가 다르니 섞지 마세요(PyJWT: jwt.ExpiredSignatureError / jwt.InvalidTokenError, jose: JWTError).

⚠️ python-multipart이 필요하다

OAuth2PasswordRequestForm은 폼 데이터를 파싱합니다. 그래서 python-multipart 패키지가 설치돼 있지 않으면 /token 호출 시 RuntimeError가 납니다. 설치 목록에 python-multipart를 꼭 포함하세요.

6✅ 셀프 체크

Q 인증과 인가의 차이를 한 문장으로?

인증은 “너 누구냐”(신원 확인), 인가는 “너 이거 할 권한 있냐”(권한 확인)입니다. 인증이 먼저 끝나야 인가를 따질 수 있습니다.

Q 패스워드를 sha256으로 해시해서 저장하면 안 되나요?

안 됩니다. sha256은 너무 빨라 무차별 대입·레인보우 테이블에 취약하고 솔트도 자동으로 붙지 않습니다. bcrypt(또는 argon2)는 의도적으로 느리고 솔트를 자동 포함해 같은 비밀번호라도 해시가 매번 다릅니다. 비밀번호엔 전용 알고리즘을 쓰세요.

Q JWT payload에 사용자 비밀번호를 넣어도 되나요?

안 됩니다. payload는 암호화가 아니라 base64일 뿐이라 누구나 디코딩해 읽을 수 있습니다. 서명은 위변조만 막지 내용을 가리지 않습니다. 보통 sub(식별자)와 exp 정도만 담습니다.
