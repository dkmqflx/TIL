의존성으로 보호하기
그리고 Redis

발급한 토큰을 매 요청마다 검증하는 일은 Depends 하나로 깔끔하게 처리됩니다. NestJS의 Guard에 정확히 대응하죠. 더해서 세션·캐시의 단골 도구 Redis를 익힙니다.

“

get_current_user를 의존성으로 만들면, 보호하고 싶은 라우트에 Depends(get_current_user) 한 줄만 더하면 된다. 권한(역할) 체크도 의존성으로 쌓아 올린다. Redis는 디스크 대신 메모리에 두는 초고속 키-값 저장소 — 세션·캐시·TTL의 단골이다.

  • get_current_user 의존성으로 토큰을 검증하고 유저를 주입한다
  • 역할 기반 권한 체크 의존성(require_admin)을 쌓는다
  • JWT(무상태) 방식과 세션(상태) 방식을 비교한다
  • Redis의 set/get/expire와 TTL을 이해한다

1get_current_user — 토큰을 유저로

Lesson 1에서 만든 oauth2_scheme이 Authorization: Bearer ... 헤더에서 토큰 문자열을 꺼내옵니다. 그 토큰을 디코딩해 유저 객체로 바꾸는 의존성이 get_current_user입니다.

deps.py
import jwt
from fastapi import Depends, HTTPException, status

def get_current_user(token: str = Depends(oauth2_scheme)):
    cred_exc = HTTPException(
        status_code=status.HTTP_401_UNAUTHORIZED,
        detail="인증 정보를 확인할 수 없습니다",
        headers={"WWW-Authenticate": "Bearer"},  # 표준 권장 헤더
    )
    try:
        payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
        username = payload.get("sub")
    except jwt.ExpiredSignatureError:
        raise HTTPException(401, "토큰이 만료되었습니다")
    except jwt.InvalidTokenError:        # 서명 위변조 등
        raise cred_exc
    user = get_user(username)
    if user is None:
        raise cred_exc
    return user

이제 보호하고 싶은 라우트는 파라미터 한 줄만 추가하면 됩니다. 의존성이 실패(401)하면 라우트 본문은 아예 실행되지 않습니다.

routers/me.py
@router.get("/me")
def read_me(user: User = Depends(get_current_user)):
    # 여기 도달했다 = 토큰 검증 통과. user 는 신뢰 가능
    return {"username": user.username, "role": user.role}

2권한(역할) 체크 — 의존성을 쌓다

인증을 넘었다고 다 되는 건 아닙니다(= 인가). “관리자만”처럼 역할을 따지려면, get_current_user 위에 또 하나의 의존성을 얹습니다.

deps.py
def require_admin(user: User = Depends(get_current_user)):
    if user.role != "admin":
        raise HTTPException(403, "관리자 권한이 필요합니다")  # 401 아님 → 403
    return user

@router.delete("/users/{id}")
def delete_user(id: int, admin: User = Depends(require_admin)):
    ...  # admin 만 도달
💡 401 vs 403

401 Unauthorized는 “너 누군지 모르겠다”(인증 실패), 403 Forbidden은 “누군진 알겠는데 권한이 없다”(인가 실패)입니다. 토큰이 없거나 깨졌으면 401, 토큰은 멀쩡한데 역할이 부족하면 403을 돌려주세요.

3NestJS Guard와 나란히

이 패턴은 NestJS의 Guard와 발상이 같습니다. “라우트 실행 전에 통과 여부를 판정하는 문지기”죠.

Nest NestJS — Guard
auth.guard.ts
@Injectable()
class AuthGuard {
  canActivate(ctx) {
    const req = ctx.switchToHttp()
                   .getRequest();
    req.user = verify(req.headers.auth);
    return !!req.user;   // false → 401
  }
}

@UseGuards(AuthGuard)
@Get('me')
me() { ... }
Fast FastAPI — Depends
me.py
# Guard = 의존성 함수
@router.get("/me")
def me(
    user = Depends(get_current_user),
):
    return user

@UseGuards ↔ Depends(...). 검증된 유저를 파라미터로 주입받는 점이 더 명시적입니다. 라우터 전체에 걸려면 APIRouter(dependencies=[Depends(...)])를 씁니다.

4JWT(무상태) vs 세션(상태)

“로그인 상태를 어디에 두느냐”에 두 갈래가 있습니다. JWT는 정보를 토큰 안에, 세션은 서버(보통 Redis)에 둡니다.

기준JWT (무상태)세션 (상태)
상태 저장저장 안 함 — 토큰 안에 정보서버(Redis 등)에 세션 저장
확장성좋음 — 서버 간 공유 불필요저장소 공유 필요(보통 Redis)
강제 로그아웃·무효화어렵다 — 만료까지 유효쉽다 — 세션 삭제하면 끝
요청당 비용서명 검증(빠름, DB 조회 X)세션 조회(저장소 접근)
🧭 현실의 절충

실무에선 둘을 섞기도 합니다. 짧은 수명의 JWT access 토큰으로 빠르게 검증하고, refresh 토큰이나 토큰 블랙리스트를 Redis에 두어 강제 무효화를 보완하는 식이죠. “무상태는 무효화가 약하다”는 약점을 Redis로 메우는 것입니다.

5Redis — 메모리에 두는 초고속 저장소

Redis는 데이터를 디스크가 아니라 메모리(RAM)에 두는 키-값 저장소입니다. 그래서 매우 빠르고, 세션·캐시·OTP·레이트리밋 같은 “짧게 자주 쓰는” 데이터의 단골입니다.

🗄️ 비유 — 책상 위 vs 창고

디스크(예: PostgreSQL)는 창고입니다. 모든 걸 영구 보관하지만 꺼내오는 데 시간이 걸리죠. Redis는 책상 위입니다 — 자주 쓰는 물건을 손 닿는 곳에 둬서 즉시 집어 쓸 수 있지만, 공간이 제한적이라 영구 보관용은 아닙니다. “자주 + 빠르게 + 잠깐”이면 책상 위(Redis)에 둡니다.

핵심 명령은 set/get과 TTL(Time To Live, 만료 시간)입니다. TTL을 주면 그 시간이 지나 자동으로 사라집니다 — 세션·OTP에 딱 맞죠.

cache.py — redis.asyncio
import redis.asyncio as redis

r = redis.from_url("redis://localhost:6379", decode_responses=True)

async def demo():
    await r.set("session:42", "민수", ex=1800)  # 값 저장 + 30분 TTL
    name = await r.get("session:42")            # "민수" (30분 안이면)
    await r.expire("session:42", 600)         # TTL 을 10분으로 갱신
    await r.delete("session:42")               # 즉시 무효화(=로그아웃)
명령의미흔한 용도
set(key, val, ex=초)값 저장 + 만료시간세션, 캐시, OTP
get(key)값 조회(없으면 None)세션/캐시 읽기
expire(key, 초)TTL 갱신활동 시 세션 연장
delete(key)즉시 삭제로그아웃, OTP 1회 소비
📝 설치와 클라이언트

파이썬 클라이언트는 redis(redis-py) 패키지입니다(uv add redis). 동기식은 import redis, FastAPI의 async 라우트에선 import redis.asyncio as redis를 써 이벤트 루프를 막지 않습니다(Week 4의 async 원리와 연결). 실행하려면 Redis 서버(예: docker run -p 6379:6379 redis)가 떠 있어야 합니다.

복습

6✅ 셀프 체크

Q 보호된 라우트를 만드는 가장 간단한 방법은?

토큰을 검증해 유저를 반환하는 get_current_user 의존성을 만들고, 라우트 함수에 user = Depends(get_current_user) 파라미터를 추가하면 됩니다. 검증이 실패하면 본문은 실행되지 않고 401이 반환됩니다.

Q NestJS Guard는 FastAPI의 무엇에 대응하나요?

의존성 함수(Depends)입니다. @UseGuards(AuthGuard)가 “라우트 전에 통과 판정”을 하듯, Depends(get_current_user)도 라우트 실행 전에 검증하고 유저를 주입합니다. 역할 체크는 require_admin처럼 의존성을 한 겹 더 쌓습니다.

Q JWT 방식의 약점과, Redis로 어떻게 보완하나요?

JWT는 무상태라 한 번 발급하면 만료 전 강제 무효화(즉시 로그아웃)가 어렵습니다. Redis에 토큰 블랙리스트나 refresh 토큰을 두고 TTL로 관리하면, 무효화가 필요한 토큰을 빠르게 걸러내 이 약점을 보완할 수 있습니다.
