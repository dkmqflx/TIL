OTP & Background Tasks
일회용 코드와 후처리

문자로 받는 일회용 인증번호(OTP)는 Redis의 TTL과 찰떡궁합입니다. 그리고 그 코드를 메일·문자로 발송하는 느린 작업은 응답을 막지 않도록 백그라운드로 넘깁니다.

“

OTP는 Redis에 코드 + 짧은 TTL로 저장하고, 검증에 성공하면 즉시 삭제해 재사용을 막는다. 발송처럼 느린 후처리는 BackgroundTasks로 넘겨 응답을 먼저 돌려주되, 정말 무겁고 긴 작업은 별도 워커(Celery 등)가 맡는다.

  • OTP를 생성해 Redis에 TTL과 함께 저장하고, 검증 후 삭제한다
  • OTP 생성·검증 API의 흐름을 그린다
  • BackgroundTasks로 응답 후 후처리를 비동기로 넘긴다
  • BackgroundTasks의 한계와 워커(Celery)가 필요한 지점을 안다

1OTP — 한 번 쓰고 버리는 코드

OTP(One-Time Password)는 “문자로 온 6자리 인증번호”처럼 딱 한 번, 짧은 시간만 유효한 코드입니다. 본인 확인, 2단계 인증 등에 쓰이죠. 두 가지 성질이 핵심입니다 — 짧은 수명과 1회 소비. 이 둘은 Redis의 TTL과 delete로 깔끔하게 구현됩니다.

⏳ 비유 — 3분 안에 쓰는 보관함 번호

OTP는 무인 택배함의 임시 비밀번호와 같습니다. 발급되면 3분만 유효하고(TTL), 한 번 문을 열면 그 번호는 폐기됩니다(delete). 시간이 지나면 저절로 사라져 따로 청소할 필요도 없죠 — Redis가 TTL로 알아서 지워줍니다.

otp.py — 생성
import secrets
import redis.asyncio as redis

r = redis.from_url("redis://localhost:6379", decode_responses=True)
OTP_TTL = 180  # 3분

async def issue_otp(phone: str) -> str:
    code = f"{secrets.randbelow(1000000):06d}"  # 000000~999999, 보안 난수
    await r.set(f"otp:{phone}", code, ex=OTP_TTL)   # 코드 + 3분 TTL
    return code
otp.py — 검증
async def verify_otp(phone: str, code: str) -> bool:
    saved = await r.get(f"otp:{phone}")   # 만료됐으면 None
    if saved is None or saved != code:
        return False
    await r.delete(f"otp:{phone}")        # 1회 소비 — 재사용 차단
    return True
⚠️ OTP 보안 기본기

① 코드는 random이 아니라 secrets(암호학적 난수)로 생성합니다. ② 검증 성공 시 반드시 삭제해 재사용을 막습니다. ③ 무차별 대입을 막으려면 시도 횟수 제한(예: 5회 후 잠금)을 두세요 — 이것도 Redis 카운터(incr + TTL)로 구현합니다.

routers/otp.py — API
@router.post("/otp/request")
async def request_otp(phone: str, bg: BackgroundTasks):
    code = await issue_otp(phone)
    bg.add_task(send_sms, phone, code)   # 발송은 백그라운드로!
    return {"message": "인증번호를 발송했습니다"}  # code 는 응답에 넣지 않음

@router.post("/otp/verify")
async def check_otp(phone: str, code: str):
    if not await verify_otp(phone, code):
        raise HTTPException(400, "인증번호가 틀렸거나 만료되었습니다")
    return {"verified": True}

2BackgroundTasks — 응답 먼저, 일은 나중에

OTP 문자 발송, 가입 환영 메일, 알림 푸시 — 이런 작업은 사용자가 결과를 기다릴 필요가 없습니다. 그런데 발송이 2초 걸린다고 응답을 2초 늦추면 손해죠. FastAPI의 BackgroundTasks는 응답을 먼저 돌려주고, 등록한 작업을 그 뒤에 실행합니다.

🍽️ 비유 — 손님 보내고 나서 설거지

식당에서 손님이 식사를 마치면 먼저 보내드리고(응답 반환), 설거지는 그 다음에 합니다(백그라운드 작업). 손님을 설거지 끝날 때까지 붙잡아두지 않죠. add_task로 “이따 할 일”을 적어두면, 응답이 나간 뒤 FastAPI가 처리합니다.

main.py — 패턴
from fastapi import FastAPI, BackgroundTasks

def send_welcome_email(to: str):
    ...  # 느린 I/O — 메일 서버 호출 등

@app.post("/signup")
def signup(email: str, bg: BackgroundTasks):
    user = create_user(email)
    bg.add_task(send_welcome_email, email)  # 등록만 — 지금 실행 X
    return {"id": user.id}                   # 응답 먼저 반환 → 그 후 메일 발송

3BackgroundTasks의 한계 — 워커가 필요할 때

BackgroundTasks는 가볍고 편하지만, 같은 서버 프로세스 안에서 실행됩니다. 그래서 한계가 분명합니다.

⚠️ 무겁고 긴 작업엔 별도 워커(Celery 등)를 써라

BackgroundTasks는 ① 작업이 서버 프로세스에 묶여 있어 서버가 죽으면 작업도 사라지고(재시도·내구성 없음), ② 장시간·고부하 작업(영상 인코딩, 대량 배치)은 응답 워커의 자원을 잠식합니다. 이런 경우엔 Celery + Redis/RabbitMQ나 ARQ 같은 독립 작업 큐/워커로 분리하세요. 기준은 단순합니다 — “짧고 가벼운 후처리”는 BackgroundTasks, “무겁거나 반드시 완료돼야 하는 작업”은 워커.

4✅ 셀프 체크

Q OTP의 “1회용·짧은 수명”을 Redis로 어떻게 구현하나요?

코드를 set(key, code, ex=180)으로 TTL과 함께 저장하면 3분 뒤 자동 만료됩니다(짧은 수명). 검증에 성공하면 delete로 즉시 지워 재사용을 막습니다(1회 소비). 코드는 secrets로 생성하고, 시도 횟수 제한도 함께 두는 게 안전합니다.

Q BackgroundTasks는 언제 쓰고, 언제 피해야 하나요?

메일·문자 발송처럼 짧고 가벼운 후처리로 응답을 지연시키고 싶지 않을 때 씁니다. 반면 영상 인코딩·대량 배치처럼 무겁거나, 실패 시 재시도·내구성이 필요한 작업은 서버 프로세스에 묶이는 BackgroundTasks 대신 Celery/ARQ 같은 별도 워커로 분리합니다.

Q OTP 발송 응답에 생성한 코드를 같이 내려주면 편하지 않나요?

절대 안 됩니다. 코드를 응답으로 내려주면 통신을 가로채는 누구나 인증을 통과할 수 있어 OTP의 의미가 사라집니다. 코드는 오직 본인 채널(문자·메일)로만 보내고, 서버 응답엔 “발송했습니다” 정도만 담습니다.
