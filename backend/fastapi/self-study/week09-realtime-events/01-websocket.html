WebSocket
서버가 먼저 말을 거는 채널

지금까지의 HTTP는 클라이언트가 물어봐야만 서버가 답했습니다. WebSocket은 한 번 연결해두면 양쪽이 아무 때나 메시지를 던질 수 있는 지속 회선입니다. 채팅·알림·실시간 대시보드의 토대죠.

“

HTTP가 편지 주고받기(매번 봉투 새로 붙여 한 번에 한 통)라면, WebSocket은 전화 통화다. 한 번 연결되면 끊을 때까지 양방향으로 자유롭게 말한다. 서버가 먼저 말을 걸 수 있다는 게 핵심이다.

  • HTTP와 WebSocket의 통신 모델 차이를 설명한다
  • @app.websocket으로 엔드포인트를 만들고 accept → 수신/송신 루프를 돌린다
  • WebSocketDisconnect로 연결 종료를 안전하게 처리한다
  • ConnectionManager로 다중 접속을 관리하고 broadcast로 채팅을 구현한다

1HTTP vs WebSocket — 편지 vs 전화

HTTP는 요청-응답 모델입니다. 클라이언트가 요청을 보내면 서버가 한 번 답하고 연결이 끝납니다. 다음 데이터를 받으려면 다시 요청해야 하죠. "새 알림 있나요?"를 1초마다 물어보는 폴링(polling)이 그래서 등장했지만, 빈손으로 돌아오는 요청이 대부분이라 낭비가 큽니다.

WebSocket은 처음 한 번 HTTP로 핸드셰이크한 뒤 연결을 그대로 유지합니다. 그 회선 위로 서버와 클라이언트가 양방향으로 아무 때나 메시지를 보냅니다. 서버에 새 데이터가 생기면 서버가 먼저 밀어줄 수 있습니다(push).

📞 비유 — 편지 vs 전화 통화

HTTP는 편지입니다. 할 말이 있을 때마다 봉투를 새로 붙여(연결을 새로 맺어) 한 통 보내고, 답장이 오면 끝. 상대 소식을 알려면 또 편지를 써야 합니다. WebSocket은 전화 통화입니다. 한 번 다이얼해 연결되면 끊기 전까지 양쪽이 자유롭게 말합니다. 상대가 먼저 "여보세요, 새 소식!" 하고 말을 걸 수도 있죠. 이게 실시간의 본질입니다.

항목HTTPWebSocket
방향단방향 (요청→응답)양방향 (full-duplex)
연결요청마다 맺고 끊음한 번 맺고 유지
서버가 먼저 보내기불가 (폴링으로 흉내)가능 (push)
오버헤드매 요청 헤더 반복최초 1회 핸드셰이크 후 가벼움
주소 스킴http:// / https://ws:// / wss://

2언제 WebSocket을 쓰는가

모든 API를 WebSocket으로 바꿀 필요는 없습니다. 서버가 먼저, 자주, 작은 메시지를 밀어줘야 하는 상황에 쓰세요. 일반 CRUD는 HTTP가 더 단순하고 캐싱·로깅·인증 생태계도 잘 갖춰져 있습니다.

상황예시
실시간 채팅메신저, 고객 상담, 협업 댓글
알림 push"새 주문이 들어왔습니다" 즉시 표시
실시간 대시보드주가·트래픽·센서 값이 끊임없이 갱신
협업 편집여러 명이 같은 문서를 동시에 수정

3가장 작은 WebSocket 엔드포인트

FastAPI는 @app.websocket 데코레이터로 WebSocket 라우트를 만듭니다. 함수 안에서 먼저 await websocket.accept()로 핸드셰이크를 수락하고, 그다음 수신/송신 루프를 돕니다. WebSocket 핸들러는 거의 항상 async def입니다.

main.py
from fastapi import FastAPI, WebSocket, WebSocketDisconnect

app = FastAPI()

@app.websocket("/ws")
async def echo(websocket: WebSocket):
    await websocket.accept()          # 핸드셰이크 수락 (필수!)
    try:
        while True:
            text = await websocket.receive_text()   # 클라이언트 메시지 대기
            await websocket.send_text(f"echo: {text}")  # 되돌려 보냄
    except WebSocketDisconnect:
        print("클라이언트 연결 종료")        # 정상적인 종료 신호
⚠️ accept()를 빠뜨리면 연결이 안 된다

HTTP 라우트와 달리 WebSocket은 await websocket.accept()를 호출해야 비로소 연결이 성립합니다. 빠뜨리면 클라이언트 쪽에서 연결이 거부됩니다. 인증을 먼저 확인한 뒤 통과한 경우에만 accept()하고, 막을 땐 await websocket.close(code=1008)로 거절하는 패턴이 흔합니다.

4WebSocketDisconnect — 끊김을 다루기

전화는 언젠가 끊깁니다. 사용자가 탭을 닫거나, 네트워크가 끊기거나, 새로고침하면 연결이 종료되죠. 이때 receive_text()가 WebSocketDisconnect 예외를 던집니다. 이 예외를 잡아 정리(cleanup)하지 않으면, 끊긴 연결이 관리 목록에 남아 다음 broadcast 때 에러를 일으킵니다.

그래서 루프는 try / except WebSocketDisconnect로 감싸고, 예외가 나면 해당 연결을 목록에서 제거하는 것이 기본 패턴입니다. 다음 절의 ConnectionManager가 바로 그 정리를 책임집니다.

5ConnectionManager — 다중 접속 관리

채팅방에는 여러 명이 들어옵니다. "한 명이 보낸 메시지를 접속한 모두에게 뿌린다(broadcast)"를 하려면, 현재 살아있는 연결들을 어딘가 모아둬야 합니다. 그 역할을 하는 작은 클래스가 ConnectionManager입니다. 활성 연결 리스트를 들고, 연결/해제/방송 세 가지만 합니다.

main.py
class ConnectionManager:
    def __init__(self):
        self.active: list[WebSocket] = []   # 살아있는 연결들

    async def connect(self, ws: WebSocket):
        await ws.accept()
        self.active.append(ws)

    def disconnect(self, ws: WebSocket):
        self.active.remove(ws)        # 목록에서 제거 (정리)

    async def broadcast(self, message: str):
        for connection in self.active:
            await connection.send_text(message)  # 모두에게 전송

manager = ConnectionManager()
📝 broadcast는 왜 async이고 await가 필요한가

send_text는 네트워크로 데이터를 내보내는 I/O 작업이라 await 대상입니다. 그래서 이를 호출하는 broadcast도 async def여야 하고, 리스트를 돌며 각 연결마다 await connection.send_text(...)로 보냅니다. Week 4에서 배운 "I/O는 await로 양보한다"가 그대로 적용됩니다.

6간단 채팅 예제 — 다 합치기

이제 라우트 하나로 채팅방이 완성됩니다. 접속하면 manager에 등록하고, 메시지를 받을 때마다 모두에게 방송하고, 끊기면 목록에서 빼고 퇴장을 알립니다.

main.py
@app.websocket("/ws/chat/{name}")
async def chat(websocket: WebSocket, name: str):
    await manager.connect(websocket)            # accept + 목록 등록
    await manager.broadcast(f"🟢 {name} 님 입장")
    try:
        while True:
            text = await websocket.receive_text()
            await manager.broadcast(f"{name}: {text}")  # 모두에게
    except WebSocketDisconnect:
        manager.disconnect(websocket)         # 정리
        await manager.broadcast(f"🔴 {name} 님 퇴장")
💡 NestJS와 비교하면

NestJS는 @WebSocketGateway() + @SubscribeMessage()로 게이트웨이를 선언하고 보통 Socket.IO 위에서 돕니다. FastAPI는 데코레이터 하나(@app.websocket)와 순수 WebSocket 프로토콜로 더 얇게 시작합니다. 대신 방(room)·재연결·폴백 같은 편의 기능은 Socket.IO가 풍부하니, 규모가 커지면 python-socketio를 얹는 선택지도 있습니다.

⚠️ 메모리 안의 리스트는 한 프로세스에만 산다

manager.active는 그 서버 프로세스의 메모리에 있는 리스트입니다. 서버를 여러 대(워커 여러 개)로 띄우면, A 서버에 붙은 사용자와 B 서버에 붙은 사용자는 서로의 메시지를 못 받습니다. 이 스케일아웃 문제를 푸는 것이 다음 강의의 메시지 브로커(Redis Pub/Sub)입니다.

복습

7✅ 셀프 체크

Q HTTP 폴링 대신 WebSocket을 쓰면 뭐가 좋아지나요?

폴링은 "새 데이터 있나요?"를 주기적으로 묻느라 빈손 요청이 대부분이라 낭비가 큽니다. WebSocket은 연결을 유지한 채 서버가 데이터가 생긴 순간 먼저 push하므로, 지연이 작고 불필요한 트래픽이 사라집니다. 채팅·알림처럼 서버발 이벤트가 잦은 경우에 특히 유리합니다.

Q WebSocketDisconnect를 처리하지 않으면 어떻게 되나요?

연결이 끊겨도 manager의 활성 목록에 죽은 연결이 그대로 남습니다. 그 상태로 broadcast하면 끊긴 소켓에 send_text를 시도하다 예외가 나고, 방송 전체가 깨질 수 있습니다. except WebSocketDisconnect에서 disconnect()로 목록에서 제거해야 합니다.

Q broadcast 함수는 왜 async로 만들어야 하나요?

각 연결에 메시지를 보내는 send_text가 네트워크 I/O라서 await 대상이기 때문입니다. await를 쓰려면 그 함수 자체가 async def여야 합니다. 리스트를 돌며 연결마다 await connection.send_text(...)로 보냅니다.
