◆ Week 9 — Lesson 02

메시지 브로커
발행하면, 듣는 쪽이 받는다

서비스끼리 직접 함수를 호출하면 단단히 묶입니다(결합). 이벤트를 발행하고 관심 있는 쪽이 구독하게 만들면, 서로를 몰라도 협력할 수 있습니다. 그 중계소가 메시지 브로커입니다.

“

브로커는 방송국이다. 발행자(publisher)는 "누가 듣는지" 모른 채 채널에 방송하고, 구독자(subscriber)는 관심 채널만 골라 듣는다. 서로를 직접 호출하지 않으니 결합이 풀리고(디커플링), 비동기로 흘러간다.

  • 동기 직접 호출과 이벤트 발행의 차이, 그리고 디커플링의 의미를 안다
  • 메시지 브로커가 왜 비동기·버퍼링·결합도 측면에서 유리한지 설명한다
  • Redis Pub/Sub의 publish / subscribe 기본 흐름을 작성한다
  • WebSocket과 결합해 여러 서버 인스턴스 간 broadcast(스케일아웃)를 구현한다

1이벤트 기반 아키텍처 — 결합도를 낮추기

주문이 들어오면 ①재고 차감, ②알림 발송, ③통계 집계를 해야 한다고 합시다. 동기 직접 호출 방식은 주문 서비스가 이 세 서비스를 차례로 호출합니다.

order_service.py — 직접 호출 (강결합)
def create_order(order):
    save(order)
    inventory.decrease(order)   # 재고 서비스를 직접 안다
    notifier.send(order)        # 알림 서비스를 직접 안다
    stats.record(order)        # 통계 서비스를 직접 안다
    # 알림 서비스가 느리거나 죽으면? 주문 전체가 막힌다.

문제는 두 가지입니다. 주문 서비스가 세 서비스를 전부 알아야 하고(강결합), 하나라도 느리거나 죽으면 주문 응답이 통째로 지연/실패합니다. 새 구독자(예: "배송 예측")를 추가하려면 주문 코드를 또 고쳐야 하죠.

이벤트 발행 방식은 주문 서비스가 그냥 "order.created" 이벤트 하나만 브로커에 던집니다. 누가 듣는지는 모릅니다. 재고·알림·통계는 각자 그 이벤트를 구독해 알아서 처리합니다.

order_service.py — 이벤트 발행 (느슨한 결합)
def create_order(order):
    save(order)
    broker.publish("order.created", order)  # 던지고 끝. 누가 듣든 모름
    # 구독자가 몇이든, 추가/제거돼도 이 코드는 안 바뀐다.
📻 비유 — 방송국 / 우체국

브로커는 방송국입니다. DJ(발행자)는 "지금 몇 명이 듣고 있는지" 모르고 그냥 채널로 방송합니다. 청취자(구독자)는 자기가 관심 있는 주파수(채널)만 맞춰 듣습니다. DJ와 청취자는 서로를 직접 알 필요가 없죠. 우체국으로 봐도 좋습니다. 편지를 우체통(채널)에 넣으면, 그 주소를 구독한 사람이 받습니다. 보내는 사람은 받는 사람의 사정을 신경 쓰지 않습니다.

2왜 메시지 브로커인가 — 3가지 이득

이득의미
비동기발행자는 처리 완료를 기다리지 않는다. 주문 응답이 알림 발송 속도에 묶이지 않음
버퍼링구독자가 잠깐 느리거나 트래픽이 몰려도, 브로커가 메시지를 받아 흘려보낸다(완충)
디커플링발행자와 구독자가 서로를 몰라도 된다. 구독자를 자유롭게 추가/제거

3Redis Pub/Sub — 가장 쉬운 시작

Redis는 인메모리 데이터 저장소이자 가벼운 메시지 브로커입니다. publish(channel, msg)로 채널에 메시지를 던지고, subscribe(channel)로 채널을 구독합니다. FastAPI(비동기)에서는 비동기 클라이언트인 redis.asyncio를 씁니다.

📝 redis.asyncio를 쓰세요 (옛 aioredis 아님)

예전엔 aioredis라는 별도 패키지를 썼지만, 지금은 redis-py에 흡수되어 import redis.asyncio as redis가 표준입니다. 옛 튜토리얼에서 aioredis를 보면 구버전입니다. 설치는 uv add redis 하나면 됩니다.

publisher — 발행
import redis.asyncio as redis

r = redis.from_url("redis://localhost:6379")

async def publish_message(text: str):
    await r.publish("chat", text)   # "chat" 채널로 발행
subscriber — 구독 루프
async def subscribe_loop():
    pubsub = r.pubsub()
    await pubsub.subscribe("chat")          # "chat" 채널 구독
    async for message in pubsub.listen():
        if message["type"] == "message":    # 실제 메시지만 (구독 확인 신호 제외)
            data = message["data"].decode()
            print("받음:", data)
⚠️ listen()은 메시지 외 신호도 흘려보낸다

pubsub.listen()이 처음 내보내는 항목은 "구독 성공(subscribe)" 같은 제어 메시지입니다. 그래서 message["type"] == "message"인 것만 골라 처리해야 합니다. 이 가드를 빠뜨리면 엉뚱한 신호를 데이터로 착각합니다.

4WebSocket × Redis — 여러 서버로 스케일아웃

지난 강의의 마지막 경고를 기억하시나요? ConnectionManager.active는 한 프로세스 메모리에만 있어서, 서버를 여러 대 띄우면 A 서버 사용자와 B 서버 사용자가 서로 못 봅니다. Redis Pub/Sub이 이 벽을 허뭅니다.

핵심 아이디어: 채팅 메시지를 직접 broadcast하지 말고, Redis 채널에 publish한다. 모든 서버 인스턴스는 같은 Redis 채널을 구독하고 있다가, 메시지가 오면 자기에게 붙은 WebSocket들에게만 뿌린다. 그러면 어느 서버에 붙었든 모두가 같은 메시지를 받습니다.

main.py — 다중 서버 채팅
# 1) 클라이언트 메시지 → Redis 채널로 발행 (직접 broadcast 안 함)
@app.websocket("/ws/chat")
async def chat(ws: WebSocket):
    await ws.accept()
    manager.active.append(ws)
    try:
        while True:
            text = await ws.receive_text()
            await r.publish("chat", text)   # Redis로!
    except WebSocketDisconnect:
        manager.active.remove(ws)

# 2) 서버 시작 시: Redis 구독 → 받은 메시지를 내 소켓들에 broadcast
async def redis_to_clients():
    pubsub = r.pubsub()
    await pubsub.subscribe("chat")
    async for msg in pubsub.listen():
        if msg["type"] == "message":
            await manager.broadcast(msg["data"].decode())

5Pub/Sub vs 큐 — 그리고 더 강력한 브로커

한 줄로 짚고 갑니다. Redis Pub/Sub은 메시지를 보존하지 않습니다. 발행하는 순간 듣고 있던 구독자에게만 전달되고, 그때 끊겨 있던 구독자는 그 메시지를 영영 못 받습니다(fire-and-forget). 반면 큐(queue)는 소비될 때까지 메시지를 쌓아두므로, 잠깐 죽었던 워커도 나중에 처리할 수 있습니다.

도구특징적합한 곳
Redis Pub/Sub가볍고 빠름 · 보존 안 함실시간 알림·채팅 fan-out
RabbitMQ큐·ACK·재시도·라우팅 풍부작업 처리 보장이 중요한 비동기 작업
Kafka로그 보존·대용량 스트림·재처리이벤트 스트리밍·데이터 파이프라인
💡 유실되면 안 되는 메시지라면

"알림이 한 번쯤 빠져도 괜찮다"면 Pub/Sub으로 충분합니다. 하지만 "결제 처리 이벤트는 절대 잃으면 안 된다"면 보존·ACK·재시도를 제공하는 RabbitMQ나 Kafka, 혹은 Redis Streams를 고려하세요. 메시지의 유실 허용 여부가 도구 선택의 첫 갈림길입니다.

복습

6✅ 셀프 체크

Q 직접 호출 대신 이벤트를 발행하면 결합도가 왜 낮아지나요?

발행자는 채널에 이벤트만 던질 뿐, 누가 구독하는지 알지 못하기 때문입니다. 구독자를 추가/제거해도 발행자 코드는 바뀌지 않습니다. 또 발행자가 구독자의 처리 완료를 기다리지 않으므로(비동기), 느린 구독자 때문에 발행자가 막히지도 않습니다.

Q WebSocket 채팅에 Redis Pub/Sub을 왜 끼워 넣나요?

WebSocket 연결 목록은 각 서버 프로세스 메모리에만 있어, 서버를 여러 대 띄우면 다른 서버에 붙은 사용자끼리 메시지가 안 닿습니다. 메시지를 Redis 채널에 publish하고 모든 서버가 그 채널을 구독해 자기 소켓들에 broadcast하면, 어느 서버에 붙었든 전부 같은 메시지를 받습니다(스케일아웃).

Q Redis Pub/Sub과 메시지 큐의 결정적 차이는?

Pub/Sub은 메시지를 보존하지 않습니다. 발행 시점에 듣고 있던 구독자에게만 전달되고, 끊겨 있던 구독자는 그 메시지를 못 받습니다. 큐는 소비될 때까지 메시지를 쌓아두어, 잠시 죽었던 워커도 복귀 후 처리할 수 있습니다. 유실이 곤란하면 RabbitMQ/Kafka/Redis Streams를 씁니다.
