◆ Week 9 — Lesson 03

성능, 추측하지 말고
측정하라

"느린 것 같다"는 느낌이 아니라 숫자로 말해야 합니다. 부하 테스트로 RPS와 지연을 재고, I/O 바운드 작업에서 async가 처리량을 어떻게 끌어올리는지 확인한 뒤, 흔한 병목을 하나씩 걷어냅니다.

“

성능은 측정이 먼저다. 부하를 걸어 RPS(처리량)와 p95 지연을 보고, 병목이 어디인지 숫자로 짚는다. I/O 바운드라면 async 전환이 같은 서버로 훨씬 많은 동시 요청을 처리하게 해준다.

  • 부하 테스트 도구(locust·wrk·ab)로 RPS와 지연(p50/p95/p99)을 측정한다
  • sync def·event-loop 블로킹·async def의 동시성 차이를 정확히 구분한다
  • I/O 바운드에서 async 전환의 처리량 효과를 수치로 이해한다
  • 흔한 병목(동기 라이브러리·N+1·캐싱·커넥션 풀·워커 수)을 개선한다

1왜 측정하는가 — 감이 아니라 숫자

"이 엔드포인트가 느린 것 같아"는 디버깅의 출발점이 못 됩니다. 어디가, 얼마나, 어떤 부하에서 느린지 모르면 엉뚱한 곳을 고치게 되죠. 그래서 부하 테스트(load test)로 동시 요청을 쏟아붓고 다음 지표를 봅니다.

지표의미
RPS (requests/sec)초당 처리한 요청 수 = 처리량(throughput). 높을수록 좋다
p50 지연요청의 절반이 이 시간 안에 끝남 (중앙값)
p95 / p99 지연95% / 99%가 이 시간 안에 끝남. 꼬리 지연(tail latency) — 사용자 체감에 중요
💡 평균이 아니라 p95·p99를 보라

평균 지연은 소수의 느린 요청을 가립니다. 평균 50ms라도 p99가 2초면, 100명 중 1명은 2초를 기다린다는 뜻이죠. 사용자 경험은 꼬리(p95/p99)가 결정합니다. 그래서 성능 리포트는 백분위수로 읽습니다.

도구특징언제
ab (ApacheBench)설치 간단 · 단일 URL 빠른 측정"이 엔드포인트 RPS 대충 보자"
wrk가볍고 매우 고성능 · 멀티스레드높은 부하로 한계 RPS 측정

# ApacheBench: 총 1000 요청, 동시 50

# wrk: 4스레드, 동시 연결 100, 10초 동안
wrk -t4 -c100 -d10s http://localhost:8000/async
locustfile.py — 시나리오 부하
from locust import HttpUser, task

class ApiUser(HttpUser):
    @task
    def hit_async(self):
        self.client.get("/async")   # locust -f locustfile.py 로 실행, 웹 UI에서 사용자 수 조절

3sync def vs async def — 정확히 무슨 일이 일어나나

흔한 오해부터 바로잡습니다. "sync def 엔드포인트는 서버 전체를 멈춘다"는 틀린 말입니다. FastAPI는 동작이 다음 세 가지로 갈립니다.

작성 방식실제 동작동시성
def (sync) 엔드포인트FastAPI가 외부 스레드풀에서 실행 (기본 약 40개)제한적 (워커 수만큼)
async def 안에서 블로킹 호출
(예: time.sleep, 동기 DB)		이벤트 루프 전체가 멈춤 😱최악 — 모두 막힘
async def + await (async 드라이버)I/O 대기 중 루프가 다른 요청 처리매우 높음 ✅
⚠️ 가장 위험한 건 async def 안의 블로킹

sync def는 스레드풀로 보내져 동시성이 "제한"될 뿐 멈추진 않습니다. 정말 치명적인 건 async def 안에서 time.sleep(1)이나 동기 DB 호출 같은 블로킹 코드를 쓰는 경우입니다. 이건 이벤트 루프 단일 스레드를 통째로 잡아, 그동안 다른 모든 요청이 얼어붙습니다. async 함수 안에서는 반드시 await 가능한 비동기 버전(asyncio.sleep, async DB 드라이버)을 쓰세요.

4데모 — I/O 바운드에서 async의 효과

외부 API 호출이나 DB 조회처럼 기다리는 시간이 대부분인 I/O 바운드 작업을 가정합시다. 각 요청이 100ms를 "그냥 대기"한다고 할 때, 두 방식을 비교합니다. (아래는 code/main.py의 데모 엔드포인트 쌍입니다.)

main.py — 블로킹 vs 비동기
import asyncio, time

# ❌ async 함수 안에서 블로킹 → 이벤트 루프가 멈춘다
@app.get("/blocking")
async def blocking():
    time.sleep(0.1)        # 루프 전체를 막는 100ms
    return {"ok": True}

# ✅ await로 양보 → 대기 중 다른 요청 처리
@app.get("/async")
async def good_async():
    await asyncio.sleep(0.1)  # 100ms 동안 루프를 양보
    return {"ok": True}

같은 100ms 대기지만 결과는 극과 극입니다. /blocking은 요청을 사실상 한 줄로 세워 처리하므로 RPS가 ~10에 머뭅니다(1초/0.1초). /async는 대기 중 다른 요청을 받아 수백 개를 겹쳐 처리하므로 RPS가 수백~수천으로 치솟습니다. Week 4에서 배운 "I/O는 await로 양보"가 처리량으로 직결되는 장면입니다.

📝 아래 숫자는 측정값이 아니라 설명용 예시

다음 그래프의 RPS 값은 원리를 보여주기 위한 illustrative(예시) 수치입니다. 실제 값은 하드웨어·워커 수·작업 성격에 따라 달라지니, 꼭 여러분 환경에서 직접 측정하세요(2절 도구 사용). 핵심은 절대값이 아니라 경향입니다.

💡 async가 항상 이기는 건 아니다 — CPU 바운드

위 효과는 대기가 많은 I/O 바운드에서의 이야기입니다. 이미지 처리·암호 해싱 같은 CPU 바운드는 양보할 대기 시간이 없어 async로 바꿔도 처리량이 늘지 않습니다. 오히려 단일 이벤트 루프를 점유해 해롭죠. CPU 작업은 run_in_executor(프로세스/스레드)나 별도 워커·작업 큐로 빼야 합니다.

5흔한 병목 & 개선

측정으로 느린 지점을 찾았다면, 다음 순서로 의심해보세요.

병목증상개선
동기 라이브러리async 앱인데 처리량이 안 오름async 드라이버로 전환 (httpx, async DB), 불가피하면 run_in_executor
N+1 쿼리목록 1개당 쿼리 N번 추가 발생JOIN·selectinload 등으로 한 번에 로드
반복 계산/조회같은 결과를 매번 다시 계산캐싱(Redis) — 결과를 잠시 저장해 재사용
커넥션 풀 부족부하 시 DB 연결 대기로 지연 급증풀 크기 튜닝(pool_size, max_overflow)
워커 수CPU 코어를 못 쓰거나, 너무 많아 메모리 압박코어 수에 맞춰 --workers 조정
terminal — 워커 늘려 띄우기
# 멀티 프로세스로 CPU 코어 활용 (CPU 바운드·전체 처리량에 유효)
uvicorn main:app --workers 4
# 운영에선 보통 gunicorn + uvicorn worker 조합을 많이 쓴다
⚠️ 개선 후엔 다시 측정하라 (1절로 복귀)

"고쳤으니 빨라졌겠지"도 추측입니다. 변경 전후로 같은 부하 테스트를 돌려 RPS·p95를 비교해야 개선이 실제였는지, 혹은 다른 병목으로 옮겨갔을 뿐인지 알 수 있습니다. 측정 → 개선 → 재측정의 루프가 성능 작업의 전부입니다.

복습

6✅ 셀프 체크

Q "sync def 엔드포인트는 서버 전체를 멈춘다"가 왜 틀린 말인가요?

FastAPI는 sync def 엔드포인트를 외부 스레드풀(기본 약 40개)에서 실행하므로, 동시성이 워커 수만큼 제한될 뿐 멈추지는 않습니다. 진짜 멈추는 경우는 async def 안에서 time.sleep 같은 블로킹 코드를 쓸 때입니다. 이때 이벤트 루프가 통째로 잡혀 다른 모든 요청이 얼어붙습니다.

Q I/O 바운드에서 async가 처리량을 올리는 원리는?

요청이 외부 응답을 기다리는 동안 await로 이벤트 루프에 제어를 양보하고, 루프는 그 사이 다른 요청을 처리합니다. 대기 시간이 겹쳐 처리되므로 같은 서버로 훨씬 많은 동시 요청을 소화합니다. 단, 양보할 대기가 없는 CPU 바운드에는 효과가 없습니다.

Q 성능 리포트에서 평균보다 p95·p99를 보는 이유는?

평균은 소수의 느린 요청을 가립니다. 평균이 낮아도 p99가 크면 일부 사용자는 매우 오래 기다린다는 뜻이고, 체감 품질은 이 꼬리 지연이 좌우합니다. 그래서 백분위수(p50/p95/p99)로 분포를 보고, 개선 전후를 같은 부하로 비교합니다.
