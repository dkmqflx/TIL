pytest 심화
Mocking · Fixture · TestClient

"테스트가 통과하니 배포해도 된다"고 말하려면, 테스트가 빠르고 격리되고 반복 가능해야 합니다. pytest의 fixture·mocking과 FastAPI의 TestClient + dependency_overrides로 그 토대를 세웁니다.

“

좋은 테스트는 매번 같은 결과를 낸다. 그러려면 외부 세계(실 DB·외부 API)를 끊어내야 한다. fixture로 셋업/티어다운을 재사용하고, dependency_overrides로 진짜 DB를 테스트 DB로 갈아끼우고, mock으로 외부 호출을 가짜 응답으로 대체한다.

  • fixture로 테스트 셋업/티어다운을 재사용하고 conftest.py로 공유한다
  • TestClient로 GET/POST/PATCH/DELETE를 상태코드·본문까지 검증한다
  • dependency_overrides로 get_db를 테스트용 sqlite로 교체한다
  • monkeypatch/Mock으로 외부 의존성을 격리한다

1테스트 피라미드 — 무엇을 얼마나

테스트는 다 같은 무게가 아닙니다. 단위(unit) > 통합(integration) > e2e 순으로 아래쪽을 두텁게 쌓는 게 정석입니다. 단위 테스트는 빠르고 많고, 위로 갈수록 느리고 적게. 우리가 오늘 쓰는 TestClient 테스트는 "앱+DB를 함께 도는" 통합 테스트 영역입니다.

🎭 비유 — Mock은 대역 배우

액션 영화에서 위험한 장면은 스턴트 대역이 대신합니다. 외부 결제 API·날씨 API처럼 "진짜로 부르면 느리고·돈 들고·매번 답이 다른" 의존성은, 테스트에서 가짜 응답을 주는 대역(Mock)으로 바꿔칩니다. 덕분에 테스트는 네트워크 없이도 항상 같은 결과를 냅니다.

2fixture — 셋업/티어다운을 재사용

모든 테스트가 "DB 연결 → 테이블 생성 → ... → 정리"를 반복하면 코드가 더럽습니다. pytest fixture는 그 준비물(setup)과 뒷정리(teardown)를 함수 하나로 묶어 재사용하게 해줍니다. yield 위는 셋업, 아래는 티어다운입니다.

test 예시
import pytest

@pytest.fixture
def client():
    Base.metadata.create_all(bind=test_engine)   # setup: 테이블 생성
    app.dependency_overrides[get_db] = override_get_db
    with TestClient(app) as c:
        yield c                               # 테스트에 c 를 넘겨줌
    app.dependency_overrides.clear()         # teardown: 정리
    Base.metadata.drop_all(bind=test_engine)

def test_create(client):                       # 인자 이름 = fixture 이름
    res = client.post("/todos", json={"title": "장보기"})
    assert res.status_code == 201
💡 conftest.py — 여러 파일이 공유하는 fixture

같은 client·db_session fixture를 여러 테스트 파일에서 쓰고 싶다면 conftest.py에 정의하세요. pytest는 그 디렉터리(및 하위)의 모든 test_*.py에서 import 없이 자동으로 인식합니다. 공유 셋업의 단일 출처가 됩니다.

Nest NestJS — Jest + supertest
todos.e2e-spec.ts
beforeAll(async () => {
  const mod = await Test
    .createTestingModule({ imports: [AppModule] })
    .compile();
  app = mod.createNestApplication();
  await app.init();
});

it('POST /todos', () =>
  request(app.getHttpServer())
    .post('/todos').expect(201));
Fast FastAPI — pytest + TestClient
test_main.py
@pytest.fixture
def client():
    with TestClient(app) as c:
        yield c

def test_create(client):
    res = client.post("/todos",
        json={"title": "x"})
    assert res.status_code == 201

beforeAll/afterAll ↔ fixture의 yield 전후. supertest ↔ TestClient.

3TestClient — CRUD를 실제로 두드려보기

TestClient는 httpx 기반으로, 진짜 서버를 띄우지 않고 ASGI 앱을 메모리에서 직접 호출합니다. 빠르고, 네트워크가 필요 없죠. CRUD마다 상태코드와 응답 본문을 모두 확인하는 게 핵심입니다.

test_main.py
def test_get_todo(client):
    created = client.post("/todos", json={"title": "운동"}).json()
    res = client.get(f"/todos/{created['id']}")
    assert res.status_code == 200          # 상태코드 확인
    assert res.json()["title"] == "운동"    # 본문 확인

def test_patch_todo(client):
    cid = client.post("/todos", json={"title": "독서"}).json()["id"]
    res = client.patch(f"/todos/{cid}", json={"done": True})
    assert res.json()["done"] is True

def test_delete_todo(client):
    cid = client.post("/todos", json={"title": "삭제"}).json()["id"]
    assert client.delete(f"/todos/{cid}").status_code == 204
    assert client.get(f"/todos/{cid}").status_code == 404  # 삭제 후 없음

4dependency_overrides — 진짜 DB를 테스트 DB로

테스트가 운영 DB를 건드리면 재앙입니다. FastAPI의 app.dependency_overrides는 의존성(get_db)을 통째로 다른 함수로 갈아끼우는 공식 메커니즘입니다. 운영에선 get_db가 PostgreSQL 세션을 주지만, 테스트에선 별도 sqlite를 주는 함수로 바꿉니다.

test_main.py
from sqlalchemy.pool import StaticPool

# 테스트용 in-memory sqlite — 커넥션을 공유해야 테이블이 유지된다
test_engine = create_engine(
    "sqlite://",
    connect_args={"check_same_thread": False},
    poolclass=StaticPool,                  # 핵심! 단일 커넥션 재사용
)
TestingSessionLocal = sessionmaker(bind=test_engine)

def override_get_db():
    db = TestingSessionLocal()
    try:
        yield db
    finally:
        db.close()

app.dependency_overrides[get_db] = override_get_db   # 갈아끼우기
⚠️ in-memory sqlite의 함정 — StaticPool

"sqlite://"(in-memory)는 커넥션마다 DB가 새로 생깁니다. 기본 풀을 쓰면 매 요청이 새 커넥션 → 방금 만든 테이블이 사라져 CRUD 테스트가 알 수 없는 이유로 실패합니다. 반드시 poolclass=StaticPool + check_same_thread=False로 단 하나의 커넥션을 공유하게 하세요. (대안: sqlite:///./test.db 파일 사용.)

5Mocking — 외부 의존성 격리

DB는 테스트 DB로 바꿨지만, "외부 날씨 API·결제 API" 같은 네트워크 호출은 어떻게 할까요? 함수 자체를 가짜로 갈아끼웁니다. pytest의 monkeypatch 또는 표준 라이브러리 unittest.mock을 씁니다.

test_main.py
def call_weather_api(city):       # 실제론 외부 HTTP 호출
    ...

def summarize_weather(city):
    data = call_weather_api(city)
    return f"{city}: {data['temp']}도"

def test_summary(monkeypatch):
    # 외부 API 대신 가짜 응답을 주는 "대역"을 주입
    monkeypatch.setattr("test_main.call_weather_api",
                        lambda c: {"temp": 21})
    assert summarize_weather("서울") == "서울: 21도"
도구방식언제
monkeypatch속성/함수/환경변수를 테스트 동안만 교체pytest 기본, 간단한 교체
unittest.mock.MagicMock가짜 객체 + 호출 여부·인자 검증"몇 번 불렸나"까지 확인
mocker (pytest-mock)mock을 fixture로 깔끔하게mock을 자주 쓸 때
dependency_overridesFastAPI 의존성 자체를 교체DB·인증 등 Depends 대상
🧭 "어디를" patch 하느냐가 핵심

mock은 정의된 곳이 아니라 사용되는 곳을 패치해야 먹힙니다. summarize_weather가 test_main 모듈에서 call_weather_api를 부른다면 "test_main.call_weather_api"를 패치하세요. 이걸 헷갈리면 "분명 mock 했는데 진짜가 불린다"는 함정에 빠집니다.

복습

6✅ 셀프 체크

Q 테스트에서 실제 DB 대신 테스트 DB를 쓰게 하는 FastAPI 메커니즘은?

app.dependency_overrides[get_db] = override_get_db입니다. Depends(get_db)를 쓰는 모든 엔드포인트가 자동으로 테스트용 세션을 받게 됩니다. 테스트가 끝나면 .clear()로 원복합니다.

Q in-memory sqlite로 TestClient를 돌렸더니 "테이블이 없다"고 합니다. 왜죠?

기본 커넥션 풀에서는 요청마다 새 커넥션이 잡히고, in-memory DB는 커넥션마다 새로 만들어지기 때문입니다. create_engine("sqlite://", connect_args={"check_same_thread": False}, poolclass=StaticPool)로 단일 커넥션을 공유시키면 해결됩니다.

Q monkeypatch와 dependency_overrides는 언제 무엇을 쓰나요?

외부 API 호출 같은 일반 함수/속성은 monkeypatch(또는 unittest.mock)로 교체합니다. Depends로 주입되는 FastAPI 의존성(DB 세션, 현재 사용자 등)은 dependency_overrides로 교체합니다.
