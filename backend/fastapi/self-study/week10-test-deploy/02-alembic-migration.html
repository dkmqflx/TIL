Alembic 마이그레이션
DB 스키마의 버전 관리

개발 초기엔 create_all()로 충분했지만, 운영 중 "컬럼 하나 추가"가 필요한 순간 그 한계가 드러납니다. 스키마 변경을 안전하게 이력으로 남기는 도구 — Alembic을 익힙니다.

“

create_all()은 없는 테이블만 만들 뿐, 이미 있는 테이블의 컬럼을 바꾸지 못한다. Alembic은 스키마 변경을 버전 파일(rev)로 기록하고 upgrade/downgrade로 앞뒤로 옮긴다 — 한마디로 DB 스키마용 git이다.

  • create_all()의 한계와 마이그레이션이 필요한 이유를 안다
  • alembic init → env.py 연결 → revision → upgrade 흐름을 익힌다
  • 마이그레이션 파일의 upgrade/downgrade 구조를 읽는다
  • autogenerate의 한계를 알고 생성 결과를 검토한다

1왜 마이그레이션인가

Week 7에서 Base.metadata.create_all(engine)로 테이블을 만들었습니다. 그런데 이 함수는 "존재하지 않는 테이블만 새로 만들" 뿐, 이미 있는 테이블에 컬럼을 더하거나 타입을 바꾸지는 못합니다. 운영 DB에 데이터가 쌓인 상태에서 "할 일에 우선순위 컬럼 추가"가 필요해지면, create_all()은 아무 일도 하지 않습니다.

🌿 비유 — Alembic은 DB 스키마용 git

코드는 git으로 커밋·이력 관리하고, 문제가 생기면 이전 커밋으로 되돌립니다. DB 스키마도 똑같이 다뤄야 합니다. Alembic의 각 마이그레이션 파일은 하나의 커밋이고, upgrade는 앞으로 가는 변경, downgrade는 되돌리는 변경입니다. alembic upgrade head는 "최신 커밋까지 따라잡기"인 셈이죠.

2시작하기 — init과 env.py 연결

alembic init으로 마이그레이션 작업 폴더를 만들고, env.py에 우리 모델의 metadata를 연결합니다. 이 연결이 있어야 autogenerate가 "모델 ↔ 실제 DB"의 차이를 감지합니다.

terminal
# 마이그레이션 폴더(alembic/) + alembic.ini 생성
$ alembic init alembic
alembic/env.py
# 우리 모델의 Base 를 가져와 metadata 를 연결한다
from app.database import Base
from app import models          # 모든 모델이 Base 에 등록되도록 import

target_metadata = Base.metadata   # ← autogenerate 가 이걸 기준으로 비교

# DB URL 은 alembic.ini 의 sqlalchemy.url 또는 env 변수로 주입
💡 모델 import를 빠뜨리지 말 것

autogenerate는 target_metadata에 등록된 모델만 봅니다. models.py를 어딘가에서 import하지 않으면 그 테이블이 metadata에 안 잡혀, Alembic이 "이 테이블 지워야겠네"라고 오판할 수 있습니다. env.py에서 모델 모듈을 확실히 import하세요.

3변경 만들기 — revision & upgrade

모델을 수정한 뒤 --autogenerate로 마이그레이션을 생성하면, Alembic이 "모델과 현재 DB의 차이"를 읽어 변경 스크립트 초안을 만들어 줍니다. 그다음 upgrade head로 적용합니다.

terminal
# 1) 모델 변경 후, 차이를 읽어 마이그레이션 파일 생성
$ alembic revision --autogenerate -m "add priority to todos"

# 2) 최신 버전까지 적용 (DB 에 실제 반영)
$ alembic upgrade head

# 되돌리기: 직전 버전으로 한 칸
$ alembic downgrade -1

# 현재 버전 / 이력 확인
$ alembic current
$ alembic history

생성된 마이그레이션 파일은 upgrade()(앞으로)와 downgrade()(되돌리기) 두 함수로 이뤄집니다. down_revision이 부모(이전) 버전을 가리켜 사슬처럼 연결됩니다.

alembic/versions/3f9a..._add_priority.py
revision = "3f9a1c"
down_revision = "a1b2c3"      # 이전 버전 = 부모 (사슬 연결)

def upgrade():
    op.add_column("todos",
        sa.Column("priority", sa.Integer(), server_default="1"))

def downgrade():
    op.drop_column("todos", "priority")   # upgrade 의 정확한 역연산

4NestJS TypeORM 마이그레이션과 나란히

Nest NestJS — TypeORM
terminal + migration
# 엔티티 변경 후 생성
$ typeorm migration:generate \
    src/migrations/AddPriority
$ typeorm migration:run
$ typeorm migration:revert

// 1700000000-AddPriority.ts
async up(q): { /* ALTER … */ }
async down(q): { /* 되돌리기 */ }
Fast FastAPI — Alembic
terminal + migration
# 모델 변경 후 생성
$ alembic revision --autogenerate \
    -m "add priority"
$ alembic upgrade head
$ alembic downgrade -1

# versions/..._add_priority.py
def upgrade(): ...
def downgrade(): ...

migration:generate ↔ revision --autogenerate, up/down ↔ upgrade/downgrade. 개념이 1:1로 맞습니다.

5autogenerate의 한계 — 반드시 검토

autogenerate는 강력하지만 만능이 아닙니다. 테이블·컬럼 추가/삭제 같은 굵직한 변경은 잘 잡지만, 일부 변경은 놓치거나 위험하게 생성합니다.

⚠️ 생성된 마이그레이션은 사람이 꼭 검토

autogenerate가 못 잡거나 부정확한 대표 사례: 컬럼 타입 변경(detect 옵션 없으면 무시), 컬럼/테이블 이름 변경(drop+add로 오인 → 데이터 유실!), 일부 제약·인덱스, 서버 기본값 등. 운영 데이터가 걸린 변경일수록 생성된 upgrade()/downgrade()를 눈으로 확인하고 수정한 뒤 적용하세요. "git diff 없이 머지"하지 않듯이요.

🧭 create_all과 Alembic은 함께 쓰지 않는다

Alembic을 도입했다면 앱 시작 시의 Base.metadata.create_all()은 제거하세요. 둘이 동시에 스키마를 만들면 "이미 테이블이 있는데 Alembic은 처음부터 만들려 한다"는 충돌이 납니다. 스키마의 단일 진실 공급원(SSOT)은 마이그레이션 이력이어야 합니다.

6✅ 셀프 체크

Q create_all()로는 왜 컬럼 추가가 안 되나요?

create_all()은 "존재하지 않는 테이블을 생성"하는 함수일 뿐, 이미 존재하는 테이블의 스키마(컬럼 추가·타입 변경 등)는 건드리지 않습니다. 변경 이력을 다루려면 Alembic 같은 마이그레이션 도구가 필요합니다.

Q autogenerate가 만든 마이그레이션을 그대로 믿어도 되나요?

아니요. 타입 변경을 놓치거나, 컬럼 이름 변경을 drop+add(데이터 유실)로 오인할 수 있습니다. 생성된 upgrade()/downgrade()를 반드시 검토하고, 필요하면 직접 수정한 뒤 적용해야 합니다.

Q 배포한 변경에 문제가 생겼습니다. 어떻게 되돌리나요?

alembic downgrade -1로 직전 버전으로, 또는 alembic downgrade <revision>로 특정 버전까지 되돌립니다. 각 마이그레이션의 downgrade()가 upgrade()의 역연산을 정확히 담고 있어야 안전하게 롤백됩니다.
