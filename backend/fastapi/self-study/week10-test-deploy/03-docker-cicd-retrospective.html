“

"내 컴퓨터에선 됐는데요"를 끝내는 게 Docker다. 멀티스테이지로 가볍게 굽고, compose로 app+DB+redis를 한 번에 띄운다. GitHub Actions가 푸시마다 lint·test를 돌려 깨진 코드를 막고, 통과하면 배포로 이어진다.

1Dockerfile — 멀티스테이지로 가볍게

핵심 아이디어는 빌드 도구(uv, 캐시)는 빌더 스테이지에만 두고, 최종 이미지엔 슬림 베이스 + 만들어진 가상환경(.venv)만 복사하는 것입니다. 그래야 이미지가 작고, 공격 표면도 줄어듭니다.

Dockerfile
# ---- 1) builder: 의존성 설치 전용 ----
FROM python:3.12-slim AS builder
COPY --from=ghcr.io/astral-sh/uv:latest /uv /uvx /bin/   # uv 바이너리만 가져옴
WORKDIR /app
COPY pyproject.toml uv.lock ./
RUN uv sync --frozen --no-dev --no-install-project   # 락대로, 개발의존성 제외
COPY . .
RUN uv sync --frozen --no-dev

# ---- 2) final: 실행 전용 (uv 없음, 복사만) ----
FROM python:3.12-slim AS final
WORKDIR /app
COPY --from=builder /app/.venv /app/.venv   # 가상환경만 가져옴 → 경량
COPY --from=builder /app /app
ENV PATH="/app/.venv/bin:$PATH"
EXPOSE 8000
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
💡 레이어 캐시 — 락 파일을 먼저 복사하는 이유

소스보다 pyproject.toml + uv.lock을 먼저 복사하고 의존성을 설치하면, 소스만 바뀐 빌드에서는 의존성 레이어가 캐시에서 재사용됩니다. 빌드가 수십 초 → 수 초로 줄죠. --frozen은 락 파일 그대로 설치(재현성), 최종 스테이지에 uv를 두지 않는 게 경량화의 핵심입니다.

2docker-compose — app + postgres + redis

운영 앱은 혼자 살지 않습니다. DB(postgres)·캐시/브로커(redis)와 함께 떠야 하죠. docker-compose로 한 파일에 묶어 docker compose up 한 번에 띄웁니다. 컨테이너 네트워크 안에서는 서비스 이름이 곧 호스트명이 됩니다(db, redis).

docker-compose.yml
services:
  app:
    build: .
    ports: ["8000:8000"]
    environment:
      DATABASE_URL: postgresql+psycopg://app:secret@db:5432/appdb
      REDIS_URL: redis://redis:6379/0
    depends_on: [db, redis]
  db:
    image: postgres:16-alpine
    environment: { POSTGRES_USER: app, POSTGRES_PASSWORD: secret, POSTGRES_DB: appdb }
  redis:
    image: redis:7-alpine

3GitHub Actions — 푸시마다 자동 검사

아무리 좋은 테스트도 돌리지 않으면 무용입니다. CI는 그걸 자동화합니다. push·pull_request 때마다 GitHub의 러너가 코드를 받아 ruff(lint) + pytest(test)를 실행하고, 하나라도 실패하면 머지를 막습니다.

.github/workflows/ci.yml
name: CI
on:
  push: { branches: [main] }
  pull_request: { branches: [main] }
jobs:
  lint-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: astral-sh/setup-uv@v5
        with: { enable-cache: true }
      - run: uv python install
      - run: uv sync --frozen
      - run: uv run ruff check .      # lint
      - run: uv run pytest -v          # test

4배포 — 어디에 올릴까

이미지를 만들었다면 올릴 곳을 정합니다. 손이 가는 정도와 통제력은 반비례합니다.

방식설명특징
PaaS (Railway·Render·Fly.io)이미지/리포만 연결하면 알아서 빌드·배포가장 쉬움, 초기 추천
VM (EC2·Lightsail)서버에 직접 docker compose로 기동통제력↑, 운영 부담↑
컨테이너 오케스트레이션 (ECS·k8s)여러 인스턴스 자동 확장·롤링 배포대규모, 복잡도↑
🧭 CI 다음의 CD, 그리고 마이그레이션

CI(검사)가 통과하면 CD(배포)가 이어집니다. 배포 직후에는 보통 alembic upgrade head를 실행해 DB 스키마를 코드와 맞춥니다(Lesson 02). 환경 변수·시크릿은 이미지에 굽지 말고 플랫폼의 시크릿 저장소로 주입하세요.

5🎉 우리가 쌓아온 여정

처음 "FastAPI가 뭐지?"에서 시작해 여기까지 왔습니다. 한 줄기로 꿰어 보면 이렇습니다 — 큰 그림(W1)으로 생태계를 잡고, 타입 힌트(W1)가 런타임까지 살아남는다는 FastAPI의 토대를 이해했습니다. Request/HTTP(W2)로 요청을 받고, Pydantic(W3)으로 데이터를 검증·응답 설계하고, async(W4~)로 "왜 빠른가"를 원리로 설명할 수 있게 됐죠. DB(W5·W7)로 데이터를 영속화하고, API 구조화(W6)로 라우터·서비스 계층을 나누고, 인증(W8)으로 지키고, 실시간/이벤트(W9)로 살아 움직이게 했습니다. 그리고 마지막으로 테스트와 배포(W10)로 "확신을 가지고 세상에 내보내는" 능력까지. 이제 당신은 읽고 쓸 줄 아는 백엔드 개발자입니다.

🏔️ 비유 — 정상에서 보는 풍경

프론트엔드(React/NestJS)라는 베이스캠프에서 출발해, 한 주 한 주 능선을 올랐습니다. 처음엔 낯설던 파이썬 타입 힌트도, async도, ORM도 이제 손에 익은 장비가 됐죠. 정상에서 돌아보니 길이 하나로 이어져 있습니다 — 그게 진짜 실력입니다.

7✅ 셀프 체크

Q Dockerfile을 멀티스테이지로 나누면 무엇이 좋아지나요?

빌드 도구(uv·캐시·빌드 산출물)는 빌더 스테이지에만 남고, 최종 이미지엔 슬림 베이스와 완성된 가상환경(.venv)만 복사됩니다. 이미지 크기가 줄고, 공격 표면도 작아지며, 시작도 빨라집니다.

Q CI에서 lint와 test를 자동화하는 이유는?

사람이 깜빡해도 GitHub Actions가 push/PR마다 ruff와 pytest를 강제 실행해, 깨진 코드가 main에 머지되는 것을 막습니다. "통과한 코드만 배포로 넘어간다"는 안전망이 생깁니다.

Q 배포 직후 DB 스키마는 어떻게 코드와 맞추나요?

배포 파이프라인에서 alembic upgrade head를 실행합니다(Week 10 Lesson 02). 그러면 최신 마이그레이션까지 적용되어 운영 DB 스키마가 코드와 일치하게 됩니다.
