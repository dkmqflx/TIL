Raw SQL vs TypeORM

TypeORM은 객체와 관계형 DB를 매핑하는 ORM입니다. SQL을 직접 쓰지 않고 TypeScript 객체로 DB를 다룰 수 있게 해줍니다.

❌ Raw SQL (직접 작성)
// 반복적인 코드, SQL 의존적
const result = await dataSource.query(
  `INSERT INTO categories (name) VALUES ($1)
   RETURNING id`,
  [category.name]
);
const id = result[0].id;
✅ TypeORM (객체 중심)
// 간결하고 객체 지향적
const saved = await this.categoryRepository
  .save(category);
const id = saved.id;

Entity 정의

TypeORM에서 테이블 구조는 Entity 클래스로 정의합니다.

typescript
// user.entity.ts
import {
  Entity,
  PrimaryGeneratedColumn,
  Column,
  CreateDateColumn,
  UpdateDateColumn,
} from 'typeorm';

@Entity('users')
export class User {
  @PrimaryGeneratedColumn()
  id: number;

  @Column({ length: 50 })
  name: string;

  @Column({ unique: true })
  email: string;

  @Column()
  age: number;

  @CreateDateColumn({ name: 'created_at' })
  createdAt: Date;

  @UpdateDateColumn({ name: 'updated_at' })
  updatedAt: Date;
}

자주 쓰는 컬럼 데코레이터

데코레이터설명
@PrimaryGeneratedColumn()AUTO_INCREMENT PK
@PrimaryGeneratedColumn('uuid')UUID PK
@Column()일반 컬럼
@Column({ nullable: true })NULL 허용
@Column({ unique: true })UNIQUE 제약
@Column({ default: 0 })기본값 설정
@CreateDateColumn()INSERT 시 자동 설정
@UpdateDateColumn()UPDATE 시 자동 갱신
@VersionColumn()낙관적 락 버전 관리 (Part 5)

Setter 대신 메서드 사용

TypeScript에서도 public setter 남용은 데이터 무결성을 해칩니다. 의미 있는 메서드로 대체하세요.

❌ 나쁜 예 — public setter
@Entity()
export class Post {
  @Column()
  title: string;

  // 어디서든 마음대로 변경 가능
  setTitle(title: string) {
    this.title = title; // 빈 문자열? null?
  }
}

// 호출부
post.setTitle('');    // 빈 제목?
post.setTitle(null);  // null?
✅ 좋은 예 — 의미 있는 메서드
@Entity()
export class Post {
  @Column({ nullable: false })
  title: string;

  // 비즈니스 로직이 명확
  updateTitle(newTitle: string): void {
    if (!newTitle?.trim()) {
      throw new BadRequestException(
        '제목은 필수입니다.'
      );
    }
    if (newTitle.length > 100) {
      throw new BadRequestException(
        '제목은 100자를 초과할 수 없습니다.'
      );
    }
    this.title = newTitle;
  }
}
장점: 유효성 검증이 Entity 안에 모임 → 어디서 호출해도 일관된 규칙이 적용됩니다. Controller/Service에서 중복 검증이 필요 없어집니다.

delete / remove - 데이터 삭제 DELETE

TypeORM에는 삭제 방법이 두 가지입니다. 가장 큰 차이는 "DB에서 먼저 조회하느냐"입니다.

delete()remove()
DB 조회 필요 여부❌ 없음 (빠름)✅ 먼저 findOne() 필요
존재 여부 확인result.affected로 확인findOne()에서 직접 확인
cascade 이벤트 발동❌ 발동 안 됨✅ @BeforeRemove 등 발동
언제 사용단순 삭제 (ID로 바로 삭제)삭제 전 엔티티 검증·가공이 필요할 때

방법 1: delete() — ID(조건)로 바로 삭제

DB에서 먼저 조회하지 않고 조건만으로 DELETE를 실행합니다. DB 요청이 1번이라 빠릅니다.

typescript
// DB 요청: 1번
// DELETE FROM users WHERE id = 1
const result = await this.userRepository.delete(1);
// ※ id 대신 조건 객체도 가능: delete({ email: 'test@test.com' })

// affected: 실제로 삭제된 행 수
// 삭제할 데이터가 애초에 없었다면 affected === 0
if (result.affected === 0) {
  throw new NotFoundException('삭제할 데이터가 없습니다.');
}

// delete()는 삭제된 엔티티 객체를 반환하지 않음
// → 삭제 후 해당 데이터가 필요하면 remove()를 써야 함

방법 2: remove() — 엔티티 객체를 받아 삭제

먼저 엔티티를 조회한 뒤, 그 객체를 넘겨서 삭제합니다. DB 요청이 2번(조회 + 삭제)입니다.

typescript
// DB 요청: 2번 (SELECT → DELETE)

// 1) 먼저 조회 — 없으면 여기서 예외를 던짐
const user = await this.userRepository.findOne({ where: { id: 1 } });
if (!user) throw new NotFoundException();

// 2) 조회한 엔티티 객체를 넘겨서 삭제
// DELETE FROM users WHERE id = 1
await this.userRepository.remove(user);

// remove()의 장점: 삭제 전 user 객체를 코드에서 활용 가능
// 예) 삭제 전에 user.name을 로그로 남기거나, 응답으로 반환하거나
실무 선택 기준
• 단순히 "ID로 삭제"만 하면 된다면 → delete(id) (더 빠름)
• 삭제 전에 권한 확인, 로그 기록, 또는 삭제된 데이터를 응답으로 반환해야 한다면 → findOne() 후 remove()
조건 없이 delete({})를 호출하면 테이블 전체가 삭제됩니다. WHERE 조건을 반드시 확인하세요.

연관관계가 필요한 이유

지금까지는 users 한 테이블만 다뤘습니다. 실제 서비스는 여러 테이블이 서로 연결됩니다 (게시글 → 카테고리, 주문 → 회원). TypeORM은 이 관계를 객체 그래프로 표현해, FK를 직접 조회·연결하는 수고를 없애줍니다.

❌ Raw 쿼리 방식
// Post 조회 후 Category 따로 조회
async findPost(id: number) {
  const post = await dataSource.query(
    'SELECT * FROM posts WHERE id = $1', [id]
  );
  // Category를 따로 조회해서 수동으로 연결
  const category = await dataSource.query(
    'SELECT * FROM categories WHERE id = $1',
    [post[0].category_id]
  );
  post[0].category = category[0];
  return post[0];
}
✅ TypeORM 연관관계
// 객체 그래프 탐색으로 바로 접근
async findPost(id: number) {
  const post = await this.postRepository
    .findOne({
      where: { id },
      relations: { category: true },
    });

  // 바로 접근 가능
  const categoryName = post.category.name;
  return post;
}

@ManyToOne — 가장 많이 쓰는 연관관계

① "N:1 관계"가 뭔가요?

쉽게 말하면 "여러 개(N)가 하나(1)를 가리키는 관계"입니다.

예시로 이해하기
유튜브 영상(N) → 채널(1)
댓글(N) → 게시글(1)
게시글(N) → 카테고리(1)

공통점: "여러 개"쪽이 "하나"의 ID를 가짐
DB 테이블로 보면
posts 테이블이 category_id라는 컬럼을 가집니다.
categories 테이블에는 아무 변화 없음.

즉, "여러 개(N)"쪽 테이블에 FK 컬럼이 생깁니다.
posts 테이블 (category_id = FK 컬럼)
idtitlecategory_id

2TypeORM 연관관계1 → "백엔드"
3React 기초2 → "프론트엔드"

② LAZY/EAGER가 뭔가요?

Post를 조회할 때 Category 정보를 같이 가져올지 말지의 차이입니다. 프론트엔드 비유로 이해해보면:

EAGER — 항상 같이 가져옴
GET /posts/1 요청 시
Post + Category를 항상 JOIN해서 반환

마치 API가 항상 category 정보를
응답에 포함시켜 주는 것과 같음

⚠️ 필요 없을 때도 매번 JOIN → 느려짐
relations 명시 — 필요할 때만
GET /posts/1 → Post만
GET /posts/1?include=category → 같이

필요할 때만 relations를 넣어서
명시적으로 함께 가져옴

✅ 실무에서 가장 많이 쓰는 방식
LAZY는 실무에서 쓰지 마세요. TypeORM의 lazy 로딩은 Promise<Category>라는 특이한 타입을 요구하고, 설정 실수 시 디버깅이 매우 어렵습니다. relations 옵션으로 명시적으로 쓰는 게 훨씬 안전합니다.

③ 실제 코드 — 실무 권장 패턴

typescriptpost.entity.ts
@Entity('posts')
export class Post {
  @PrimaryGeneratedColumn()
  id: number;

  @Column()
  title: string;

  // @ManyToOne: "Post가 N, Category가 1"
  // () => Category : 연결할 대상 엔티티
  // (category) => category.posts : 반대쪽(Category)에서 이 관계를 뭐라 부르는지
  @ManyToOne(() => Category, (category) => category.posts, {
    nullable: false,  // category 없는 post는 존재 불가 (NOT NULL)
  })
  @JoinColumn({ name: 'category_id' })  // DB 컬럼명을 category_id로 지정
  category: Category;

  @Column({ name: 'category_id' })
  categoryId: number;  // ID만 필요할 때 편하게 꺼내 쓸 수 있음
}
typescriptpost.service.ts
// category 정보가 필요 없을 때 → 그냥 조회
const post = await this.postRepository.findOne({
  where: { id: 1 },
  // category 정보 없음 → post.category = undefined

// category 정보가 필요할 때 → relations 명시

엔티티에 eager: true를 설정하면 relations를 쓰지 않아도 매번 자동으로 JOIN합니다.

방식동작언제 쓰나주의점
eager: true 항상 자동 JOIN 이 관계가 항상 필요한 경우 불필요한 조회도 항상 JOIN → 성능 주의
relations 명시 해당 쿼리에서만 JOIN 필요할 때만 가져오고 싶을 때 가장 일반적인 실무 패턴 ✅
아무것도 없음 category 미포함 ID만 필요한 경우 (categoryId) post.category는 undefined
실무 팁: eager: true는 "이 엔티티를 조회하면 항상 이 관계가 필요하다"는 확신이 있을 때만 씁니다. 대부분의 경우는 relations로 명시하는 게 더 유연합니다.

INNER JOIN — 교집합만 반환

양쪽 테이블 모두 데이터가 있는 행만 반환합니다. 주문이 없는 사용자는 제외됩니다.

typescript
// 실행되는 SQL:
// SELECT u.name, u.email, o.product, o.price
// FROM users u INNER JOIN orders o ON o.user_id = u.id

const result = await this.userRepository
  // users 테이블을 기준으로 쿼리 시작, 'u'는 별칭(alias)
  // → 이후 u.name, u.email처럼 짧게 참조 가능
  .createQueryBuilder('u')

  // 'u.orders' → User 엔티티에 선언된 orders 프로퍼티를 가리킴
  //
  // user.entity.ts:
  //   @OneToMany(() => Order, (order) => order.user)
  //   orders: Order[];  ← 여기서 'u.orders'가 이 프로퍼티를 참조
  //
  // TypeORM은 이 데코레이터를 읽어서
  //   1) 대상 테이블: Order 엔티티 → orders 테이블
  //   2) FK 컬럼: Order의 @ManyToOne 쪽에서 user_id를 찾아 ON 조건 자동 생성
  //
  // 두 번째 인자 'o'는 이 joined 테이블에 붙이는 별칭
  // → 이후 select에서 o.product, o.price처럼 참조 가능
  //
  // AndSelect: JOIN한 데이터를 결과에 포함 (없으면 필터링에만 쓰고 결과 미포함)
  .innerJoinAndSelect('u.orders', 'o')

  // 반환할 컬럼 지정 — u.는 users, o.는 orders 테이블
  // 생략하면 SELECT * (모든 컬럼)
  .select(['u.name', 'u.email', 'o.product', 'o.price'])

  // 쿼리 실행 → User[] 배열로 반환
  // getOne()이면 단건(User | null), getRawMany()면 집계용 원시 데이터
  .getMany();

// 실행되는 SQL:
// SELECT u.name AS name, COUNT(o.id) AS orderCount, SUM(o.price) AS totalPrice
// FROM users u LEFT JOIN orders o ON o.user_id = u.id
// GROUP BY u.id

const result = await this.userRepository
  .createQueryBuilder('u')

  // leftJoinAndSelect와 달리 AndSelect가 없음
  // → 집계 함수 결과를 addSelect로 직접 지정하므로 AndSelect 불필요
  .leftJoin('u.orders', 'o')

  // select(): 첫 번째 인자 = SQL 표현식, 두 번째 인자 = 결과 객체의 키 이름(alias)
  // → 결과: { name: '김철수', ... }
  .select('u.name', 'name')

  // addSelect(): select()에 컬럼을 추가 (select()를 또 쓰면 앞 것이 덮어씌워짐)
  // COUNT(o.id): o.id가 null이 아닌 행의 수 → 주문 없는 사용자는 0
  .addSelect('COUNT(o.id)', 'orderCount')

  // SUM(o.price): price 합계 → 주문 없는 사용자는 null
  .addSelect('SUM(o.price)', 'totalPrice')

  // GROUP BY: u.id 기준으로 묶음
  // → 같은 사용자의 여러 주문 행을 하나로 집계
  .groupBy('u.id')

  // getRawMany(): 집계 결과는 User 엔티티 구조와 맞지 않으므로 원시 객체 배열로 반환

① 서브쿼리 실행

② 결과를 조건으로 전달

INNER JOIN과 서브쿼리의 차이
둘 다 "주문한 사용자만 조회"라는 같은 결과를 냅니다.

typescript
// 실행되는 SQL:
// SELECT * FROM users u
// WHERE u.id IN (SELECT DISTINCT o.user_id FROM orders o)

// ── 1단계: 서브쿼리 정의 ──
const subQuery = this.orderRepository
  .createQueryBuilder('o')

  // orders 테이블에서 user_id만 선택
  .select('o.user_id')

  // DISTINCT: 같은 user_id가 여러 번 나와도 중복 제거
  // → 주문을 1건 이상 한 사용자 id 목록
  .distinct(true);

// ── 2단계: 서브쿼리를 메인 쿼리의 IN 조건에 삽입 ──
const users = await this.userRepository
  .createQueryBuilder('u')

  // subQuery.getQuery(): 서브쿼리를 SQL 문자열로 변환 (실행하지는 않음)
  // → WHERE u.id IN (SELECT DISTINCT o.user_id FROM orders o)
  // INNER JOIN으로도 같은 결과를 낼 수 있지만,
  // 서브쿼리는 JOIN 없이 users 테이블 컬럼만 반환하고 싶을 때 더 명확
  .where(`u.id IN (${subQuery.getQuery()})`)

  .getMany();
프론트 관점에서: JOIN은 서버에서 여러 API를 호출해 데이터를 합치는 과정을 DB 레벨에서 한 번에 처리하는 것과 같습니다. getRawMany()는 집계/서브쿼리처럼 Entity 형태로 매핑이 불가능한 경우에 사용합니다.

DTO 프로젝션 — 필요한 컬럼만 골라서 조회

보통 find()나 getMany()는 Entity의 모든 컬럼을 가져옵니다. 그런데 목록 화면에는 id, name, price 세 개만 필요한데 불필요한 컬럼까지 전부 가져오면 낭비입니다. 필요한 컬럼만 골라 가져오는 것이 DTO 프로젝션입니다.

getMany() — 전체 컬럼 조회
const products = await this.productRepository
  .createQueryBuilder('p')
  .getMany();

// SELECT p.id, p.name, p.price,
//        p.description,   ← 안 쓰는데도 가져옴
//        p.stock,         ← 안 쓰는데도 가져옴
//        p.created_at,    ← 안 쓰는데도 가져옴
//        ...
// FROM products p
getRawMany() — 필요한 컬럼만
const products = await this.productRepository
  .createQueryBuilder('p')
  .select(['p.id', 'p.name', 'p.price'])
  .getRawMany();

// SELECT p.id, p.name, p.price
// FROM products p
//
// → 딱 3개만 가져옴

JOIN한 다른 테이블 컬럼도 함께 가져올 수 있습니다. AS로 원하는 이름을 붙입니다.

typescript
// 상품 목록에 판매자 이름과 카테고리 이름도 같이 보여주고 싶을 때
const list = await this.productRepository
  .createQueryBuilder('p')
  .innerJoin('p.seller', 'seller')      // seller 테이블 JOIN
  .innerJoin('p.category', 'category')  // category 테이블 JOIN
  .select([
    'p.id        AS id',
    'p.name      AS name',
    'p.price     AS price',
    'seller.name   AS "sellerName"',    // AS로 원하는 키 이름 지정

  ])
  .getRawMany();

LIMIT / OFFSET - 페이지네이션

DB에 데이터가 10,000건 있을 때 전부 한 번에 가져오면 느리고 낭비입니다. LIMIT과 OFFSET으로 잘라서 가져오는 것이 페이지네이션입니다.

LIMIT (take)
한 번에 가져올 행의 수.
프론트에서 보내는 size 값과 동일.
OFFSET (skip)
앞에서 몇 개를 건너뛸지.
"2페이지면 10개를 넘기고 그 다음부터 가져와"

전체 데이터가 30건이고 한 페이지에 10건씩 보여준다면:

findAndCount() — 목록 + 전체 개수 한 번에

프론트에서 "전체 몇 페이지인지" 계산하려면 전체 데이터 수(total)가 필요합니다.
findAndCount()는 목록과 전체 개수를 쿼리 2번으로 한 번에 가져옵니다.

typescript
async findAll(page: number, size: number) {
  // findAndCount(): [data, total] 튜플로 반환
  // 내부적으로 SELECT ... LIMIT/OFFSET 과 SELECT COUNT(*) 두 쿼리를 실행
  const [data, total] = await this.userRepository.findAndCount({
    order: { createdAt: 'DESC' },
    take: size,                  // LIMIT: 한 페이지에 가져올 수
    skip: (page - 1) * size,     // OFFSET: 건너뛸 수
  });

  return {
    data,                              // 현재 페이지 목록
    total,                             // 전체 데이터 수 (예: 30)
    page,                              // 현재 페이지 번호
    totalPages: Math.ceil(total / size), // 전체 페이지 수 (예: 3)
    // Math.ceil: 소수점 올림 → 31건이면 ceil(31/10) = 4페이지
  };
}

트랜잭션 처리

트랜잭션이란 여러 DB 작업을 하나의 묶음(논리적 단위)으로 처리하는 것입니다.
묶음 안의 작업이 모두 성공하면 반영(commit), 하나라도 실패하면 전부 취소(rollback)합니다.

ACID — 트랜잭션의 4대 성질

Atomicity (원자성)
All or Nothing. 부분 성공 없음.
Consistency (일관성)
트랜잭션 전후로 DB가 일관된 상태 유지.
Isolation (격리성)
동시에 실행되는 트랜잭션끼리 간섭 없음. (→ Part 5 Lock)
Durability (지속성)
커밋된 트랜잭션은 장애가 나도 유지.

왜 필요한가? — 트랜잭션 없는 경우의 문제

예시: 쇼핑몰에서 주문할 때 "주문 생성"과 "재고 차감" 두 작업이 필요합니다.

상황주문 생성재고 차감결과
정상 ✔ 성공 ✔ 성공 정상 주문
트랜잭션 없음 ✔ 성공 ✘ 실패 (재고 없음) 주문은 생성됐는데 재고는 그대로 → 데이터 불일치
트랜잭션 있음 ✔ 성공 ✘ 실패 → 롤백 주문 생성도 함께 취소 → 데이터 일관성 유지
프론트 관점에서: 트랜잭션은 여러 개의 fetch() 호출을 묶어서 "하나라도 실패하면 나머지도 모두 취소"되도록 보장하는 것과 같습니다. 단, 이 보장을 DB 레벨에서 처리한다는 차이가 있습니다.

트랜잭션 핵심 용어

용어의미
BEGIN트랜잭션 시작 — "지금부터 묶음 시작"
COMMIT모든 작업 성공 → DB에 최종 반영
ROLLBACK하나라도 실패 → 모든 작업 취소, 이전 상태로 되돌림
manager트랜잭션 안에서 사용하는 EntityManager. 같은 트랜잭션 안의 작업임을 보장
방법 1: dataSource.transaction() ✅ 권장
콜백 안에서 예외가 나면 자동 롤백. BEGIN/COMMIT/ROLLBACK을 직접 쓸 필요 없음.
방법 2: QueryRunner
BEGIN/COMMIT/ROLLBACK을 직접 제어. 트랜잭션 중간에 분기가 필요하거나 savepoint 등 고급 기능이 필요할 때.

방법 1: dataSource.transaction() (권장)

typescript
import { DataSource } from 'typeorm';

@Injectable()
export class OrderService {
  // DataSource: DB 연결 자체를 나타내는 객체
  // Repository는 특정 Entity 전용이지만, DataSource는 트랜잭션처럼 DB 전체를 다룰 때 사용
  constructor(private readonly dataSource: DataSource) {}

  async placeOrder(productId: number, quantity: number) {
    // transaction() 콜백 안의 모든 작업이 하나의 트랜잭션으로 묶임
    // 콜백이 정상 종료 → 자동 COMMIT
    // 콜백에서 예외 발생 → 자동 ROLLBACK (이전에 성공한 작업도 모두 취소)
    await this.dataSource.transaction(async (manager) => {

      // manager를 통해 작업해야 같은 트랜잭션 안에서 실행됨
      // this.orderRepository.save()처럼 Repository를 직접 쓰면 트랜잭션 밖이 됨 — 주의!
      await manager.save(Order, { productId, quantity }); // ← 1단계: 주문 생성

      const result = await manager.decrement(             // ← 2단계: 재고 차감
        Stock, { productId }, 'quantity', quantity
      );

      if (result.affected === 0) {
        // 여기서 예외를 던지면 1단계(주문 생성)도 함께 롤백됨
        throw new BadRequestException('재고가 부족합니다.');
      }
      // 모두 성공 → 콜백 종료 → 자동 COMMIT
    });
  }
}

방법 2: QueryRunner — 직접 제어

트랜잭션의 시작/커밋/롤백을 직접 코드로 작성합니다. 절차가 복잡한 대신 세밀한 제어가 가능합니다.

typescript
async transfer(fromId: number, toId: number, amount: number) {
  // QueryRunner: DB 커넥션을 직접 잡고 트랜잭션을 수동으로 제어하는 객체
  const queryRunner = this.dataSource.createQueryRunner();

  await queryRunner.connect();          // DB 커넥션 풀에서 커넥션 획득
  await queryRunner.startTransaction(); // BEGIN — 트랜잭션 시작

  try {
    // queryRunner.manager를 통해 작업 → 같은 트랜잭션 안에서 실행됨
    await queryRunner.manager.decrement(Account, { id: fromId }, 'balance', amount); // 출금
    await queryRunner.manager.increment(Account, { id: toId },   'balance', amount); // 입금

    await queryRunner.commitTransaction(); // 모두 성공 → COMMIT (DB에 최종 반영)

  } catch (err) {
    // 중간에 하나라도 실패하면 ROLLBACK → 출금/입금 모두 취소
    await queryRunner.rollbackTransaction();
    throw err; // 에러를 다시 던져서 호출한 쪽에서도 실패를 알 수 있게 함

  } finally {
    // 성공/실패 여부와 관계없이 반드시 커넥션을 반납
    // 안 하면 커넥션 풀이 고갈되어 다른 요청들이 DB를 못 씀
    await queryRunner.release();
  }
}
트랜잭션 안에서는 반드시 manager 또는 queryRunner.manager를 통해 작업해야 합니다.
this.userRepository.save()처럼 주입받은 Repository를 직접 쓰면 트랜잭션 밖에서 실행되어 롤백이 되지 않습니다.

독립 트랜잭션 패턴

"주문은 실패해도 알림 로그는 남겨야 한다"처럼, 메인 트랜잭션과 별개로 커밋되어야 하는 작업은 새 QueryRunner로 독립 실행합니다.

typescript
@Injectable()
export class OrderService {
  constructor(
    private readonly dataSource: DataSource,
    private readonly notificationService: NotificationService,
  ) {}

  async createOrder(dto: CreateOrderDto): Promise<Order> {
    // 메인 트랜잭션
    const order = await this.dataSource.transaction(async (manager) => {
      // ... 주문 생성 로직
    });

    // 독립 트랜잭션으로 알림 저장 (메인 실패해도 영향 없음)
    try {
      await this.notificationService.logNewOrder(order.id);
    } catch (e) {
      // 알림 실패해도 주문은 이미 커밋됨
      console.error('알림 저장 실패:', e.message);
    }

    return order;
  }
}

@Injectable()
export class NotificationService {
  constructor(private readonly dataSource: DataSource) {}

  // 항상 새 QueryRunner로 독립 트랜잭션 실행
  async logNewOrder(orderId: number): Promise<void> {
    const qr = this.dataSource.createQueryRunner();
    await qr.connect();
    await qr.startTransaction();
    try {
      await qr.manager.save(NotificationLog, { orderId });
      await qr.commitTransaction();
    } catch (err) {
      await qr.rollbackTransaction();
      throw err;
    } finally {
      await qr.release();
    }
  }
}

① 프론트엔드 비유로 먼저 이해하기

typescript
const products = await this.productRepository
  .createQueryBuilder('p')
  .leftJoinAndSelect('p.seller', 'seller')      // seller JOIN + 결과에 포함
  .leftJoinAndSelect('p.category', 'category')  // category JOIN + 결과에 포함
  .where('p.categoryId = :categoryId', { categoryId: 1 })  // 카테고리 필터
  .getMany();

// DB 쿼리 1번:
// SELECT p.*, seller.*, category.*
// FROM products p

Race Condition & 동시성 제어

① Race Condition이란?

서버는 사용자 요청을 동시에 여러 개 처리합니다. 그런데 여러 요청이 같은 데이터를 동시에 수정하면 엉뚱한 결과가 나옵니다. 이게 Race Condition입니다.

예시: 잔액 10,000원 계좌에서 두 사람이 동시에 5,000원씩 출금
시간요청 A (ATM 1번)요청 B (ATM 2번)DB 잔액
T1잔액 읽음 → 10,000원10,000원
T2잔액 읽음 → 10,000원10,000원
T310,000 - 5,000 = 5,000원 저장5,000원
T410,000 - 5,000 = 5,000원 저장 (A의 결과 덮어씀!)5,000원
10,000원이 빠져나갔는데 잔액은 5,000원만 줄었다 → 5,000원 사라짐!
왜 이런 일이 생기나요?
A와 B 둘 다 "10,000원"을 읽은 뒤 각자 계산했기 때문입니다.
A가 먼저 저장해서 5,000원이 됐는데, B가 그걸 모르고 자기 계산 결과(5,000원)를 덮어씁니다.
결국 5,000원이 두 번 빠져나간 것처럼 동작해야 하는데 한 번만 반영됩니다.

② 해결 방법 2가지 — 락(Lock)

락은 "내가 이 데이터를 쓰고 있는 동안 다른 사람은 손대지 마"라고 표시해두는 것입니다.

🔄 낙관적 락 (Optimistic Lock)
가정: 충돌이 드물다
방식: 미리 잠그지 않고, 저장할 때 "내가 읽은 뒤 누가 수정했나?" 확인
충돌 시: 예외 발생 → 처음부터 재시도

✅ 잠금이 없어서 빠름
✅ 쿠폰 발급, 이벤트 응모처럼 충돌이 적을 때
🔐 비관적 락 (Pessimistic Lock)
가정: 충돌이 자주 일어난다
방식: 읽는 순간 잠금 → 내가 끝낼 때까지 다른 요청은 대기
충돌 시: 앞 요청이 끝날 때까지 자동으로 기다림

✅ 데이터 정확도 100% 보장
✅ 계좌 이체, 재고 차감처럼 충돌이 잦을 때

낙관적 락 — 버전 번호로 충돌 감지

@VersionColumn()을 엔티티에 달면 TypeORM이 자동으로 버전을 관리합니다. 저장할 때 "내가 읽었던 버전과 지금 DB 버전이 같은지" 확인합니다.

낙관적 락이 충돌을 잡는 흐름
시간요청 A요청 BDB 버전
T1읽음: 잔액 10,000 / version=1version=1
T2읽음: 잔액 10,000 / version=1version=1
T3저장: WHERE version=1 → version=2로 갱신 ✅ 성공version=2
T4저장: WHERE version=1 → DB는 이미 version=2 ❌ 0건 갱신version=2
B는 예외 발생 → 처음부터 재시도 (다시 읽으면 version=2를 보게 됨)
typescriptaccount.entity.ts
@Entity('accounts')
export class Account {
  @PrimaryGeneratedColumn()
  id: number;

  @Column()
  balance: number;

  // 이 컬럼만 추가하면 낙관적 락 적용 완료
  // TypeORM이 save() 시 자동으로 버전 체크 + 버전 증가를 처리
  @VersionColumn()
  version: number;

  withdraw(amount: number): void {
    if (this.balance < amount) throw new BadRequestException('잔액 부족');
    this.balance -= amount;
  }
}
typescriptaccount.service.ts — 낙관적 락
async withdraw(accountId: number, amount: number): Promise<void> {
  // 1) 계좌 조회 (이때 version도 함께 읽힘, 예: version=1)
  const account = await this.accountRepository.findOneOrFail({
    where: { id: accountId },
  });

  // 2) 잔액 변경 (아직 DB에는 반영 안 됨, 메모리에서만)
  account.withdraw(amount);

  // 3) 저장 시 TypeORM이 자동으로 아래 SQL을 실행:
  //    UPDATE accounts
  //    SET balance = 5000, version = 2      ← version + 1
  //    WHERE id = 1 AND version = 1         ← 내가 읽었던 version 조건
  //
  //    만약 다른 요청이 먼저 저장해서 DB version이 이미 2라면
  //    WHERE version = 1 조건이 맞지 않아 0건 갱신 → 예외 발생
  await this.accountRepository.save(account);
}

// 충돌 시 재시도하는 버전
async withdrawWithRetry(accountId: number, amount: number): Promise<void> {
  for (let attempt = 1; attempt <= 3; attempt++) {
    try {
      await this.withdraw(accountId, amount);
      return;  // 성공하면 종료
    } catch (err) {
      // 낙관적 락 충돌 예외인 경우만 재시도
      if (err.name === 'OptimisticLockVersionMismatchError') {
        if (attempt === 3) {
          // 3번 모두 실패하면 사용자에게 알림
          throw new ConflictException('잠시 후 다시 시도해주세요.');
        }
        // 100ms, 200ms, 300ms 대기 후 재시도 (충돌 피하기 위해)
        await new Promise((res) => setTimeout(res, 100 * attempt));
      } else {
        throw err;  // 다른 에러는 그대로 던짐
      }
    }
  }
}

비관적 락 — 읽는 순간 잠금

낙관적 락이 "일단 작업하고 나중에 충돌 확인"이라면, 비관적 락은 "처음부터 잠그고 시작"입니다. 읽는 순간 DB가 해당 행을 잠가서 내가 커밋할 때까지 다른 요청은 기다려야 합니다.

❌ 락 없이 동시 출금 (Race Condition)
A, B 둘 다 잔액 10,000원을 읽음
A: 10,000 - 5,000 = 5,000 저장
B: 10,000 - 5,000 = 5,000 저장 (덮어씀)

결과: 10,000원이 빠져나갔는데 잔액 5,000원
→ 5,000원 소실
✅ 비관적 락으로 순서 보장
A가 읽으면서 행을 잠금
B는 A가 끝날 때까지 대기
A: 10,000 - 5,000 = 5,000 저장 후 해제
B: 5,000 읽음 → 5,000 - 5,000 = 0 저장

결과: 잔액 0원
→ 두 번 모두 정확히 반영
비관적 락 단계별 흐름
시간요청 A요청 BDB 상태
T1SELECT ... FOR UPDATE → 잠금 획득🔒 A가 잠금 보유
T2SELECT ... FOR UPDATE 시도 → 잠겨있어서 대기 중...🔒 B 대기
T3잔액 수정 (5,000원) → 저장 → 커밋✅ 잠금 해제
T4이제 읽을 수 있음 → 잔액 5,000원 읽음B가 잠금 획득
T5잔액 수정 (0원) → 저장 → 커밋✅ 완료
typescriptaccount.service.ts — 비관적 락
// ⚠️ 비관적 락은 반드시 트랜잭션 안에서 사용해야 함
// 이유: 락은 트랜잭션이 커밋/롤백될 때 자동으로 해제됨
//       트랜잭션 없이 락을 걸면 언제 해제할지 알 수 없음
async withdrawWithLock(accountId: number, amount: number): Promise<void> {
  await this.dataSource.transaction(async (manager) => {

    // lock: { mode: 'pessimistic_write' } 옵션 하나만 추가하면 됨
    // 실행되는 SQL:
    //   SELECT * FROM accounts WHERE id = ? FOR UPDATE
    //
    // "FOR UPDATE"의 의미:
    //   이 행을 읽는 순간 잠금 → 내 트랜잭션이 끝날 때까지
    //   다른 요청은 이 행을 읽지도 못하고 대기
    const account = await manager.findOne(Account, {
      where: { id: accountId },
      lock: { mode: 'pessimistic_write' },
    });

    if (!account) throw new NotFoundException('계좌 없음');

    // 이 시점부터 커밋 전까지는 나만 이 계좌 수정 가능
    // 다른 요청들은 모두 이 transaction이 끝날 때까지 대기 중
    account.withdraw(amount);

    await manager.save(account);
    // ↑ 커밋 발생 → FOR UPDATE 잠금 자동 해제
    // → 대기 중이던 다음 요청이 이제 읽기 시작
  });
}
// ⚠️ 주의: 트랜잭션이 길어질수록 다른 요청의 대기 시간도 늘어남
// 락을 잡는 구간을 최대한 짧게 유지하는 것이 중요

데드락 — 서로 기다리다 영원히 멈추는 상황

비관적 락을 쓸 때 주의할 점입니다. A가 1번 계좌를 잠그고 2번을 기다리는데, B는 2번을 잠그고 1번을 기다리면 — 둘 다 영원히 기다리게 됩니다.

❌ 데드락 발생하는 코드
// 요청 A: 계좌1 → 계좌2 이체
// 요청 B: 계좌2 → 계좌1 이체 (동시 실행)

// A 실행 순서: 계좌1 락 → 계좌2 락
// B 실행 순서: 계좌2 락 → 계좌1 락

// A가 계좌1 잠금 획득 ✅
// B가 계좌2 잠금 획득 ✅
// A가 계좌2 잠금 시도 → B가 갖고 있어서 대기...
// B가 계좌1 잠금 시도 → A가 갖고 있어서 대기...
// → 서로 무한 대기 = 데드락
✅ 항상 작은 ID 먼저 잠그기
async transfer(fromId: number, toId: number, amount: number) {
  // 핵심: 어떤 이체든 항상 ID가 작은 계좌를 먼저 잠금
  // A (1→2): 계좌1 먼저 → 계좌2
  // B (2→1): 계좌1 먼저 → 계좌2  ← 순서가 같아짐!
  const firstId  = Math.min(fromId, toId);
  const secondId = Math.max(fromId, toId);

  await this.dataSource.transaction(async (manager) => {
    // 두 요청 모두 같은 순서로 잠그니까
    // B는 A가 계좌1 락 해제할 때까지 순서대로 기다림
    const first  = await manager.findOne(Account,
      { where: { id: firstId  }, lock: { mode: 'pessimistic_write' } });
    const second = await manager.findOne(Account,
      { where: { id: secondId }, lock: { mode: 'pessimistic_write' } });

    // fromId/toId에 맞게 from/to 변수 재할당
    const from = fromId === firstId ? first : second;
    const to   = toId   === firstId ? first : second;

    from.withdraw(amount);
    to.deposit(amount);
    await manager.save([from, to]);
  });
}

typescript
it('동시 출금 시 잔액이 정확해야 한다', async () => {
  // 초기 잔액 10,000원으로 계좌 생성
  const account = await accountRepository.save({ balance: 10000 });

  // 10개의 요청을 동시에 보냄 (각 2,000원 출금)
  // Promise.all → 10개가 거의 동시에 실행됨
  const requests = Array.from({ length: 10 }, () =>
    accountService.withdrawWithLock(account.id, 2000)
      .catch(() => {})  // 실패한 요청은 무시 (잔액 부족 등)
  );
  await Promise.all(requests);

  const result = await accountRepository.findOneOrFail({
    where: { id: account.id },
  });

  // 락이 제대로 동작했다면:
  // 10,000 / 2,000 = 5번만 성공 → 잔액 0원
  // 나머지 5번은 "잔액 부족" 예외로 실패
  expect(result.balance).toBe(0);  // 음수가 되면 Race Condition 발생한 것
});

users.service.ts

typescript
@Injectable()
export class UsersService {
  constructor(
    @InjectRepository(User)
    private readonly userRepository: Repository<User>,
  ) {}

  // 전체 조회 (페이지네이션)
  async findAll(page: number, size: number) {
    const [data, total] = await this.userRepository.findAndCount({
      order: { id: 'DESC' },
      take: size,
      skip: (page - 1) * size,
    });
    return { data, total, page, totalPages: Math.ceil(total / size) };
  }

  // 단건 조회
  async findOne(id: number): Promise<User> {
    const user = await this.userRepository.findOne({ where: { id } });
    if (!user) throw new NotFoundException(`사용자를 찾을 수 없습니다. id=${id}`);
    return user;
  }

  // 생성
  async create(dto: CreateUserDto): Promise<User> {
    const existing = await this.userRepository.findOne({ where: { email: dto.email } });
    if (existing) throw new ConflictException('이미 사용 중인 이메일입니다.');
    const user = this.userRepository.create(dto);
    return this.userRepository.save(user);
  }

  // 수정
  async update(id: number, dto: UpdateUserDto): Promise<User> {
    const user = await this.findOne(id);  // 없으면 자동 404
    Object.assign(user, dto);
    return this.userRepository.save(user);
  }

  // 삭제
  async remove(id: number): Promise<void> {
    const result = await this.userRepository.delete(id);
    if (result.affected === 0) {
      throw new NotFoundException(`삭제할 사용자가 없습니다. id=${id}`);
    }
  }
}

@Controller('users')
export class UsersController {
  constructor(private readonly usersService: UsersService) {}

  @Get()                              // GET /users?page=1&size=10
  findAll(
    @Query('page', ParseIntPipe) page = 1,
    @Query('size', ParseIntPipe) size = 10,
  ) {
    return this.usersService.findAll(page, size);
  }

  @Get(':id')                         // GET /users/1
  findOne(@Param('id', ParseIntPipe) id: number) {
    return this.usersService.findOne(id);
  }

  @Post()                             // POST /users
  create(@Body() dto: CreateUserDto) {
    return this.usersService.create(dto);
  }

  @Patch(':id')                       // PATCH /users/1
  update(
    @Param('id', ParseIntPipe) id: number,
    @Body() dto: UpdateUserDto,
  ) {
    return this.usersService.update(id, dto);
  }

  @Delete(':id')                      // DELETE /users/1
  remove(@Param('id', ParseIntPipe) id: number) {
    return this.usersService.remove(id);
  }
}
