백엔드의 지도
HTTP · REST

프론트엔드가 매일 쓰던 HTTP/REST를 '서버 입장'에서 다시 본다. fetch를 보내던 쪽이 아니라, 받아서 응답을 만드는 쪽으로.

"

클라이언트–서버·HTTP·REST는 이미 절반은 안다. 프론트에서 fetch('/posts')를 보내 본 적 있다면, 남은 절반은 단 하나 — 서버가 무엇을 받고(Request) 무엇을 돌려주는가(Response)로 머릿속 지도를 뒤집는 것뿐이다.

  • 클라이언트–서버 모델을 한 문장으로 설명할 수 있다
  • HTTP 요청·응답이 각각 어떤 조각(라인·헤더·바디)으로 이뤄지는지 안다
  • REST의 자원·메서드 원칙과 메서드↔CRUD↔상태코드 매핑을 외운다
  • V8 → Node.js → Express → NestJS 스택에서 우리 코드가 어디 서는지 안다

1클라이언트와 서버 — 방향만 바뀐다

웹은 결국 요청(Request)과 응답(Response)의 끝없는 주고받음입니다. 요청을 보내는 쪽이 Client, 받아서 처리하고 돌려주는 쪽이 Server. 지금까지 당신은 왼쪽(요청을 쏘는 쪽)에 있었고, 이제 오른쪽(응답을 만드는 쪽)으로 자리를 옮깁니다.

🍔 비유 — 식당 주문

손님(클라이언트)은 주방에 직접 들어가지 않습니다. 주문서(HTTP 요청)를 카운터에 내고, 주방(서버)이 요리해서 접시(HTTP 응답)를 돌려주죠. 손님은 주방이 가스불을 쓰는지 인덕션을 쓰는지 몰라도 됩니다. 프론트 개발자였던 당신은 지금까지 "주문하는 손님"이었고, 백엔드 개발자가 된다는 건 이제 주방에 들어가 요리를 만드는 쪽이 된다는 뜻입니다.

2HTTP 요청/응답 해부 — 봉투 안을 열어 보기

브라우저 네트워크 탭에서 늘 보던 그것입니다. 요청과 응답은 모두 ① 시작 라인 ② 헤더(메타데이터) ③ 빈 줄 ④ 바디(본문)라는 동일한 골격을 가집니다. 서버가 하는 일은 결국 "이 봉투를 뜯어 읽고, 같은 형식의 봉투를 만들어 돌려주는 것"입니다.

🧭 메서드는 요청에, 상태코드는 응답에

요청의 첫 줄에는 메서드(GET/POST…)가, 응답의 첫 줄에는 상태코드(200/404…)가 옵니다. "무엇을 하려는가"는 클라이언트가 메서드로 말하고, "그래서 어떻게 됐는가"는 서버가 상태코드로 답합니다.

3REST — 명사는 URL, 동사는 메서드

API 설계 스타일은 여럿이지만 웹의 사실상 표준은 REST입니다. 핵심 아이디어는 하나 — 세상의 모든 것을 자원(Resource)으로 보고, 자원에 주소(URL)를 주고, 행동은 HTTP 메서드로 표현한다. SNS의 "게시글(post)" 자원을 예로 들면 다음과 같이 매핑됩니다.

HTTP 메서드CRUDURL(주소)대표 성공 코드
GETRead/posts · /posts/42200 OK
POSTCreate/posts201 Created
PUTUpdate(전체)/posts/42200 OK
PATCHUpdate(부분)/posts/42200 OK
DELETEDelete/posts/42204 No Content
📮 비유 — 같은 주소, 다른 행동

주소 /posts/42는 "42번 게시글"이라는 집 주소 하나입니다. 그 집에 대해 GET은 "안을 들여다봐", PATCH는 "내용을 고쳐", DELETE는 "허물어". 주소(명사)와 행동(동사)을 분리하는 것이 REST의 우아함입니다. 그래서 POST /createPost가 아니라 POST /posts라고 씁니다.

방금 만든 REST 엔드포인트에 직접 요청을 쏴 보는 가장 빠른 방법은 curl입니다. 터미널이 곧 클라이언트가 됩니다.

terminal — curl
# 게시글 1건 조회 (GET)
curl http://localhost:3000/posts/42

# 새 게시글 생성 (POST) — 헤더로 형식 알리고, -d 로 바디 전송
curl -X POST http://localhost:3000/posts \
  -H "Content-Type: application/json" \
  -d '{"content":"첫 글!"}'

# -i 를 붙이면 응답의 상태 라인 + 헤더까지 함께 본다
curl -i http://localhost:3000/posts

4Node.js & NestJS의 위치 — 우리 코드는 어디 서나

"NestJS 서버"라고 부르지만, 실제로는 여러 층이 쌓인 탑입니다. 맨 아래 V8 엔진이 자바스크립트를 실행하고, Node.js가 거기에 네트워크·파일 같은 바깥 세계를 붙이고, Express가 HTTP 라우팅을 정리하고, 그 위에 NestJS가 구조(모듈·DI)를 얹습니다. 우리가 짜는 코드는 가장 꼭대기 층에 산다는 걸 그려 두면, 다음 레슨에서 "왜 Nest가 Express를 감싸나"가 자연스럽게 이어집니다.

💡 프론트 경험과 연결

브라우저에서 자바스크립트를 돌리던 엔진(V8)과 같은 엔진이 Node.js 안에서 돕니다. 다른 건 "바깥세상에 닿는 능력" — 브라우저는 DOM·fetch를, Node는 네트워크 소켓·파일 시스템을 줍니다. 그래서 같은 JS/TS 문법으로 이제 서버를 짤 수 있는 겁니다.

5✅ 셀프 체크 — 면접관처럼 물어보기

Q REST를 한 문장으로 설명해 보세요.

"모든 것을 자원(Resource)으로 보고, 각 자원에 URL(명사)을 부여하고, 그 자원에 대한 행동은 HTTP 메서드(동사)로 표현하는" API 설계 스타일입니다. 그래서 POST /createPost가 아니라 POST /posts처럼, 주소에는 명사만 두고 동작은 메서드로 구분합니다.

Q GET과 POST의 멱등성(idempotency) 차이는?

GET은 안전(safe)하면서 멱등(idempotent)합니다 — 서버 상태를 바꾸지 않고, 몇 번을 보내도 결과가 같습니다. 반면 POST는 안전하지도 멱등하지도 않습니다 — 호출할 때마다 새 자원이 생기므로 두 번 보내면 글이 두 개 생깁니다. (참고로 PUT·DELETE는 안전하진 않지만 멱등합니다: 같은 요청을 반복해도 최종 상태가 동일.) "안전 = 변경 없음"과 "멱등 = 반복해도 결과 동일"은 다른 개념임을 구분하는 게 포인트입니다.

Q 404와 500은 각각 누구의 문제인가요?

404 Not Found는 4xx = 클라이언트 쪽 문제입니다 — 요청한 자원이 없다(주소가 틀렸거나 삭제됨). 500 Internal Server Error는 5xx = 서버 쪽 문제입니다 — 서버 코드가 처리 중 터졌다는 뜻이라, 백엔드 개발자가 로그를 봐야 합니다. 앞자리만 보면 책임 소재가 갈립니다: 4xx는 "요청을 고쳐라", 5xx는 "서버를 고쳐라".
