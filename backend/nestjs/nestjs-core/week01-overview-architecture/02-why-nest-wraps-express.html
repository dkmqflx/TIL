왜 NestJS는
Express를 감싸나

raw http → Express → NestJS. 같은 엔드포인트를 세 번 만들어 보며, 각 계층이 무엇을 대신 해 주는지를 눈으로 확인한다.

“

NestJS는 무에서 만든 서버가 아니다. Express 위에 구조(DI·모듈·데코레이터)를 얹은 프레임워크다. 순수 http로는 라우팅·파싱을 손으로 챙겨야 하고, Express는 그걸 덜어 주지만 "코드를 어떻게 조직할지"는 안 알려 준다 — 그 빈칸을 채우는 게 Nest다.

  • 순수 Node http 서버가 무엇을 손으로 챙겨야 하는지 안다
  • Express가 라우팅·파싱·헬퍼로 무엇을 덜어 주는지 안다
  • NestJS가 그 위에 더하는 구조적 가치(DI·모듈·데코레이터)를 설명한다
  • "언제 Nest를 택하나"를 판단 기준으로 말할 수 있다

1순수 Node http — 전부 손으로

가장 밑바닥부터 시작합니다. Node에는 외부 라이브러리 없이도 서버를 띄울 수 있는 내장 http 모듈이 있습니다. 하지만 라우팅도, 응답 형식도 전부 직접 챙겨야 합니다.

🍔 비유 — 재료부터 직접 키우는 식당

순수 http는 밀을 직접 빻아 빵을 굽고, 소를 키워 패티를 만드는 식당입니다. 못 할 건 없지만, 손님(요청) 하나하나에 "이게 햄버거 주문인지 음료 주문인지"를 일일이 손으로 구분(라우팅)하고, 접시에 담는 형식(Content-Type·상태코드)도 매번 직접 적어야 합니다. 주문 한 종류 늘 때마다 if가 한 줄씩 늘죠.

raw-http.js
const http = require("node:http");

const server = http.createServer((req, res) => {
  // 라우팅을 직접 분기 — 메서드와 URL을 손으로 검사한다.
  if (req.method === "GET" && req.url === "/posts") {
    res.writeHead(200, { "Content-Type": "application/json" });
    res.end(JSON.stringify({ message: "hello" }));
    return;
  }

  // 위에서 처리되지 않은 모든 요청 = 404. 이것도 손으로 챙긴다.
  res.writeHead(404, { "Content-Type": "application/json" });
  res.end(JSON.stringify({ message: "Not Found" }));
});

server.listen(3000);   // node raw-http.js
📝 손으로 챙겨야 하는 것들

req.method/req.url 분기, 바디 스트림 직접 읽기, Content-Type 헤더, JSON 직렬화, 404 폴백… 라우트가 늘수록 if가 늘고, 코드는 금세 거대한 분기 덩어리가 됩니다. 비즈니스 로직보다 배관(plumbing)이 더 많아지는 지점입니다.

2Express — 배관을 덜어 준다

같은 동작을 Express로 옮기면, 위에서 손으로 하던 라우팅 분기·응답 헬퍼를 프레임워크가 대신 해 줍니다. 같은 엔드포인트가 어떻게 짧아지는지 보세요 — 아래는 위 코드와 완전히 같은 동작입니다.

express-server.js
const express = require("express");   // npm i express
const app = express();

// 메서드 + 경로를 한 줄로 선언. status/json도 헬퍼로 간결해진다.
app.get("/posts", (req, res) => {
  res.json({ message: "hello" });
});

// 어떤 라우트에도 안 걸리면 여기로. (404 처리도 한 줄)
app.use((req, res) => {
  res.status(404).json({ message: "Not Found" });
});

app.listen(3000);   // node express-server.js
💡 한 번뿐인 before/after

writeHead(200, {...}) + JSON.stringify가 res.json(...) 한 줄로, 손수 짜던 라우팅 분기가 app.get("/posts", ...)로 줄었습니다. Express가 HTTP의 반복 작업(라우팅·파싱·응답 헬퍼)을 추상화해 준 것이죠. 하지만 Express는 여기까지입니다 — "파일을 어떻게 나누고, 로직을 어떻게 조직할지"는 여전히 당신 몫입니다.

3NestJS가 더해 주는 것 — 구조

Express는 "어떻게든 동작하는" 서버를 빠르게 만들어 줍니다. 문제는 커질 때입니다. 라우트가 수십 개가 되고 서비스끼리 얽히면, 자유로움이 곧 "각자 알아서"라는 무질서가 됩니다. NestJS는 바로 그 위에 한 층을 더 얹습니다 — 코드를 조직하는 일관된 구조를.

Nest가 "구조"로 더해 주는 것을 한 줄씩 풀면:

Nest가 더하는 것해결하는 문제
DI 컨테이너의존성을 직접 new 하지 않고 주입받아 → 결합도↓·테스트↑ (Week 3)
Module기능 단위로 코드를 묶어 경계를 명확히 → 큰 앱도 길을 잃지 않음
@Decorator@Controller/@Get으로 라우트를 선언적으로 → 의도가 코드에 드러남
요청 파이프라인Guard·Pipe·Interceptor·Filter로 횡단 관심사를 한 곳에 (Week 8·11)
🧭 "감싼다(wrap)"의 진짜 의미

NestJS는 기본적으로 내부에서 Express를 HTTP 어댑터로 사용합니다. 즉 Nest 코드를 짜면 그 아래에서 결국 Express가 돈다는 뜻입니다. (원하면 Fastify 어댑터로 갈아끼울 수도 있죠.) 그래서 "Nest냐 Express냐"는 양자택일이 아니라, Express를 깔고 그 위에 구조를 입을 것인가의 문제입니다.

4그래서, 언제 Nest를?

프레임워크는 공짜가 아닙니다. Nest는 배워야 할 개념(데코레이터·모듈·DI)이 있고, 작은 스크립트엔 과합니다. 선택 기준을 정리하면:

💡 선택 가이드

Express(또는 raw)가 어울릴 때: 엔드포인트 몇 개짜리 프로토타입, 단순 프록시·웹훅 수신기, 일회성 스크립트. NestJS가 빛날 때: 여러 도메인(User·Post·Comment…)이 얽히고, 인증·검증·트랜잭션 같은 횡단 관심사가 필요하고, 여러 명이 오래 유지보수할 서비스. 한마디로 "오래 키울 제품"이면 Nest입니다 — 우리가 만들 SNS 백엔드가 정확히 그 경우입니다.

⚠️ "Nest가 느리다"는 오해

Nest가 Express를 감싸므로 "한 겹 더 느리지 않냐"고 묻곤 합니다. 추상화 비용은 사실상 무시할 수준이고, 대부분의 병목은 프레임워크가 아니라 DB 쿼리·네트워크 I/O입니다. 정말 극한 처리량이 필요하면 Nest를 Fastify 어댑터로 돌릴 수도 있습니다 — 구조는 그대로 둔 채로요.

복습

5✅ 셀프 체크 — 면접관처럼 물어보기

Q NestJS가 내부적으로 기본 사용하는 HTTP 라이브러리는?

Express입니다. Nest는 기본 HTTP 어댑터로 Express를 사용하고, 그 위에 DI·모듈·데코레이터 같은 구조를 얹습니다. 그래서 "Nest는 Express를 감싼다(wrap)"고 말합니다. 필요하면 Fastify 어댑터로 교체할 수도 있어, Nest 코드 구조는 유지한 채 하부 HTTP 엔진만 바꾸는 것도 가능합니다.

Q 데코레이터·DI가 주는 이점은?

데코레이터는 @Controller/@Get처럼 "이 클래스·메서드가 무엇인지"를 선언적으로 드러내, 라우팅 분기 코드 없이 의도가 곧 코드가 됩니다. DI(의존성 주입)는 필요한 객체를 직접 new 하지 않고 주입받게 해, ① 결합도가 낮아지고 ② 테스트 시 가짜(mock)로 갈아끼우기 쉬우며 ③ 객체 생성·수명을 컨테이너가 일관되게 관리합니다. 둘 다 "코드를 조직하는 규칙"을 강제해 큰 앱에서 길을 잃지 않게 합니다.

Q Express만으로 규모가 커지면 무엇이 힘들어지나요?

Express는 "어떻게 조직할지"를 강제하지 않아, 프로젝트마다·사람마다 구조가 제각각이 됩니다. 라우트가 수십 개로 늘면 ① 파일·폴더 구조에 정답이 없어 일관성이 무너지고, ② 의존성을 손으로 require/new 해 결합도가 높아지며, ③ 인증·검증·로깅 같은 횡단 관심사가 라우트마다 흩뿌려지고, ④ 테스트하려면 실제 객체를 직접 끼워야 해 번거로워집니다. Nest의 모듈·DI·파이프라인은 바로 이 무질서를 막으려는 장치입니다.
