프로젝트 셋업 · SNS 컨셉
요청 라이프사이클

nest new로 골격을 만들고, 우리가 12주 동안 만들 인스타그램 SNS 도메인을 정의하고, 요청이 컨트롤러에 닿기까지 거치는 단계를 외운다.

“

nest CLI로 골격을 생성하고 → 인스타그램 SNS 도메인(User · Post · Comment · Follow)을 그려 보고 → 모든 요청이 컨트롤러 앞에서 거치는 Request Life Cycle(Middleware → Guard → Interceptor → Pipe → Handler → …)을 머릿속에 새긴다. 이 셋이 앞으로 12주의 좌표축이다.

  • Nest CLI로 프로젝트를 만들고 기본 폴더 구조를 읽는다
  • 우리가 만들 인스타그램 SNS의 도메인 모델(엔티티·관계)을 개관한다
  • 요청이 거치는 Request Life Cycle의 순서를 외운다

1nest new — 골격과 폴더 구조

Nest는 CLI가 골격을 대신 만들어 줍니다. 빈 폴더에서 시작해 손으로 설정 파일을 만들 필요가 없습니다 — nest new 한 번이면 실행 가능한 최소 앱이 떨어집니다.

terminal
# CLI 전역 설치 후 프로젝트 생성
npm i -g @nestjs/cli
nest new my-sns          # 패키지 매니저 고르면 골격 생성

cd my-sns
npm run start:dev        # watch 모드 → http://localhost:3000

생성된 src/ 안에는 아래 핵심 파일들이 들어 있습니다. 세 덩어리(진입점 · 루트 모듈 · 컨트롤러)가 골격의 전부입니다.

src/ — nest new가 만들어 준 골격(요약)
// ── main.ts ── 진입점: AppModule을 루트로 앱을 만들고 포트를 연다
import { NestFactory } from "@nestjs/core";
import { AppModule } from "./app.module";

async function bootstrap() {
  const app = await NestFactory.create(AppModule);
  await app.listen(3000);
}
bootstrap();

// ── app.module.ts ── 루트 모듈: 컨트롤러·프로바이더를 묶는 "조립 설명서"
@Module({
  imports: [],
  controllers: [AppController],
  providers: [AppService],   // ← DI 컨테이너가 연결 (Week 3 심화)
})
export class AppModule {}
🍔 비유 — 밀키트(meal kit)

nest new는 밀키트를 받는 것과 같습니다. 손질된 재료(폴더 구조), 레시피 카드(설정 파일), 조립 설명서(app.module.ts)가 박스째 도착하죠. 당신은 처음부터 장을 보지 않고, 바로 요리(기능 추가)부터 시작할 수 있습니다. @Module은 "이 키트에 무슨 재료(컨트롤러·서비스)가 들어가는지" 적은 그 조립 설명서입니다.

📝 순수 툴링은 콜아웃으로

환경설정·디버거·PgAdmin·Postman 같은 순수 툴링은 이 자료에서 별도 레슨으로 다루지 않습니다. 대신 해당 도구가 실제로 필요해지는 주차(예: PostgreSQL은 Week 4, 엔드포인트 테스트는 Week 2)에서 짧은 콜아웃으로 그때그때 안내합니다. 지금은 "골격이 어떻게 생겼나"만 눈에 익히면 충분합니다.

2인스타그램 SNS 컨셉 — 우리가 만들 것

12주 동안 우리는 단편 예제가 아니라 하나의 인스타그램형 SNS 백엔드를 향해 갑니다. 도메인을 먼저 그려 두면, 매 주차의 기능이 이 그림의 어느 부분을 채우는지 한눈에 보입니다. 핵심 엔티티는 넷 — User · Post · Comment · Follow입니다.

3Request Life Cycle — 요청이 거치는 관문들

클라이언트의 요청은 컨트롤러 핸들러에 곧장 도착하지 않습니다. 여러 관문을 순서대로 통과합니다. 이 순서를 외워 두면, 앞으로 "검증은 어디서? 인증은 어디서? 로깅은 어디서?"를 정확히 어느 단계에 끼울지 바로 답할 수 있습니다.

🧭 순서를 외우는 한 문장

"미들웨어가 문을 열고 → 가드가 신분증을 검사하고 → 인터셉터가 가로채고 → 파이프가 짐을 검색한 뒤 → 핸들러가 일을 한다." 각 단계는 앞으로 별도 주차에서 깊게 다룹니다 — Pipe·Guard·Decorator는 Week 8, Interceptor·Exception Filter·Middleware는 Week 11. 지금은 순서만 확실히 잡아 두세요.

⚠️ Guard와 Pipe의 자리를 헷갈리지 말 것

"통과 여부"(인증/인가)는 Guard가 핸들러 이전에 결정하고, "입력의 모양"(검증·변환)은 Pipe가 핸들러 직전에 다듬습니다. 인증을 Pipe에서, 검증을 Guard에서 하려 하면 자리가 어긋나 동작이 꼬입니다. 막을지 말지 → Guard, 바꿀지 말지 → Pipe로 기억하세요.

복습

4✅ 셀프 체크 — 면접관처럼 물어보기

Q Request Life Cycle의 순서를 나열해 보세요.

Middleware → Guard → Interceptor(before) → Pipe → Route Handler → Interceptor(after) → (응답) 순서입니다. 그리고 이 흐름 어디서든 예외가 발생하면 Exception Filter가 잡아 응답으로 변환합니다. 한 줄 요약: "문 열고(Middleware) → 신분 검사(Guard) → 가로채고(Interceptor) → 입력 검사(Pipe) → 처리(Handler) → 응답 가공(Interceptor)".

Q Guard와 Middleware의 차이는?

둘 다 핸들러 이전 단계지만 목적과 정보량이 다릅니다. Middleware는 가장 앞단의 범용 전처리(로깅·CORS·바디 파싱 등)로, raw req/res에 접근하지만 "어떤 핸들러로 갈지(라우트 컨텍스트)"는 아직 모릅니다. Guard는 더 뒤에서 실행되며 ExecutionContext를 받아 "지금 어떤 컨트롤러·핸들러로 가는지"를 알고, true/false(또는 예외)로 접근을 허용/차단합니다 — 즉 인증·인가에 특화돼 있습니다. 요약하면 Middleware는 "범용 전처리", Guard는 "라우트를 아는 통과 검문소"입니다.

Q nest new가 만들어 주는 핵심 파일은?

src/ 안의 세 덩어리가 핵심입니다. ① main.ts — 진입점으로 NestFactory.create(AppModule)로 앱을 만들고 app.listen()으로 포트를 엽니다. ② app.module.ts — 루트 모듈로 @Module({ controllers, providers })에 구성요소를 등록합니다. ③ app.controller.ts(+ app.service.ts) — 라우트 핸들러와 로직. 그 밖에 tsconfig·nest-cli.json·package.json 같은 설정과 기본 테스트 파일도 함께 생성됩니다.
