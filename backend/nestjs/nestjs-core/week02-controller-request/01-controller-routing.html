Controller &
데코레이터 라우팅

@Controller·@Get·@Post… 데코레이터만으로 URL을 메서드에 연결한다. 라우팅 분기 코드가 사라지고, 의도가 선언으로 드러나는 순간을 직접 확인한다.

"

Express에서 app.get('/posts', handler)라고 직접 등록하던 라우트가, NestJS에서는 클래스 위에 올린 데코레이터로 바뀐다. 코드가 "어떻게 연결하는가"가 아니라 "무엇인가"를 말하기 시작하는 것 — 그게 데코레이터 라우팅의 핵심이다.

  • @Controller('posts')가 URL prefix를 어떻게 정의하는지 안다
  • @Get/@Post/@Patch/@Delete 메서드 데코레이터와 경로 파라미터를 쓸 수 있다
  • 라우트 순서와 와일드카드 사용 시 주의사항을 설명할 수 있다
  • 핸들러가 반환한 값이 HTTP 응답으로 직렬화되는 과정을 이해한다

1@Controller('posts') prefix & 라우팅 개념

NestJS의 컨트롤러는 URL 공간의 한 구역을 담당하는 우체국 창구입니다. @Controller('posts')는 "이 클래스는 /posts로 시작하는 모든 URL을 담당한다"고 선언합니다. 그 안의 각 메서드가 세부 경로를 맡습니다.

🍔 비유 — 우편물 분류실

대형 우체국(NestJS 앱)에는 여러 창구(컨트롤러)가 있습니다. "posts 창구"(@Controller('posts'))는 "/posts"라고 적힌 우편물만 받습니다. 창구 안에는 담당 직원들(핸들러 메서드)이 앉아 있고, 각자 담당 업무가 붙어 있습니다 — @Get() 직원은 "조회" 우편물, @Post() 직원은 "신규 등록" 우편물을 처리합니다. 우편물이 도착하면 주소를 보고 창구를 찾고, 창구에서 업무 태그를 보고 직원을 찾는 두 단계 매칭이 일어납니다.

posts.controller.ts — 기본 구조
import { Controller, Get, Post, Patch, Delete } from '@nestjs/common';

@Controller('posts')           // → 이 클래스의 모든 라우트는 /posts 로 시작
export class PostsController {

  @Get()                       // GET /posts
  findAll() { return []; }

  @Get(':id')                  // GET /posts/:id
  findOne() { return {}; }

  @Post()                      // POST /posts
  create() { return {}; }

  @Patch(':id')               // PATCH /posts/:id
  update() { return {}; }

  @Delete(':id')              // DELETE /posts/:id
  remove() { return {}; }
}

2@Get/@Post/@Patch/@Delete 메서드 데코레이터

메서드 데코레이터는 HTTP 메서드 + 하위 경로를 동시에 선언합니다. 인자가 없으면 /(컨트롤러 prefix 그 자체), 문자열을 넘기면 prefix/문자열이 최종 경로가 됩니다.

데코레이터HTTP 메서드예: prefix='posts'일 때대표 용도
@Get()GETGET /posts목록 조회
@Get(':id')GETGET /posts/:id단건 조회
@Post()POSTPOST /posts생성
@Patch(':id')PATCHPATCH /posts/:id부분 수정
@Delete(':id')DELETEDELETE /posts/:id삭제
💡 PUT vs PATCH — SNS 프로젝트에선 PATCH를

PUT은 자원 전체를 교체(기존 필드 전부 보내야 함), PATCH는 일부만 수정합니다. SNS 게시글 수정처럼 "content만 바꿀게요"는 PATCH가 적합합니다. 이 커리큘럼에서는 PATCH를 기본으로 씁니다.

3라우트 path · 와일드카드 · 순서 주의

경로 파라미터(:id)는 어떤 문자열도 매칭합니다. 그래서 더 구체적인 경로를 먼저 선언해야 합니다 — :id가 먼저 있으면 뒤에 오는 구체적인 경로를 가로챌 수 있기 때문입니다.

라우트 순서 주의 예시
@Controller('posts')
export class PostsController {

  // ✅ 올바른 순서: 구체적인 경로를 :id보다 앞에 선언
  @Get('popular')              // GET /posts/popular ← 먼저 매칭됨
  findPopular() { return []; }

  @Get(':id')                  // GET /posts/:id ← popular가 아니면 여기
  findOne() { return {}; }

  // ⚠️ 와일드카드: 드물게만 사용, 순서 더욱 중요
  @Get('ab*cd')               // abXYcd, abcd, ab123cd 모두 매칭
  wildcard() { return 'wildcard'; }
}
⚠️ :id가 'popular'를 삼킨다

@Get(':id')를 @Get('popular')보다 먼저 선언하면, GET /posts/popular 요청이 findPopular가 아닌 findOne으로 라우팅됩니다 — :id가 문자열 'popular'도 잡아버리기 때문입니다. 더 구체적인 경로를 항상 위에 두세요.

4핸들러 반환값이 응답이 되는 과정

NestJS는 핸들러가 반환한 값을 자동으로 HTTP 응답으로 변환합니다. 별도로 res.json()을 호출할 필요가 없습니다.

반환값 → HTTP 응답 자동 변환
@Get()
findAll() {
  return [{ id: 1, content: '첫 글' }];
  // → 200 OK + Content-Type: application/json
  // → Body: [{"id":1,"content":"첫 글"}]
}

@Post()
create() {
  return { id: 2, content: '새 글' };
  // → 201 Created (POST는 기본 201)
}

// 상태코드를 명시하고 싶을 땐 @HttpCode
@Delete(':id')
@HttpCode(204)
remove() { return; }   // 204 No Content
🧭 객체/배열 → JSON, 문자열 → text/plain

반환값이 객체나 배열이면 Nest가 JSON으로 직렬화하고 Content-Type: application/json을 붙입니다. 문자열이면 text/html로 그대로 내보냅니다. Promise나 Observable을 반환해도 자동으로 기다렸다가 그 결과를 직렬화합니다.

복습

5✅ 셀프 체크 — 면접관처럼 물어보기

Q @Controller('posts')와 @Get(':id')를 같이 쓰면 최종 URL은?

GET /posts/:id입니다. @Controller에 넘긴 문자열이 prefix(공통 앞 경로)가 되고, 메서드 데코레이터의 인자가 하위 경로가 됩니다. 두 경로가 /로 합쳐집니다. 따라서 @Controller('api/v1/posts')처럼 버전 prefix를 클래스 레벨에서 한 번에 관리할 수도 있습니다.

Q 컨트롤러에 비즈니스 로직(DB 조회, 계산 등)을 바로 넣으면 안 되는 이유는?

컨트롤러의 책임은 요청을 받고, 적절한 곳에 위임하고, 응답을 돌려주는 것뿐입니다. 여기에 DB 조회나 복잡한 계산을 넣으면 ① 컨트롤러가 비대해져 테스트하기 어렵고, ② 같은 로직을 다른 곳(예: 배치·WebSocket)에서 재사용하기 어렵고, ③ Week 3에서 배울 Service(서비스 계층)로의 역할 분리가 불가능해집니다. 컨트롤러는 항상 "얇게" 유지하고, 로직은 Service에게 위임합니다.

Q @Post() 핸들러가 반환한 값의 기본 상태코드는 몇인가요? 이유는?

기본값은 201 Created입니다. REST 관례상 POST는 새 자원을 생성하므로, "만들어졌다"는 의미의 201이 의미적으로 맞습니다. GET·PATCH·DELETE 등 나머지 메서드의 기본값은 200 OK입니다. 기본값을 바꾸고 싶으면 @HttpCode(204) 데코레이터를 핸들러 위에 추가하면 됩니다.
