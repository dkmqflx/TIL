Query · Param · Body
요청 데이터 받기

하나의 HTTP 요청 안에는 여러 조각이 있다 — 경로 변수, 쿼리스트링, 바디, 헤더. 데코레이터 하나씩이 그 조각과 1:1로 매핑된다.

"

Express에서는 req.params.id, req.query.page, req.body로 직접 꺼냈다. NestJS에서는 핸들러 파라미터에 데코레이터를 달면 Nest가 알아서 꺼내 준다 — 코드가 요청의 어느 부분을 쓰는지 파라미터 선언만으로 드러난다.

  • @Param('id')로 경로 변수를 추출하는 방법을 안다
  • @Query()로 쿼리스트링 전체 또는 특정 키를 받는 방법을 안다
  • @Body()로 요청 바디를 받고, 어떤 메서드에서 주로 쓰이는지 안다
  • 목록·단건·생성·수정·삭제 REST API 세트를 한눈에 이해한다

1HTTP 요청 해부 — 데이터가 어디서 오나

요청 데이터는 네 군데에 나뉘어 담깁니다. 각 위치마다 목적이 다르고, NestJS는 그 각각에 대응하는 데코레이터를 제공합니다.

🍔 비유 — 택배 운송장

택배 한 박스를 받는다고 상상해 보세요. 주소(URL 경로)에는 "서울시 강남구 42번지"처럼 어디로 갈지가 적혀 있고, 운송 메모(쿼리스트링)에는 "문 앞 놓아주세요" 같은 옵션이, 포장 안쪽(바디)에는 실제 물건이, 운송장 헤더(Header)에는 "취급주의", "냉장" 같은 메타 정보가 담깁니다. HTTP 요청도 정확히 이 구조입니다.

2데코레이터 빠른 참조표

데코레이터요청의 어느 부분예시 / 반환 타입
@Param('id')URL 경로 변수 :id'42' — 항상 string
@Param()경로 변수 전체 객체{ id: '42' }
@Query('page')쿼리스트링 특정 키'2' — 항상 string
@Query()쿼리스트링 전체 객체{ page: '2', limit: '10' }
@Body()요청 바디 (JSON)파싱된 객체 전체
@Body('content')바디의 특정 키'새 글'
@Headers()요청 헤더 전체{ authorization: '…' }
@Headers('authorization')특정 헤더'Bearer …'

3코드로 보는 각 데코레이터

posts.controller.ts — 데코레이터 사용 예
import {
  Controller, Get, Post, Patch, Delete,
  Param, Query, Body, Headers,
} from '@nestjs/common';

@Controller('posts')
export class PostsController {

  // GET /posts?page=2&limit=10
  @Get()
  findAll(@Query() query: { page: string; limit: string }) {
    return { page: query.page, limit: query.limit };
  }

  // GET /posts/42
  @Get(':id')
  findOne(@Param('id') id: string) {
    return { id };   // id는 항상 string — 숫자가 필요하면 +id 또는 ParseIntPipe
  }

  // POST /posts
  @Post()
  create(@Body() body: { content: string }) {
    return { content: body.content };
  }

  // PATCH /posts/42
  @Patch(':id')
  update(@Param('id') id: string, @Body() body: { content: string }) {
    return { id, content: body.content };
  }

  // DELETE /posts/42
  @Delete(':id')
  remove(@Param('id') id: string) {
    return { deleted: id };
  }
}
💡 @Param 값은 항상 string — ParseIntPipe

URL에서 꺼낸 값은 항상 문자열입니다. 숫자 ID가 필요하다면 @Param('id', ParseIntPipe) id: number처럼 Pipe로 변환하세요. Pipe는 Week 8에서 자세히 다루지만, 지금은 "타입 변환이 필요하면 Pipe"로만 기억해 두면 충분합니다.

5✅ 셀프 체크 — 면접관처럼 물어보기

Q @Param과 @Query의 선택 기준은 무엇인가요?

@Param(경로 변수)은 자원을 식별할 때 씁니다 — "42번 게시글"처럼 특정 자원을 콕 찍어야 할 때 URL 경로 안에 넣습니다. @Query(쿼리스트링)은 동일한 자원 집합에 필터·정렬·페이지를 적용할 때 씁니다 — "게시글 목록의 2번째 페이지, 10개씩" 같은 선택적 조건이라면 ?page=2&limit=10으로 전달합니다. 규칙: 없으면 자원 자체를 찾을 수 없는 값이면 Param, 없어도 자원은 찾을 수 있지만 결과를 좁히는 값이면 Query.

Q @Body()는 어떤 HTTP 메서드에서 주로 쓰이나요? GET에서도 쓸 수 있나요?

@Body()는 POST·PUT·PATCH처럼 "생성/수정" 동작에서 주로 씁니다 — 클라이언트가 서버에 데이터를 보낼 때 바디에 담기 때문입니다. GET 요청에는 관례적으로 바디를 쓰지 않습니다. HTTP 스펙이 금지하지는 않지만, 많은 프록시와 클라이언트(브라우저 fetch·curl 기본 동작 등)가 GET 바디를 무시하거나 제거합니다. GET에서 필터/검색 조건을 보내야 한다면 Query String으로 전달하는 것이 올바른 REST 관례입니다.
