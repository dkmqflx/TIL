DB 없이도 컨트롤러의 CRUD 모양은 완성할 수 있다. 그리고 자원이 없을 때 throw new NotFoundException() 한 줄이면, Nest 내장 Exception Filter가 알아서 404 JSON 응답을 만들어 준다. 예외를 HTTP 코드로 변환하는 것도 프레임워크의 일이다.

  • 인메모리 배열로 CRUD(목록·단건·생성·수정·삭제)를 완성한다
  • NotFoundException 등 NestJS 내장 HTTP 예외를 던지는 법을 안다
  • 예외 → HTTP 응답(상태코드 + JSON 바디) 변환 흐름을 이해한다
  • 인메모리 저장의 한계와 Week 4 DB 연동의 필요성을 안다

1인메모리 배열로 CRUD 구현

실제 DB 없이 클래스 내부 배열에 게시글을 쌓으면, 컨트롤러의 역할과 CRUD 흐름을 군더더기 없이 이해할 수 있습니다. Week 4에서 TypeORM+PostgreSQL로 교체하기 전까지 이 구조로 작동합니다.

posts.controller.ts — 인메모리 CRUD
import {
  Controller, Get, Post, Patch, Delete,
  Param, Query, Body,
  NotFoundException,
} from '@nestjs/common';

/** 게시글 타입 (임시 인메모리용) */
interface Post {
  id:      number;
  content: string;
}

@Controller('posts')
export class PostsController {
  private posts: Post[] = [];
  private nextId = 1;

  // GET /posts?page=1
  @Get()
  findAll(@Query('page') page = '1') {
    return { page: +page, data: this.posts };
  }

  // GET /posts/:id
  @Get(':id')
  findOne(@Param('id') id: string) {
    const post = this.posts.find(p => p.id === +id);
    if (!post) throw new NotFoundException(`Post #${id} not found`);
    return post;
  }

  // POST /posts
  @Post()
  create(@Body() body: { content: string }) {
    const post: Post = { id: this.nextId++, content: body.content };
    this.posts.push(post);
    return post;
  }

  // PATCH /posts/:id
  @Patch(':id')
  update(@Param('id') id: string, @Body() body: { content: string }) {
    const post = this.posts.find(p => p.id === +id);
    if (!post) throw new NotFoundException(`Post #${id} not found`);
    post.content = body.content;
    return post;
  }

  // DELETE /posts/:id
  @Delete(':id')
  remove(@Param('id') id: string) {
    const idx = this.posts.findIndex(p => p.id === +id);
    if (idx === -1) throw new NotFoundException(`Post #${id} not found`);
    this.posts.splice(idx, 1);
    return { deleted: +id };
  }
}
⚠️ 인메모리 저장의 한계 — Week 4 예고

배열로 저장하면 서버를 재시작할 때마다 데이터가 사라집니다. 또 서버 인스턴스가 여러 개면 각자 독립된 배열을 가져 데이터가 불일치합니다. 실제 서비스에서는 PostgreSQL 같은 영속 DB가 반드시 필요합니다. Week 4에서 Docker로 PostgreSQL을 띄우고 TypeORM으로 연결하면 이 배열을 DB 테이블로 교체합니다.

2NestJS 내장 HTTP 예외

NestJS는 자주 쓰이는 HTTP 에러 상황마다 미리 만들어진 예외 클래스를 제공합니다. throw new NotFoundException() 한 줄이면 내장 Exception Filter가 알아서 JSON 응답을 만들어 돌려줍니다.

예외 클래스HTTP 상태코드대표 사용 상황
NotFoundException404 Not Found자원을 찾을 수 없을 때
BadRequestException400 Bad Request잘못된 입력값
UnauthorizedException401 Unauthorized인증되지 않은 요청
ForbiddenException403 Forbidden권한 없음
ConflictException409 Conflict중복 데이터(이미 존재)
InternalServerErrorException500 Internal Server Error예상치 못한 서버 오류
🧭 모두 HttpException을 상속

위 클래스들은 전부 @nestjs/common의 HttpException을 상속합니다. 표에 없는 상황은 throw new HttpException('Custom message', 418)처럼 직접 상태코드를 지정할 수 있습니다. 전체 내장 예외 목록은 공식 문서 Built-in HTTP exceptions에서 확인하세요.

3예외 → HTTP 응답 변환 흐름

핸들러에서 예외를 던지면, Nest의 내장 Exception Filter가 잡아서 일정한 형식의 JSON 응답으로 바꿉니다. 개발자가 직접 res.status(404).json({...})를 쓸 필요가 없습니다.

🍔 비유 — 주방 → 안내 데스크

주방(핸들러)에서 "이 메뉴는 없습니다"라고 알리면, 안내 데스크(Exception Filter)가 손님에게 정해진 양식으로 "404: 해당 메뉴를 찾을 수 없습니다"라고 안내합니다. 주방은 무슨 알림판을 쓸지 몰라도 됩니다 — 알림 규격은 안내 데스크가 통일해서 처리합니다.

💡 메시지 커스터마이즈

throw new NotFoundException('Post #42 not found')처럼 문자열을 넘기면 응답 JSON의 message 필드가 그 값으로 채워집니다. 두 번째 인자로 설명(description)을 넣을 수도 있습니다: new NotFoundException('찾을 수 없음', 'POSTS_NOT_FOUND'). 클라이언트가 에러 원인을 정확히 파악할 수 있도록 구체적인 메시지를 담는 것이 좋습니다.

Q throw new NotFoundException()이 만드는 응답 형태는?

NestJS 내장 Exception Filter가 다음과 같은 JSON을 응답합니다. 

{
  "statusCode": 404,
  "message": "Not Found",     // 또는 throw 시 넘긴 커스텀 메시지
  "error": "Not Found"
}
HTTP 상태코드는 404, Content-Type: application/json이 붙습니다. 메시지를 지정하지 않으면 Nest 기본값인 "Not Found"가 들어가고, 문자열을 넘기면 그 문자열로 message가 채워집니다.

Q NestJS 내장 예외 클래스 전체 목록은 어디서 확인하나요?

공식 문서 docs.nestjs.com → Exception Filters → Built-in HTTP exceptions에서 확인할 수 있습니다. 또는 IDE에서 @nestjs/common의 타입 선언을 열고 HttpException을 상속한 클래스들을 보면 됩니다 — BadRequestException, UnauthorizedException, ForbiddenException, NotFoundException, MethodNotAllowedException, NotAcceptableException, RequestTimeoutException, ConflictException, GoneException, PayloadTooLargeException, UnsupportedMediaTypeException, UnprocessableEntityException, InternalServerErrorException, NotImplementedException, BadGatewayException, ServiceUnavailableException, GatewayTimeoutException 등이 있습니다.
