Service로
로직 분리

컨트롤러는 요청·응답만, 비즈니스 로직은 서비스로. 책임 분리가 테스트·재사용·유지보수를 어떻게 바꾸는지 직접 확인한다.

"

컨트롤러가 DB 조회, 계산, 유효성 검사를 모두 떠안으면 테스트도 재사용도 어려워진다. NestJS는 서비스(Service)라는 계층을 두어 @Injectable() 클래스에 로직을 몰아넣고, 컨트롤러는 "받아서 위임하고 돌려주는" 역할만 하게 만든다.

  • @Injectable() 데코레이터의 의미와 서비스 클래스의 역할을 설명할 수 있다
  • 컨트롤러는 얇게(요청·응답 처리만), 로직은 서비스에 두는 이유를 말할 수 있다
  • 책임 분리로 얻는 테스트 용이성·재사용성을 구체적으로 설명할 수 있다
  • PostsService에 CRUD 로직을 옮기고, 컨트롤러에서 호출하는 코드를 작성할 수 있다

1@Injectable() 서비스란

서비스(Service)는 비즈니스 로직을 담는 클래스입니다. @Injectable()을 붙이면 "NestJS DI 컨테이너가 이 클래스의 인스턴스를 만들어 관리해도 된다"고 선언하는 것입니다. 컨테이너가 인스턴스를 생성하고, 필요한 곳에 주입(inject)합니다.

🍔 비유 — 식당의 홀과 주방

홀 직원(컨트롤러)은 손님의 주문을 받아 주방에 전달하고, 완성된 음식을 손님에게 가져다줍니다. 요리를 직접 하지 않습니다. 주방(서비스)은 레시피와 조리 과정을 모두 알고 있습니다. 손님이 무슨 테이블에 앉았는지, HTTP로 왔는지 WebSocket으로 왔는지는 알 필요가 없습니다. 이 분업 덕분에 ① 주방 레시피를 고쳐도 홀 직원은 바뀌지 않고 ② 테스트 시에는 가짜 주문을 주방에만 넣어볼 수 있습니다.

posts.service.ts — @Injectable 기본 구조
import { Injectable, NotFoundException } from '@nestjs/common';

@Injectable()            // ← DI 컨테이너에 등록 선언
export class PostsService {
  private posts = [];
  private nextId = 1;

  findAll() { return this.posts; }

  findOne(id: number) {
    const post = this.posts.find(p => p.id === id);
    if (!post) throw new NotFoundException(`Post #${id} not found`);
    return post;
  }

  create(content: string) {
    const post = { id: this.nextId++, content };
    this.posts.push(post);
    return post;
  }
}
🧭 @Injectable()의 정확한 의미

@Injectable()은 "이 클래스를 직접 new하지 말고, NestJS 컨테이너가 만들어서 제공한다"는 표시입니다. 컨테이너는 이 클래스의 인스턴스를 딱 하나 만들어(기본 싱글턴) 필요한 곳마다 재사용합니다. @Controller(), @Module()도 내부적으로 같은 방식으로 동작하지만, 서비스는 @Injectable()을 명시해야 등록이 됩니다.

2컨트롤러는 얇게 — 요청·응답만

컨트롤러의 역할은 딱 세 가지입니다: ① HTTP 요청을 받고 ② 서비스에 위임하고 ③ 결과를 응답으로 돌려준다. 비즈니스 로직이 컨트롤러에 들어오기 시작하면, 같은 로직을 WebSocket이나 배치에서 재사용하려면 복사·붙여넣기가 필요해집니다.

posts.controller.ts — 얇은 컨트롤러, 서비스에 위임
import { Controller, Get, Post, Patch, Delete, Param, Body } from '@nestjs/common';
import { PostsService } from './posts.service';

@Controller('posts')
export class PostsController {
  // ① 생성자에서 서비스를 주입받는다 (Week 3 Lesson 02에서 상세 설명)
  constructor(private readonly postsService: PostsService) {}

  @Get()
  findAll() {
    return this.postsService.findAll();   // 로직은 서비스가
  }

  @Get(':id')
  findOne(@Param('id') id: string) {
    return this.postsService.findOne(+id);
  }

  @Post()
  create(@Body() body: { content: string }) {
    return this.postsService.create(body.content);
  }

  @Patch(':id')
  update(@Param('id') id: string, @Body() body: { content: string }) {
    return this.postsService.update(+id, body.content);
  }

  @Delete(':id')
  remove(@Param('id') id: string) {
    return this.postsService.remove(+id);
  }
}

3책임 분리의 효과 — 재사용·테스트

서비스로 로직을 분리하면 세 가지 이점이 즉시 생깁니다.

이점설명예
재사용같은 서비스를 여러 컨트롤러·게이트웨이에서 호출 가능REST + WebSocket이 PostsService.create()를 공유
단위 테스트HTTP 없이 서비스 메서드만 직접 테스트new PostsService().create('글')로 순수 로직 검증
유지보수DB를 인메모리 → TypeORM으로 바꿀 때 컨트롤러 손댈 필요 없음서비스 내부만 교체하면 끝
💡 컨트롤러 테스트 vs 서비스 테스트

컨트롤러 테스트는 HTTP 레이어(파라미터 파싱, 상태코드)를 검증하고, 서비스 테스트는 비즈니스 로직(없는 게시글에 예외 던지기, ID 증가 규칙 등)을 검증합니다. 둘을 분리하면 각 테스트가 작고 명확해집니다. NestJS는 @nestjs/testing의 TestingModule로 서비스를 독립 실행할 수 있습니다.

4✅ 셀프 체크 — 면접관처럼 물어보기

Q 로직을 서비스로 빼면 좋은 점을 두 가지만 말해보세요.

① 재사용성 — REST 컨트롤러와 WebSocket 게이트웨이가 같은 서비스 메서드를 호출할 수 있습니다. 로직을 한 곳에서만 고치면 됩니다. ② 단위 테스트 용이성 — HTTP 요청·응답 없이도 서비스 메서드를 new PostsService().findOne(1)처럼 직접 호출해서 로직만 검증할 수 있습니다. 컨트롤러를 거치지 않으니 테스트 코드가 간결해집니다.

Q @Injectable()을 붙이지 않으면 어떻게 되나요?

NestJS DI 컨테이너가 해당 클래스를 주입 가능한 프로바이더로 인식하지 못합니다. 다른 클래스(컨트롤러 등)에서 생성자에 타입으로 요청해도 컨테이너가 인스턴스를 만들어 줄 수 없어 런타임 에러가 발생합니다. @Injectable()은 "컨테이너야, 이 클래스를 네가 관리해도 돼"라고 알리는 메타데이터 선언입니다.

Q 컨트롤러에서 직접 DB 조회를 해도 동작은 되는데, 왜 서비스 계층을 나누나요?

동작은 하지만 세 가지 문제가 생깁니다: ① 재사용 불가 — WebSocket 게이트웨이나 배치에서 같은 로직이 필요하면 컨트롤러 코드를 복사해야 합니다. ② 테스트 어려움 — 컨트롤러 테스트에 DB 연결까지 필요하게 됩니다. ③ 변경 파급 범위 — DB를 바꾸면 컨트롤러도 바꿔야 합니다. 서비스 계층을 나누면 각 클래스가 하나의 이유로만 변경되는 단일 책임 원칙(SRP)을 지킬 수 있습니다.
