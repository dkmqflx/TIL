DI & IoC
원리

new PostsService()를 직접 쓰지 않는다. NestJS IoC 컨테이너가 인스턴스를 만들고, 필요한 곳에 주입(inject)해 준다. 이 역전이 테스트와 유연성을 만든다.

"

의존성을 직접 생성(new)하는 대신, 프레임워크(컨테이너)가 생성·관리·주입하게 맡기는 것 — 이게 IoC(제어의 역전)다. NestJS는 생성자 파라미터의 타입을 읽어 어떤 인스턴스를 주입할지 결정한다.

  • 의존성을 직접 new하지 않고 주입받는 이유를 설명할 수 있다
  • IoC(제어의 역전) 개념을 한 문장으로 설명할 수 있다
  • 생성자 주입(constructor injection) 문법을 작성할 수 있다
  • 생성자 주입이 단위 테스트를 쉽게 만드는 이유를 말할 수 있다

1의존성을 "직접 new 하지 않는다"

컨트롤러가 서비스를 쓰려면 서비스 인스턴스가 필요합니다. 가장 단순한 방법은 new PostsService()를 컨트롤러 안에서 호출하는 것이지만, NestJS는 이 방식을 권장하지 않습니다.

❌ 직접 new — 하지 말 것
@Controller('posts')
export class PostsController {
  // ❌ 직접 생성: 어떤 PostsService를 쓸지 컨트롤러가 결정
  private postsService = new PostsService();  // 강한 결합
}
✅ 생성자 주입 — NestJS 방식
@Controller('posts')
export class PostsController {
  // ✅ 생성자에서 받기만 한다 — 컨테이너가 만들어서 넣어 준다
  constructor(private readonly postsService: PostsService) {}
}

차이는 단순해 보이지만 의미가 큽니다 — 누가 인스턴스를 만드는가의 제어권이 컨트롤러에서 컨테이너로 넘어갑니다. 이것이 IoC(Inversion of Control, 제어의 역전)입니다.

🍔 비유 — 재료를 직접 사러 가지 않는 요리사

과거에는 요리사가 직접 시장에 가서 재료를 골라 사왔습니다(직접 new). 이제는 재료 담당 팀(IoC 컨테이너)이 "오늘 메뉴에 필요한 재료를 미리 준비해 두었습니다"라며 요리대에 올려놓습니다. 요리사는 받아서 요리만 합니다. 재료 공급처(구현체)를 바꾸고 싶을 때도 요리사 코드를 건드리지 않고, 재료 담당 팀의 목록만 수정합니다.

2IoC 컨테이너가 인스턴스를 만들고 주입

NestJS는 애플리케이션이 시작될 때 모든 providers를 분석하고, TypeScript의 타입 정보(emitDecoratorMetadata)를 이용해 어디에 무엇을 주입해야 하는지 계산합니다. 이후 인스턴스를 생성하고 생성자 파라미터에 넣어 줍니다.

3생성자 주입 — constructor injection

NestJS에서 가장 권장하는 주입 방식은 생성자 주입입니다. TypeScript의 private readonly 단축 문법을 쓰면 생성자 파라미터 선언과 클래스 프로퍼티 할당이 동시에 이루어집니다.

생성자 주입 전체 예시
import { Controller, Get } from '@nestjs/common';
import { PostsService } from './posts.service';

@Controller('posts')
export class PostsController {
  /**
   * PostsService 타입을 보고 컨테이너가 인스턴스를 찾아 주입.
   * private readonly = 선언 + 할당 동시. 외부에서 변경 불가.
   */
  constructor(private readonly postsService: PostsService) {}

  @Get()
  findAll() {
    return this.postsService.findAll();
  }
}

// 테스트할 때: 가짜(Mock) 서비스를 생성자에 직접 넘길 수 있다
// const ctrl = new PostsController(new FakePostsService());
💡 생성자 주입이 테스트를 쉽게 만드는 이유

생성자 주입을 쓰면 테스트 코드에서 가짜(Mock) 서비스를 직접 넘길 수 있습니다. new PostsController(new MockPostsService())처럼 DB 연결 없이도 컨트롤러만 고립시켜 테스트할 수 있습니다. 반면 new PostsService()를 내부에서 직접 생성하면, 그 구현체에 강하게 묶여 가짜 객체를 끼워 넣을 방법이 없습니다.

4프로바이더 토큰 — 타입이 키다

컨테이너는 토큰(token)으로 어떤 인스턴스를 주입할지 결정합니다. 가장 기본적이고 흔한 경우는 클래스 자체가 토큰이 되는 방식입니다. 생성자 파라미터에 PostsService 타입을 쓰면, 컨테이너가 해당 클래스로 등록된 인스턴스를 찾아 주입합니다.

클래스가 토큰 — 가장 기본 형태
// 모듈의 providers 배열에 PostsService를 등록하면:
@Module({ providers: [PostsService] })

// 이것은 아래의 축약 표현이다:
@Module({
  providers: [{
    provide:  PostsService,   // 토큰 (키)
    useClass: PostsService,   // 실제 구현체
  }]
})

// provide: PostsService 토큰으로 useClass: PostsService 인스턴스가 주입된다.
🧭 useValue / useFactory는 Week 후반에

프로바이더는 클래스 외에도 useValue(상수값), useFactory(팩토리 함수)를 토큰에 연결할 수 있습니다. 이 방식은 환경 설정이나 외부 클라이언트 주입 때 활용합니다. 지금은 "클래스 = 토큰 = 주입 대상"이라는 90% 케이스만 확실히 잡아두세요.

복습

5✅ 셀프 체크 — 면접관처럼 물어보기

Q IoC(제어의 역전)를 한 문장으로 설명해 보세요.

"객체가 자신의 의존성을 직접 생성하는 대신, 외부(컨테이너·프레임워크)가 생성하고 주입해 주도록 제어권을 넘기는 것"입니다. 컨트롤러가 new PostsService()를 직접 부르지 않고, NestJS 컨테이너가 대신 만들어 생성자에 넘겨주는 것이 그 예입니다.

Q 생성자 주입이 단위 테스트를 쉽게 만드는 이유는?

생성자 주입을 쓰면 테스트 코드에서 가짜(Mock) 인스턴스를 직접 넘길 수 있기 때문입니다. new PostsController(new MockPostsService())처럼 DB 없이도 컨트롤러의 HTTP 처리 로직만 독립적으로 테스트할 수 있습니다. 반면 내부에서 new PostsService()를 직접 호출하면 구현체에 강하게 묶여 테스트 시 대체할 방법이 없습니다.

Q NestJS가 어떤 인스턴스를 주입할지 어떻게 결정하나요?

TypeScript의 데코레이터 메타데이터(emitDecoratorMetadata)를 통해 생성자 파라미터의 타입 정보를 읽습니다. 예를 들어 constructor(private postsService: PostsService)를 보면 PostsService라는 클래스(토큰)를 키로 컨테이너에서 대응하는 인스턴스를 찾아 주입합니다. 모듈의 providers 배열에 등록된 클래스만 주입받을 수 있습니다.
