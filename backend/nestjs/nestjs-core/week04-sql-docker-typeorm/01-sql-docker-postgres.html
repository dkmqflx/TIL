SQL 기본기
Docker로 PostgreSQL

관계형 데이터베이스의 핵심 개념을 빠르게 정리하고, Docker Compose 한 파일로 PostgreSQL을 로컬에서 바로 띄운다. TypeORM을 배우기 전에 DB가 실제로 무슨 일을 하는지 손으로 느껴 보자.

"

TypeORM은 결국 SQL을 대신 써 주는 번역가다. 번역가를 잘 쓰려면 원어(SQL)를 조금은 알아야 한다. 그리고 그 DB는 docker compose up 한 줄로 어디서든 동일하게 켤 수 있어야 한다.

  • 테이블·행·열·PK가 무엇인지 한 문장으로 설명할 수 있다
  • SELECT / INSERT / UPDATE / DELETE 구문의 기본 형태를 안다
  • Docker 이미지와 컨테이너의 차이를 설명할 수 있다
  • docker-compose.yml을 작성해 PostgreSQL 컨테이너를 띄울 수 있다

1SQL 기초 — 테이블, PK, 4가지 명령

관계형 DB는 데이터를 표(Table)에 저장합니다. 각 표는 열(Column)과 행(Row)으로 이루어지고, 모든 행을 고유하게 식별하는 열이 기본 키(Primary Key, PK)입니다. SQL은 이 표를 읽고 쓰는 언어입니다.

SQL 명령CRUD예시
SELECTReadSELECT * FROM posts WHERE id = 1;
INSERTCreateINSERT INTO posts (title) VALUES ('첫 글');
UPDATEUpdateUPDATE posts SET title = '수정' WHERE id = 1;
DELETEDeleteDELETE FROM posts WHERE id = 1;
SQL — posts 테이블 예제
-- 테이블 생성
CREATE TABLE posts (
  id      SERIAL PRIMARY KEY,
  title   TEXT    NOT NULL,
  content TEXT
);

-- 행 추가
INSERT INTO posts (title, content)
  VALUES ('첫 글', '안녕하세요!');

-- 전체 조회
SELECT * FROM posts;

-- 특정 행만 수정
UPDATE posts SET title = '수정된 글' WHERE id = 1;

-- 삭제
DELETE FROM posts WHERE id = 1;
💡 PK는 절대 중복·NULL 불가

SERIAL PRIMARY KEY는 PostgreSQL에서 자동 증가하는 정수 PK를 만드는 가장 간단한 방법입니다. TypeORM의 @PrimaryGeneratedColumn()이 이 SQL을 대신 생성합니다.

2Docker — 이미지 vs 컨테이너

Docker를 이해하는 핵심 개념은 딱 두 가지입니다. 이미지(Image)는 실행 가능한 패키지 — 프로그램 코드·환경·설정을 모두 담은 청사진입니다. 컨테이너(Container)는 그 이미지를 실제로 실행한 프로세스입니다. 같은 이미지에서 여러 컨테이너를 만들 수 있고, 컨테이너를 지워도 이미지는 남습니다.

🍔 비유 — 어디서든 똑같이 데우는 밀폐 도시락

이미지는 레시피대로 이미 완성된 밀폐 도시락입니다. 서울 팀원 PC에서든, CI 서버에서든, 클라우드에서든 같은 도시락을 꺼내면 같은 맛이 납니다. 컨테이너는 그 도시락을 전자레인지에 넣고 돌린 상태 — 실제로 열이 나고 먹을 수 있는 실행 중인 인스턴스입니다. 도시락을 버려도(컨테이너 삭제) 레시피(이미지)는 사라지지 않습니다. 다시 꺼내 돌리면 그만입니다.

3docker-compose.yml로 PostgreSQL 띄우기

Docker Compose는 여러 컨테이너를 하나의 파일로 선언하고, docker compose up 한 줄로 모두 시작합니다. 실무에서는 보통 DB · 앱 · 캐시 서버를 함께 정의합니다. 지금은 PostgreSQL 하나만 띄워 봅니다.

docker-compose.yml
version: '3.9'

services:
  postgres:
    image: postgres:16
    container_name: nestjs_postgres
    environment:
      POSTGRES_USER: admin
      POSTGRES_PASSWORD: secret
      POSTGRES_DB: nestjs_db
    ports:
      - "5432:5432"       # 호스트:컨테이너
    volumes:
      - postgres_data:/var/lib/postgresql/data  # 데이터 영속화

volumes:
  postgres_data:
terminal — 컨테이너 시작 / 중지
# 백그라운드로 컨테이너 시작 (-d: detached)
docker compose up -d

# 실행 중인 컨테이너 확인
docker compose ps

# 컨테이너 중지 (데이터는 volume에 유지)
docker compose down
💡 volumes: 가 없으면 컨테이너를 지울 때 DB 데이터도 사라진다

volumes 선언으로 데이터 디렉터리를 Docker 관리 볼륨에 마운트하면, 컨테이너를 재시작하거나 삭제 후 재생성해도 데이터가 보존됩니다. docker compose down -v를 쓰면 볼륨까지 삭제됩니다 — 개발 환경 초기화 시에만 사용하세요.

4VSCode Postgres 익스플로러로 연결 확인

컨테이너가 뜨면 VSCode의 PostgreSQL 익스텐션(또는 Database Client)으로 시각적으로 확인할 수 있습니다. 다음 정보를 입력합니다.

🧭 VSCode 연결 정보

Host: localhost  /  Port: 5432  /  Database: nestjs_db
Username: admin  /  Password: secret
이는 docker-compose.yml의 POSTGRES_* 환경변수와 일치해야 합니다. 연결이 성공하면 테이블 탐색기에서 TypeORM이 생성한 테이블을 바로 확인할 수 있습니다.

5✅ 셀프 체크 — 면접관처럼 물어보기

Q Docker 이미지와 컨테이너의 차이를 설명해 보세요.

이미지는 실행 가능한 청사진 — 파일 시스템·환경 변수·실행 명령이 담긴 읽기 전용 패키지입니다. 컨테이너는 그 이미지를 바탕으로 실제 실행 중인 격리된 프로세스입니다. 같은 이미지에서 여러 컨테이너를 만들 수 있고, 컨테이너를 삭제해도 이미지는 남아 있어 언제든 새 컨테이너를 만들 수 있습니다.

Q docker run postgres:16이 아니라 docker-compose.yml을 쓰는 이유는 무엇인가요?

docker run은 명령줄에 모든 옵션(포트·환경 변수·볼륨)을 직접 넣어야 해서 길어지고 실수하기 쉽습니다. Compose 파일은 같은 설정을 선언적으로 코드로 관리할 수 있어 팀 전체가 동일한 환경을 공유하고, 버전 관리(git)도 됩니다. 또한 여러 서비스(DB + 앱 + Redis)를 동시에 관리하기 쉽고, docker compose up 한 줄로 전체를 시작할 수 있습니다.

Q ports: "5432:5432"에서 앞의 숫자와 뒤의 숫자는 각각 무엇을 의미하나요?

형식은 호스트포트:컨테이너포트입니다. 뒤의 5432는 컨테이너 내부에서 PostgreSQL이 수신하는 포트이고, 앞의 5432는 호스트(내 PC)에서 그 컨테이너 포트로 포워딩되는 포트입니다. 앞을 5433으로 바꾸면 localhost:5433으로 접속할 수 있습니다. 이렇게 컨테이너 내부 포트와 호스트 포트를 분리하면 여러 DB 컨테이너를 충돌 없이 띄울 수 있습니다.
