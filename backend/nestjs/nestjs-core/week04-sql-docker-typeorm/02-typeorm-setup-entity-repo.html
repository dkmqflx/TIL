TypeORM 설정
Entity · Repository

TypeOrmModule.forRoot()로 DB에 연결하고, @Entity + @Column으로 테이블을 클래스로 정의하고, @InjectRepository()로 Repository를 주입받는다.

"

TypeORM의 역할은 객체 ↔ 테이블 번역이다. Entity 클래스 = 표 양식, Repository = 창구 직원 — 이 두 비유만 머릿속에 있으면 데코레이터 문법이 자연스럽게 읽힌다.

  • TypeOrmModule.forRoot()에 넘기는 연결 옵션을 설명할 수 있다
  • @Entity, @PrimaryGeneratedColumn, @Column이 하는 일을 안다
  • TypeOrmModule.forFeature([Post])가 필요한 이유를 설명할 수 있다
  • @InjectRepository(Post)로 Repository를 주입받는 패턴을 쓸 수 있다

1TypeOrmModule.forRoot() — DB 연결 설정

NestJS에서 TypeORM을 쓰려면 루트 모듈(AppModule)에 TypeOrmModule.forRoot()를 등록합니다. 여기에 DB 종류·호스트·포트·인증 정보·동기화 옵션을 넘깁니다.

app.module.ts — TypeORM 연결 설정
import { Module } from '@nestjs/common';
import { TypeOrmModule } from '@nestjs/typeorm';
import { PostsModule } from './posts/posts.module';
import { Post } from './posts/post.entity';

@Module({
  imports: [
    TypeOrmModule.forRoot({
      type:       'postgres',
      host:       'localhost',
      port:       5432,
      username:   'admin',
      password:   'secret',
      database:   'nestjs_db',
      entities:   [Post],        // 등록할 Entity 목록
      synchronize: true,         // 개발 환경에서만! 스키마 자동 동기화
    }),
    PostsModule,
  ],
})
export class AppModule {}
⚠️ synchronize: true는 개발 환경 전용입니다

이 옵션이 켜지면 앱 시작 시 Entity 정의에 맞게 DB 스키마가 자동으로 변경됩니다. 편리하지만 운영 DB에서 사용하면 데이터 손실이 발생할 수 있습니다. 프로덕션에서는 반드시 false로 두고 마이그레이션을 사용하세요.

2@Entity + @Column — 클래스로 테이블 정의

TypeORM에서 DB 테이블은 TypeScript 클래스로 표현됩니다. @Entity()를 붙인 클래스가 하나의 테이블이 되고, 프로퍼티에 @Column()을 붙이면 그게 열(Column)이 됩니다.

🍔 비유 — Entity는 표 양식, ORM은 표↔객체 번역가

Entity 클래스는 도서관의 도서 카드 양식처럼, "이 테이블에는 어떤 항목(컬럼)이 있고 각 항목의 타입은 무엇인가"를 선언합니다. ORM(TypeORM)은 이 양식을 보고 SQL CREATE TABLE을 만들고, 나중에 DB에서 가져온 행(Row)을 TypeScript 객체로, 또는 반대로 TypeScript 객체를 SQL로 변환해 주는 번역가입니다. 번역가 덕분에 개발자는 SQL 대신 객체만 다루면 됩니다.

post.entity.ts
import { Entity, PrimaryGeneratedColumn, Column } from 'typeorm';

@Entity()
export class Post {
  @PrimaryGeneratedColumn()
  id: number;       // SERIAL PRIMARY KEY — 자동 증가 PK

  @Column()
  title: string;    // NOT NULL TEXT

  @Column({ nullable: true })
  content: string;  // NULL 허용 TEXT
}

3forFeature + @InjectRepository — Repository 주입

Repository는 특정 Entity에 대한 DB 작업 창구입니다. NestJS에서 이를 쓰려면 두 단계가 필요합니다. ① 해당 모듈에 TypeOrmModule.forFeature([Post])를 등록해 Repository를 DI 컨테이너에 제공하고, ② Service 생성자에 @InjectRepository(Post)를 붙여 주입받습니다.

posts.module.ts — Repository 등록
import { Module } from '@nestjs/common';
import { TypeOrmModule } from '@nestjs/typeorm';
import { Post } from './post.entity';
import { PostsController } from './posts.controller';
import { PostsService } from './posts.service';

@Module({
  imports: [
    TypeOrmModule.forFeature([Post]),  // Post Repository를 이 모듈에 제공
  ],
  controllers: [PostsController],
  providers:   [PostsService],
})
export class PostsModule {}
posts.service.ts — Repository 주입받기
import { Injectable } from '@nestjs/common';
import { InjectRepository } from '@nestjs/typeorm';
import { Repository } from 'typeorm';
import { Post } from './post.entity';

@Injectable()
export class PostsService {
  constructor(
    @InjectRepository(Post)
    private postsRepo: Repository<Post>,
  ) {}

  // 이제 this.postsRepo.find() 등으로 DB 접근 가능
}
💡 forRoot vs forFeature

TypeOrmModule.forRoot()는 앱 전체에 DB 연결을 한 번만 설정합니다(AppModule). TypeOrmModule.forFeature([Post])는 특정 모듈에서 Post Repository를 쓸 수 있도록 제공합니다. forFeature 없이 @InjectRepository를 쓰면 NestJS가 Repository를 찾지 못해 에러가 납니다.

4✅ 셀프 체크 — 면접관처럼 물어보기

Q @Entity()를 붙인 클래스는 어떤 역할을 하나요?

@Entity()는 TypeORM에게 "이 TypeScript 클래스는 DB 테이블과 매핑된다"고 알리는 데코레이터입니다. 클래스 이름(또는 전달한 문자열)이 테이블 이름이 되고, @Column()을 붙인 프로퍼티들이 열(Column)이 됩니다. synchronize: true일 때 앱 시작 시 이 정의대로 테이블이 자동 생성·수정됩니다.

Q Repository는 어떻게 주입받나요? @InjectRepository가 필요한 이유는?

Repository를 주입받으려면 두 단계가 필요합니다. ① 해당 기능 모듈에 TypeOrmModule.forFeature([Post])를 등록해 NestJS DI 컨테이너가 Post Repository를 제공할 수 있도록 합니다. ② Service 생성자 파라미터에 @InjectRepository(Post)를 붙입니다. @InjectRepository가 필요한 이유는 Repository가 일반 클래스가 아니라 TypeORM이 동적으로 생성하는 특수 토큰이기 때문입니다 — 타입 정보만으로는 NestJS가 어떤 Repository를 주입해야 할지 알 수 없어서, 데코레이터로 "Post 테이블의 Repository를 달라"고 명시합니다.

Q synchronize: true를 프로덕션에서 쓰면 안 되는 이유는?

이 옵션은 앱 시작 시 Entity 정의를 기준으로 DB 스키마를 자동으로 변경합니다. 개발 중에는 편리하지만, 프로덕션 DB에서는 컬럼 삭제·타입 변경이 자동으로 일어날 수 있어 기존 데이터 손실의 위험이 있습니다. 프로덕션에서는 synchronize: false로 설정하고, TypeORM Migration을 사용해 변경 사항을 명시적으로 관리해야 합니다.
