Repository CRUD
Find / Create / Save / Delete

TypeORM Repository의 핵심 메서드를 통해 CRUD를 구현한다. 특히 create()와 save()의 역할 분리를 정확히 이해하는 것이 중요하다.

"

create()는 메모리에 인스턴스를 만들 뿐이고, 실제 DB에 저장하는 건 save()다. 이 둘을 구분하는 것이 TypeORM을 처음 쓸 때 가장 자주 실수하는 지점이다.

  • find()와 findOne()의 차이 및 where 옵션 사용법을 안다
  • create()가 메모리 인스턴스를 만들고, save()가 DB에 반영함을 설명할 수 있다
  • save()로 기존 레코드를 업데이트하는 방법을 안다
  • delete()와 remove()의 차이를 설명할 수 있다

1find / findOne — 조회

find()는 조건에 맞는 모든 행을 배열로 반환합니다. findOne()은 조건에 맞는 첫 번째 행 하나(또는 없으면 null)를 반환합니다. 둘 다 { where: ... } 옵션으로 필터링합니다.

posts.service.ts — find / findOne
// 전체 조회 — 배열 반환
async findAll(): Promise<Post[]> {
  return this.postsRepo.find();
}

// 단건 조회 — where 옵션으로 조건 지정
async findOne(id: number): Promise<Post> {
  return this.postsRepo.findOne({ where: { id } });
}

// 복합 조건 — title로 조회
async findByTitle(title: string): Promise<Post[]> {
  return this.postsRepo.find({ where: { title } });
}

2create + save — 생성

create()는 Entity 인스턴스를 메모리에만 만듭니다. 이 시점에는 DB에 아무 일도 일어나지 않습니다. save()를 호출해야 비로소 SQL INSERT가 실행되어 DB에 저장됩니다.

🍔 비유 — 주문서 작성(create)과 주방 전달(save)

create()는 손님이 주문서를 작성하는 것입니다 — 아직 카운터에 내지 않은 상태라 주방(DB)은 아무것도 모릅니다. save()는 그 주문서를 카운터에 제출하는 것 — 이제 주방이 요리(INSERT/UPDATE)를 시작합니다. 주문서만 쓰고 내지 않으면 음식이 나올 리 없습니다.

⚠️ create()는 메모리 인스턴스만 만들고, 실제 DB 반영은 save()가 한다

const post = this.postsRepo.create({ title: '새 글' })만 실행하면 DB에는 아무것도 저장되지 않습니다. 반드시 await this.postsRepo.save(post)를 호출해야 INSERT가 실행됩니다. 이 순서를 빠뜨리면 "저장했는데 DB에 없는" 버그가 생깁니다.

posts.service.ts — create + save
async create(title: string, content: string): Promise<Post> {
  // 1단계: 메모리에 Post 인스턴스 생성 (DB 미반영)
  const post = this.postsRepo.create({ title, content });

  // 2단계: DB에 INSERT 실행 → id 포함한 Post 반환
  return this.postsRepo.save(post);
}

3save로 업데이트

save()는 INSERT와 UPDATE 둘 다 담당합니다. 전달한 객체에 id가 있으면 해당 행을 UPDATE하고, 없으면 INSERT합니다.

posts.service.ts — save로 업데이트
async update(id: number, title: string): Promise<Post> {
  // 1. 기존 행 조회
  const post = await this.postsRepo.findOne({ where: { id } });

  // 2. 메모리에서 값 변경
  post.title = title;

  // 3. save() — id가 있으므로 UPDATE 실행
  return this.postsRepo.save(post);
}

4delete / remove — 삭제

delete()는 조건(id 또는 where 객체)을 받아 SQL DELETE를 실행합니다. 결과 객체를 반환할 뿐, Entity 인스턴스를 필요로 하지 않습니다. remove()는 Entity 인스턴스를 받아 삭제하고, 삭제된 인스턴스를 반환합니다.

Repository 메서드SQL특징
find(options?)SELECT *배열 반환
findOne(options)SELECT * LIMIT 1단건 또는 null
create(dto)(없음)메모리 인스턴스만 생성
save(entity)INSERT 또는 UPDATEid 없으면 INSERT, 있으면 UPDATE
delete(id/where)DELETE조건만 주면 됨. 인스턴스 불필요
remove(entity)DELETE인스턴스 필요. 삭제된 entity 반환
posts.service.ts — delete
// delete: id만 알면 됨 (인스턴스 불필요)
async remove(id: number): Promise<void> {
  await this.postsRepo.delete(id);
}
🧭 실무에서는 delete()를 더 많이 씁니다

remove()는 먼저 findOne()으로 인스턴스를 가져와야 합니다 — DB 조회가 한 번 더 발생합니다. delete(id)는 id만 있으면 바로 DELETE를 실행하므로 더 효율적입니다. 단, Cascade 삭제(관계 Entity 함께 삭제)가 필요하면 remove()를 써야 합니다.

5✅ 셀프 체크 — 면접관처럼 물어보기

Q create()와 save()의 차이를 설명해 보세요.

create()는 주어진 데이터로 메모리에 Entity 인스턴스만 생성합니다. 이 시점에는 DB에 아무런 SQL도 실행되지 않습니다. save()를 호출해야 비로소 TypeORM이 INSERT(새 레코드) 또는 UPDATE(기존 레코드)를 실행해 DB에 실제로 반영합니다. create()만 부르고 save()를 빠뜨리면 데이터가 저장되지 않는 버그가 생깁니다.

Q findOne({ where: { id } })에서 where 옵션은 어떻게 활용하나요?

where 옵션은 SQL의 WHERE 절에 해당합니다. 객체의 키-값 쌍으로 조건을 지정합니다. 예를 들어 { where: { id: 1 } }는 WHERE id = 1, { where: { title: '제목', id: 5 } }는 WHERE title = '제목' AND id = 5입니다. TypeORM의 In, Like, MoreThan 같은 연산자를 함께 쓰면 더 복잡한 조건도 표현할 수 있습니다.

Q save()는 언제 INSERT를 하고 언제 UPDATE를 하나요?

save()에 전달하는 Entity 인스턴스에 PK(id)가 없거나 DB에 존재하지 않으면 INSERT, 이미 DB에 존재하는 id가 있으면 UPDATE를 실행합니다. 내부적으로 TypeORM은 PK 값을 기준으로 SELECT를 먼저 실행해 존재 여부를 확인합니다. 그래서 update() 시 먼저 findOne()으로 가져와 값을 수정한 후 save()를 호출하는 패턴이 일반적입니다.
