Column 옵션
Enum · Embedding

@Column()에 숨겨진 옵션들 — type, nullable, default, unique, length — 을 완전히 정리하고, Enum 컬럼과 Entity Embedding으로 공통 필드를 묶는 법을 배운다.

"

@Column()은 단순한 표시가 아니다 — 옵션을 얼마나 정확히 선언하느냐가 DB 스키마 품질을 결정한다. 빈 @Column()과 @Column({ type: 'varchar', length: 100, nullable: false, default: '' })은 완전히 다른 테이블을 만든다.

  • @Column()의 주요 옵션(type, nullable, default, unique, length, select, update)을 설명할 수 있다
  • Enum 컬럼을 만들고, TypeScript enum과 연결하는 방법을 안다
  • Entity Embedding으로 여러 Entity에 걸친 공통 필드를 하나의 클래스로 묶을 수 있다

1@Column 옵션 완전 정리

@Column()에 옵션 객체를 전달하면 해당 컬럼의 DB 타입, 제약 조건, 기본값을 세밀하게 제어할 수 있습니다. 옵션을 생략하면 TypeORM이 TypeScript 타입에서 추론하지만, 명시적으로 선언하는 것이 더 안전합니다.

🍔 비유 — 입력 칸의 규칙

온라인 가입 폼을 생각해 보세요. 각 입력 칸에는 규칙이 있습니다 — "이 칸은 필수입니다(nullable: false)", "이 칸은 비워두면 '게스트'가 자동으로 들어갑니다(default: 'guest')", "이 칸은 중복 이메일을 허용하지 않습니다(unique: true)", "이 칸은 최대 50자까지만 됩니다(length: 50)". @Column()의 옵션이 바로 DB 수준에서 그 칸의 규칙을 정의하는 것입니다.

옵션타입설명예시
typestringDB 컬럼 타입 명시'varchar', 'int', 'text', 'boolean'
nullablebooleanNULL 허용 여부 (기본 false)nullable: true
defaultanyDB 수준 기본값default: 0, default: 'guest'
uniquebooleanUNIQUE 제약 추가unique: true
lengthnumbervarchar 최대 길이length: 100
selectbooleanfalse면 기본 조회에서 제외select: false
updatebooleanfalse면 최초 INSERT 이후 수정 불가update: false
namestringDB 컬럼 이름 직접 지정name: 'created_at'
commentstringDB 컬럼 주석comment: '이메일 주소'
user.entity.ts — @Column 옵션 예시
import { Entity, PrimaryGeneratedColumn, Column } from 'typeorm';

@Entity()
export class User {
  @PrimaryGeneratedColumn()
  id: number;

  // 최대 100자 varchar, NULL 불가, UNIQUE
  @Column({ type: 'varchar', length: 100, nullable: false, unique: true })
  email: string;

  // 기본 SELECT에서 제외 (비밀번호 해시 등 민감 정보)
  @Column({ select: false })
  password: string;

  // 기본값 0, 한 번 저장되면 update 불가
  @Column({ type: 'int', default: 0, update: false })
  loginCount: number;

  // NULL 허용, DB 이름은 'nick_name'으로
  @Column({ type: 'varchar', length: 50, nullable: true, name: 'nick_name' })
  nickname: string | null;
}
⚠️ select: false와 update: false의 동작을 정확히 이해하세요

select: false인 컬럼은 find() 등 일반 조회에서 자동으로 빠집니다. 반드시 필요할 때는 QueryBuilder에서 .addSelect('user.password')를 명시해야 합니다. update: false는 TypeORM ORM 레이어에서만 보호합니다 — 직접 SQL을 실행하거나 QueryBuilder를 쓰면 여전히 변경이 가능합니다. "DB 수준 잠금"이 아닌 "ORM 수준 가이드"임을 기억하세요.

2Enum 컬럼 — @Column({ type: 'enum', enum: ... })

Enum 컬럼은 컬럼에 들어올 수 있는 값을 미리 제한합니다. TypeScript의 enum과 연결하면 타입 안전성까지 확보할 수 있습니다. PostgreSQL에서는 실제로 ENUM 타입이 생성됩니다.

post.entity.ts — Enum 컬럼
import { Entity, PrimaryGeneratedColumn, Column } from 'typeorm';

// TypeScript enum 정의
export enum PostStatus {
  DRAFT     = 'DRAFT',
  PUBLISHED = 'PUBLISHED',
  ARCHIVED  = 'ARCHIVED',
}

@Entity()
export class Post {
  @PrimaryGeneratedColumn()
  id: number;

  @Column()
  title: string;

  // type: 'enum' + enum: 열거형 클래스 지정
  @Column({
    type: 'enum',
    enum: PostStatus,
    default: PostStatus.DRAFT,
  })
  status: PostStatus;
}
💡 Enum 컬럼이 유용한 이유

상태값처럼 "정해진 값 중 하나만 허용"되는 필드에 Enum을 쓰면 세 가지 이점이 있습니다. ① DB 수준에서 유효하지 않은 값이 들어오는 것을 막아줍니다. ② TypeScript 타입으로 자동완성과 오류 감지가 됩니다. ③ 코드 전체에서 'PUBLISHED' 같은 문자열 리터럴 대신 PostStatus.PUBLISHED를 써서 오타를 방지합니다. 단, PostgreSQL Enum은 이미 존재하는 값을 삭제하거나 이름을 바꾸면 마이그레이션이 복잡해질 수 있으므로 주의하세요.

3Entity Embedding — 공통 필드를 묶어 재사용

여러 Entity에 똑같은 컬럼 묶음(예: address 관련 필드)이 반복된다면, Embedded Entity로 공통 필드를 하나의 클래스로 추출하고 @Column(() => Embedded)로 삽입할 수 있습니다. DB에서는 같은 테이블에 일반 컬럼으로 합쳐집니다.

address.embedded.ts + user.entity.ts — Embedding 예시
// --- 공통 필드를 담은 임베딩 클래스 ---
import { Column } from 'typeorm';

export class Address {
  @Column({ type: 'varchar', length: 100, nullable: true })
  city: string;

  @Column({ type: 'varchar', length: 200, nullable: true })
  street: string;
}

// --- Address를 포함하는 Entity ---
import { Entity, PrimaryGeneratedColumn, Column } from 'typeorm';
import { Address } from './address.embedded';

@Entity()
export class User {
  @PrimaryGeneratedColumn()
  id: number;

  @Column()
  name: string;

  // () => Address: 이 프로퍼티 아래로 Address의 컬럼들이 펼쳐짐
  @Column(() => Address)
  address: Address;
}
// 결과 DB 컬럼: id, name, address_city, address_street
🧭 Embedding vs 상속의 차이

Entity Embedding은 공통 필드를 같은 테이블에 펼쳐 합치는 방식입니다. 상속(다음 레슨)은 기본 클래스를 extends해서 코드를 재사용합니다. Embedding은 "주소"처럼 논리적 묶음이 있는 필드에, 상속은 createdAt/updatedAt처럼 모든 Entity에 공통으로 들어가는 메타 필드에 어울립니다.

4✅ 셀프 체크 — 면접관처럼 물어보기

Q nullable: false와 default: ''를 동시에 주면 어떻게 되나요?

nullable: false는 컬럼이 NULL을 허용하지 않는다는 뜻입니다. default: ''는 값을 주지 않을 때 빈 문자열을 기본값으로 씁니다. 둘을 함께 쓰면, 값을 명시하지 않아도 빈 문자열이 자동으로 채워져 NOT NULL 제약을 지킵니다. 반면 nullable: true에 default가 없으면 값을 생략할 경우 NULL이 저장됩니다. "필수 필드이지만 빈 값은 허용"이라면 nullable: false + default: '' 조합이 적합합니다.

Q Enum 컬럼을 쓸 때의 장점과 주의점을 설명하세요.

장점: ① DB에서 허용 값을 제한해 데이터 무결성을 보장합니다. ② TypeScript enum 타입으로 자동완성과 컴파일 오류 감지가 됩니다. ③ 'PUBLISHED' 같은 문자열 리터럴 오타를 막습니다. 주의점: PostgreSQL ENUM은 값을 이미 생성한 후 삭제하거나 이름을 바꾸는 마이그레이션이 복잡합니다. 가능하면 Enum 값은 추가만 하고 수정·삭제는 신중하게 해야 합니다. 또한 synchronize: true로 자동 동기화할 때 Enum 변경이 예상대로 반영되지 않는 경우가 있으니 마이그레이션으로 직접 관리하는 것이 안전합니다.

Q select: false인 컬럼을 조회하는 방법은?

select: false로 선언된 컬럼은 find()와 같은 일반 조회에서 결과에 포함되지 않습니다. 그 값이 꼭 필요할 때는 QueryBuilder를 사용해서 .select(['user.id', 'user.email']).addSelect('user.password')처럼 해당 컬럼을 명시적으로 추가해야 합니다. 비밀번호 해시 같은 민감한 필드를 기본 조회에서 자동으로 제외시키는 데 주로 활용합니다.
