Table Inheritance
BaseModel 상속

@CreateDateColumn · @UpdateDateColumn으로 타임스탬프를 자동화하고, 추상 BaseModel을 만들어 모든 Entity에 공통 필드를 상속한다. TypeORM 상속 전략의 종류도 간단히 살펴본다.

"

실무 코드베이스에서 createdAt과 updatedAt을 직접 채워 넣는 Entity는 없다. BaseModel을 한 번 만들어 두면 모든 Entity가 공통 필드를 자동으로 물려받는다 — 코드 중복 없이, 자동으로.

  • @CreateDateColumn과 @UpdateDateColumn이 언제 어떻게 값을 채우는지 설명할 수 있다
  • 추상 BaseModel 클래스를 만들고, Entity가 이를 상속하는 패턴을 구현할 수 있다
  • TypeORM의 단일 테이블 상속(STI)과 구체 테이블 상속(CTI)의 차이를 설명할 수 있다

1@CreateDateColumn / @UpdateDateColumn — 자동 타임스탬프

TypeORM이 제공하는 두 특수 데코레이터는 @Column()을 쓰지 않고도 날짜/시간 컬럼을 자동 관리합니다. 직접 값을 채워 넣을 필요가 없습니다.

post.entity.ts — 자동 타임스탬프
import {
  Entity, PrimaryGeneratedColumn, Column,
  CreateDateColumn, UpdateDateColumn,
} from 'typeorm';

@Entity()
export class Post {
  @PrimaryGeneratedColumn()
  id: number;

  @Column()
  title: string;

  // INSERT 시 자동으로 현재 시각 기록, 이후 수정 불가
  @CreateDateColumn()
  createdAt: Date;

  // INSERT 및 UPDATE 시마다 현재 시각으로 자동 갱신
  @UpdateDateColumn()
  updatedAt: Date;
}
데코레이터값이 채워지는 시점이후 수정
@CreateDateColumn()INSERT (최초 저장)TypeORM이 자동으로 변경하지 않음
@UpdateDateColumn()INSERT + save() 호출마다매번 현재 시각으로 갱신
💡 직접 값을 넣지 않아도 됩니다

두 데코레이터는 TypeORM이 내부적으로 관리합니다. post.createdAt = new Date()처럼 직접 할당하거나, DTO에서 createdAt 필드를 받을 필요가 없습니다. save()를 호출하면 알아서 채워집니다.

2추상 BaseModel — 공통 필드를 상속으로 재사용

대부분의 Entity는 id, createdAt, updatedAt을 공통으로 가집니다. 이 세 필드를 추상 클래스 BaseModel에 한 번만 선언하고, 나머지 Entity들이 extends로 상속받으면 코드 중복을 없앨 수 있습니다.

🍔 비유 — 서류의 공통 머리말

회사에서 작성하는 모든 서류(계약서, 보고서, 결재문서)에는 공통으로 들어가는 항목이 있습니다 — 문서번호(id), 작성일시(createdAt), 최종수정일시(updatedAt). 이 항목을 매번 각 서류 양식에 새로 그리는 대신, "표준 머리말 양식"(BaseModel)을 한 번 만들어 두고 모든 서류가 그 양식을 물려받아(extends) 쓰는 것과 같습니다.

base.entity.ts — 추상 BaseModel
import {
  PrimaryGeneratedColumn,
  CreateDateColumn,
  UpdateDateColumn,
} from 'typeorm';

// @Entity() 없음 — 테이블을 직접 만들지 않는 추상 클래스
export abstract class BaseModel {
  @PrimaryGeneratedColumn()
  id: number;

  @CreateDateColumn()
  createdAt: Date;

  @UpdateDateColumn()
  updatedAt: Date;
}
post.entity.ts — BaseModel 상속
import { Entity, Column } from 'typeorm';
import { BaseModel } from './base.entity';

@Entity()
export class Post extends BaseModel {
  // id, createdAt, updatedAt은 BaseModel에서 상속
  @Column({ type: 'varchar', length: 200 })
  title: string;

  @Column({ type: 'text', nullable: true })
  content: string | null;
}

// User도 마찬가지로 extends BaseModel 하면 동일한 id/createdAt/updatedAt을 가짐
⚠️ BaseModel에는 @Entity()를 붙이지 않습니다

abstract class BaseModel에 @Entity()를 붙이면 TypeORM이 base_model이라는 실제 테이블을 만들려고 합니다. 추상 기반 클래스에는 @Entity()가 없어야 합니다 — 테이블을 만드는 건 상속받은 구체 Entity(User, Post)의 역할입니다.

3상속 전략 — 단일 테이블 vs 구체 테이블

TypeORM은 Entity 상속에 두 가지 전략을 지원합니다. 위에서 본 BaseModel 패턴은 TypeScript 추상 클래스 상속이지 TypeORM의 공식 상속 전략이 아닙니다. TypeORM 공식 상속 전략은 아래 두 가지입니다.

전략데코레이터DB 구조특징
단일 테이블 상속 (STI) @TableInheritance({ column: 'type' }) 하나의 테이블에 모든 서브클래스 컬럼 + 구분자 컬럼 JOIN 없음. 컬럼이 많아질수록 NULL이 늘어남
구체 테이블 상속 (CTI) @Entity() on each child class 자식마다 별도 테이블. 공통 컬럼 반복 테이블이 분리. 공통 쿼리가 번거로움
🧭 실무에서는 BaseModel 추상 클래스 패턴이 가장 흔합니다

공식 상속 전략(STI/CTI)보다는, abstract class BaseModel을 만들고 각 Entity가 TypeScript extends로 상속하는 패턴이 실무에서 가장 널리 쓰입니다. TypeORM이 각 Entity를 독립된 테이블로 생성하면서도, 공통 필드 선언은 BaseModel 한 곳에만 있어 코드 중복이 없습니다.

복습

4✅ 셀프 체크 — 면접관처럼 물어보기

Q BaseModel을 상속받는 이유는 무엇인가요?

거의 모든 Entity가 가져야 하는 id, createdAt, updatedAt을 한 번만 선언하고 여러 Entity에 재사용하기 위해서입니다. BaseModel에 선언해 두면, 새로운 Entity를 만들 때 extends BaseModel만 하면 세 필드를 자동으로 갖게 됩니다. 타임스탬프 컬럼 추가/수정이 필요할 때도 BaseModel 한 곳만 고치면 됩니다. 코드 중복을 줄이고 일관성을 유지하는 것이 핵심 이유입니다.

Q createdAt을 자동으로 기록하는 방법은?

@CreateDateColumn() 데코레이터를 사용합니다. 이 데코레이터가 붙은 컬럼은 save()로 Entity를 최초 INSERT할 때 TypeORM이 자동으로 현재 시각을 채워 넣습니다. 개발자가 post.createdAt = new Date()처럼 직접 값을 지정할 필요가 없습니다. 이후 save()를 다시 호출해도(UPDATE) createdAt은 변경되지 않습니다.

Q @UpdateDateColumn과 @CreateDateColumn의 차이는?

@CreateDateColumn()은 INSERT 시 딱 한 번 현재 시각을 기록하고 이후에는 TypeORM이 변경하지 않습니다. @UpdateDateColumn()은 INSERT 시 기록하고, 이후 save()를 통해 해당 Entity가 수정될 때마다 현재 시각으로 덮어씁니다. "언제 만들어졌나"는 createdAt, "마지막으로 언제 수정됐나"는 updatedAt으로 구분됩니다.
