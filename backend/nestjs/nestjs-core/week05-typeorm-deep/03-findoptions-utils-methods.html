FindManyOptions
유틸 · 메서드

find()에 전달하는 FindManyOptions(where, select, order, relations, take/skip)를 완전 정리하고, TypeORM 연산자 유틸리티와 count/exists/findAndCount 메서드를 익힌다.

"

find({ where: { id } })가 전부인 줄 알았다면 — 아직 절반도 못 쓴 것이다. where 안에 연산자(MoreThan, Like, In…)를 조합하면 복잡한 SQL 조건도 TypeScript 객체 하나로 표현된다.

  • FindManyOptions의 핵심 키(where, select, order, relations, take, skip)를 설명할 수 있다
  • MoreThan, Like, ILike, In, Between, IsNull 등 TypeORM 연산자를 상황에 맞게 쓸 수 있다
  • count, exists, findAndCount의 차이와 용도를 안다

1FindManyOptions 핵심 키

repository.find(options)에 전달하는 객체가 FindManyOptions입니다. findOne()에는 FindOneOptions(같은 구조에서 take/skip만 없음)를 씁니다.

🍔 비유 — 도서관 검색 조건

도서관에서 책을 찾을 때 사서에게 조건을 전달합니다. "2023년 이후에 출간된(where: { publishedAt: MoreThan(...) }) 'JavaScript' 관련 책 중에서(where: { title: ILike('%JavaScript%') }), 제목과 저자만 알려주고(select), 최신순으로(order), 10권씩 2페이지를 주세요(take: 10, skip: 10)." FindManyOptions는 이 검색 조건 전체를 하나의 객체로 묶은 것입니다.

posts.service.ts — FindManyOptions 핵심 키
import { Repository } from 'typeorm';

const posts = await postsRepo.find({
  // 조회할 컬럼만 선택 (SQL: SELECT id, title FROM ...)
  select: { id: true, title: true },

  // 필터 조건 (SQL: WHERE ... )
  where: { isPublished: true },

  // 연관 Entity를 함께 불러오기 (SQL: JOIN)
  relations: { author: true, comments: true },

  // 정렬 (SQL: ORDER BY createdAt DESC)
  order: { createdAt: 'DESC' },

  // 페이지네이션: 최대 10개, 앞 20개 건너뜀
  take: 10,   // SQL: LIMIT 10
  skip: 20,   // SQL: OFFSET 20
});
키SQL 대응설명
whereWHERE필터 조건. 객체 또는 연산자 함수 사용
selectSELECT 컬럼...가져올 컬럼만 지정
relationsJOIN연결된 Entity를 함께 조회
orderORDER BY정렬 기준과 방향('ASC'/'DESC')
takeLIMIT최대 조회 개수 (페이지 크기)
skipOFFSET건너뛸 행 수 (페이지 오프셋)

2TypeORM 연산자 유틸리티

where 절에서 단순 동등 비교 이외의 조건이 필요할 때, TypeORM이 제공하는 연산자 함수를 사용합니다. typeorm 패키지에서 직접 임포트합니다.

posts.service.ts — 연산자 활용 예시
import {
  MoreThan, LessThan, Like, ILike,
  In, Between, IsNull,
} from 'typeorm';

// 조회수 100 초과
postsRepo.find({ where: { viewCount: MoreThan(100) } });

// 제목에 'NestJS' 포함 (대소문자 구분, SQL LIKE)
postsRepo.find({ where: { title: Like('%NestJS%') } });

// 제목에 'nestjs' 포함 (대소문자 무시, PostgreSQL ILIKE)
postsRepo.find({ where: { title: ILike('%nestjs%') } });

// 상태가 'DRAFT' 또는 'PUBLISHED' 중 하나
postsRepo.find({ where: { status: In(['DRAFT', 'PUBLISHED']) } });

// 조회수가 10~100 사이
postsRepo.find({ where: { viewCount: Between(10, 100) } });

// 삭제일시가 NULL인 행만 (소프트 삭제 미적용 행)
postsRepo.find({ where: { deletedAt: IsNull() } });
연산자SQL 대응용도
MoreThan(n)> nn보다 큰 값
MoreThanOrEqual(n)>= nn 이상인 값
LessThan(n)< nn보다 작은 값
LessThanOrEqual(n)<= nn 이하인 값
Like(pattern)LIKE pattern패턴 매칭 (대소문자 구분)
ILike(pattern)ILIKE pattern패턴 매칭 (대소문자 무시, PG 전용)
In([...])IN (...)배열 중 하나
Between(a, b)BETWEEN a AND b범위 내
IsNull()IS NULLNULL인 값
Not(value)!= value같지 않은 값 (다른 연산자와 조합 가능)

3자주 쓰는 메서드 — count · exists · findAndCount

find() 외에도 개수 확인과 페이지네이션에 자주 쓰는 Repository 메서드가 있습니다.

posts.service.ts — count / exists / findAndCount
import { ILike } from 'typeorm';

// 1. count — 조건에 맞는 행 개수만 반환 (SQL: SELECT COUNT(*))
const total: number = await postsRepo.count({
  where: { status: 'PUBLISHED' },
});

// 2. exists — 조건에 맞는 행이 하나라도 있으면 true (TypeORM 0.3.x+)
const hasDraft: boolean = await postsRepo.exists({
  where: { status: 'DRAFT', authorId: 5 },
});

// 3. findAndCount — [데이터 배열, 총 개수] 튜플 반환
const [posts, totalCount]: [Post[], number] =
  await postsRepo.findAndCount({
    where: { title: ILike('%NestJS%') },
    order: { createdAt: 'DESC' },
    take: 10,
    skip: 0,
  });

console.log(`총 ${totalCount}건 중 ${posts.length}건 조회`);
💡 findAndCount는 페이지네이션의 핵심

findAndCount(options)는 [데이터 배열, 조건에 맞는 총 개수] 튜플을 한 번의 쿼리로(내부적으로 두 번의 SQL) 반환합니다. 페이지네이션 API 응답에서 "현재 페이지 데이터"와 "전체 몇 건인지"를 동시에 클라이언트에 내려줄 수 있어 매우 유용합니다. 페이지네이션 일반화는 Week 10에서 깊이 다룹니다.

메서드반환 타입SQL주용도
count(options?)numberSELECT COUNT(*)총 개수 확인
exists(options?)booleanSELECT EXISTS(...)존재 여부만 확인
findAndCount(options?)[T[], number]SELECT + COUNT페이지네이션
🧭 relations 옵션 주의 — 필요할 때만 로드하세요

relations: { author: true }처럼 연관 Entity를 로드하면 내부적으로 LEFT JOIN이 실행됩니다. 불필요하게 relations를 로드하면 N+1 문제나 과도한 JOIN으로 성능이 저하될 수 있습니다. 연관 데이터가 실제로 필요한 쿼리에만 relations를 주고, 그렇지 않으면 생략하는 것이 좋습니다. Week 6에서 관계(Relation)를 상세히 다룹니다.

4✅ 셀프 체크 — 면접관처럼 물어보기

Q relations 옵션은 어떤 역할을 하나요?

relations 옵션은 현재 Entity와 연결된 다른 Entity를 함께 불러오는 기능입니다. 내부적으로 SQL LEFT JOIN이 실행됩니다. 예를 들어 find({ relations: { author: true } })를 쓰면 각 Post에 연결된 author(User) 데이터도 함께 조회됩니다. 단, 불필요하게 쓰면 JOIN 비용과 N+1 문제가 생길 수 있어 실제로 연관 데이터가 필요할 때만 명시합니다.

Q findAndCount가 반환하는 것은 무엇인가요? 언제 씁니까?

findAndCount(options)는 [조건에 맞는 데이터 배열, 조건에 맞는 전체 행 수] 형태의 튜플을 반환합니다. 예: [Post[], 42]. 페이지네이션 API를 만들 때 가장 많이 씁니다 — take: 10, skip: 0으로 첫 페이지 10건을 가져오면서 동시에 "총 42건이 있다"는 정보도 함께 얻어 클라이언트에 내려줄 수 있습니다. find() + count()를 따로 두 번 호출하는 것보다 코드가 간결합니다.

Q Like와 ILike의 차이를 설명하세요.

Like('%nest%')는 SQL의 LIKE 연산자에 대응합니다 — 대소문자를 구분합니다. ILike('%nest%')는 PostgreSQL의 ILIKE 연산자에 대응합니다 — 대소문자를 무시합니다. 따라서 ILike('%nest%')는 'NestJS', 'NESTJS', 'nestjs' 모두 매칭합니다. 사용자 검색처럼 대소문자 구분 없이 찾아야 할 때는 ILike를, 정확한 케이스로 매칭해야 할 때는 Like를 씁니다. 단, ILike는 PostgreSQL 전용입니다.
