관계 이론
SQL Relations

Foreign Key로 테이블이 서로를 참조하는 방법, JOIN으로 흩어진 데이터를 합치는 법, 그리고 1:1 · 1:N · N:M 세 가지 관계 모델을 직관으로 이해한다.

"

한 테이블에 모든 것을 몰아넣으면 중복이 폭발한다. 관계형 DB의 핵심 아이디어는 단순하다 — 데이터는 나눠서 저장하고, 필요할 때 JOIN으로 다시 연결한다. FK(Foreign Key)가 그 연결 고리다.

  • Foreign Key(FK)가 무엇이고, 참조 무결성을 어떻게 보장하는지 설명할 수 있다
  • INNER JOIN · LEFT JOIN SQL을 작성하고 결과 행을 예측할 수 있다
  • 1:1 · 1:N · N:M 관계가 각각 어떤 상황에서 쓰이는지 설명할 수 있다
  • N:M 관계에 왜 중간(조인) 테이블이 필요한지 이해한다

1Foreign Key — 테이블 간의 연결 고리

Foreign Key(외래 키, FK)는 한 테이블의 컬럼이 다른 테이블의 PK(기본 키)를 참조하도록 선언하는 제약입니다. 이를 통해 DB가 참조 무결성(Referential Integrity)을 강제합니다 — 존재하지 않는 행을 가리키는 FK를 삽입하려 하면 오류가 발생합니다.

🍔 비유 — 명함첩의 참조

명함첩에는 각 사람의 카드(users 테이블)가 있고, 게시판엔 각 글의 쪽지(posts 테이블)가 있습니다. 쪽지에는 작성자 칸에 "카드 번호"만 적혀 있죠. 실제로 글을 누가 썼는지 알고 싶으면 그 번호로 명함첩을 찾아봅니다. 이 "카드 번호"가 FK입니다. FK가 보장하는 것은 단 하나 — 카드 번호에 해당하는 카드가 반드시 명함첩에 존재한다는 것입니다.

SQL — FK 제약 선언
-- posts 테이블에 author_id 컬럼과 FK 제약 추가
ALTER TABLE posts
  ADD COLUMN author_id INT,
  ADD CONSTRAINT fk_posts_author
    FOREIGN KEY (author_id)
    REFERENCES users(id)
    ON DELETE CASCADE;

-- 존재하지 않는 user(id=999)를 참조하려 하면 오류 발생
INSERT INTO posts (title, author_id) VALUES ('Test', 999);
-- ERROR: insert or update on table "posts" violates foreign key constraint
🧭 FK가 보장하는 것 — 참조 무결성

FK 제약이 선언되면 DB는 두 가지를 강제합니다. ① posts.authorId에 삽입하는 값은 users.id에 반드시 존재해야 합니다. ② ON DELETE CASCADE처럼 연쇄 동작을 선언하면, 참조 대상 행이 삭제될 때 자동으로 처리(연쇄 삭제·NULL 처리)됩니다. 이 보장을 참조 무결성이라고 합니다.

2JOIN — 흩어진 데이터를 연결하기

FK로 나눠 저장한 데이터를 다시 합칠 때 JOIN을 사용합니다. 가장 자주 쓰는 두 가지는 INNER JOIN(양쪽 모두 존재하는 행만)과 LEFT JOIN(왼쪽 행은 모두, 오른쪽은 없으면 NULL)입니다.

SQL — JOIN 예시
-- INNER JOIN: 작성자가 있는 게시글만 (author_id IS NOT NULL)
SELECT
  p.id        AS post_id,
  p.title,
  u.nickname  AS author_name
FROM  posts p
INNER JOIN users u ON p.author_id = u.id;

-- LEFT JOIN: 작성자 없는 게시글도 포함 (author_name = NULL)
SELECT
  p.id        AS post_id,
  p.title,
  u.nickname  AS author_name
FROM  posts p
LEFT JOIN users u ON p.author_id = u.id;

3관계 종류 모델링 — 1:1 · 1:N · N:M

테이블 간 관계는 카디널리티(한쪽이 몇 개와 연결될 수 있는가)에 따라 세 가지로 분류합니다. 상황에 맞는 관계 모델을 선택하는 것이 설계의 핵심입니다.

관계예시FK 위치N:M 특이사항
1:1User ↔ ProfileFK 소유 쪽(보통 두 번째)—
1:NUser → PostN쪽(posts.authorId)—
N:MPost ↔ Tag중간 테이블 양쪽 FK중간(조인) 테이블 필수
💡 N:M에 중간 테이블이 필요한 이유

한 컬럼에 여러 값(배열)을 저장할 수 없는 관계형 DB의 특성상, "Post의 tag_ids = [1, 2, 3]" 같은 형태는 불가능합니다. 대신 조인 테이블을 두어 (postId, tagId) 쌍을 행 하나씩으로 저장합니다. 이렇게 하면 N:M이 두 개의 1:N 관계로 분해되어 FK 제약을 유지할 수 있습니다. TypeORM의 @ManyToMany + @JoinTable이 이 중간 테이블을 자동으로 생성해 줍니다.

복습

4✅ 셀프 체크 — 면접관처럼 물어보기

Q Foreign Key가 보장하는 것은 무엇인가요?

참조 무결성(Referential Integrity)입니다. FK가 선언된 컬럼에 삽입하는 값은 참조 대상 테이블의 해당 컬럼에 반드시 존재해야 합니다. 예를 들어 posts.authorId = 999를 삽입하려 할 때, users.id = 999가 없으면 DB가 오류를 반환합니다. 이로써 "존재하지 않는 사용자가 쓴 게시글" 같은 불일치 데이터가 생기는 것을 방지합니다.

Q N:M 관계에서 왜 중간(조인) 테이블이 필요한가요?

관계형 DB는 한 컬럼에 배열 형태로 여러 외래 키를 저장하는 것을 지원하지 않습니다. posts.tagIds = [1, 2, 3] 같은 형태는 FK 제약을 걸 수 없고 정규화도 되지 않습니다. 대신 중간 테이블에 (postId, tagId) 쌍을 행 하나씩으로 저장합니다. 이렇게 하면 N:M이 사실상 두 개의 1:N 관계로 분해되어 FK 제약을 그대로 유지하면서 연결 정보를 표현할 수 있습니다.

Q INNER JOIN과 LEFT JOIN의 차이는 무엇인가요?

INNER JOIN은 두 테이블 모두에 매칭 행이 있을 때만 결과를 반환합니다. 매칭되지 않는 왼쪽 행은 제외됩니다. LEFT JOIN은 왼쪽 테이블의 모든 행을 보존하되, 오른쪽에 매칭 행이 없으면 오른쪽 컬럼값을 NULL로 채웁니다. TypeORM에서 relations: { author: true } 옵션을 사용하면 내부적으로 LEFT JOIN이 실행됩니다 — 작성자가 없는 게시글도 조회 결과에 포함되기 때문입니다.
