Relation Options
쿼리 · Author 적용

관계 데코레이터에 붙는 옵션(cascade · eager · onDelete)의 의미와 위험 요소를 이해하고, relations 옵션·QueryBuilder로 관계 데이터를 포함 조회하며, Post에 author 관계를 실제 적용한다.

"

관계 옵션은 편의와 위험이 함께 온다. eager: true는 항상 JOIN하고, cascade: true는 연쇄 저장하고, onDelete: 'CASCADE'는 연쇄 삭제한다. 모두 "언제나"가 아니라 "필요할 때만" 켜야 한다.

  • cascade · eager · onDelete 각 옵션이 DB와 ORM에 미치는 효과를 설명할 수 있다
  • repo.find({ relations: { author: true } })와 QueryBuilder leftJoinAndSelect의 차이를 안다
  • Post 엔티티에 author 관계를 선언하고 데이터를 생성·조회하는 코드를 작성할 수 있다
  • eager 남용이 성능 문제로 이어지는 이유를 설명할 수 있다

1관계 옵션 — cascade · eager · onDelete

관계 데코레이터의 세 번째 인수(옵션 객체)에 동작을 제어하는 옵션을 선언합니다. 각 옵션은 독립적으로 설정할 수 있으며, 모두 기본값은 비활성화입니다.

🍔 비유 — 패키지 배송 규칙

주문(Post)을 처리할 때 관련 물건들을 어떻게 다루느냐를 결정하는 규칙입니다. cascade는 "주문을 저장할 때 연관 물건도 함께 저장해라". eager는 "주문을 꺼낼 때 항상 연관 물건도 함께 꺼내라". onDelete: 'CASCADE'는 "주문이 취소되면 연관 배송 정보도 자동으로 삭제해라". 편리하지만 무심코 켜 두면 예상치 못한 삭제나 과도한 조회가 일어납니다.

post.entity.ts — 관계 옵션 예시
import {
  Entity, PrimaryGeneratedColumn, Column,
  CreateDateColumn, UpdateDateColumn,
  ManyToOne, JoinColumn,
} from 'typeorm';
import { User } from './user.entity';

@Entity()
export class Post {
  @PrimaryGeneratedColumn()
  id: number;

  @Column({ type: 'varchar', length: 200 })
  title: string;

  @Column({ type: 'text', nullable: true })
  content: string | null;

  @CreateDateColumn()
  createdAt: Date;

  @UpdateDateColumn()
  updatedAt: Date;

  /**
   * author 관계 옵션:
   * - onDelete: 'CASCADE' → User가 삭제되면 이 Post도 자동 삭제 (DB 레벨)
   * - nullable: true      → author 없는 게시글도 허용
   * - eager: false(기본)  → find() 시 자동 JOIN 안 함 (명시적 relations로만)
   */
  @ManyToOne(() => User, (user) => user.posts, {
    onDelete: 'CASCADE',
    nullable: true,
  })
  @JoinColumn({ name: 'authorId' })
  author: User | null;
}
옵션레벨기본값효과
cascadeORMfalse부모 save() 시 연관 엔티티도 함께 INSERT/UPDATE
eagerORMfalsefind() 호출마다 자동으로 LEFT JOIN (항상)
onDeleteDB FK'NO ACTION'참조 대상 삭제 시 DB가 처리: CASCADE · SET NULL · RESTRICT
nullableDB 컬럼trueFK 컬럼이 NULL을 허용할지 여부
⚠️ eager: true 남용 — 항상 JOIN되어 성능 저하

eager: true를 선언하면 해당 관계는 모든 find() 호출에서 자동으로 LEFT JOIN됩니다. 게시글 목록 100개를 가져오는 API에서도 작성자 정보 100개가 항상 JOIN된다는 뜻입니다. 필요 없는 경우에도 쿼리 비용이 증가하고, 데이터가 많아질수록 성능 저하가 심해집니다. 권장 패턴: eager는 끄고, 관계 데이터가 필요한 쿼리에서만 relations: { author: true }로 명시해 주세요.

2관계 포함 조회 — relations 옵션과 QueryBuilder

eager: false(기본값)일 때 관계 데이터를 포함해 조회하는 방법은 두 가지입니다. ① find({ relations: … })은 간단한 1단계 관계에 적합하고, ② QueryBuilder leftJoinAndSelect는 복잡한 조건이나 중첩 관계에 적합합니다.

posts.service.ts — relations 옵션과 QueryBuilder
import { Injectable } from '@nestjs/common';
import { InjectRepository } from '@nestjs/typeorm';
import { Repository } from 'typeorm';
import { Post } from './post.entity';

@Injectable()
export class PostsService {
  constructor(
    @InjectRepository(Post)
    private readonly postsRepo: Repository<Post>,
  ) {}

  // ① find + relations 옵션 — 단순 관계 조회에 적합
  async findAllWithAuthor(): Promise<Post[]> {
    return this.postsRepo.find({
      relations: { author: true },  // author 포함 LEFT JOIN
      order: { createdAt: 'DESC' },
    });
  }

  // ② QueryBuilder — 복잡한 조건·정렬·중첩 JOIN에 적합
  async findPublishedWithAuthor(): Promise<Post[]> {
    return this.postsRepo
      .createQueryBuilder('post')
      .leftJoinAndSelect('post.author', 'author')  // author JOIN
      .where('post.content IS NOT NULL')
      .orderBy('post.createdAt', 'DESC')
      .getMany();
  }

  // author 포함 단건 조회
  async findOneWithAuthor(id: number): Promise<Post | null> {
    return this.postsRepo.findOne({
      where: { id },
      relations: { author: true },
    });
  }
}

3Post에 author 관계 적용 — 데이터 생성 · 조회

실제 SNS 앱 컨텍스트에서 User와 Post를 연결하고, 게시글을 작성자와 함께 저장하고 조회하는 전체 흐름을 작성합니다.

posts.service.ts — author 관계 포함 생성/조회
import { Injectable, NotFoundException } from '@nestjs/common';
import { InjectRepository } from '@nestjs/typeorm';
import { Repository } from 'typeorm';
import { Post } from './post.entity';
import { User } from '../users/user.entity';

@Injectable()
export class PostsService {
  constructor(
    @InjectRepository(Post)
    private readonly postsRepo: Repository<Post>,
    @InjectRepository(User)
    private readonly usersRepo: Repository<User>,
  ) {}

  // author 관계 포함 게시글 생성
  async createPost(
    authorId: number,
    title: string,
    content: string,
  ): Promise<Post> {
    const author = await this.usersRepo.findOneOrFail({ where: { id: authorId } });

    const post = this.postsRepo.create({ title, content, author });
    return this.postsRepo.save(post);
  }

  // author 포함 전체 조회 (relations 명시)
  async findAll(): Promise<Post[]> {
    return this.postsRepo.find({
      relations: { author: true },
      order: { createdAt: 'DESC' },
    });
  }

  // author 포함 단건 조회
  async findOne(id: number): Promise<Post> {
    const post = await this.postsRepo.findOne({
      where: { id },
      relations: { author: true },
    });
    if (!post) throw new NotFoundException(`Post #${id} not found`);
    return post;
  }
}
🧭 cascade가 위험할 수 있는 이유

cascade: true를 선언하면 부모 엔티티를 save()할 때 연관된 자식 엔티티도 자동으로 INSERT/UPDATE됩니다. 의도치 않은 자식 데이터가 수정되거나, 삭제(cascade: ['remove'])의 경우 관련 데이터 전체가 사라질 수 있습니다. 트랜잭션 범위가 예상보다 넓어져 디버깅도 어려워집니다. cascade는 명확한 의도가 있을 때(예: 폼 전체를 한번에 저장)만 사용하고, 기본값(false)을 유지하는 것이 안전합니다.

💡 eager vs 명시적 relations — 선택 기준

eager: true는 "이 관계는 언제나 필요하다"는 보장이 있을 때만 사용합니다. 예: Profile은 User를 조회할 때 거의 항상 함께 필요한 경우. 반면 Post의 author처럼 일부 API에서만 필요한 관계는 eager: false(기본값)를 유지하고, 필요한 엔드포인트에서만 relations: { author: true }를 명시하세요. 이렇게 하면 불필요한 JOIN을 피해 성능을 최적화할 수 있습니다.

4✅ 셀프 체크 — 면접관처럼 물어보기

Q eager: true와 명시적 relations 옵션의 차이는 무엇인가요?

eager: true는 엔티티 선언부에 한 번 설정하면 해당 관계가 모든 find() 호출에서 자동으로 LEFT JOIN됩니다 — 명시하지 않아도 항상 조인됩니다. 반면 명시적 relations: { author: true }는 그 쿼리 한 번에만 조인이 발생합니다. eager는 편리하지만 항상 JOIN하므로 불필요한 경우 성능 비용이 생깁니다. 권장: eager: false(기본)를 유지하고, 관계 데이터가 필요한 쿼리에서만 relations를 명시하세요.

Q cascade가 위험할 수 있는 이유는 무엇인가요?

cascade: true를 설정하면 부모 엔티티를 save()할 때 연관 자식 엔티티도 자동으로 INSERT/UPDATE됩니다. 문제는 의도치 않은 쓰기가 발생할 수 있다는 것입니다 — 예를 들어 Post를 수정할 때 author 객체가 함께 바뀌어 User 정보까지 덮어쓰일 수 있습니다. cascade: ['remove']가 활성화되어 있으면 부모 삭제 시 자식도 연쇄 삭제됩니다. onDelete: 'CASCADE'가 DB 레벨의 삭제 연쇄라면, cascade는 ORM 레벨의 쓰기 연쇄로, 범위를 예측하기 어려워 디버깅이 복잡해집니다.

Q onDelete: 'CASCADE'와 onDelete: 'SET NULL'은 각각 언제 쓰나요?

onDelete: 'CASCADE'는 참조 대상(예: User)이 삭제될 때 참조하는 행(예: Post)도 함께 삭제됩니다. User를 삭제하면 그 User의 모든 게시글도 사라져야 하는 경우에 적합합니다. onDelete: 'SET NULL'은 참조 대상이 삭제될 때 FK 컬럼을 NULL로 설정합니다 — User가 탈퇴해도 게시글은 남겨두되 작성자 정보만 지우는 경우에 적합합니다. 이때 FK 컬럼은 반드시 nullable: true여야 합니다.
