Session vs JWT
토큰 이론 · 해싱

두 가지 인증 방식의 트레이드오프를 비교하고, Access / Refresh Token의 역할과 수명을 이해한다. 비밀번호는 왜 반드시 해시로 저장해야 하는지, bcrypt가 어떻게 salt를 다루는지 익힌다.

"

인증의 핵심은 "서버가 사용자를 기억하는 방법"이다. 세션은 서버 메모리에 기록하고, JWT는 서명된 토큰을 클라이언트에 넘겨 서버가 아무것도 저장하지 않는다. 비밀번호는 절대 평문으로 저장하지 않고, bcrypt 해시로만 보관한다.

  • 세션과 JWT의 차이를 "서버 저장 여부"와 "확장성"으로 설명할 수 있다
  • Access Token과 Refresh Token의 역할·수명을 구분하여 설명할 수 있다
  • bcrypt가 salt를 해시 안에 내장한다는 것을 이해하고, bcrypt.hash/compare 흐름을 설명할 수 있다
  • 비밀번호를 평문(또는 단순 MD5)으로 저장하면 안 되는 이유를 구체적으로 설명할 수 있다

1세션 vs JWT — 서버 저장 여부가 갈린다

세션(Session)은 로그인 성공 시 서버가 세션 저장소(메모리·Redis)에 사용자 정보를 기록하고, 클라이언트에게는 session ID만 쿠키로 전달합니다. 반면 JWT(JSON Web Token)는 클라이언트에게 서명된 토큰을 발급하고, 서버는 아무것도 저장하지 않습니다. 이 차이가 확장성과 구현 복잡도 전반에 영향을 줍니다.

🍔 비유 — 입장 팔찌

페스티벌 입장을 생각해 보세요. 세션 방식은 입장 시 직원이 명단에 이름을 적고 번호표를 줍니다 — 재입장할 때마다 직원이 명단을 보고 확인합니다. JWT 방식은 직원이 처음 한 번 확인 후 위조 불가능한 팔찌(서명된 토큰)를 채워 줍니다. 이후 재입장할 때는 팔찌를 보여주기만 하면 됩니다 — 명단을 뒤지거나 직원에게 물어볼 필요가 없습니다. 서버가 "명단(세션 저장소)"을 유지하지 않는 것이 JWT가 stateless(무상태)라고 불리는 이유입니다.

서버가 세션 저장소를 유지하지 않는다는 뜻입니다. 대신 토큰 안에 클레임(sub, email, type 등)이 내장되어 있고, 서버는 비밀키로 서명을 검증하는 것만으로 토큰의 진위와 만료 여부를 판단합니다. 수평 확장(서버 인스턴스 추가)이 쉬운 이유가 바로 이것입니다 — 어떤 서버 인스턴스에 요청이 도달해도, 같은 비밀키가 있으면 검증이 가능합니다.

2Access Token vs Refresh Token — 역할과 수명

JWT를 두 종류로 분리하는 이유는 보안과 사용성의 균형 때문입니다. 하나의 토큰만 쓰면, 유효 기간이 길면 탈취 위험이 높고 짧으면 자주 로그인해야 합니다.

구분Access TokenRefresh Token
역할API 요청 인증 (매 요청 헤더)Access Token 재발급 전용
수명짧게 (예: 1시간 ~ 수분)길게 (예: 7일 ~ 30일)
전송 빈도모든 API 요청마다Access Token 만료 시에만
탈취 위험짧은 수명으로 피해 최소화안전한 곳에 보관 필요
payload.type"access""refresh"
💡 두 토큰은 payload의 type 필드로 구분한다

Access Token과 Refresh Token은 발급 시 payload에 type: "access" / type: "refresh"를 명시합니다. 재발급 엔드포인트에서는 받은 토큰의 type을 확인해 "refresh"일 때만 새 access token을 발급합니다. access token을 refresh 엔드포인트에 넘겨도 거부되어야 합니다.

3비밀번호 해싱 — bcrypt와 salt

비밀번호는 절대 평문으로 저장해서는 안 됩니다. DB가 유출되어도 원본 비밀번호를 알 수 없도록 단방향 해시로 변환해 저장합니다. NestJS 생태계에서는 bcrypt가 사실상 표준입니다.

⚠️ 비밀번호 평문 저장 절대 금지

"DB가 해킹당하면 어차피 끝 아닌가?"라는 생각은 잘못된 것입니다. 대부분의 사람이 여러 사이트에 같은 비밀번호를 씁니다. 평문 유출 → 타 사이트 피해(credential stuffing)로 이어집니다. MD5나 SHA-1도 너무 빠르기 때문에 사용 금지입니다 — rainbow table과 GPU 대량 연산으로 쉽게 역산됩니다. bcrypt의 cost factor(saltRounds)가 높을수록 해시 계산이 느려져 대량 시도 공격을 막습니다.

bcrypt는 해시할 때 무작위 salt를 생성하고, 그 salt를 해시 결과물 안에 내장합니다. 별도의 salt 컬럼을 저장할 필요가 없고, bcrypt.compare가 내장된 salt를 자동으로 추출해 비교합니다.

auth.service.ts — bcrypt 해싱 · 비교
import * as bcrypt from 'bcrypt';

const ROUNDS = 10;  // cost factor — 높을수록 느리고 안전

// 회원가입: 비밀번호 해싱 후 저장
const hashedPw = await bcrypt.hash(password, ROUNDS);
// hashedPw 안에 salt가 내장됨 → DB에 hashedPw만 저장

// 로그인: 입력 비밀번호 vs 저장된 해시 비교
const isMatch = await bcrypt.compare(inputPassword, storedHashedPw);
// bcrypt.compare 가 해시에서 salt를 자동 추출해 재해싱 후 비교
if (!isMatch) throw new UnauthorizedException('비밀번호가 올바르지 않습니다');

4✅ 셀프 체크 — 면접관처럼 물어보기

Q JWT가 "stateless"라는 게 무슨 뜻인가요?

서버가 로그인 상태를 어디에도 저장하지 않는다는 뜻입니다. 세션 방식은 서버 메모리나 Redis에 세션 데이터를 기록해야 하지만, JWT는 서명된 토큰을 클라이언트에게 발급하고 서버는 아무것도 저장하지 않습니다. 이후 요청이 들어오면 서버는 비밀키로 서명만 검증해 진위를 확인합니다. 서버 인스턴스가 몇 개이든 같은 비밀키를 공유하면 어느 인스턴스에서도 검증할 수 있어 수평 확장이 용이합니다.

Q Refresh Token이 따로 필요한 이유는 무엇인가요?

보안과 사용성을 동시에 확보하기 위해서입니다. Access Token만 쓴다면 ① 만료 기간을 짧게 설정하면 사용자가 자주 로그인해야 하고 ② 길게 설정하면 탈취 시 오랫동안 악용될 수 있습니다. Refresh Token을 도입하면, Access Token은 짧게(수분~수십 분) 유지해 탈취 피해를 최소화하고, Refresh Token은 길게(수 일~수 주) 유지해 사용자 경험을 유지합니다. Access Token이 만료되면 Refresh Token으로 조용히 새 Access Token을 발급받으면 되기 때문입니다.

Q bcrypt.compare는 어떻게 salt 없이도 비교할 수 있나요?

bcrypt 해시 결과물($2b$10$...) 안에 salt가 내장되어 있기 때문입니다. bcrypt.compare(입력PW, 저장된해시)를 호출하면 내부적으로 ① 해시에서 salt를 추출 → ② 입력 비밀번호를 같은 salt로 재해싱 → ③ 결과를 저장된 해시와 비교합니다. 따라서 별도의 salt 컬럼을 저장할 필요가 없고, 코드도 간결해집니다.
