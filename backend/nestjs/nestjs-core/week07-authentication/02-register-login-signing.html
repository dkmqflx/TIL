토큰 signing

회원가입 시 bcrypt로 비밀번호를 해싱하고, 로그인 시 bcrypt.compare로 검증한 뒤 jwtService.sign으로 Access/Refresh Token을 발급하는 전체 흐름을 구현한다.

"

로그인은 세 단계다: ① 이메일로 유저 찾기 → ② bcrypt로 비밀번호 검증 → ③ 토큰 서명·발급. 비밀번호는 검증 후 즉시 잊는다 — JWT payload에는 절대 넣지 않는다.

  • registerWithEmail이 비밀번호를 bcrypt 해싱 후 저장한다는 흐름을 설명할 수 있다
  • authenticateWithEmailAndPassword의 두 단계(유저 조회·비밀번호 검증)를 설명할 수 있다
  • jwtService.sign(payload, options)으로 토큰을 발급하는 코드를 작성할 수 있다
  • JWT payload에 넣어야 할 것과 넣으면 안 되는 것을 구분할 수 있다

1회원가입 — registerWithEmail

회원가입의 핵심 규칙은 단 하나 — 비밀번호는 절대 평문으로 저장하지 않는다. 저장 전 반드시 bcrypt로 해싱해서 해시값만 DB에 보관합니다.

🍔 비유 — 금고에 지문 보관

은행이 비밀번호를 그대로 파일에 적어 두지 않듯, 우리도 비밀번호를 그대로 저장하지 않습니다. 대신 지문처럼 복원할 수 없는 방식(bcrypt 해시)으로 변환해서 보관합니다. 나중에 본인이 왔을 때 "지문이 일치하는지"만 확인하면 됩니다 — 원본을 알 필요가 없습니다.

auth.service.ts — registerWithEmail
import * as bcrypt from 'bcrypt';

const ROUNDS = 10;

async registerWithEmail(user: {
  email: string;
  nickname: string;
  password: string;
}) {
  // 1. 비밀번호 해싱 (salt는 bcrypt가 자동 생성·내장)
  const hashedPw = await bcrypt.hash(user.password, ROUNDS);

  // 2. 해시된 비밀번호로 유저 생성 — 평문은 이 시점에 버려짐
  const newUser = await this.usersRepository.save({
    email: user.email,
    nickname: user.nickname,
    password: hashedPw,  // 해시만 저장
  });

  // 3. 토큰 발급 (회원가입 즉시 로그인 처리)
  return this.loginWithEmail(newUser);
}
⚠️ 이미 가입된 이메일 중복 처리

실제 구현에서는 save() 전에 usersRepository.findOne({ where: { email } })으로 이미 가입된 이메일인지 확인하고, 중복이면 ConflictException을 던져야 합니다. DB에 unique 제약이 있더라도 명시적 체크로 사용자에게 명확한 오류 메시지를 제공하는 것이 좋습니다.

2로그인 검증 — authenticateWithEmailAndPassword

로그인 검증은 이메일로 유저 조회 → bcrypt.compare로 비밀번호 확인의 두 단계입니다. 이 함수는 검증만 담당하고, 토큰 발급은 별도 함수에 위임합니다.

auth.service.ts — authenticateWithEmailAndPassword + loginWithEmail
async authenticateWithEmailAndPassword(user: {
  email: string;
  password: string;
}) {
  // 1. 이메일로 유저 조회
  const existingUser = await this.usersRepository.findOne({
    where: { email: user.email },
    // select: { password: true } — 필요에 따라 비밀번호 필드 명시 조회
  });

  if (!existingUser) {
    throw new UnauthorizedException('존재하지 않는 사용자입니다');
  }

  // 2. 입력 비밀번호 vs 저장된 해시 비교
  const isMatch = await bcrypt.compare(user.password, existingUser.password);

  if (!isMatch) {
    throw new UnauthorizedException('비밀번호가 올바르지 않습니다');
  }

  return existingUser;  // 검증 통과 — 유저 객체 반환
}

async loginWithEmail(user: { email: string; id: number }) {
  return {
    accessToken: this.signToken(user, false),
    refreshToken: this.signToken(user, true),
  };
}

3토큰 서명 — signToken + 엔드포인트

jwtService.sign(payload, options)으로 토큰을 발급합니다. payload에는 최소한의 클레임만 넣고, isRefresh 플래그로 수명을 구분합니다.

💡 JWT payload에는 최소 정보만 — 비밀번호 절대 금지

payload는 서명은 되지만 암호화되지 않습니다. Base64로 디코딩하면 누구나 읽을 수 있습니다. 비밀번호(해시 포함)는 절대 넣으면 안 되고, sub(유저 ID), email, type("access"/"refresh") 정도의 최소 정보만 넣는 것이 원칙입니다.

auth.service.ts — signToken + 엔드포인트
// auth.service.ts
signToken(user: { email: string; id: number }, isRefresh: boolean): string {
  const payload = {
    sub: user.id,     // 유저 식별자 (subject)
    email: user.email,
    type: isRefresh ? 'refresh' : 'access',
    // ⚠️ 비밀번호(평문·해시 모두)는 절대 payload에 포함하지 않는다
  };

  return this.jwtService.sign(payload, {
    secret: JWT_SECRET,
    expiresIn: isRefresh ? '7d' : '1h',  // refresh: 7일 / access: 1시간
  });
}

// auth.controller.ts
@Post('register/email')
async registerEmail(@Body() body: RegisterEmailDto) {
  return this.authService.registerWithEmail(body);
}

@Post('login/email')
async loginEmail(@Body() body: LoginEmailDto) {
  const user = await this.authService.authenticateWithEmailAndPassword(body);
  return this.authService.loginWithEmail(user);
}

4✅ 셀프 체크 — 면접관처럼 물어보기

Q 로그인 검증 단계는 어떻게 되나요?

두 단계입니다. ① 이메일로 유저 조회 — usersRepository.findOne({ where: { email } })으로 DB에서 유저를 찾습니다. 없으면 즉시 UnauthorizedException을 던집니다. ② bcrypt.compare로 비밀번호 검증 — 입력한 비밀번호와 DB에 저장된 해시를 비교합니다. 일치하지 않으면 역시 UnauthorizedException을 던집니다. 두 단계 모두 통과하면 유저 객체를 반환하고, 이후 토큰을 발급합니다. 보안상 "이메일이 없다" / "비밀번호가 틀렸다"를 구분해 응답하지 않는 것이 좋습니다 — 공격자에게 정보를 주지 않기 위해서입니다.

Q jwtService.sign의 payload에 무엇을 넣어야 하나요?

payload에는 최소한의 식별 정보만 넣습니다: sub(유저 ID), email, type("access" 또는 "refresh"). 절대 넣으면 안 되는 것은 비밀번호(평문은 물론 해시도)입니다. JWT payload는 서명되어 있을 뿐 암호화되지 않아 Base64 디코딩으로 누구나 읽을 수 있습니다. 또한 민감한 개인정보(주민번호·카드번호 등)도 넣지 않습니다. payload가 크면 매 요청 헤더 크기도 커지므로, "이 요청이 누구의 것인지"를 알 수 있는 최소 정보만 담습니다.

Q 회원가입 직후 토큰을 즉시 발급하는 이유는 무엇인가요?

회원가입 자체가 "이 사람이 맞다"는 검증을 완료한 시점이기 때문입니다. 회원가입 후 별도로 다시 로그인 폼을 채우는 것은 사용자 경험을 불필요하게 방해합니다. registerWithEmail이 성공하면 내부적으로 loginWithEmail을 호출해 Access/Refresh Token을 즉시 반환합니다. 이렇게 하면 프론트엔드는 회원가입 응답으로 받은 토큰을 바로 저장하고 인증 상태로 전환할 수 있습니다.
