재발급(refresh) 로직

Authorization 헤더에서 Basic / Bearer 토큰을 파싱하고, jwtService.verify로 검증한 뒤, Refresh Token으로 새 Access Token을 발급하는 재발급 흐름을 구현한다.

"

토큰을 헤더에서 꺼내는 것도, 재발급도 타입 확인이 핵심이다. Refresh Token으로만 Access Token을 재발급하고, Access Token으로 재발급을 요청하면 거부해야 한다.

  • Authorization: Basic xxx와 Authorization: Bearer xxx의 형식을 파싱하는 로직을 설명할 수 있다
  • jwtService.verify(token, options)로 서명 검증 + 만료 확인을 한 번에 한다는 것을 안다
  • 재발급 시 payload의 type이 "refresh"인지 반드시 확인해야 하는 이유를 설명할 수 있다
  • Access Token 만료 → Refresh Token으로 재발급 → 새 Access Token 수신 흐름을 그릴 수 있다

1Authorization 헤더 파싱 — extractTokenFromHeader

클라이언트는 토큰을 Authorization 헤더에 담아 보냅니다. 형식은 두 가지입니다. Basic은 이메일/비밀번호를 Base64 인코딩한 것이고, Bearer는 JWT 토큰 자체입니다. 두 형식 모두 "<type> <credential>" 구조이므로, 공백으로 분리해 각 부분을 추출합니다.

🍔 비유 — 봉투 형식 규칙

인가 헤더는 "종류 + 자격증명" 형식입니다. "Bearer jwt.token.here"는 "팔찌(Bearer) 방식으로 이 토큰(jwt.token.here)을 제시한다"는 뜻입니다. 서버는 먼저 봉투 앞에 붙은 종류 레이블을 확인하고, 그 다음 봉투 안의 내용을 처리합니다. 종류가 다르면(Basic인데 Bearer로 처리) 파싱 방법이 달라집니다.

auth.service.ts — extractTokenFromHeader
extractTokenFromHeader(header: string, isBearer: boolean): string {
  // Authorization 헤더 형식: "Bearer jwt..." or "Basic base64..."
  const [type, token] = header.split(' ');
  const expectedType = isBearer ? 'Bearer' : 'Basic';

  if (type !== expectedType || !token) {
    throw new UnauthorizedException('잘못된 Authorization 헤더 형식입니다');
  }

  return token;
}

// Basic 헤더 → 이메일/비밀번호 디코딩
decodeBasicToken(base64: string): { email: string; password: string } {
  const decoded = Buffer.from(base64, 'base64').toString('utf8');
  const [email, ...rest] = decoded.split(':');
  return { email, password: rest.join(':') };
}

2토큰 검증 — jwtService.verify

jwtService.verify(token, { secret })는 두 가지를 한 번에 합니다: ① 서명 검증(위조 여부)과 ② 만료 확인(exp 클레임). 하나라도 실패하면 예외를 던집니다.

auth.service.ts — verifyToken
verifyToken(token: string) {
  try {
    return this.jwtService.verify(token, {
      secret: JWT_SECRET,  // 서명 검증용 비밀키
    });
    // 성공 시 payload 반환 { sub, email, type, iat, exp }
  } catch (e) {
    throw new UnauthorizedException('토큰이 유효하지 않거나 만료되었습니다');
  }
}
🧭 jwtService.verify vs jwtService.decode

verify는 서명 검증 + 만료 확인을 모두 합니다 — 검증 실패 시 예외를 던집니다. decode는 서명 검증 없이 payload를 읽기만 합니다 — 위조된 토큰도 디코딩됩니다. 인증 흐름에서는 반드시 verify를 사용해야 합니다. decode는 만료된 토큰의 payload를 읽어야 하는 특수 상황에서만 사용합니다.

3Access Token 재발급 — refresh 흐름

Access Token이 만료되면 클라이언트는 Refresh Token을 제시해 새 Access Token을 발급받습니다. 재발급 시 반드시 payload.type === "refresh"를 확인해야 합니다 — Access Token으로 재발급을 요청하는 것을 막기 위해서입니다.

⚠️ Refresh Token 재발급 시 반드시 type 확인

재발급 엔드포인트에 Access Token이 들어오면 서명 자체는 유효할 수 있습니다. 그러므로 jwtService.verify 후 payload의 type이 "refresh"인지를 명시적으로 확인해야 합니다. type !== "refresh"이면 즉시 UnauthorizedException을 던집니다. 이 타입 체크 없이는 Access Token으로도 재발급이 가능해지는 보안 취약점이 생깁니다.

auth.service.ts — rotateAccessToken (refresh → new access)
async rotateAccessToken(refreshToken: string): Promise<{ accessToken: string }> {
  // 1. 서명 검증 + 만료 확인
  const payload = this.verifyToken(refreshToken);

  // 2. Refresh Token 타입 확인 (핵심 보안 체크)
  if (payload.type !== 'refresh') {
    throw new UnauthorizedException('Refresh Token으로만 재발급이 가능합니다');
  }

  // 3. 새 Access Token 발급 (Refresh Token은 갱신하지 않음)
  const newAccessToken = this.signToken(
    { id: payload.sub, email: payload.email },
    false,  // isRefresh = false → access token
  );

  return { accessToken: newAccessToken };
}

// auth.controller.ts
@Post('token/access')
async refreshAccessToken(@Headers('authorization') authHeader: string) {
  const refreshToken = this.authService.extractTokenFromHeader(authHeader, true);
  return this.authService.rotateAccessToken(refreshToken);
}

4✅ 셀프 체크 — 면접관처럼 물어보기

Q Authorization: Bearer <token> 헤더의 형식을 설명해 보세요.

형식은 "인증 방식 + 공백 + 자격증명"입니다. Bearer는 인증 방식(토큰 소지자 인증), 그 뒤 공백 이후가 JWT 토큰 자체입니다. 파싱 시 header.split(' ')로 분리하면 ['Bearer', 'eyJhbGci...']가 됩니다. 첫 번째 요소로 타입을 확인하고(type === 'Bearer'), 두 번째 요소가 실제 토큰입니다. Basic의 경우에는 두 번째 부분이 email:password를 Base64로 인코딩한 값입니다.

Q 재발급 요청 시 서버는 무엇을 검증하나요?

두 가지를 순서대로 검증합니다. ① 서명 유효성 + 만료 여부 — jwtService.verify(token, { secret })로 토큰이 위조되지 않았고 만료되지 않았음을 확인합니다. 실패하면 즉시 401. ② payload.type === "refresh" 확인 — Refresh Token으로 발급된 것인지 타입을 검사합니다. Access Token을 재발급 엔드포인트에 가져오면 서명은 유효하더라도 이 타입 체크에서 거부됩니다. 두 단계 모두 통과해야만 새 Access Token을 발급합니다.

Q jwtService.verify와 jwtService.decode의 차이는 무엇인가요?

verify는 서명 검증 + 만료 확인을 모두 수행합니다 — 검증 실패 시 예외를 던지고, 성공 시 payload를 반환합니다. decode는 서명 검증 없이 payload만 추출합니다 — 위조되거나 만료된 토큰도 디코딩됩니다. 따라서 인증 흐름에서는 반드시 verify를 사용해야 합니다. decode는 예를 들어 만료된 토큰의 sub를 읽어야 하는 특수 상황(예: 만료된 토큰으로 로그아웃 처리)에서만 제한적으로 사용합니다.
