Pipe
검증 · 변환 파이프라인

핸들러(컨트롤러 메서드)가 실행되기 직전, 들어오는 파라미터를 검증(validation)하거나 변환(transformation)하는 것이 Pipe의 역할이다. NestJS 내장 파이프부터 직접 만드는 Custom Pipe까지 완전히 이해한다.

"

Pipe는 두 가지만 한다 — 들어온 값이 올바른가 검증하거나, 필요한 타입으로 변환하거나. 핸들러는 파이프를 통과한 "깨끗한" 값만 받는다.

  • NestJS가 내장 제공하는 ParseIntPipe, ParseUUIDPipe 등 주요 파이프를 파라미터 데코레이터에 적용할 수 있다
  • PipeTransform 인터페이스의 transform(value, metadata)를 구현해 Custom Pipe를 만들 수 있다
  • DefaultValuePipe로 기본값을 부여하고, 여러 파이프를 동시에 파라미터에 적용할 수 있다
  • Pipe가 요청 라이프사이클에서 Guard 이후, 핸들러 직전에 실행된다는 순서를 안다

1내장 파이프 — ParseIntPipe · ParseUUIDPipe

URL 파라미터나 쿼리스트링은 항상 문자열로 들어옵니다. @Param('id')가 수신하는 값은 "42"지, 숫자 42가 아닙니다. NestJS 내장 파이프는 이 변환을 선언적으로 처리해 줍니다.

🍔 비유 — 정수기 필터

수도꼭지에서 나오는 물(요청 값)에는 불순물(잘못된 타입·형식)이 섞여 있을 수 있습니다. 정수기 필터(Pipe)가 중간에서 물을 걸러, 통과시키거나(검증 통과) 막거나(예외 throw) 변환(문자열 → 숫자)해서 핸들러에 전달합니다. 핸들러는 "깨끗한 물"만 받을 수 있고, 필터를 우회할 방법이 없습니다.

posts.controller.ts — 내장 파이프 적용
import { Controller, Get, Param, ParseIntPipe, ParseUUIDPipe } from '@nestjs/common';

@Controller('posts')
export class PostsController {

  // ParseIntPipe: "42" → 42, "abc" → 400 Bad Request 자동
  @Get(':id')
  getPost(@Param('id', ParseIntPipe) id: number) {
    return { id }; // id는 이미 number 타입으로 보장됨
  }

  // ParseUUIDPipe: UUID 형식 검증 (v4 기본), 형식 틀리면 400
  @Get('by-uuid/:uuid')
  getByUuid(@Param('uuid', ParseUUIDPipe) uuid: string) {
    return { uuid };
  }
}
🧭 주요 내장 파이프 목록

ParseIntPipe / ParseFloatPipe / ParseBoolPipe — 숫자·불린 변환. ParseUUIDPipe — UUID 형식 검증. ParseArrayPipe — 쿼리 배열 파싱. ParseEnumPipe — Enum 값 검증. ValidationPipe — DTO 전체 검증(Week 9에서 상세히). 이 파이프들은 검증 실패 시 BadRequestException(400)을 자동으로 던집니다.

2Custom Pipe — PipeTransform 구현

내장 파이프로 해결이 안 되는 검증·변환이 필요할 때 Custom Pipe를 직접 만듭니다. PipeTransform 인터페이스를 구현하고 transform(value, metadata) 메서드만 작성하면 됩니다.

positive-int.pipe.ts — Custom Pipe 예제
import { PipeTransform, Injectable, ArgumentMetadata, BadRequestException } from '@nestjs/common';

@Injectable()
export class PositiveIntPipe implements PipeTransform {
  transform(value: any, metadata: ArgumentMetadata) {
    // value: 파이프에 들어온 실제 값
    // metadata.type: 'body' | 'query' | 'param' | 'custom'
    // metadata.data: 데코레이터에 전달된 키 이름 (예: 'id')

    const num = Number(value);

    if (!Number.isInteger(num) || num <= 0) {
      throw new BadRequestException(`'${metadata.data}'는 양의 정수여야 합니다`);
    }

    return num; // 변환된 값을 반환 — 핸들러는 이 값을 받는다
  }
}

// 사용 — 컨트롤러에서
@Get(':id')
getPost(@Param('id', PositiveIntPipe) id: number) {
  return { id };
}
💡 transform의 반환값이 핸들러가 받는 값

transform에서 return한 값이 핸들러 파라미터로 주입됩니다. 검증만 하고 원본을 그대로 쓰고 싶다면 return value, 변환이 필요하면 변환 후 반환합니다. 예외를 던지면 핸들러는 실행되지 않습니다.

3DefaultValuePipe + 다중 파이프 동시 적용

쿼리 파라미터처럼 없을 수도 있는 값에 기본값을 주려면 DefaultValuePipe를 먼저 적용하고, 그 뒤에 변환 파이프를 붙입니다. 파이프는 배열 순서대로 왼쪽부터 순차 실행됩니다.

posts.controller.ts — DefaultValuePipe + 다중 파이프
import { Controller, Get, Query, DefaultValuePipe, ParseIntPipe } from '@nestjs/common';

@Controller('posts')
export class PostsController {

  @Get()
  getPosts(
    // page가 없으면 → DefaultValuePipe가 '1' 적용 → ParseIntPipe가 1로 변환
    @Query('page', new DefaultValuePipe(1), ParseIntPipe) page: number,
    @Query('limit', new DefaultValuePipe(20), ParseIntPipe) limit: number,
  ) {
    return { page, limit }; // ?page=2 → page=2, limit=20
  }
}
⚠️ 파이프는 Guard 이후에 실행된다

NestJS 요청 라이프사이클 순서: Middleware → Guard → Interceptor(pre) → Pipe → Handler. Guard가 요청을 차단하면 파이프는 아예 실행되지 않습니다. 따라서 "인증이 필요한 라우트"에서 Guard가 먼저 401을 던지면, 파이프의 파라미터 검증까지 도달하지 않습니다.

복습

4✅ 셀프 체크 — 면접관처럼 물어보기

Q NestJS Pipe의 두 가지 역할은 무엇인가요?

두 가지입니다. ① 검증(Validation) — 들어온 값이 올바른 형식·범위인지 확인하고, 맞지 않으면 예외(BadRequestException)를 던집니다. ② 변환(Transformation) — 들어온 값을 원하는 타입·형태로 바꿔서 핸들러에 전달합니다(예: "42" → 42). ParseIntPipe는 두 역할을 동시에 수행합니다 — 문자열이 정수로 변환 가능한지 검증하고, 동시에 숫자로 바꿔서 반환합니다.

Q Pipe는 요청 라이프사이클의 어디에서 동작하나요?

Pipe는 핸들러(컨트롤러 메서드) 직전에 실행됩니다. 정확한 순서는 Middleware → Guard → Interceptor(pre) → Pipe → Handler입니다. Guard가 먼저 요청의 통과 여부를 판단하고, Pipe는 통과된 요청의 파라미터를 핸들러에 넘기기 직전 검증·변환합니다. Pipe가 예외를 던지면 핸들러는 실행되지 않습니다.

Q transform(value, metadata)에서 metadata는 어떤 정보를 담고 있나요?

metadata는 ArgumentMetadata 타입으로, 세 필드를 가집니다. type: 파라미터의 출처 — 'body', 'query', 'param', 'custom' 중 하나입니다. metatype: TypeScript 타입 메타데이터(예: Number, String). data: 데코레이터에 전달한 키 이름(예: @Param('id')라면 'id'). 이 정보를 활용하면 파라미터 출처나 이름에 따라 다른 검증 로직을 적용하는 범용 파이프를 만들 수 있습니다.
