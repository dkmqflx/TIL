Guard
CanActivate · 인증 가드

CanActivate 인터페이스를 구현해 요청이 핸들러에 도달하기 전 통과 여부를 판단하는 Guard를 만든다. BasicTokenGuard부터 BearerTokenGuard·AccessTokenGuard 적용 흐름까지.

"

Guard는 단 하나의 질문만 한다 — "이 요청이 핸들러에 들어갈 자격이 있는가?" true면 통과, false거나 예외면 차단. 무엇을 처리할지가 아니라 들여보낼지를 결정하는 것이 Guard의 역할이다.

  • CanActivate 인터페이스와 canActivate(context: ExecutionContext)의 반환값 의미를 설명할 수 있다
  • ExecutionContext를 통해 HTTP 요청 객체를 꺼내는 방법(context.switchToHttp().getRequest())을 안다
  • BasicTokenGuard가 Authorization: Basic 헤더를 파싱하는 흐름을 설명할 수 있다
  • BearerTokenGuard / AccessTokenGuard를 컨트롤러나 라우트 레벨에 적용할 수 있다

1CanActivate 인터페이스 — 통과 / 차단

Guard는 CanActivate 인터페이스를 구현합니다. 핵심은 단 하나의 메서드 — canActivate(context: ExecutionContext). 이 메서드가 true(또는 Promise<true>)를 반환하면 요청이 통과되고, false나 예외를 던지면 403 Forbidden으로 차단됩니다.

🍔 비유 — 클럽 입구 보안요원

클럽 입구의 보안요원은 음악을 틀거나 음식을 내오지 않습니다. 오직 입장 가능 여부만 판단합니다. 초대장이 있으면 true — 통과. 없으면 false — 차단. 보안요원은 "안에서 무슨 일이 벌어지는지(핸들러 로직)"에 관심이 없고, 입장 자격만 봅니다. NestJS Guard도 마찬가지로, 인증·인가만 담당하고 비즈니스 로직은 건드리지 않습니다.

basic-token.guard.ts — CanActivate 기본 구조
import { CanActivate, ExecutionContext, Injectable, UnauthorizedException } from '@nestjs/common';

@Injectable()
export class BasicTokenGuard implements CanActivate {
  async canActivate(context: ExecutionContext): Promise<boolean> {
    // 1. HTTP 요청 객체 꺼내기
    const req = context.switchToHttp().getRequest();

    // 2. Authorization 헤더 확인
    const rawToken = req.headers['authorization'];
    if (!rawToken) {
      throw new UnauthorizedException('Authorization 헤더가 없습니다');
    }

    // 3. "Basic base64encoded" 형식 파싱
    const [type, token] = rawToken.split(' ');
    if (type !== 'Basic' || !token) {
      throw new UnauthorizedException('올바르지 않은 Basic 토큰 형식입니다');
    }

    // 4. Base64 디코딩 → email:password 추출
    const decoded = Buffer.from(token, 'base64').toString('utf8');
    const [email, password] = decoded.split(':');

    // 5. 요청 객체에 credentials 저장 (이후 핸들러에서 사용 가능)
    req.user = { email, password };

    return true; // 통과
  }
}
🧭 Guard vs Middleware — 결정적 차이

Middleware는 Guard보다 먼저 실행되지만, 라우트·핸들러 정보를 알지 못합니다(req, res, next만 있음). Guard는 Middleware 이후에 실행되지만 ExecutionContext를 통해 현재 핸들러가 어떤 컨트롤러·메서드인지 알 수 있습니다. 이 정보로 "이 라우트에 어떤 Role이 필요한가" 같은 메타데이터 기반 인가 로직을 구현할 수 있습니다. 인증/인가 로직은 Middleware보다 Guard가 적합한 이유입니다.

2BearerTokenGuard — JWT Bearer 검증

실제 프로젝트에서 가장 많이 쓰는 패턴입니다. Authorization: Bearer <token> 헤더를 파싱하고, JWT를 검증해 request.user에 사용자 정보를 주입합니다. 코드는 code/bearer-token.guard.ts에서 확인하세요.

guard 적용 — 컨트롤러/라우트 레벨
import { Controller, Get, UseGuards } from '@nestjs/common';
import { AccessTokenGuard } from './auth/guard/access-token.guard';

// 컨트롤러 전체에 적용
@UseGuards(AccessTokenGuard)
@Controller('posts')
export class PostsController {

  @Get()
  getPosts() {
    // Guard를 통과했으므로 req.user가 보장된다
    return [];
  }

  // 특정 라우트만 다른 Guard 적용 (오버라이드)
  @Post('register')
  @UseGuards(BasicTokenGuard)
  register() {
    return {};
  }
}

3AccessTokenGuard — 토큰 타입까지 검증

BearerTokenGuard는 토큰 자체의 서명 유효성만 확인합니다. AccessTokenGuard는 여기서 한 단계 더 나아가, payload의 type 필드가 "access"인지까지 검사합니다. Refresh Token으로 일반 API를 호출하면 거부됩니다.

access-token.guard.ts — BearerTokenGuard 확장
import { CanActivate, ExecutionContext, Injectable, UnauthorizedException } from '@nestjs/common';

@Injectable()
export class AccessTokenGuard implements CanActivate {
  async canActivate(context: ExecutionContext): Promise<boolean> {
    const req = context.switchToHttp().getRequest();

    // BearerTokenGuard와 동일한 헤더 파싱 + jwtService.verify 과정 생략
    // (실제 구현에서는 BearerTokenGuard를 상속하거나 AuthService 주입)

    const payload = req.tokenPayload; // BearerTokenGuard가 미리 세팅한 값

    if (payload?.type !== 'access') {
      throw new UnauthorizedException('Access Token이 아닙니다');
    }

    return true;
  }
}
💡 Guard 등록 방법 세 가지

① 라우트 레벨: @UseGuards(Guard)를 메서드 위에. ② 컨트롤러 레벨: @UseGuards(Guard)를 클래스 위에(전체 적용). ③ 전역(Global): app.useGlobalGuards(new Guard()) — 단, 이 방식은 DI가 안 되므로 실무에서는 APP_GUARD 토큰을 이용한 프로바이더 등록을 선호합니다.

4✅ 셀프 체크 — 면접관처럼 물어보기

Q Guard와 Middleware의 차이점은 무엇인가요?

두 가지 핵심 차이가 있습니다. ① 실행 시점: Middleware가 먼저 실행되고, Guard가 그 이후에 실행됩니다. ② 컨텍스트 접근: Middleware는 req, res, next만 알고, 어떤 컨트롤러·메서드가 처리할지 모릅니다. Guard는 ExecutionContext를 통해 현재 핸들러의 클래스명·메서드명·메타데이터(예: @Roles() 데코레이터 값)까지 접근할 수 있습니다. 따라서 "역할(Role) 기반 접근 제어"처럼 라우트 정보가 필요한 인가 로직은 Middleware가 아닌 Guard로 구현합니다.

Q canActivate가 false를 반환하면 어떻게 되나요?

false를 반환하면 NestJS가 자동으로 403 Forbidden 응답을 반환합니다. 단, 실무에서는 보통 false 반환보다 UnauthorizedException을 직접 던지는 방식을 선호합니다 — 더 명확한 401 상태코드와 커스텀 메시지를 함께 전달할 수 있기 때문입니다. canActivate는 boolean 외에도 Promise<boolean>, Observable<boolean>을 반환할 수 있습니다 — 비동기 JWT 검증처럼 I/O가 필요한 경우 async/await를 활용합니다.

Q BasicTokenGuard와 BearerTokenGuard는 어느 엔드포인트에 각각 적용하나요?

BasicTokenGuard는 로그인/회원가입 엔드포인트처럼 아직 토큰이 없는 상태에서 사용자를 식별할 때 적용합니다. Authorization: Basic base64(email:password) 헤더로 자격증명을 전달합니다. BearerTokenGuard / AccessTokenGuard는 이미 로그인한 사용자가 JWT를 가지고 있는 상태, 즉 인증이 필요한 일반 API 엔드포인트에 적용합니다. Authorization: Bearer jwt.token.here 형식입니다. 재발급 엔드포인트에는 Refresh Token을 검증하는 별도 Guard를 사용해야 합니다.
