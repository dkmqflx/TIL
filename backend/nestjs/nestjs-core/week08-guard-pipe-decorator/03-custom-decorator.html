Custom Decorator
@User 파라미터 데코레이터

createParamDecorator로 request.user를 핸들러 파라미터에 바로 주입하는 @User() 데코레이터를 구현한다. data 인자로 특정 필드만 골라 받는 패턴도 함께.

"

Guard가 request.user를 채워두면, @User() 데코레이터가 그걸 꺼내 핸들러에 바로 꽂아준다. 두 개가 인증과 추출을 분업하는 구조다.

  • createParamDecorator((data, ctx) => ...) 기본 구조를 작성할 수 있다
  • @User() 파라미터 데코레이터로 request.user 전체를 핸들러에 주입할 수 있다
  • @User('email')처럼 data 인자로 특정 필드만 추출하는 패턴을 구현할 수 있다
  • Guard → request.user 세팅 → Custom Decorator 추출 흐름을 그릴 수 있다

1createParamDecorator 기본

@Body(), @Param()처럼 핸들러 파라미터에 값을 주입하는 데코레이터를 직접 만들 수 있습니다. createParamDecorator에 콜백을 전달하면 되고, 콜백의 반환값이 파라미터로 주입됩니다.

🍔 비유 — 택배 상자에서 필요한 것만 꺼내기

request.user는 Guard가 배달해 놓은 택배 상자입니다. 상자 안에는 이메일, 닉네임, 역할 등 여러 정보가 들어 있습니다. @User()는 "상자 전체를 꺼내라"이고, @User('email')는 "상자 안에서 이메일만 꺼내라"입니다. 핸들러는 매번 req.user.email처럼 길게 쓰는 대신, 원하는 항목을 콕 집어 파라미터로 받을 수 있습니다.

user.decorator.ts — createParamDecorator 기본
import { createParamDecorator, ExecutionContext } from '@nestjs/common';

// data: @User('email') 처럼 전달한 인자 ('email' 또는 undefined)
// ctx: ExecutionContext — request 객체 접근에 사용
export const User = createParamDecorator(
  (data: string | undefined, ctx: ExecutionContext) => {
    const req = ctx.switchToHttp().getRequest();
    return req.user; // request.user 전체 반환
  },
);

2@User() 데코레이터 구현 — request.user 추출

Guard가 request.user에 인증된 사용자 정보를 저장해 두었다면, @User()는 그 객체를 꺼내 핸들러 파라미터로 바로 주입합니다. 실제 코드는 code/user.decorator.ts에서 확인하세요.

posts.controller.ts — @User() 사용
import { Controller, Get, Post, UseGuards } from '@nestjs/common';
import { AccessTokenGuard } from './auth/guard/access-token.guard';
import { User } from './auth/decorator/user.decorator';
import { UserModel } from './users/entity/users.entity';

@UseGuards(AccessTokenGuard)
@Controller('posts')
export class PostsController {

  @Post()
  createPost(
    @User() user: UserModel,        // request.user 전체
  ) {
    return { authorId: user.id };
  }

  @Get('my')
  getMyPosts(
    @User('email') email: string,   // request.user.email 만
  ) {
    return { email };
  }
}

3data 인자로 특정 필드만 뽑기

데코레이터에 인자를 전달하면 data 파라미터로 받을 수 있습니다. @User('email')처럼 필드 이름을 넘기면 user[data]를 반환하고, 인자 없이 @User()만 쓰면 user 전체를 반환하는 패턴이 가장 흔합니다.

user.decorator.ts — data 인자로 필드 선택 (전체 구현)
import { createParamDecorator, ExecutionContext } from '@nestjs/common';
import { UserModel } from '../users/entity/users.entity';

export const User = createParamDecorator(
  (data: keyof UserModel | undefined, ctx: ExecutionContext) => {
    const req = ctx.switchToHttp().getRequest();
    const user = req.user as UserModel;

    // data가 있으면 해당 필드만, 없으면 user 전체 반환
    return data ? user?.[data] : user;
  },
);

// 사용 예시
// @User()            → UserModel 전체 객체
// @User('id')        → user.id (number)
// @User('email')     → user.email (string)
// @User('nickname')  → user.nickname (string)
💡 Guard와 Custom Decorator는 한 팀

Custom Decorator 혼자서는 request.user가 비어 있으면 아무것도 꺼낼 수 없습니다. Guard가 JWT를 검증하고 request.user를 채워두는 것이 선행되어야 합니다. 이 두 개의 분업이 NestJS 인증 흐름의 핵심 패턴입니다 — Guard: "누구인지 확인해서 기록", Custom Decorator: "기록된 걸 꺼내서 전달".

복습

4✅ 셀프 체크 — 면접관처럼 물어보기

Q 파라미터 데코레이터가 핸들러에 무엇을 넣어주나요?

createParamDecorator의 콜백 함수가 반환하는 값이 핸들러 파라미터로 주입됩니다. @User()의 경우 콜백이 req.user를 반환하므로, 핸들러는 user: UserModel 파라미터로 인증된 사용자 객체를 바로 받습니다. 콜백에서 undefined를 반환하면 파라미터가 undefined가 됩니다. 이 메커니즘은 NestJS 내장 @Body(), @Query(), @Param()도 동일하게 동작합니다.

Q data 인자는 어디에 쓰나요?

data는 데코레이터를 사용할 때 전달하는 인자입니다. @User('email')이라고 쓰면 data === 'email'이 됩니다. @User()처럼 인자 없이 쓰면 data === undefined입니다. 이를 활용해 return data ? user?.[data] : user처럼 조건 분기를 만들면, 하나의 데코레이터로 전체 객체 또는 특정 필드를 선택적으로 꺼낼 수 있습니다. TypeScript에서 keyof UserModel 타입을 data에 지정하면 자동완성과 타입 안전성을 함께 얻을 수 있습니다.

Q Guard 없이 @User()를 사용하면 어떻게 되나요?

request.user가 undefined이므로 @User()의 반환값도 undefined가 됩니다. Guard는 JWT를 검증한 뒤 req.user = payload처럼 요청 객체에 사용자 정보를 직접 저장합니다. 이 선행 작업 없이 데코레이터만 쓰면 핸들러가 undefined를 받아 TypeError가 발생하거나 잘못된 동작을 할 수 있습니다. 따라서 @User()를 쓰는 라우트에는 반드시 @UseGuards(AccessTokenGuard) 등 Guard가 함께 있어야 합니다.
