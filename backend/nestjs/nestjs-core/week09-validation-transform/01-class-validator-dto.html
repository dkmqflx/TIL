Class Validator + DTO
요청 본문 검증 파이프라인

DTO(Data Transfer Object)로 요청 본문의 구조와 타입을 정의하고, class-validator 데코레이터와 전역 ValidationPipe로 검증을 선언적으로 처리한다. PickType·PartialType으로 DTO를 재사용해 중복을 없앤다.

"

DTO는 요청 본문의 계약서다. 컨트롤러에 도달하기 전, ValidationPipe가 이 계약을 위반한 요청을 자동으로 400으로 돌려보낸다. 비즈니스 로직 안에서 if (!email) 같은 방어 코드가 사라진다.

  • DTO 클래스로 요청 본문의 형태와 타입을 명시적으로 정의할 수 있다
  • @IsString, @IsEmail, @Length, @IsInt, @IsOptional 등 주요 검증 데코레이터를 적용할 수 있다
  • 전역 ValidationPipe의 whitelist·transform 옵션이 각각 무엇을 하는지 설명할 수 있다
  • PickType·PartialType(@nestjs/mapped-types)으로 DTO를 재사용하는 패턴을 구현할 수 있다

1DTO(Data Transfer Object) 개념

DTO는 요청 본문의 모양(shape)과 타입을 하나의 클래스로 선언한 객체입니다. Express의 req.body는 타입이 없는 any지만, NestJS에서 DTO를 쓰면 컨트롤러 파라미터부터 타입 안전성이 보장됩니다.

🍔 비유 — 입국 심사대

공항 입국 심사대는 여권을 보고 "서류 형식이 맞는가?"를 먼저 확인합니다. 이름 칸이 비어 있거나, 만료된 여권을 들고 있으면 입국 심사관(비즈니스 로직)에게 도달하지 않고 바로 돌려보냅니다. DTO + ValidationPipe가 바로 이 입국 심사대입니다. 올바른 형식의 요청만 컨트롤러(비즈니스 로직) 안으로 통과됩니다.

전체 예제는 code/create-post.dto.ts에서 확인하세요.

create-post.dto.ts — 기본 DTO 정의
import { IsString, IsInt, IsOptional, Length, Min, Max } from 'class-validator';

export class CreatePostDto {
  @IsString()
  @Length(1, 100)
  title: string;

  @IsString()
  @Length(1, 5000)
  content: string;

  @IsOptional()  // 없어도 통과, 있으면 아래 규칙 적용
  @IsInt()
  @Min(1)
  @Max(5)
  rating?: number;
}
🧭 주요 class-validator 데코레이터

@IsString() — 문자열. @IsInt() — 정수. @IsEmail() — 이메일 형식. @IsUrl() — URL 형식. @Length(min, max) — 문자열 길이 범위. @Min(n) / @Max(n) — 숫자 범위. @IsOptional() — 필드 자체가 없으면 검증 건너뜀. 이 중 @IsOptional()은 관례적으로 다른 데코레이터보다 먼저(위에) 선언합니다.

2검증 데코레이터 + 컨트롤러 연결

DTO 클래스를 컨트롤러의 @Body() 파라미터 타입으로 지정하면, ValidationPipe가 자동으로 요청 본문을 DTO 클래스에 매핑하고 데코레이터 규칙을 검증합니다.

posts.controller.ts — DTO를 Body 타입으로 지정
import { Controller, Post, Body } from '@nestjs/common';
import { CreatePostDto } from './dto/create-post.dto';

@Controller('posts')
export class PostsController {

  @Post()
  createPost(@Body() dto: CreatePostDto) {
    // ValidationPipe 통과 후: dto.title은 string, dto.content는 string
    // dto.rating은 number | undefined (IsOptional이므로)
    return dto;
  }
}

3전역 ValidationPipe — whitelist · transform 옵션

매 컨트롤러마다 파이프를 붙이는 대신 main.ts에서 전역 등록합니다. 두 옵션이 핵심입니다.

main.ts — 전역 ValidationPipe 등록
import { NestFactory } from '@nestjs/core';
import { ValidationPipe } from '@nestjs/common';
import { AppModule } from './app.module';

async function bootstrap() {
  const app = await NestFactory.create(AppModule);

  app.useGlobalPipes(
    new ValidationPipe({
      // whitelist: DTO에 데코레이터가 없는 프로퍼티를 자동 제거
      // → 공격자가 임의 키를 넣어도 서비스 레이어까지 전달되지 않음
      whitelist: true,

      // transform: 요청 본문을 DTO 클래스 인스턴스로 변환
      // → dto instanceof CreatePostDto === true
      // → @Type(() => Number) 없이도 쿼리 스트링 숫자 자동 변환
      transform: true,
    }),
  );

  await app.listen(3000);
}
bootstrap();
💡 whitelist vs forbidNonWhitelisted

whitelist: true 는 DTO에 없는 키를 조용히 제거합니다. 여기에 forbidNonWhitelisted: true 를 추가하면 허용되지 않은 키가 있을 때 400 오류를 반환합니다. 프로덕션에서는 whitelist: true만으로도 충분히 안전하며, 엄격한 API는 forbidNonWhitelisted: true도 함께 씁니다.

4PickType · PartialType으로 DTO 재사용

@nestjs/mapped-types의 유틸리티 함수로 기존 DTO에서 일부 필드를 뽑거나 전체를 선택적으로 만들어 중복을 제거합니다. 검증 데코레이터도 함께 상속됩니다.

create-post.dto.ts — PickType · PartialType 활용
import { IsString, IsInt, IsOptional, Length, Min, Max } from 'class-validator';
import { PartialType, PickType } from '@nestjs/mapped-types';

export class CreatePostDto {
  @IsString()
  @Length(1, 100)
  title: string;

  @IsString()
  @Length(1, 5000)
  content: string;

  @IsOptional()
  @IsInt()
  @Min(1) @Max(5)
  rating?: number;
}

// PickType: 필요한 필드만 골라 새 DTO 생성, 검증 데코레이터 포함 상속
export class UpdatePostTitleDto extends PickType(CreatePostDto, ['title', 'content'] as const) {}

// PartialType: 모든 필드를 Optional로 변환 (PATCH 엔드포인트에 적합)
export class UpdatePostDto extends PartialType(CreatePostDto) {}
⚠️ @nestjs/swagger의 것이 아닌 @nestjs/mapped-types를 임포트

PickType, PartialType은 @nestjs/mapped-types와 @nestjs/swagger 두 패키지에 모두 있습니다. Swagger를 쓰지 않는 프로젝트에서 실수로 swagger 것을 임포트하면 의존성 에러가 납니다. 반드시 @nestjs/mapped-types를 사용하세요.

5✅ 셀프 체크 — 면접관처럼 물어보기

Q DTO와 ValidationPipe가 없으면 어떤 문제가 생기나요?

두 가지 문제가 생깁니다. ① 타입 불안정 — req.body는 any이므로 서비스 레이어에서 직접 if (!body.title || typeof body.title !== 'string') 같은 방어 코드를 반복 작성해야 합니다. ② 보안 위험 — 공격자가 임의의 키를 본문에 실어 보내도 그대로 서비스에 전달됩니다. DTO + ValidationPipe(whitelist: true)를 쓰면 타입 안전성과 화이트리스트 필터링을 한꺼번에 해결합니다.

Q PickType이 해결하는 문제는 무엇인가요?

DTO 중복 선언 문제를 해결합니다. 예를 들어 Create DTO에 title, content, rating이 있을 때, Update DTO에서 title만 바꾸는 엔드포인트가 필요하다면 기존 필드를 복사해 새 클래스를 만들기 쉽습니다. PickType(CreatePostDto, ['title', 'content'])을 쓰면 필드와 검증 데코레이터가 함께 상속되어 단 한 줄로 해결됩니다. Create DTO에서 @Length를 수정하면 자동으로 Update DTO에도 반영됩니다.

Q ValidationPipe의 transform: true옵션은 무엇을 하나요?

두 가지 변환을 수행합니다. ① 클래스 인스턴스화 — 요청 본문(plain object)을 DTO 클래스 인스턴스로 변환합니다. transform: false(기본)이면 dto instanceof CreatePostDto가 false입니다. ② 원시 타입 변환 — 쿼리 스트링·URL 파라미터처럼 항상 문자열로 오는 값을 DTO에 선언된 타입(number, boolean)으로 자동 변환합니다. 이를 통해 @Type(() => Number) 데코레이터를 생략할 수 있습니다.
