Validation Message 일반화
일관된 에러 메시지 패턴

검증 데코레이터에 커스텀 메시지를 직접 넣는 방식의 한계를 이해하고, ValidationArguments를 받는 메시지 생성 함수로 일반화해 프로젝트 전체에서 일관된 에러 메시지를 반환한다.

"

에러 메시지를 함수로 만들면 DTO마다 같은 문구를 복붙할 필요가 없다. 필드 이름·제약 값을 ValidationArguments에서 꺼내 쓰면, 메시지 하나가 모든 DTO를 커버한다.

  • 검증 데코레이터의 message 옵션에 문자열이 아닌 함수를 전달할 수 있다
  • ValidationArguments 타입의 주요 필드(property, value, constraints)를 활용할 수 있다
  • 메시지 생성 함수를 별도 파일로 분리해 DTO 전반에서 재사용할 수 있다

1데코레이터별 커스텀 메시지

검증 데코레이터는 두 번째 인수(옵션 객체)의 message 필드로 에러 메시지를 커스터마이징할 수 있습니다. 가장 단순한 방식은 문자열을 그대로 넣는 것입니다.

🍔 비유 — 붕어빵 틀

에러 메시지를 DTO마다 직접 적으면 "제목은 1~100자" / "본문은 1~5000자" 처럼 내용만 다른 문구가 파일마다 흩어집니다. 붕어빵 틀(메시지 생성 함수)을 만들면 "틀"에 재료(필드 이름, 제약 값)만 넣으면 모양이 같은 붕어빵(일관된 메시지)이 나옵니다. 틀을 바꾸면 전체 붕어빵이 바뀝니다.

create-post.dto.ts — 문자열 직접 지정 (기본)
import { IsString, Length } from 'class-validator';

export class CreatePostDto {
  @IsString({ message: 'title은 문자열이어야 합니다' })
  @Length(1, 100, { message: 'title은 1~100자 사이여야 합니다' })
  title: string;

  @IsString({ message: 'content는 문자열이어야 합니다' })
  @Length(1, 5000, { message: 'content는 1~5000자 사이여야 합니다' })
  content: string;
}
// 문제: DTO가 늘어날수록 동일 패턴의 메시지가 반복됨
🧭 기본 에러 메시지

message를 지정하지 않으면 class-validator가 영문 기본 메시지를 반환합니다(예: "title must be a string"). 프론트엔드에서 한국어 메시지가 필요하거나, 일관된 포맷이 필요하면 message를 커스터마이징하는 것이 좋습니다.

2메시지 생성 함수로 일반화

message 옵션에 문자열 대신 함수를 전달할 수 있습니다. 함수는 ValidationArguments를 받아 동적으로 메시지를 생성합니다.

validation-messages.ts — 재사용 가능한 메시지 유틸
import { ValidationArguments } from 'class-validator';

// args.property: 검증 중인 필드 이름 ('title', 'content' 등)
// args.value: 실제로 들어온 값
// args.constraints: 데코레이터에 전달한 제약 [min, max] 등

export const stringValidationMessage = (args: ValidationArguments) =>
  `${args.property}는 문자열이어야 합니다`;

export const lengthValidationMessage = (args: ValidationArguments) => {
  const [min, max] = args.constraints as [number, number];
  return `${args.property}는 ${min}~${max}자 사이여야 합니다`;
};

export const emailValidationMessage = (args: ValidationArguments) =>
  `${args.property}는 유효한 이메일 형식이어야 합니다`;
create-post.dto.ts — 메시지 함수 적용 후
import { IsString, Length } from 'class-validator';
import { stringValidationMessage, lengthValidationMessage } from './validation-messages';

export class CreatePostDto {
  @IsString({ message: stringValidationMessage })
  @Length(1, 100, { message: lengthValidationMessage })
  title: string;

  @IsString({ message: stringValidationMessage })
  @Length(1, 5000, { message: lengthValidationMessage })
  content: string;
}
// 결과: title에 숫자를 넣으면 → "title는 문자열이어야 합니다"
// "abc"를 title에 넣고 200자면 → "title는 1~100자 사이여야 합니다"
💡 메시지 함수를 별도 파일로 분리하는 이유

프로젝트에 DTO 파일이 늘어날수록 메시지 패턴이 중복됩니다. validation-messages.ts 하나에 함수를 모아두면 메시지 문구를 바꿀 때 한 파일만 수정하면 됩니다. 국제화(i18n)가 필요할 때도 이 파일 하나만 교체하면 됩니다.

복습

3✅ 셀프 체크 — 면접관처럼 물어보기

Q 메시지를 일반화(함수화)하면 어떤 점이 좋은가요?

세 가지 이점이 있습니다. ① 중복 제거 — 같은 패턴의 메시지를 DTO마다 복사할 필요 없이 함수 하나로 커버합니다. ② 일관성 — 프로젝트 전체의 에러 메시지가 동일한 포맷을 유지합니다. ③ 유지보수성 — 메시지 문구나 포맷을 바꿀 때 함수 한 곳만 수정하면 모든 DTO에 적용됩니다. i18n 적용도 이 함수만 교체하면 됩니다.

Q ValidationArguments로 어떤 정보를 알 수 있나요?

ValidationArguments에는 네 가지 주요 필드가 있습니다. property: 현재 검증 중인 필드 이름(예: 'title'). value: 요청에서 실제로 들어온 값. constraints: 데코레이터에 전달된 제약 값 배열(예: @Length(1, 100)이면 [1, 100]). object: 검증 중인 DTO 인스턴스 전체. targetName: DTO 클래스 이름. 이를 활용하면 "어떤 필드에서, 어떤 값이, 어떤 규칙을 어겼는가"를 메시지에 동적으로 반영할 수 있습니다.

Q message 옵션에 함수를 넣으면 언제 호출되나요?

검증이 실패했을 때만 호출됩니다. 검증을 통과한 필드에 대해서는 메시지 함수가 전혀 실행되지 않습니다. 따라서 메시지 함수 안에서 비용이 큰 연산을 해도 정상 요청 성능에 영향을 주지 않습니다. 단, 반드시 문자열을 반환해야 하며 비동기 함수(async)는 지원하지 않습니다.
