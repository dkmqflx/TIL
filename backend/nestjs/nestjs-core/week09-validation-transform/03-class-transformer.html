Class Transformer
직렬화 제어 — Exclude · Expose

class-transformer의 @Exclude()·@Expose() 데코레이터로 엔티티 직렬화(응답)에서 민감한 필드를 제거하고, ClassSerializerInterceptor를 전역 적용해 선언 한 번으로 모든 응답을 제어한다.

"

@Exclude()는 선언이고, ClassSerializerInterceptor는 실행이다. 둘 다 있어야 비밀번호가 응답에서 빠진다. 어느 하나만으로는 동작하지 않는다.

  • @Exclude()·@Expose() 데코레이터의 역할과 차이를 설명할 수 있다
  • ClassSerializerInterceptor를 전역 등록해 모든 응답에 직렬화 규칙을 적용할 수 있다
  • toPlainOnly·toClassOnly 방향 옵션이 각각 어떤 경우에 쓰이는지 설명할 수 있다
  • @Expose()로 getter 같은 계산된 필드를 응답에 포함시킬 수 있다

1@Exclude() · @Expose() 직렬화 제어

class-transformer는 클래스 인스턴스를 일반 객체(plain object)로 변환(직렬화)하거나, 그 반대(역직렬화)를 할 때 어떤 필드를 포함할지 데코레이터로 선언할 수 있습니다.

🍔 비유 — 출력 전 민감정보 가리개

병원 영수증 출력 화면에서 환자 주민번호는 가려지고(마스킹), 이름·진료비는 그대로 표시됩니다. 화면(응답)을 위한 마스킹이지, 내부 DB에는 주민번호가 그대로 존재합니다. @Exclude({ toPlainOnly: true })가 이 역할입니다 — 응답(직렬화) 시에만 제거하고, 저장·처리 시에는 그대로 유지합니다.

전체 예제는 code/user.entity.ts에서 확인하세요.

user.entity.ts — @Exclude + @Expose 적용
import { Entity, PrimaryGeneratedColumn, Column } from 'typeorm';
import { Exclude, Expose } from 'class-transformer';

@Entity()
export class User {
  @PrimaryGeneratedColumn()
  id: number;

  @Column({ unique: true })
  email: string;

  @Column()
  nickname: string;

  // toPlainOnly: true → 응답(직렬화) 시에만 제거
  // 저장(역직렬화) 시에는 그대로 포함 → DB 저장 정상 동작
  @Exclude({ toPlainOnly: true })
  @Column()
  password: string;

  // @Expose(): 클래스에 @Exclude()가 기본 적용된 상태에서 명시적으로 포함
  // 또는 getter(계산된 필드)를 응답에 포함시킬 때 사용
  @Expose()
  get displayName(): string {
    return `${this.nickname} <${this.email}>`;
  }
}
⚠️ @Exclude()만으로는 비밀번호가 응답에서 제거되지 않는다

@Exclude()는 선언일 뿐입니다. 실제로 직렬화를 수행하는 ClassSerializerInterceptor가 등록되어 있어야 응답에서 제거됩니다. 또한 핸들러가 TypeORM 엔티티 클래스 인스턴스를 반환해야 합니다 — { ...user }처럼 plain object로 spread하면 인스턴스 정보가 사라져 데코레이터가 동작하지 않습니다.

2ClassSerializerInterceptor 전역 적용

main.ts에서 전역 인터셉터로 등록하면 모든 핸들러의 응답에 class-transformer 직렬화 규칙이 자동 적용됩니다.

main.ts — ClassSerializerInterceptor 전역 등록
import { NestFactory, Reflector } from '@nestjs/core';
import { ClassSerializerInterceptor, ValidationPipe } from '@nestjs/common';
import { AppModule } from './app.module';

async function bootstrap() {
  const app = await NestFactory.create(AppModule);

  app.useGlobalPipes(new ValidationPipe({ whitelist: true, transform: true }));

  // Reflector는 NestJS DI 컨테이너에서 메타데이터를 읽는 도구
  // ClassSerializerInterceptor가 내부적으로 Reflector를 필요로 한다
  app.useGlobalInterceptors(new ClassSerializerInterceptor(app.get(Reflector)));

  await app.listen(3000);
}
bootstrap();

3toPlainOnly · toClassOnly 방향 옵션

방향 옵션으로 직렬화·역직렬화 중 어느 방향에서만 규칙을 적용할지 세밀하게 제어할 수 있습니다.

user.entity.ts — 방향 옵션 비교
import { Exclude } from 'class-transformer';

export class User {
  // toPlainOnly: true
  // → 인스턴스 → plain(JSON 응답) 시에만 제거
  // → plain → 인스턴스(로그인 요청 역직렬화) 시에는 password 포함 → DB 저장 정상
  @Exclude({ toPlainOnly: true })
  password: string;

  // toClassOnly: true (반대 방향)
  // → plain → 인스턴스(역직렬화) 시에만 제거
  // → 인스턴스 → plain(응답) 시에는 포함됨
  // 사용 사례: 외부에서 받은 값을 저장하지 않고 싶을 때
  @Exclude({ toClassOnly: true })
  temporaryToken?: string;

  // 옵션 없음: 양방향 제거 (직렬화·역직렬화 모두에서 제외)
  @Exclude()
  internalFlag: boolean;
}
💡 비밀번호 필드는 반드시 toPlainOnly: true

옵션 없이 @Exclude()만 쓰면 역직렬화(요청 본문 → 클래스 인스턴스) 시에도 제외됩니다. 회원가입·로그인 시 password가 엔티티에 담기지 않아 해싱 후 저장이 되지 않는 문제가 생깁니다. 비밀번호처럼 "응답에서는 숨기되 입력은 받아야 하는" 필드에는 반드시 { toPlainOnly: true }를 사용하세요.

복습

4✅ 셀프 체크 — 면접관처럼 물어보기

Q 응답 JSON에서 비밀번호를 제거하려면 무엇이 필요한가요?

두 가지가 반드시 함께 있어야 합니다. ① @Exclude({ toPlainOnly: true }) — 엔티티 클래스의 password 필드에 선언합니다. 응답(직렬화) 시에만 제거하고, 입력(역직렬화) 시에는 포함합니다. ② ClassSerializerInterceptor — main.ts에 전역 등록해야 실제로 직렬화가 수행됩니다. 또한 핸들러가 { ...user } 같은 plain object가 아니라 TypeORM 엔티티 클래스 인스턴스를 반환해야 데코레이터가 동작합니다.

Q @Expose()는 언제 사용하나요?

두 가지 경우에 씁니다. ① 전체 제외 후 일부 포함 — 클래스에 @Exclude()를 기본 적용하고(excludeExtraneousValues: true), @Expose()로 표시한 필드만 응답에 포함시키는 화이트리스트 방식입니다. ② Getter(계산된 필드) 포함 — TypeScript의 get 접근자는 기본적으로 직렬화에서 제외됩니다. @Expose()를 붙이면 displayName, fullName 같은 계산된 값을 응답에 포함시킬 수 있습니다. 이 방식으로 DB에 저장하지 않는 뷰 전용 필드를 응답에 추가할 수 있습니다.

Q ClassSerializerInterceptor를 사용할 때 핸들러가 plain object를 반환하면 어떻게 되나요?

@Exclude()·@Expose() 데코레이터가 동작하지 않습니다. ClassSerializerInterceptor는 내부적으로 instanceToPlain()을 호출하는데, 이 함수는 클래스 인스턴스의 메타데이터(데코레이터 정보)를 읽습니다. { ...entity }처럼 spread하거나 Object.assign({}, entity)로 복사하면 프로토타입 체인이 끊겨 메타데이터가 사라집니다. 따라서 핸들러에서는 TypeORM Repository의 find()·findOne()이 반환하는 클래스 인스턴스를 그대로 반환해야 합니다.
