Cursor Pagination
오프셋의 한계 → 커서로 해결

전통적인 OFFSET 페이지네이션의 데이터 밀림·성능 문제를 이해하고, 마지막으로 본 항목의 id를 커서로 다음 묶음을 조회하는 패턴을 TypeORM으로 구현한다.

"

오프셋은 페이지 번호를 기억하고, 커서는 어디까지 읽었는지를 기억한다. 실시간으로 데이터가 추가·삭제되는 피드에서는 커서가 훨씬 안정적이다.

  • 오프셋 페이지네이션의 데이터 밀림 문제와 큰 OFFSET의 성능 비용을 설명할 수 있다
  • 커서(cursor)가 "마지막으로 본 항목의 id"임을 이해하고 다음 페이지 조회에 적용할 수 있다
  • where: { id: MoreThan(cursor) } + order: { id: 'ASC' }로 커서 쿼리를 작성할 수 있다
  • take+1 트릭으로 hasMore를 판정하고 응답 메타를 구성할 수 있다

1오프셋(OFFSET) 페이지네이션의 한계

OFFSET 방식은 LIMIT 20 OFFSET 40처럼 "몇 번째 행부터" 를 DB에 알려주는 고전적 방법입니다. 직관적이지만 두 가지 치명적 약점이 있습니다.

⚠️ 오프셋의 두 가지 약점

① 데이터 밀림 (Duplicate / Skip) — 사용자가 2페이지를 보는 동안 다른 사람이 새 글을 올리면, DB 행 순서가 바뀌어 같은 글을 두 번 보거나 한 글을 놓칩니다. 실시간 피드에서 피할 수 없습니다.
② 큰 OFFSET 성능 비용 — OFFSET 10000이면 PostgreSQL이 앞의 10,000 행을 읽고 버립니다. 페이지가 깊어질수록 조회 속도가 선형적으로 느려집니다.

2커서(Cursor) 개념 — 책갈피 방식

커서 페이지네이션은 페이지 번호 대신 마지막으로 받은 항목의 id를 기억합니다. 다음 요청 시 "이 id보다 큰 항목들을 N개 주세요"라고 묻습니다.

🍔 비유 — 책갈피

책을 읽다 멈출 때 페이지 번호가 아니라 책갈피(현재 읽은 문장)를 꽂습니다. 다음에 읽을 때는 책갈피가 꽂힌 바로 그 자리 다음부터 읽으면 됩니다. 책의 앞부분에 새 페이지가 추가되더라도 책갈피 위치는 변하지 않습니다. 커서 페이지네이션의 cursor.after가 바로 이 책갈피입니다.

3MoreThan + order ASC로 커서 쿼리 구현

TypeORM에서 커서 쿼리는 MoreThan 연산자와 ASC 정렬의 조합으로 구현합니다. 두 옵션은 항상 쌍으로 사용해야 합니다.

posts.service.ts — 커서 방식 paginatePosts
import { Injectable } from '@nestjs/common';
import { InjectRepository } from '@nestjs/typeorm';
import { Repository, MoreThan } from 'typeorm';
import { Post } from './entities/post.entity';

@Injectable()
export class PostsService {
  constructor(
    @InjectRepository(Post)
    private readonly postsRepository: Repository<Post>,
  ) {}

  async paginatePosts(cursor: number | undefined, take: number = 20) {
    // take+1 개 조회 → 한 개 더 있으면 hasMore = true
    const results = await this.postsRepository.find({
      where: cursor !== undefined
        ? { id: MoreThan(cursor) }
        : {},
      // MoreThan 은 ASC 와 쌍이어야 한다.
      // DESC + MoreThan 이면 "이미 지나간 행" 을 조회해 빈 결과가 된다.
      order: { id: 'ASC' },
      take: take + 1,
    });

    const hasMore = results.length > take;
    const items = hasMore ? results.slice(0, take) : results;

    // 커서는 슬라이스된 마지막 항목의 id (probe 항목이 아님)
    const nextCursor = items.length > 0
      ? items[items.length - 1].id
      : null;

    return {
      data: items,
      meta: {
        cursor: { after: nextCursor },
        hasMore,
      },
    };
  }
}
💡 take+1 트릭 — 왜 하나 더 조회하나

별도의 count() 쿼리 없이 "다음 페이지 존재 여부"를 판정하는 영리한 방법입니다. N개가 필요하면 N+1개를 요청합니다. 결과가 N+1개이면 더 있다(hasMore = true). N개 이하이면 마지막 페이지(hasMore = false). 응답에는 반드시 N개로 잘라서(slice) 보내야 합니다. nextCursor는 잘린 N개 중 마지막 항목의 id입니다.

4응답 메타 — cursor.after · hasMore

커서 페이지네이션의 응답은 data배열과 함께 다음 요청에 쓸 meta를 포함합니다. 클라이언트는 hasMore가 true일 때 cursor.after를 다음 요청의 쿼리 파라미터로 전달합니다.

응답 JSON 예시
{
  "data": [
    { "id": 43, "title": "첫 글" },
    // ... 총 20개
    { "id": 62, "title": "스물 번째 글" }
  ],
  "meta": {
    "cursor": { "after": 62 },  // 클라이언트가 다음 요청 시 전달할 커서
    "hasMore": true            // false 이면 더 이상 페이지 없음
  }
}
🧭 마지막 페이지 판단 — hasMore: false

결과 개수가 take와 같거나 적으면 hasMore = false입니다. data가 비어 있으면([]) 이미 마지막을 지나쳤습니다. 클라이언트는 hasMore: false를 받으면 "더 보기" 버튼을 숨기거나 무한 스크롤을 멈춥니다.

5✅ 셀프 체크 — 면접관처럼 물어보기

Q 커서 페이지네이션이 오프셋보다 나은 점은 무엇인가요?

두 가지 이점이 있습니다. ① 데이터 안정성 — 커서는 "마지막으로 본 항목의 id"를 기준으로 조회하므로, 조회 사이에 새 글이 삽입되거나 삭제되어도 결과가 밀리거나 중복되지 않습니다. OFFSET은 행 번호 기반이라 삽입·삭제 시 결과가 흔들립니다. ② 성능 — WHERE id > cursor는 인덱스를 타는 범위 스캔이므로 페이지가 깊어져도 일정한 속도를 유지합니다. OFFSET은 앞 행을 모두 읽고 버리므로 깊은 페이지일수록 느려집니다.

Q hasMore = false를 어떻게 판단하나요? 별도 count() 쿼리가 필요한가요?

별도 count() 쿼리는 필요 없습니다. take+1 트릭을 사용합니다. 필요한 항목 수(take)보다 하나 더(take+1) 조회합니다. 결과가 take+1개이면 다음 페이지가 있는 것(hasMore = true)이고, take개 이하이면 마지막 페이지(hasMore = false)입니다. 응답에는 항상 take개만 잘라서 보내야 합니다.

Q MoreThan(cursor)를 쓸 때 왜 order: { id: 'ASC' }가 필수인가요?

MoreThan(cursor)는 "cursor보다 큰 id"를 조건으로 설정합니다. ASC 정렬이어야 cursor 다음 항목들이 올라오는 방향과 쿼리 범위가 일치합니다. DESC로 정렬하면 "cursor보다 큰 id"는 이미 앞 페이지에서 소비한 항목들이므로 결과가 비어 버립니다. 커서 기반 앞 방향 페이지네이션에서 MoreThan + ASC는 항상 쌍으로 사용해야 합니다.
