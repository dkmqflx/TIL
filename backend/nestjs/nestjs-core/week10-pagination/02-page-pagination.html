Page Pagination
페이지 번호로 바로 점프

page·take 파라미터로 skip = (page - 1) × take를 계산하고, TypeORM findAndCount로 전체 개수와 현재 묶음을 한 번에 받아 총 페이지 수를 응답에 포함한다.

"

페이지 번호 방식은 총 페이지 수를 알 수 있어 "3 / 15 페이지" 같은 UI가 가능하다. 커서로는 전체 개수를 집계해야 하므로 추가 쿼리가 필요하다. 관리자 화면·검색 결과처럼 임의 페이지 점프가 필요한 곳에 어울린다.

  • page·take 파라미터를 받아 skip = (page - 1) * take를 계산할 수 있다
  • TypeORM findAndCount가 [items, total]을 한 번에 반환함을 이해한다
  • totalPages = Math.ceil(total / take) 공식으로 총 페이지 수를 계산할 수 있다
  • 페이지 방식과 커서 방식의 적합한 사용 시나리오를 구분할 수 있다

1page · take 파라미터와 skip 계산

페이지 번호 방식은 "page 2, 한 페이지에 20개"라는 직관적인 파라미터로 동작합니다. DB 입장에서는 OFFSET + LIMIT로 번역됩니다.

🍔 비유 — 책의 쪽 번호

사전에서 원하는 단어를 찾을 때 "250쪽으로 바로 펼치기"가 가능합니다. 책갈피(커서)가 없어도 쪽 번호만 알면 어느 페이지든 즉시 열 수 있습니다. 단, 중간에 페이지가 삽입되면 기존 쪽 번호가 전부 밀립니다. 페이지 번호 기반 페이지네이션이 이와 같습니다. 임의 페이지 점프가 필요하지만 실시간 데이터 정합성이 덜 중요한 경우에 적합합니다.

2findAndCount로 total 포함 응답

TypeORM의 findAndCount는 데이터와 전체 개수를 한 번의 쿼리로 반환합니다. 총 페이지 수는 Math.ceil(total / take)로 계산합니다.

posts.service.ts — 페이지 방식 paginatePosts
import { Injectable } from '@nestjs/common';
import { InjectRepository } from '@nestjs/typeorm';
import { Repository } from 'typeorm';
import { Post } from './entities/post.entity';

@Injectable()
export class PostsService {
  constructor(
    @InjectRepository(Post)
    private readonly postsRepository: Repository<Post>,
  ) {}

  async paginatePosts(page: number, take: number = 20) {
    // page는 1-indexed: page=1 → skip=0, page=2 → skip=20
    const skip = (page - 1) * take;

    // findAndCount: SELECT + COUNT(*) 를 한 번에 수행
    // 반환값: [Post[], number] — items 배열과 총 레코드 수
    const [items, total] = await this.postsRepository.findAndCount({
      skip,
      take,
      order: { createdAt: 'DESC' },
    });

    return {
      data: items,
      meta: {
        total,
        page,
        take,
        totalPages: Math.ceil(total / take),
      },
    };
  }
}
💡 findAndCount는 두 쿼리가 아닌 하나의 논리적 요청

TypeORM의 findAndCount는 내부적으로 SELECT … LIMIT … OFFSET …과 SELECT COUNT(*)를 실행하지만, 코드 상에서는 한 번의 호출로 처리됩니다. 직접 find()와 count()를 따로 호출하면 트랜잭션 없이 두 쿼리 사이에 데이터가 변경될 수 있으므로 findAndCount를 사용하는 것이 더 안전합니다.

3응답 구조 — total · totalPages 포함

페이지 방식 응답은 커서 방식과 달리 total과 totalPages를 포함합니다. 이를 통해 "3 / 15 페이지" 같은 UI 표시가 가능합니다.

응답 JSON 예시 — page=3, take=20, total=280
{
  "data": [
    { "id": 181, "title": "41번째 글" },
    // ... 총 20개
    { "id": 200, "title": "60번째 글" }
  ],
  "meta": {
    "total": 280,          // 전체 게시글 수
    "page": 3,             // 현재 페이지 (1-indexed)
    "take": 20,            // 한 페이지 항목 수
    "totalPages": 14       // Math.ceil(280 / 20) = 14
  }
}
🧭 페이지 방식 vs 커서 방식 — 선택 기준

페이지 방식이 적합한 경우: 관리자 대시보드, 검색 결과, "3 / 15 페이지" UI처럼 임의 페이지 점프가 필요하거나 총 개수를 표시해야 할 때.
커서 방식이 적합한 경우: 소셜 피드, 무한 스크롤, 실시간으로 데이터가 추가·삭제되는 환경. 데이터 밀림 없이 일관된 스트림을 제공합니다.

4✅ 셀프 체크 — 면접관처럼 물어보기

Q 페이지 방식이 커서 방식보다 적합한 경우는 어떤 경우인가요?

임의 페이지 점프와 총 페이지 수 표시가 필요한 경우입니다. 예를 들어 관리자 화면에서 "5 / 20 페이지"를 표시하고 7페이지로 바로 이동하는 시나리오, 또는 검색 결과 페이지 번호 목록(1·2·3…10)을 UI에 렌더링해야 하는 경우입니다. 커서 방식은 현재 위치에서 다음으로만 이동 가능하고 전체 개수를 모르므로 이런 UI를 구현하기 어렵습니다.

Q total은 어떻게 구하나요? 별도 쿼리가 필요한가요?

TypeORM의 findAndCount를 사용하면 별도 쿼리가 필요 없습니다. findAndCount(options)는 [items, total] 튜플을 반환합니다. items는 skip·take를 적용한 현재 페이지 항목이고, total은 skip·take를 제외한 전체 레코드 수입니다. 총 페이지 수는 Math.ceil(total / take)로 계산합니다.

Q page=1일 때 skip 값은 얼마인가요? 공식을 설명하세요.

skip = (page - 1) * take이므로 page=1이면 skip = 0입니다. page는 1-indexed(1부터 시작)이기 때문에 (1-1)=0이 되어 처음부터 조회합니다. page=2이면 skip = 1 * take(예: take=20이면 skip=20), page=3이면 skip = 40. SQL로는 OFFSET skip LIMIT take로 번역됩니다.
