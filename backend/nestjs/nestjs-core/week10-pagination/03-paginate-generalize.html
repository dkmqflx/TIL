Pagination 일반화
BasePaginationDto · paginate() · composeFindOptions

커서/페이지 분기를 하나의 paginate() 함수로 추상화하고, BasePaginationDto의 쿼리 키를 composeFindOptions로 TypeORM FindOptions에 자동 매핑한다. 한 번 작성한 유틸이 모든 리스트 엔드포인트에서 재사용된다.

"

엔드포인트마다 페이지네이션 로직을 반복하지 마라. BasePaginationDto로 쿼리를 선언하고, paginate(dto, repository) 한 줄로 커서/페이지 모드 모두를 처리한다. 일반화의 핵심 — 한 번 작성, 어디서나 재사용.

  • BasePaginationDto의 각 프로퍼티(page, where__id__more_than, order__createdAt, take)가 하는 역할을 설명할 수 있다
  • paginate() 함수가 dto.page 존재 여부로 커서/페이지 모드를 분기하는 방식을 이해한다
  • composeFindOptions가 DTO 키의 이중 언더스코어(__) 접두사를 파싱해 where·order로 변환하는 전략을 설명할 수 있다
  • overrideFindOptions로 추가 필터(예: 특정 user의 게시글)를 병합하는 방법을 이해한다

1BasePaginationDto — 공통 쿼리 프로퍼티

BasePaginationDto는 커서/페이지 페이지네이션에서 공통으로 쓰이는 쿼리 파라미터를 하나의 DTO로 선언합니다. 컨트롤러의 @Query()에 이 DTO를 사용하면 모든 파라미터가 자동 검증·변환됩니다.

🍔 비유 — 만능 페이지네이션 어댑터

콘센트 어댑터 하나로 한국·미국·유럽 플러그를 모두 꽂을 수 있듯, BasePaginationDto는 어떤 엔드포인트에 끼워도 커서와 페이지 두 가지 페이지네이션 모드를 모두 지원합니다. 각 엔드포인트마다 페이지네이션 로직을 다시 작성할 필요 없이, paginate(dto, repository) 한 줄로 끝납니다.

전체 예제는 code/base-pagination.dto.ts에서 확인하세요.

base-pagination.dto.ts — 공통 쿼리 DTO
import { IsNumber, IsOptional, IsString } from 'class-validator';

export class BasePaginationDto {
  // 커서 모드: 마지막으로 받은 항목의 id
  // where__id__more_than → composeFindOptions → { id: MoreThan(value) }
  @IsNumber()
  @IsOptional()
  where__id__more_than?: number;

  // 페이지 모드: 1-indexed 페이지 번호
  // page가 존재하면 커서 대신 skip = (page-1)*take 계산
  @IsNumber()
  @IsOptional()
  page?: number;

  // 공통 정렬 방향. 커서 모드에서는 반드시 ASC여야 MoreThan이 올바르게 동작한다.
  @IsString()
  @IsOptional()
  order__createdAt?: 'ASC' | 'DESC';

  // 한 번에 가져올 항목 수 (기본 20)
  @IsNumber()
  @IsOptional()
  take: number = 20;
}
🧭 이중 언더스코어(__) 키 규칙

프로퍼티 이름에 __(언더스코어 2개)를 구분자로 사용합니다. where__id__more_than은 세 조각으로 파싱됩니다: 접두사 where, 필드명 id, 연산자 more_than. order__createdAt은 두 조각: order, createdAt. composeFindOptions가 이 규칙으로 DTO를 TypeORM FindOptions로 변환합니다.

2공통 paginate() 함수 — 커서/페이지 분기

paginate()는 dto.page의 존재 여부로 페이지 모드와 커서 모드를 분기합니다. 두 모드 모두 composeFindOptions를 통해 DTO를 TypeORM 옵션으로 변환합니다.

전체 구현은 code/paginate.ts에서 확인하세요.

paginate.ts — 공통 paginate 함수 (핵심 골격)
import { Repository, FindManyOptions, MoreThan } from 'typeorm';
import { BasePaginationDto } from './base-pagination.dto';

export async function paginate<T>(
  dto: BasePaginationDto,
  repository: Repository<T>,
  overrideFindOptions: FindManyOptions<T> = {},
) {
  const baseOptions = composeFindOptions<T>(dto);

  // page가 있으면 페이지 모드 (skip 방식)
  if (dto.page !== undefined) {
    const take = dto.take ?? 20;
    const skip = (dto.page - 1) * take;

    const [items, total] = await repository.findAndCount({
      ...baseOptions,
      ...overrideFindOptions,
      skip,
      take,
    });

    return {
      data: items,
      meta: { total, page: dto.page, take, totalPages: Math.ceil(total / take) },
    };
  }

  // page가 없으면 커서 모드 (take+1 트릭)
  const take = dto.take ?? 20;
  const results = await repository.find({
    ...baseOptions,
    ...overrideFindOptions,
    take: take + 1,
  });

  const hasMore = results.length > take;
  const items = hasMore ? results.slice(0, take) : results;
  const lastItem = items[items.length - 1] as (T & { id: number }) | undefined;

  return {
    data: items,
    meta: { cursor: { after: lastItem?.id ?? null }, hasMore },
  };
}

3composeFindOptions — DTO를 TypeORM 옵션으로 변환

composeFindOptions는 DTO 프로퍼티 이름을 파싱해 TypeORM FindOptions의 where·order 객체로 변환합니다.

composeFindOptions 파싱 전략
function composeFindOptions<T>(dto: BasePaginationDto): FindManyOptions<T> {
  const where: Record<string, unknown> = {};
  const order: Record<string, string> = {};

  for (const [key, value] of Object.entries(dto)) {
    if (value === undefined || value === null) continue;

    if (key.startsWith('where__')) {
      // "where__id__more_than" → ['where', 'id', 'more_than']
      const [, field, op] = key.split('__');
      where[field] = op === 'more_than'
        ? MoreThan(value as number)
        : value;  // 연산자 없으면 직접 할당 (Equal)
    } else if (key.startsWith('order__')) {
      // "order__createdAt" → ['order', 'createdAt']
      const [, field] = key.split('__');
      order[field] = value as string;
    }
  }

  return {
    where: where as FindManyOptions<T>['where'],
    order: order as FindManyOptions<T>['order'],
  };
}

4overrideFindOptions — 추가 필터 병합

특정 사용자의 게시글만 조회하는 경우처럼, 공통 BasePaginationDto 외에 엔드포인트별 고정 조건을 추가할 수 있습니다. overrideFindOptions로 전달하면 composeFindOptions 결과에 spread로 병합됩니다.

posts.service.ts — overrideFindOptions 활용
import { Injectable } from '@nestjs/common';
import { InjectRepository } from '@nestjs/typeorm';
import { Repository } from 'typeorm';
import { Post } from './entities/post.entity';
import { BasePaginationDto } from './dto/base-pagination.dto';
import { paginate } from './utils/paginate';

@Injectable()
export class PostsService {
  constructor(
    @InjectRepository(Post)
    private readonly postsRepository: Repository<Post>,
  ) {}

  // 전체 게시글 — 공통 DTO만으로 충분
  async paginatePosts(dto: BasePaginationDto) {
    return paginate(dto, this.postsRepository);
  }

  // 특정 사용자의 게시글 — overrideFindOptions로 추가 조건 병합
  async paginatePostsByUser(dto: BasePaginationDto, userId: number) {
    return paginate(dto, this.postsRepository, {
      where: { author: { id: userId } },
    });
  }
}
💡 한 번 일반화하면 모든 리스트 엔드포인트에서 재사용

Posts뿐만 아니라 Comments, Followers, Notifications 등 리스트를 반환하는 모든 엔드포인트에서 같은 paginate()를 씁니다. 각 서비스는 paginate(dto, thisRepository, { where: { ... } }) 한 줄만 추가하면 됩니다. 커서/페이지 전환, total 계산, hasMore 판정이 모두 공통 함수 안에 캡슐화되어 있기 때문입니다.

5✅ 셀프 체크 — 면접관처럼 물어보기

Q 페이지네이션을 일반화하면 얻는 이점은 무엇인가요?

세 가지 이점이 있습니다. ① 코드 중복 제거 — 커서 분기, take+1 트릭, findAndCount 호출, 메타 구성 로직을 엔드포인트마다 반복하지 않고 한 함수로 관리합니다. ② 일관된 응답 형태 — 모든 리스트 엔드포인트가 같은 { data, meta } 구조를 반환하므로 클라이언트가 응답을 예측할 수 있습니다. ③ 버그 단일 수정 — 커서 처리에 버그가 있으면 paginate.ts 한 곳만 고치면 모든 엔드포인트에 반영됩니다.

Q where__id__more_than 같은 DTO 키를 TypeORM FindOptions로 파싱하는 전략을 설명하세요.

composeFindOptions에서 Object.entries(dto)로 DTO의 키-값을 순회합니다. 키를 split('__')으로 분리해 첫 조각으로 접두사를 판별합니다. where__로 시작하면 나머지를 [field, op]로 분해합니다. op가 more_than이면 TypeORM의 MoreThan(value)를 생성하고, 연산자가 없으면 값을 직접 할당합니다(Equal). order__로 시작하면 [field]를 키로 정렬 방향 값을 order 객체에 넣습니다. 최종적으로 { where, order }를 FindManyOptions로 반환합니다.

Q paginate()가 커서 모드와 페이지 모드를 어떻게 구분하나요?

dto.page의 존재 여부로 분기합니다. dto.page !== undefined이면 페이지 모드로 진입해 skip = (dto.page - 1) * take를 계산하고 findAndCount로 total을 포함한 응답을 반환합니다. dto.page가 없으면 커서 모드로 진입해 take+1을 조회하고 hasMore를 판정합니다. 두 모드 모두 composeFindOptions로 DTO를 TypeORM 옵션으로 변환하고, overrideFindOptions를 spread로 병합합니다.
