Transaction · Interceptor
All-or-Nothing을 코드로

QueryRunner로 트랜잭션 경계를 직접 제어하고, NestInterceptor로 그 경계를 핸들러를 감싸는 방식으로 자동화한다. 가장 실무적인 횡단 관심사 패턴.

"

트랜잭션은 All-or-Nothing. Interceptor는 핸들러 전·후를 감싸는 레이어. 둘을 합치면 — 핸들러가 성공하면 commit, 실패하면 rollback, 항상 release. 커넥션 누수 없이 트랜잭션 경계가 자동화된다.

  • 트랜잭션의 원자성(Atomicity) 개념과 필요한 상황을 설명할 수 있다
  • QueryRunner의 connect → startTransaction → commit/rollback → release 라이프사이클을 순서대로 작성할 수 있다
  • NestInterceptor의 intercept(context, next)와 RxJS next.handle().pipe() 구조를 이해한다
  • TransactionInterceptor가 catchError·tap으로 rollback/commit을 자동화하는 방식을 설명할 수 있다

1트랜잭션 — All-or-Nothing(원자성)

트랜잭션은 여러 DB 작업을 하나의 논리 단위로 묶어, 모두 성공하거나 모두 실패하도록 보장합니다. 일부만 성공한 채 멈추는 상황을 원천 차단합니다.

🍔 비유 — 은행 계좌 이체

A 계좌에서 1만 원을 출금하고 B 계좌에 입금하는 두 쿼리가 있습니다. 출금은 됐는데 입금이 실패한다면? 돈이 허공에 사라집니다. 트랜잭션은 두 쿼리를 묶어 둘 다 성공해야 확정(commit)하고, 하나라도 실패하면 출금도 되돌립니다(rollback). SNS에서는 이미지 파일 레코드 저장과 포스트 레코드 저장이 이 관계입니다.

2QueryRunner — 수동 트랜잭션 라이프사이클

TypeORM에서 트랜잭션을 직접 제어할 때는 DataSource에서 QueryRunner를 꺼냅니다. 라이프사이클은 반드시 이 순서를 지켜야 합니다.

QueryRunner 수동 트랜잭션 골격
const qr = dataSource.createQueryRunner();

await qr.connect();           // 1. 커넥션 풀에서 연결 획득
await qr.startTransaction();  // 2. 트랜잭션 시작 (BEGIN)

try {
  await qr.manager.save(imageEntity);   // 3. 작업 실행
  await qr.manager.save(postEntity);
  await qr.commitTransaction();          // 4a. 성공 → commit (COMMIT)
} catch (err) {
  await qr.rollbackTransaction();        // 4b. 실패 → rollback (ROLLBACK)
  throw err;
} finally {
  await qr.release();                    // 5. 항상 커넥션 반납
}
⚠️ release()는 반드시 finally에서

release()를 빠뜨리거나 try 블록 안에 넣으면 예외 발생 시 커넥션이 풀로 돌아오지 않아 커넥션 누수가 발생합니다. 수십 개의 누수가 쌓이면 DB 커넥션 풀이 소진되어 앱 전체가 멈춥니다. 항상 finally { await qr.release() }로 작성하세요.

3Interceptor — 핸들러를 감싸는 AOP 레이어

NestInterceptor는 컨트롤러 핸들러의 실행 전·후에 로직을 끼워 넣는 AOP(관점 지향 프로그래밍) 도구입니다. intercept(context, next)를 구현하고, next.handle()이 반환하는 Observable에 RxJS 연산자를 파이프합니다.

Interceptor 기본 구조 — 로깅 예시
import {
  Injectable, NestInterceptor, ExecutionContext, CallHandler,
} from '@nestjs/common';
import { Observable } from 'rxjs';
import { tap } from 'rxjs/operators';

@Injectable()
export class LoggingInterceptor implements NestInterceptor {
  intercept(context: ExecutionContext, next: CallHandler): Observable<unknown> {
    const start = Date.now();

    return next.handle().pipe(
      tap(() => {
        // 핸들러 완료 후 실행
        console.log(`처리 시간: ${Date.now() - start}ms`);
      }),
    );
  }
}
💡 next.handle()을 호출해야 핸들러가 실행된다

next.handle()은 다음 단계(최종적으로 컨트롤러 핸들러)를 실행하는 Observable입니다. 이를 호출하지 않으면 핸들러 자체가 실행되지 않습니다. .pipe(tap(...), catchError(...))로 완료·오류 시점에 각각 로직을 끼워 넣습니다.

4Transaction Interceptor — 트랜잭션 경계 자동화

Interceptor 안에서 QueryRunner를 생성하고 요청 객체에 붙이면, 핸들러가 성공/실패할 때 자동으로 commit/rollback됩니다. 핸들러는 @QueryRunner() 커스텀 데코레이터로 꺼내 씁니다.

code/transaction.interceptor.ts — 전체 구현
import {
  Injectable, NestInterceptor, ExecutionContext, CallHandler,
} from '@nestjs/common';
import { Observable, catchError, tap, throwError } from 'rxjs';
import { DataSource, QueryRunner } from 'typeorm';

@Injectable()
export class TransactionInterceptor implements NestInterceptor {
  constructor(private readonly dataSource: DataSource) {}

  async intercept(
    context: ExecutionContext,
    next: CallHandler,
  ): Promise<Observable<unknown>> {
    const req = context.switchToHttp().getRequest();

    const qr: QueryRunner = this.dataSource.createQueryRunner();
    await qr.connect();
    await qr.startTransaction();

    req.queryRunner = qr;  // 커스텀 데코레이터로 핸들러에 주입

    return next.handle().pipe(
      catchError(async (err) => {
        await qr.rollbackTransaction();
        await qr.release();
        return throwError(() => err);
      }),
      tap(async () => {
        await qr.commitTransaction();
        await qr.release();
      }),
    );
  }
}

5✅ 셀프 체크 — 면접관처럼 물어보기

Q 트랜잭션이 꼭 필요한 상황을 예를 들어 설명하세요.

두 가지 이상의 DB 쓰기가 논리적으로 하나여야 할 때 필요합니다. 예: 이미지 파일 레코드 저장(image 테이블)과 포스트 레코드 저장(post 테이블)을 동시에 해야 하는 경우. 이미지 레코드는 저장됐는데 포스트 저장에 실패하면 고아(orphan) 이미지 레코드가 남습니다. 트랜잭션으로 묶으면 포스트 저장 실패 시 이미지 레코드도 자동으로 롤백됩니다.

Q rollbackTransaction()은 언제 호출되나요?

핸들러(또는 그 안의 DB 쿼리)가 예외를 던질 때 호출됩니다. TransactionInterceptor에서는 RxJS catchError가 Observable 스트림의 에러를 잡아 rollbackTransaction()과 release()를 순서대로 실행한 뒤 throwError(() => err)로 에러를 다시 던집니다. 이를 통해 NestJS의 Exception Filter가 원래 에러를 받아 처리할 수 있습니다.

Q Interceptor와 Middleware의 차이는 무엇인가요?

두 가지 핵심 차이가 있습니다. ① 실행 위치: Middleware는 라우팅 전 가장 먼저 실행되고, Interceptor는 Guard·Pipe 이후, 핸들러 직전에 실행됩니다. ② 응답 접근: Middleware는 응답 객체에 직접 접근하지만, Interceptor는 next.handle() Observable로 핸들러 반환값을 변환하거나 에러를 처리할 수 있습니다. 트랜잭션처럼 핸들러 실행 전·후 양쪽을 제어해야 할 때 Interceptor가 적합합니다.
