Exception Filter · Middleware
에러 응답 통일과 입구 검문소

@Catch(HttpException)으로 에러 응답 형태를 통일하고, NestMiddleware로 요청이 컨트롤러에 닿기 전 공통 처리(로깅·파싱)를 삽입한다.

"

Middleware는 요청이 들어오는 입구 검문소, Exception Filter는 에러가 나가는 출구 응급실. 둘은 라이프사이클에서 정반대 지점에 위치하지만, 함께 쓰면 요청·응답 전체를 일관되게 제어할 수 있다.

  • @Catch(HttpException)과 ExceptionFilter 인터페이스로 에러 응답 구조를 통일할 수 있다
  • app.useGlobalFilters() 또는 APP_FILTER로 전역 등록하는 두 방식의 차이를 안다
  • NestMiddleware.use(req, res, next)를 구현하고 configure(consumer)로 라우트에 적용할 수 있다
  • Middleware, Guard, Interceptor, Exception Filter가 요청 라이프사이클에서 실행되는 순서를 설명할 수 있다

1Exception Filter — 에러 응답 형태 통일

NestJS 기본 에러 응답은 { statusCode, message, error } 형태입니다. 하지만 실무에서는 타임스탬프·경로 같은 디버그 정보를 함께 반환하거나, 프로젝트 전체에서 동일한 에러 포맷을 강제하고 싶습니다. ExceptionFilter로 원하는 형태를 만들 수 있습니다.

code/http-exception.filter.ts
import {
  ExceptionFilter, Catch, ArgumentsHost, HttpException,
} from '@nestjs/common';
import { Request, Response } from 'express';

@Catch(HttpException)
export class HttpExceptionFilter implements ExceptionFilter {
  catch(exception: HttpException, host: ArgumentsHost): void {
    const ctx = host.switchToHttp();
    const response = ctx.getResponse<Response>();
    const request = ctx.getRequest<Request>();
    const status = exception.getStatus();
    const exceptionResponse = exception.getResponse();

    const message =
      typeof exceptionResponse === 'string'
        ? exceptionResponse
        : (exceptionResponse as Record<string, unknown>).message ?? exception.message;

    response.status(status).json({
      statusCode: status,
      timestamp: new Date().toISOString(),
      path: request.url,
      message,
    });
  }
}
💡 @Catch()에 클래스를 넣으면 그 타입만 잡는다

@Catch(HttpException)은 HttpException과 그 하위 클래스(NotFoundException, BadRequestException 등)를 모두 잡습니다. 인수를 비우면(@Catch()) 모든 예외를 잡습니다. DB 에러처럼 HttpException이 아닌 예외도 처리하려면 @Catch()를 쓰고 내부에서 타입을 분기하면 됩니다.

2전역 등록 — useGlobalFilters vs APP_FILTER

Exception Filter를 전역으로 등록하는 방법은 두 가지입니다.

방법 1 — main.ts에서 수동 등록
// main.ts
const app = await NestFactory.create(AppModule);
app.useGlobalFilters(new HttpExceptionFilter());
await app.listen(3000);
방법 2 — APP_FILTER 프로바이더 (DI 가능)
// app.module.ts
import { APP_FILTER } from '@nestjs/core';

@Module({
  providers: [
    {
      provide: APP_FILTER,
      useClass: HttpExceptionFilter,
    },
  ],
})
export class AppModule {}
🧭 Filter는 라이프사이클 마지막 — Interceptor와 역할 구분

Exception Filter는 요청 라이프사이클에서 가장 마지막에 실행됩니다. 핸들러나 Interceptor에서 던진 예외가 필터로 흘러옵니다. Interceptor는 핸들러의 정상 흐름을 가공(tap)하거나 에러를 변환(catchError)할 수 있는 반면, Exception Filter는 최종 에러 응답 형태를 담당합니다. 두 역할은 다릅니다.

3Middleware — 요청 입구의 검문소

Middleware는 요청이 라우팅되기 전, NestJS 파이프라인의 가장 앞 단계에서 실행됩니다. Express 미들웨어와 동일한 (req, res, next) 시그니처를 사용합니다. 대표 용도는 로깅, 요청 파싱, CORS, 속도 제한입니다.

🍔 비유 — 입구 검문소 vs 출구 응급실

Middleware = 입구 검문소: 건물에 들어오는 모든 사람을 검사합니다. 통과하면 안으로 들어갑니다. Exception Filter = 출구 응급실: 안에서 다쳐서 나오는 사람을 처치합니다. 검문소를 통과 못 하면 응급실에 도달할 일도 없고, 응급실은 정상적으로 나가는 사람에게는 관여하지 않습니다. 둘은 라이프사이클에서 정반대 위치에 있습니다.

logging.middleware.ts — 요청 로깅 미들웨어
import { Injectable, NestMiddleware } from '@nestjs/common';
import { Request, Response, NextFunction } from 'express';

@Injectable()
export class LoggingMiddleware implements NestMiddleware {
  use(req: Request, res: Response, next: NextFunction): void {
    console.log(`[${new Date().toISOString()}] ${req.method} ${req.url}`);
    next();   // 반드시 next()를 호출해야 다음 단계로 진행된다
  }
}

// app.module.ts — 라우트에 적용
export class AppModule implements NestModule {
  configure(consumer: MiddlewareConsumer): void {
    consumer
      .apply(LoggingMiddleware)
      .forRoutes('*');  // 모든 라우트에 적용
  }
}

4✅ 셀프 체크 — 면접관처럼 물어보기

Q Exception Filter와 Interceptor의 에러 처리 차이를 설명하세요.

두 가지 관점의 차이가 있습니다. ① 목적: Interceptor의 catchError는 에러를 잡아 다른 Observable로 변환하거나 부가 작업(rollback 등)을 수행하고 에러를 다시 던집니다. Exception Filter는 최종적으로 HTTP 응답 형태를 결정합니다. ② 실행 순서: Interceptor의 catchError가 먼저 실행되고, 에러가 rethrow되면 Exception Filter가 받아 응답합니다. 트랜잭션 롤백은 Interceptor에서, 에러 포맷 통일은 Exception Filter에서 담당하는 것이 역할 분리의 핵심입니다.

Q Middleware는 요청 라이프사이클의 어디에서 동작하나요?

Middleware는 요청 라이프사이클에서 가장 먼저 실행됩니다. Guard, Interceptor, Pipe, Handler 모두 Middleware 이후에 실행됩니다. 이 때문에 인증 여부와 무관하게 모든 요청에 적용되어야 하는 로깅, 요청 파싱, CORS 처리 등에 적합합니다. next()를 호출하지 않으면 요청이 다음 단계로 진행되지 않습니다.

Q useGlobalFilters()와 APP_FILTER 중 어떤 방법이 DI를 지원하나요?

APP_FILTER 방식이 DI를 지원합니다. main.ts의 useGlobalFilters(new HttpExceptionFilter())는 필터를 직접 인스턴스화하므로 NestJS DI 컨테이너 밖에서 생성됩니다. 따라서 필터 내부에서 다른 서비스(ConfigService, 로거 등)를 주입받을 수 없습니다. 반면 AppModule의 providers에 { provide: APP_FILTER, useClass: HttpExceptionFilter }를 등록하면 NestJS가 인스턴스를 관리하므로 DI가 정상 작동합니다.
