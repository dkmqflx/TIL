WebSocket & Socket.IO
실시간 양방향 채팅

HTTP의 요청-응답 방식에서 벗어나, @WebSocketGateway와 @SubscribeMessage로 상시 연결·양방향 채팅 서버를 구축한다. Room으로 채팅방을 분리하고, Gateway로 그룹 broadcasting을 구현한다.

"

HTTP는 클라이언트가 먼저 말해야 서버가 답한다. WebSocket은 전화 통화처럼 한 번 연결되면 누구든 언제든 먼저 말할 수 있다. 채팅·알림·실시간 대시보드는 모두 이 연결 방식이 필요하다.

  • WebSocket이 HTTP와 어떻게 다른지(핸드셰이크·상시 연결·양방향)를 설명할 수 있다
  • @WebSocketGateway · @WebSocketServer · @SubscribeMessage의 역할을 각각 이해한다
  • socket.join(room)과 server.to(room).emit()으로 채팅방 분리·broadcasting을 구현할 수 있다
  • Chat·Message 엔티티 구조와 메시지 저장·전송 흐름을 그릴 수 있다

1WebSocket vs HTTP — 연결 방식이 다르다

HTTP는 요청 → 응답 → 연결 종료의 일회성 패턴입니다. 서버는 클라이언트가 먼저 요청하기 전까지 아무것도 보낼 수 없습니다. WebSocket은 최초 HTTP 핸드셰이크(Upgrade)로 연결을 맺은 뒤, 연결을 끊기 전까지 양쪽이 언제든지 메시지를 주고받을 수 있습니다.

🍔 비유 — 편지 vs 전화 통화

HTTP는 편지입니다. 보낼 때마다 봉투에 주소를 쓰고(요청), 상대가 답장을 써서 부쳐야(응답) 오며, 대화 한 번에 왕복 시간이 걸립니다. 반면 WebSocket은 전화 통화입니다. 한 번 연결(핸드셰이크)하면 통화가 끊기기 전까지 양쪽이 실시간으로 말할 수 있습니다. 채팅은 전화가 필요하지 편지로 하지 않습니다.

2@WebSocketGateway — NestJS의 WebSocket 진입점

NestJS에서 WebSocket 서버는 Gateway라는 특수 클래스로 구현합니다. @WebSocketGateway() 데코레이터를 달면 NestJS가 Socket.IO 서버를 만들어 줍니다. 핸들러는 @SubscribeMessage('이벤트명')으로 등록하고, 서버 자체 인스턴스는 @WebSocketServer()로 주입받습니다.

code/chats.gateway.ts
import {
  WebSocketGateway, WebSocketServer,
  SubscribeMessage, MessageBody, ConnectedSocket,
} from '@nestjs/websockets';
import { Server, Socket } from 'socket.io';

@WebSocketGateway({
  namespace: 'chats',
  cors: { origin: '*' },
})
export class ChatsGateway {
  // Socket.IO 서버 인스턴스 자동 주입
  @WebSocketServer()
  server: Server;

  // 클라이언트가 'send_message' 이벤트를 emit하면 실행
  @SubscribeMessage('send_message')
  async sendMessage(
    @MessageBody() data: { chatId: number; content: string; authorId: number },
    @ConnectedSocket() client: Socket,
  ) {
    const room = `chat-${data.chatId}`;

    // 해당 Room의 모든 소켓에 receive_message 이벤트 broadcast
    this.server.to(room).emit('receive_message', {
      chatId: data.chatId,
      content: data.content,
      authorId: data.authorId,
    });
  }
}
💡 @WebSocketServer() vs client.emit()

@WebSocketServer() server는 서버 전체 인스턴스입니다. server.to(room).emit()은 특정 Room의 모든 클라이언트에게 보냅니다. 반면 client.emit()은 해당 소켓에만 보냅니다. 채팅방 broadcast에는 server.to(room)을 사용하세요.

3Room — 채팅방 분리와 Broadcasting

Socket.IO의 Room은 소켓을 논리 그룹으로 묶는 기능입니다. 클라이언트가 enter_chat 이벤트를 보내면 서버에서 client.join(room)으로 해당 Room에 합류시키고, 이후 그 Room으로 보내는 모든 이벤트는 해당 채팅방 참가자에게만 전달됩니다.

채팅방 입장 핸들러
@SubscribeMessage('enter_chat')
async enterChat(
  @MessageBody() data: { chatId: number },
  @ConnectedSocket() client: Socket,
) {
  const room = `chat-${data.chatId}`;
  await client.join(room);          // 해당 소켓을 Room에 합류
  client.emit('enter_chat', { message: `방 ${data.chatId}에 입장` });
}

4Chat · Message 엔티티 + 메시지 저장·전송

실시간 채팅에서도 메시지는 DB에 저장되어야 합니다. Chat(채팅방)과 Message(메시지) 두 엔티티를 만들고, Gateway의 sendMessage 핸들러 안에서 ① DB 저장 → ② Room emit 순서로 처리합니다.

Chat / Message 엔티티 골격
// chat.entity.ts
@Entity()
export class Chat {
  @PrimaryGeneratedColumn() id: number;
  @OneToMany(() => Message, (m) => m.chat) messages: Message[];
  @CreateDateColumn() createdAt: Date;
}

// message.entity.ts
@Entity()
export class Message {
  @PrimaryGeneratedColumn() id: number;
  @Column() content: string;
  @ManyToOne(() => Chat, (c) => c.messages) chat: Chat;
  @ManyToOne(() => User) author: User;
  @CreateDateColumn() createdAt: Date;
}

// Gateway의 send_message 핸들러 (저장 + emit)
@SubscribeMessage('send_message')
async sendMessage(
  @MessageBody() data: { chatId: number; content: string; authorId: number },
) {
  // 1. DB 저장
  const saved = await this.messageService.saveMessage(data);

  // 2. Room broadcast
  this.server.to(`chat-${data.chatId}`).emit('receive_message', saved);
}
🧭 Socket.IO vs 순수 WebSocket

NestJS는 순수 ws와 socket.io 두 어댑터를 모두 지원합니다. Socket.IO는 자동 재연결, Room, 네임스페이스, 폴링 폴백, 브라우저 호환성을 기본 제공하므로 실무에서 훨씬 많이 사용됩니다. @nestjs/platform-socket.io를 의존성으로 추가하고 @WebSocketGateway()에 설정하면 됩니다.

5✅ 셀프 체크 — 면접관처럼 물어보기

Q WebSocket이 HTTP와 근본적으로 다른 점은 무엇인가요?

가장 큰 차이는 연결 유지 방식과 통신 방향입니다. HTTP는 클라이언트가 요청을 보낼 때마다 새 연결을 맺고 응답 후 종료하는 단방향·일회성 패턴입니다. WebSocket은 최초 HTTP 핸드셰이크(Upgrade)로 연결을 맺은 뒤 클라이언트와 서버가 모두 언제든 먼저 메시지를 보낼 수 있는 상시 양방향 채널을 유지합니다. 이 차이 때문에 실시간 채팅, 알림, 주가 업데이트처럼 서버가 먼저 데이터를 푸시해야 하는 기능에 WebSocket이 필수입니다.

Q Socket.IO의 Room을 사용하는 이유가 무엇인가요?

Room이 없으면 server.emit()이 연결된 모든 클라이언트에게 이벤트를 전송합니다. 채팅 앱에서 1번 방의 메시지가 2번 방의 사용자에게도 전달되면 안 되죠. Room은 소켓을 논리 그룹으로 묶어 server.to('chat-1').emit()이 해당 방 참가자에게만 전달되도록 격리합니다. client.join(room)으로 입장하고 client.leave(room)으로 퇴장하는 것으로 간단히 관리할 수 있습니다.

Q @WebSocketServer()로 주입받는 server와 client 객체의 차이는?

@WebSocketServer() server는 Socket.IO 서버 전체 인스턴스로, 모든 연결된 소켓에 또는 특정 Room에 이벤트를 보낼 수 있습니다. @ConnectedSocket() client는 현재 이벤트를 보낸 특정 소켓(클라이언트) 하나에 대한 참조입니다. client.emit()은 그 클라이언트에게만, client.broadcast.to(room).emit()은 본인을 제외한 Room 멤버에게, server.to(room).emit()은 본인 포함 Room 멤버 전체에게 전송합니다.
