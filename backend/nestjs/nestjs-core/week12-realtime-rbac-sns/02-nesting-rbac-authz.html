Authorization(소유권 Guard)

Posts 하위에 Comments 모듈을 중첩해 계층 라우트를 구성하고, @Roles + RolesGuard로 역할 기반 접근 제어(RBAC)를 구현한다. 전역 Private 원칙과 @IsPublic() 패턴, 그리고 소유권 기반 IsMineOrAdmin Guard까지 완성한다.

"

인증(Authentication)은 "당신이 누구냐"를 확인하고, 인가(Authorization)는 "당신이 이것을 할 수 있느냐"를 결정한다. RBAC는 역할로, 소유권 Guard는 "내 것인가"로 — 두 가지를 함께 써야 실무 인가가 완성된다.

  • 모듈 네스팅으로 posts/:postId/comments 형태의 중첩 라우트를 구성하는 방법을 안다
  • @Roles(RoleEnum.ADMIN) 데코레이터와 RolesGuard가 Reflector를 통해 메타데이터를 읽는 방식을 설명할 수 있다
  • 전역 AuthGuard를 기본 적용하고 @IsPublic()으로 공개 라우트만 예외 처리하는 패턴의 목적을 안다
  • IsPostMineOrAdminGuard가 역할 기반과 소유권 기반 인가를 결합하는 방식을 이해한다

1모듈 네스팅 — Posts 하위에 Comments 모듈

댓글(Comment)은 게시글(Post)에 종속되는 자원이므로 posts/:postId/comments처럼 중첩 라우트로 표현하는 것이 REST 원칙에 맞습니다. NestJS에서는 Comments 모듈을 독립 모듈로 만들되, 컨트롤러의 @Controller('posts/:postId/comments')와 Posts 모듈에서의 imports로 라우트 계층을 구성합니다.

중첩 라우트 컨트롤러 골격
// comments/comments.controller.ts
@Controller('posts/:postId/comments')
export class CommentsController {
  @Get()
  findAll(@Param('postId', ParseIntPipe) postId: number) {
    return this.commentsService.findAllByPost(postId);
  }

  @Post()
  create(
    @Param('postId', ParseIntPipe) postId: number,
    @Body() dto: CreateCommentDto,
  ) {
    return this.commentsService.create(postId, dto);
  }
}

// posts/posts.module.ts — CommentsModule을 imports에 포함
@Module({
  imports: [CommentsModule],
  controllers: [PostsController],
  providers: [PostsService],
})
export class PostsModule {}
💡 중첩 라우트에서 :postId 파라미터 전달

@Controller('posts/:postId/comments')를 쓰면 해당 컨트롤러의 모든 핸들러에서 @Param('postId')로 부모 자원의 ID를 받을 수 있습니다. Comments 서비스에서 DB를 조회할 때 where: { post: { id: postId } }로 필터링하면 정확히 해당 게시글의 댓글만 반환됩니다.

2RBAC — @Roles 데코레이터 + RolesGuard

RBAC(Role-Based Access Control)는 사용자의 역할(role)에 따라 라우트 접근을 제어합니다. 구현은 두 부분으로 나뉩니다: ① @Roles() 커스텀 데코레이터가 핸들러에 역할 메타데이터를 붙이고, ② RolesGuard가 요청 시 Reflector로 그 메타데이터를 읽어 request.user.role과 비교합니다.

code/roles.guard.ts — Reflector 기반 RolesGuard
import { Injectable, CanActivate, ExecutionContext } from '@nestjs/common';
import { Reflector } from '@nestjs/core';

export enum RoleEnum { USER = 'USER', ADMIN = 'ADMIN' }
export const ROLES_KEY = 'roles';
export const Roles = (...roles: RoleEnum[]) =>
  SetMetadata(ROLES_KEY, roles);

@Injectable()
export class RolesGuard implements CanActivate {
  constructor(private readonly reflector: Reflector) {}

  canActivate(context: ExecutionContext): boolean {
    // 핸들러 또는 클래스에 붙은 @Roles() 메타데이터 읽기
    const required = this.reflector.getAllAndOverride<RoleEnum[]>(
      ROLES_KEY,
      [context.getHandler(), context.getClass()],
    );

    if (!required || required.length === 0) return true;  // @Roles 없으면 통과

    const { user } = context.switchToHttp().getRequest();
    return !!user && required.includes(user.role);
  }
}

3전역 Private 기본 + @IsPublic() 예외

"모든 라우트를 기본적으로 인증 필요"로 만들고, 공개해야 하는 라우트에만 @IsPublic()을 붙이는 패턴입니다. 실수로 보호되어야 할 라우트를 열어두는 것을 방지하는 화이트리스트 방어 전략입니다.

전역 AuthGuard에서 @IsPublic() 확인
// auth.guard.ts (전역 등록)
export const IS_PUBLIC_KEY = 'isPublic';
export const IsPublic = () => SetMetadata(IS_PUBLIC_KEY, true);

@Injectable()
export class AccessTokenGuard implements CanActivate {
  constructor(private reflector: Reflector) {}

  canActivate(context: ExecutionContext): boolean {
    // @IsPublic()이 붙은 라우트는 토큰 없이도 통과
    const isPublic = this.reflector.getAllAndOverride<boolean>(
      IS_PUBLIC_KEY,
      [context.getHandler(), context.getClass()],
    );
    if (isPublic) return true;

    // 이후 토큰 검증 로직 …
    return this.validateToken(context);
  }
}

// 컨트롤러에서 공개 라우트 표시
@IsPublic()          // 로그인·회원가입은 토큰 불필요
@Post('login')
login() { ... }
🍔 비유 — 건물 출입 시스템 + 내 사물함

RBAC는 건물 출입 등급 카드입니다. 일반 직원(USER) 카드는 사무실만 열리고, 관리자(ADMIN) 카드는 서버실·금고도 열립니다. 역할이 없으면 어디도 못 들어갑니다. 소유권 Guard는 그 위에 내 사물함은 나만 열 수 있다는 규칙입니다. ADMIN 카드가 있어도 다른 사람 사물함을 막 열어선 안 될 때도 있죠. 두 규칙을 함께 써야 인가가 완전합니다.

4Authorization — IsPostMineOrAdminGuard

RBAC는 역할을 확인하지만, "이 게시글이 내 것인가"는 DB를 조회해야 압니다. IsPostMineOrAdminGuard는 ① ADMIN이면 무조건 통과 ② 그렇지 않으면 DB에서 게시글의 authorId를 확인해 본인인지 검사하는 인가 Guard입니다.

code/is-mine-or-admin.guard.ts
@Injectable()
export class IsPostMineOrAdminGuard implements CanActivate {
  constructor(
    @InjectRepository(PostEntity)
    private readonly postRepo: Repository<PostEntity>,
  ) {}

  async canActivate(context: ExecutionContext): Promise<boolean> {
    const req = context.switchToHttp().getRequest();
    const user = req.user;
    const postId = +req.params.postId;

    // ADMIN은 모든 게시글에 접근 가능
    if (user.role === RoleEnum.ADMIN) return true;

    const post = await this.postRepo.findOne({ where: { id: postId } });
    if (!post) throw new NotFoundException();

    // 본인 글이 아니면 403
    if (post.authorId !== user.id) throw new ForbiddenException();

    return true;
  }
}

// 컨트롤러 사용 예시
@UseGuards(IsPostMineOrAdminGuard)
@Delete(':postId')
remove(@Param('postId', ParseIntPipe) postId: number) {
  return this.postsService.remove(postId);
}
💡 RBAC와 소유권 인가는 별개의 레이어

RBAC(RolesGuard)는 역할 수준 — "ADMIN만 이 엔드포인트에 접근 가능" — 을 처리합니다. 소유권 Guard(IsMineOrAdminGuard)는 자원 수준 — "이 자원이 내 것이거나 ADMIN인 경우만" — 을 처리합니다. 두 가지를 @UseGuards(RolesGuard, IsPostMineOrAdminGuard)처럼 함께 걸어 복합 인가 정책을 구성할 수 있습니다. 댓글에도 동일한 구조로 IsCommentMineOrAdminGuard를 추가할 수 있습니다.

5✅ 셀프 체크 — 면접관처럼 물어보기

Q RBAC(역할 기반 접근 제어)란 무엇인가요?

RBAC는 사용자에게 역할(Role)을 부여하고, 각 역할이 수행할 수 있는 행동을 정의하는 인가 방식입니다. NestJS에서는 @Roles(RoleEnum.ADMIN) 커스텀 데코레이터로 핸들러에 필요한 역할 메타데이터를 붙이고, RolesGuard가 Reflector.getAllAndOverride()로 그 메타데이터를 읽어 request.user.role과 비교합니다. 역할이 일치하지 않으면 403 Forbidden을 반환합니다. RBAC의 장점은 역할별 권한을 한 곳에서 선언적으로 관리할 수 있어 유지보수가 쉽다는 점입니다.

Q 모든 라우트를 기본 Private로 두고 @IsPublic()을 사용하는 패턴의 목적은 무엇인가요?

실수로 보호되어야 할 라우트를 공개하는 것을 방지하는 화이트리스트 방어 전략입니다. 전통적 방식처럼 보호 라우트에만 @UseGuards(AuthGuard)를 달면, 개발자가 새 라우트를 추가할 때 Guard를 빠뜨리는 실수를 할 수 있습니다. 반대로 전역 Guard를 기본 적용하고 @IsPublic()을 붙인 라우트만 예외로 두면, 새 라우트는 자동으로 보호됩니다. 인증이 필요 없는 라우트(로그인·회원가입·공개 게시글 목록)에만 @IsPublic()을 명시적으로 달아야 공개됩니다.

Q RolesGuard와 IsPostMineOrAdminGuard는 각각 어떤 경우에 사용하나요?

두 Guard는 서로 다른 인가 레이어를 담당합니다. RolesGuard는 역할 수준: "이 엔드포인트는 ADMIN 역할을 가진 사용자만 호출할 수 있다"처럼, 특정 기능 자체에 역할 제한을 걸 때 씁니다. IsMineOrAdminGuard는 자원 소유권 수준: "이 게시글/댓글은 작성자 본인 또는 ADMIN만 수정·삭제할 수 있다"처럼, 특정 자원에 대한 접근을 자원 소유자와 관리자로 제한할 때 씁니다. 실무에서는 두 Guard를 함께 적용해 역할 + 소유권 이중 인가를 구성합니다.
