1Follower/Followee — User 자기참조 N:M

SNS의 팔로우는 User ↔ User 관계입니다. TypeORM의 @ManyToMany로 간단히 구현할 수도 있지만, 인스타그램처럼 비공개 계정 승인(isConfirmed) 같은 추가 필드가 필요하면 조인 테이블 자체를 별도 Entity(Follow)로 만들어야 합니다.

🍔 비유 — 인맥 장부 + 좋아요 카운터

팔로우 테이블은 인맥 장부입니다. 장부에는 "누가(follower) 누구를(following) 친구 신청했는지(isConfirmed=false)"와 "승인됐는지(isConfirmed=true)"를 적습니다. 단순 N:M 조인이었다면 이 메모를 적을 칸이 없습니다. followerCount 컬럼은 좋아요 카운터입니다. 승인될 때마다 +1, 취소될 때마다 -1하는 방식으로 실시간 카운터를 유지합니다. 매번 "장부를 세어봐"(COUNT 쿼리)보다 훨씬 빠릅니다.

code/follow.entity.ts
@Entity()
export class Follow {
  @PrimaryGeneratedColumn() id: number;

  @ManyToOne(() => User, (u) => u.following, { onDelete: 'CASCADE' })
  @JoinColumn({ name: 'follower_id' })
  follower: User;
  @Column() follower_id: number;

  @ManyToOne(() => User, (u) => u.followers, { onDelete: 'CASCADE' })
  @JoinColumn({ name: 'following_id' })
  following: User;
  @Column() following_id: number;

  // false = 신청 대기, true = 승인 완료 (비공개 계정 맞팔 승인)
  @Column({ default: false }) isConfirmed: boolean;
  @CreateDateColumn() createdAt: Date;
}

2팔로우 신청 → 승인 → 취소 흐름

팔로우 플로우는 세 단계입니다: ① 신청(isConfirmed=false) → ② 팔로위가 승인(isConfirmed=true + followerCount++) → ③ 취소(Follow 레코드 삭제 + followerCount--). 승인과 취소는 Follow 상태 변경과 Count 업데이트를 트랜잭션으로 묶어야 원자성이 보장됩니다(Week 11과 연계).

code/user-follow.service.ts — 핵심 흐름
// 1. 팔로우 신청
async followUser(followerId: number, followingId: number) {
  const follow = this.followRepo.create({
    follower_id: followerId, following_id: followingId, isConfirmed: false,
  });
  return this.followRepo.save(follow);
}

// 2. 팔로우 승인 — 트랜잭션으로 isConfirmed 변경 + followerCount++
async confirmFollow(followId: number, followingId: number) {
  const qr = this.dataSource.createQueryRunner();
  await qr.connect();  await qr.startTransaction();
  try {
    const follow = await qr.manager.findOneOrFail(Follow, { where: { id: followId } });
    follow.isConfirmed = true;
    await qr.manager.save(follow);
    await this.incrementFollowerCount(followingId, qr.manager);
    await qr.commitTransaction();
  } catch (e) {
    await qr.rollbackTransaction(); throw e;
  } finally { await qr.release(); }
}

3Follower/Comment Count — 별도 컬럼으로 증감 관리

팔로워 수를 조회할 때마다 COUNT(*) 쿼리를 날리면 사용자가 많아질수록 성능이 저하됩니다. 대신 User 엔티티에 followerCount 컬럼을 두고, 팔로우 승인 시 +1, 취소 시 -1로 점증 관리하는 방식을 씁니다.

followerCount 컬럼 증감 — QueryBuilder UPDATE
// followerCount +1 (Raw SQL-safe 증감, race condition 없음)
await manager
  .createQueryBuilder()
  .update('user_entity')
  .set({ followerCount: () => 'follower_count + 1' })
  .where('id = :id', { id: userId })
  .execute();

// followerCount -1 (0 미만으로 내려가지 않도록 GREATEST 사용)
await manager
  .createQueryBuilder()
  .update('user_entity')
  .set({ followerCount: () => 'GREATEST(follower_count - 1, 0)' })
  .where('id = :id', { id: userId })
  .execute();
💡 카운터 컬럼 증감 vs COUNT 쿼리 — 트레이드오프

COUNT 쿼리: 항상 정확하지만 대용량에서 느립니다. 인덱스가 있어도 테이블 스캔이 발생합니다. 별도 컬럼 증감: 읽기가 O(1)로 빠르지만, 트랜잭션 없이 쓰면 정합성이 깨질 수 있습니다. 따라서 카운터 증감은 반드시 Follow 상태 변경과 같은 트랜잭션 안에서 실행해야 합니다(Week 11 QueryRunner 패턴과 직결).

4✅ 셀프 체크 — 면접관처럼 물어보기

Q 팔로우 관계를 단순 @ManyToMany 대신 별도 Entity로 만드는 이유는 무엇인가요?

@ManyToMany는 TypeORM이 자동으로 조인 테이블을 만들지만, 그 테이블에는 두 FK 컬럼만 존재합니다. 비공개 계정의 승인 여부(isConfirmed), 팔로우 신청 시각(createdAt), 알림 여부 같은 추가 메타데이터가 필요하면 자동 조인 테이블로는 담을 방법이 없습니다. 별도 Follow 엔티티를 만들고 User를 두 개의 @ManyToOne으로 참조하면 조인 테이블에 원하는 컬럼을 자유롭게 추가할 수 있습니다. 이 패턴은 N:M 관계에 추가 속성이 필요한 모든 경우에 일반적으로 사용됩니다.

Q followerCount를 별도 컬럼으로 관리할 때 가장 중요하게 고려해야 할 점은 무엇인가요?

데이터 정합성입니다. followerCount 컬럼의 값이 실제 Follow 레코드 수와 다르면 잘못된 정보를 노출합니다. 이를 방지하기 위해 카운터 증감은 반드시 Follow 상태 변경과 같은 트랜잭션 안에서 실행해야 합니다. 팔로우 승인 쿼리가 성공했는데 카운터 업데이트가 실패하면 rollback으로 전체를 되돌려야 합니다. 트레이드오프를 요약하면: 읽기 성능(O(1))을 얻는 대신 쓰기 복잡성(트랜잭션 필수)이 증가합니다. 높은 트래픽이 예상되는 SNS 서비스에서는 이 방식이 일반적으로 채택됩니다.
