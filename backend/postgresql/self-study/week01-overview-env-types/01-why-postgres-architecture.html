왜 Postgres인가
그리고 "연결은 왜 비싼가"

Postgres의 정체성은 "연결 하나 = 프로세스 하나"라는 한 문장에 담겨 있습니다. 이 한 가지 사실이 풀링·서버리스 설계를 전부 좌우합니다.

“

Postgres는 연결 하나 = 프로세스 하나다. 그래서 연결은 비싸고, 그 한 가지 사실이 풀링·서버리스 설계를 전부 좌우한다.

  • MySQL과 다른 Postgres의 멀티프로세스 아키텍처를 이해한다
  • "연결이 비싸다"는 결과가 왜 중요한지(풀링 필요성) 안다
  • PG18 비동기 I/O가 무엇을 개선했는지 한 줄로 안다

1왜 Postgres인가

먼저 솔직하게 — MySQL도 훌륭한 DB입니다. 단순 CRUD·읽기 위주 웹 서비스에서는 어느 쪽을 골라도 큰 차이가 없습니다. 그럼에도 Postgres를 고르는 이유는 표준 SQL 준수, 확장(extension) 생태계, JSONB·RLS 같은 고유 기능, 그리고 특정 기업에 종속되지 않은 강력한 오픈소스 거버넌스 때문입니다.

핵심은 "무난함"이 아니라 특정 영역에서 압도적으로 강하다는 점입니다. 아래 표는 Postgres가 특히 빛나는 자리를 정리한 것입니다.

영역Postgres의 강점한 줄 설명
반정형 데이터JSONBJSON을 이진 형태로 저장·인덱싱. 스키마 유연성과 쿼리 성능을 모두 잡는다.
행 수준 보안RLS (Row-Level Security)"이 행은 이 사용자만" 같은 접근 제어를 DB 차원에서 강제한다.
확장성extension (예: pgvector)DB 자체를 플러그인으로 확장. 벡터 검색·지리정보 등을 모듈로 붙인다.
동시성MVCC읽기가 쓰기를 막지 않는다. 다중 사용자 환경에서 락 경합이 적다.
인덱스B-tree·GIN·GiST·BRIN 등데이터 모양에 맞춰 고를 수 있는 풍부한 인덱스 종류.
💡 "무조건 Postgres"가 아니다

도구 선택은 종교가 아닙니다. 위 영역(JSONB, RLS, 확장, 복잡한 동시성)이 필요 없다면 MySQL도 완벽히 합리적인 선택입니다. 이 코스가 Postgres를 다루는 건 이 강점들을 깊게 쓰는 법을 익히기 위해서지, MySQL이 틀려서가 아닙니다.

2아키텍처 — 연결 하나에 프로세스 하나

Postgres를 이해하는 가장 빠른 길은 "연결을 어떻게 처리하는가"를 보는 것입니다. 그리고 여기서 MySQL과 결정적으로 갈립니다.

🏢 비유 — 직원을 늘리는 회사 vs 사무실을 새로 짓는 회사

MySQL은 한 사무실(프로세스) 안에 직원(스레드)을 늘리는 회사입니다. 손님이 오면 직원 한 명을 더 붙이면 되니 가볍습니다. 반면 Postgres는 연결이 올 때마다 독립된 사무실(프로세스) 한 채를 새로 짓는 회사입니다. 사무실은 튼튼하고 서로 완전히 격리되지만, 짓는 비용이 커서 함부로 많이 못 짓습니다. 이 비유 한 장이 오늘 레슨의 전부입니다.

그림처럼 부모인 postmaster가 새 연결을 받을 때마다 자식 backend 프로세스를 하나 fork합니다. 그 연결의 모든 쿼리는 전용 backend가 처리하고, 연결이 끊기면 그 프로세스도 사라집니다.

연결 처리 모델 MySQL

thread-per-connection — 하나의 서버 프로세스 안에서 연결마다 스레드를 띄웁니다. 스레드는 가볍고 메모리를 공유하므로 새 연결을 받는 비용이 작습니다.

연결 처리 모델 PostgreSQL

process-per-connection — 연결마다 독립된 OS 프로세스(backend)를 띄웁니다. 각 backend는 자체 메모리(work_mem 등)를 가져 서로 완전히 격리되지만, 그만큼 생성 비용과 메모리 부담이 큽니다.

3그래서 — 연결은 비싸다 (실무 결과)

아키텍처 이야기가 추상적으로 들렸다면, 여기서 현실로 내려옵니다. "프로세스 하나"라는 사실은 곧바로 "연결이 비싸다"는 실무 결과로 이어집니다.

⚠️ 연결을 요청마다 새로 열면 안 된다

프로세스 생성 비용과 연결당 메모리가 더해지면, 수백 개의 동시 연결만으로도 Postgres는 무릎을 꿇습니다. 그래서 연결 풀(PgBouncer)이 사실상 필수입니다 — 미리 열어둔 소수의 연결을 여러 요청이 돌려 씁니다. 특히 서버리스(AWS Lambda 등) 환경에선 트래픽이 몰리면 인스턴스가 폭증하고, 각 인스턴스가 제각기 연결을 열어 DB를 덮치는 "connection storm"이 대표적인 사고입니다. (자세한 풀링·서버리스 대응은 Week 6.03에서 다룹니다.)

"비싸다"를 구체적인 숫자로 느껴봅시다. Postgres의 기본 max_connections는 보통 100 안팎입니다 — 즉 동시에 열 수 있는 backend 프로세스가 그 정도라는 뜻입니다.

psql — 기본 연결 한도 확인
SHOW max_connections;
--  max_connections
-- -----------------
--  100            ← 기본값. 연결 100개 = backend 프로세스 100개

숫자를 무작정 올리는 건 답이 아닙니다. max_connections를 키우면 그만큼 프로세스·메모리 부담이 커지기 때문입니다. 올바른 해법은 한도를 무리하게 늘리는 게 아니라, 풀러로 적은 연결을 알뜰하게 공유하는 것입니다.

4PG18 비동기 I/O — 한 스푼

마지막으로 최신 소식 한 스푼. 2026년 기준 우리가 다루는 PostgreSQL 18은 I/O 처리 방식에 의미 있는 개선을 가져왔습니다.

🧩 PostgreSQL 18: 비동기 I/O 도입

PG18 PG18부터 io_method 설정으로 비동기 I/O(AIO)를 지원합니다. 읽기 요청을 하나씩 기다리지 않고 줄 세워 한꺼번에 처리해, 시퀀셜 스캔·VACUUM 같은 작업에서 최대 3배까지 빨라진 벤치마크 사례가 보고됐습니다. 지금 단계에서는 "Postgres가 디스크 읽기를 더 똑똑하게 한다" 정도만 기억하면 충분합니다. 내부 동작은 뒤 주차에서 다시 만납니다.

복습

5✅ 셀프 체크

Q Postgres에서 연결이 '비싸다'고 하는 이유는?

연결 하나가 OS 프로세스 하나이기 때문입니다. 프로세스 생성 비용과 연결당 독립 메모리가 커서, 다수의 연결이 서버를 압박합니다. 그래서 미리 열어둔 연결을 돌려 쓰는 풀링이 필요합니다.

Q MySQL과 Postgres의 연결 처리 모델 차이는?

MySQL은 thread-per-connection — 한 프로세스 안에서 연결마다 스레드를 띄웁니다. Postgres는 process-per-connection — 연결마다 독립된 backend 프로세스를 띄웁니다. 후자는 격리가 강하지만 더 무겁습니다.

Q 서버리스 환경에서 Postgres 연결이 문제되는 이유는?

인스턴스가 폭발적으로 늘면 각 인스턴스가 제각기 연결을 열어 connection storm이 발생하고, DB의 max_connections 한도를 넘겨 버립니다. PgBouncer 같은 풀러나 서버리스 전용 드라이버로 연결 수를 완화합니다.
