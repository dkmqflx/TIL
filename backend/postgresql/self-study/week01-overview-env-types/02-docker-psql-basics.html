“

DB를 직접 설치하지 말고 postgres:18 컨테이너로 빌려 쓴다. 그리고 psql의 백슬래시 명령(\d 패밀리)으로 DB 속을 들여다본다.

1왜 Docker로 DB를 띄우나

"PostgreSQL을 내 맥에 직접 설치"하면 버전이 꼬이고, 다른 프로젝트와 포트가 충돌하고, 팀원마다 환경이 제각각이 됩니다. Docker는 이 문제를 격리(isolation)와 재현성(reproducibility)으로 해결합니다 — DB가 컨테이너 안에 갇혀 로컬과 섞이지 않고, 같은 설계도(이미지)를 받으면 누구나 똑같은 DB를 띄웁니다.

📦 비유 — 콘크리트 창고 짓기 vs 조립식 레고 창고

DB를 직접 설치하는 건 마당에 콘크리트로 창고를 짓는 것과 같습니다. 한 번 지으면 옮기기 힘들고, 헐 때도 흙바닥이 더럽혀집니다. Docker 컨테이너는 레고처럼 가져다 놓는 조립식 창고입니다. up으로 세우고 down으로 흔적 없이 치웁니다. "내 컴퓨터에선 됐는데?"가 사라지는 이유 — 모두가 같은 컨테이너 설계도를 받기 때문입니다.

방식설치충돌팀 공유
로컬 직접 설치OS마다 절차가 다름버전·포트 충돌 잦음"내 환경에선…" 문제
Docker 컨테이너이미지 pull 한 번컨테이너 안에 격리compose 파일 1장 공유

2docker-compose.yml & 띄우고 끄기

컨테이너 한 채를 띄우는 설계도입니다. 이미지·계정·포트·데이터 보관 위치를 선언합니다. 이번 주 실습 환경은 아래 파일로 이미 준비돼 있습니다.

docker-compose.yml
services:
  db:
    image: postgres:18          # PostgreSQL 18 공식 이미지
    container_name: pg-study
    environment:
      POSTGRES_USER: study
      POSTGRES_PASSWORD: study_pw
      POSTGRES_DB: study_db
    ports:
      - "5432:5432"          # 호스트 5432 → 컨테이너 5432
    volumes:
      - pgdata:/var/lib/postgresql  # ⚠ /data 아님 — 아래 팁 참고

volumes:
  pgdata:
💡 postgres:18은 볼륨을 상위 경로에 마운트한다

PG18 볼륨은 /var/lib/postgresql(상위)에 마운트해야 합니다. 예전 관습대로 /var/lib/postgresql/data로 마운트하면 부팅이 거부됩니다 — postgres:18 이미지가 데이터를 버전별 하위 디렉터리(예: .../18/docker)에 두도록 구조가 바뀌었기 때문입니다. 옛 자료(이미지·블로그)에는 아직 /data 접미사가 흔하니, PG18에서는 떼고 쓰세요.

설계도가 준비됐으니, 같은 폴더에서 명령 한 줄이면 컨테이너를 켜고 끌 수 있습니다.

terminal
# 백그라운드(-d, detached)로 DB 컨테이너 시작
docker compose up -d

# 잘 떴는지 확인
docker compose ps

# 종료(데이터는 volume에 보존)
docker compose down

# 데이터까지 완전 삭제
docker compose down -v
⚠️ down -v는 데이터를 증발시킨다

그냥 docker compose down은 컨테이너만 지우고 named volume(pgdata)은 남겨 데이터가 살아남습니다. 하지만 -v를 붙이면 볼륨까지 삭제돼 그 안에 쌓인 데이터가 전부 사라집니다. 실습을 깨끗이 초기화하고 싶을 때만 의식적으로 쓰세요 — 습관적으로 붙이면 안 됩니다.

3psql 접속

컨테이너가 떴으면, 그 안의 PostgreSQL에 말을 걸 차례입니다. psql은 PostgreSQL이 공식으로 제공하는 커맨드라인 클라이언트(CLI)입니다. 이미지 안에 이미 들어 있으니, 컨테이너 안에서 바로 실행하면 됩니다.

terminal — 컨테이너 안의 psql로 접속
docker compose exec db psql -U study -d study_db
# -U study   : 접속할 역할(role)
# -d study_db: 접속할 데이터베이스
# 접속되면 프롬프트가 study_db=# 로 바뀐다

접속이 됐는지 가장 확실히 확인하는 방법은 "나는 지금 어느 DB에 누구로 붙어 있나"를 물어보는 것입니다.

study_db=# — 연결 확인
SELECT current_database(), current_user, version();
 current_database | current_user |              version
------------------+--------------+----------------------------------
 study_db         | study        | PostgreSQL 18.4 (Debian ...) ...
(1 row)

4psql 메타 명령 — \d 패밀리

psql의 진짜 무기는 백슬래시(\)로 시작하는 메타 명령입니다. DB 속(테이블·스키마·역할 등)을 SQL을 길게 쓰지 않고 한 글자로 들여다볼 수 있습니다.

명령무엇을 보나
\l데이터베이스 목록
\dt현재 스키마의 테이블 목록
\d <table>특정 테이블의 구조(컬럼·타입·인덱스)
\du역할(role) 목록
\dn스키마 목록
\?psql 메타 명령 도움말
🧩 메타 명령은 SQL이 아니다

psql \dt 같은 명령은 psql 클라이언트가 해석하는 명령이지 표준 SQL이 아닙니다. 그래서 애플리케이션 드라이버(psycopg, JDBC 등)로 보내면 동작하지 않습니다. (psql은 내부적으로 이를 pg_catalog 조회 SQL로 바꿔 실행합니다 — \d 한 글자가 긴 카탈로그 쿼리의 별명인 셈입니다.)

간단한 테이블을 하나 만들어 \dt와 \d를 직접 체험해 봅시다. 아래는 실제로 실행한 결과입니다.

study_db=# — \dt 와 \d 체험
CREATE TABLE ping (id int PRIMARY KEY, msg text);

\dt
        List of tables
 Schema | Name | Type  | Owner
--------+------+-------+-------
 public | ping | table | study

\d ping
                Table "public.ping"
 Column |  Type   | Collation | Nullable | Default
--------+---------+-----------+----------+---------
 id     | integer |           | not null |
 msg    | text    |           |          |
Indexes:
    "ping_pkey" PRIMARY KEY, btree (id)

5데이터베이스 vs 스키마 vs search_path

여기서 Postgres 입문자가 가장 자주 헷갈리는 세 가지를 정리합니다. 데이터베이스, 스키마, 그리고 search_path는 서로 다른 층위입니다.

🗄️ 비유 — 서랍장(장 · 서랍 · 물건)

데이터베이스 = 장(서랍장 한 채), 스키마 = 그 안의 서랍, 테이블 = 서랍 속 물건입니다. 하나의 장(데이터베이스) 안에 여러 서랍(스키마)이 있고, 기본 서랍의 이름은 public입니다. 물건을 찾을 때 어느 서랍부터 열어볼지의 순서가 바로 search_path입니다.

지금 연결의 기본 검색 순서를 확인해 봅시다.

study_db=# — 기본 search_path
SHOW search_path;
   search_path
-----------------
 "$user", public
(1 row)

기본값 "$user", public은 "먼저 내 이름($user)과 같은 스키마를 찾고, 없으면 public을 찾으라"는 뜻입니다. SELECT * FROM ping처럼 스키마를 명시하지 않으면, 이 순서대로 ping 테이블을 찾습니다.

이 구조가 MySQL과 어떻게 다른지가 핵심입니다.

데이터베이스 모델 MySQL

MySQL에서는 database ≈ schema가 사실상 동의어입니다. 목록은 SHOW DATABASES;, 전환은 USE db;로 합니다. "한 단계"의 통(database) 안에 바로 테이블이 들어 있다고 보면 됩니다.

데이터베이스 모델 PostgreSQL

Postgres는 한 database 안에 여러 schema가 있는 "두 단계" 구조입니다(기본 스키마는 public). 목록은 \l, 전환은 \c db로 합니다. 같은 database 안에서 스키마는 \dn으로 봅니다.

⚠️ \c 로 database를 바꾸면 연결이 새로 맺어진다

Postgres에서 한 연결(connection)은 하나의 database에만 붙습니다. 그래서 psql에서 \c other_db로 다른 database로 옮기면, 기존 연결을 끊고 새 연결을 다시 맺습니다. MySQL의 USE db;가 같은 연결 안에서 대상만 바꾸는 것과 결정적으로 다릅니다 — 반면 같은 database 안의 스키마는 연결을 끊지 않고 search_path나 스키마 접두사(schema.table)로 자유롭게 넘나듭니다.

복습

6✅ 셀프 체크

Q Postgres에서 database와 schema의 관계는?

하나의 database 안에 여러 schema가 들어 있고, 기본 스키마는 public입니다(두 단계 구조). 반면 MySQL은 database와 schema가 사실상 동의어입니다. 서랍장으로 치면 database가 장, schema가 그 안의 서랍입니다.

Q 테이블 구조를 보는 psql 명령은?

\d <table>입니다(예: \d ping). 컬럼·타입·인덱스를 보여줍니다. 테이블 목록만 볼 때는 \dt를 씁니다.

Q search_path는 무엇을 결정하나?

스키마를 명시하지 않았을 때 객체(테이블·함수 등)를 어느 스키마에서 어떤 순서로 찾을지를 결정합니다. 기본값 "$user", public은 내 이름과 같은 스키마를 먼저, 없으면 public을 찾습니다.
