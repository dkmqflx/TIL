데이터 타입 총정리
IDENTITY · uuidv7() · timestamptz

Postgres의 풍부한 타입 중 앱 개발자가 매일 쓰는 것만 골라 봅니다. 자동증가 키는 IDENTITY, UUID는 PG18 내장 uuidv7(), 시간은 무조건 timestamptz.

“

Postgres의 풍부한 타입 중 앱 개발자가 매일 쓰는 것만: IDENTITY 키, PG18 네이티브 uuidv7(), 그리고 무조건 timestamptz.

  • SERIAL vs IDENTITY — 왜 IDENTITY가 표준이자 권장인지 안다
  • PG18 내장 uuidv7()로 정렬 가능한 UUID 키를 만든다
  • timestamp vs timestamptz 함정을 피하고, 돈은 numeric으로 다룬다

1자동증가 키 — SERIAL vs IDENTITY

"새 행마다 1, 2, 3… 자동으로 증가하는 기본 키"는 거의 모든 테이블에 필요합니다. Postgres에는 두 가지 방법이 있는데, 오래된 SERIAL과 SQL 표준인 GENERATED ALWAYS AS IDENTITY입니다. 아래는 같은 의도를 두 방식으로 쓴 것입니다.

study_db=# — 두 가지 자동증가 키
-- 구식: SERIAL (뒤에서 시퀀스를 몰래 만들어 연결)
CREATE TABLE old_way (id serial PRIMARY KEY, name text);

-- 권장: SQL 표준 IDENTITY
CREATE TABLE std_way (id int GENERATED ALWAYS AS IDENTITY PRIMARY KEY, name text);

SERIAL은 진짜 타입이 아니라 "int 컬럼 + 시퀀스 + 기본값 연결"의 축약입니다. 그래서 시퀀스가 별도 객체로 남아 소유권·권한이 컬럼과 따로 놀고, 컬럼을 지워도 시퀀스가 남는 등 자잘한 함정이 있습니다. GENERATED ALWAYS AS IDENTITY는 같은 일을 하되 시퀀스를 컬럼이 소유하도록 묶어 두고, ALWAYS는 사용자가 실수로 id를 직접 끼워 넣는 것까지 막아 줍니다. 새 테이블이라면 IDENTITY를 쓰세요.

자동증가 키 MySQL

MySQL은 컬럼에 AUTO_INCREMENT를 붙입니다. 테이블당 하나, 보통 PK에 답니다. 별도의 시퀀스 객체라는 개념이 없습니다.

자동증가 키 PostgreSQL

Postgres는 SQL 표준 GENERATED ALWAYS AS IDENTITY를 권장합니다(내부적으로 시퀀스를 쓰지만 컬럼이 소유). SERIAL은 같은 일을 하는 레거시 축약이라 신규 코드에선 피합니다.

2UUIDv7 (PG18) ⭐

분산 시스템이나 클라이언트 생성 ID가 필요하면 정수 대신 UUID를 키로 씁니다. 그런데 어떤 UUID냐가 성능을 가릅니다.

🧬 비유 — 무작위 UUID vs 시간이 박힌 UUID

UUIDv4는 완전 무작위입니다. 키가 인덱스(B-tree) 트리의 여기저기로 흩뿌려져 꽂히니, 새 행이 들어올 때마다 가운데 페이지를 쪼개는 페이지 분할(page split)이 잦고 캐시도 잘 안 맞습니다. UUIDv7은 앞부분에 생성 시각이 박혀 있어, "새로 만든 행은 항상 값이 더 큼" — 즉 인덱스의 맨 뒤에 순서대로 꽂힙니다. 새 책을 항상 책장 맨 끝에 꽂는 것과 같아, 분할이 줄고 캐시 지역성이 올라갑니다.

PG18에서는 함수 하나로 v7 UUID를 얻습니다. 아래는 실제로 실행한 결과입니다 — 같은 시각에 만든 세 행의 id가 생성 순서대로 정렬되는 데 주목하세요(앞쪽 019eac0e-3cef 시각 접두사가 공유됨).

study_db=# — uuidv7() (PG18 내장)
SELECT uuidv7();
                uuidv7
--------------------------------------
 019eac0e-3ce8-7ebd-b830-32e512124867

-- 기본값으로 박아 두면 INSERT마다 자동 생성
CREATE TABLE evt (
  id uuid PRIMARY KEY DEFAULT uuidv7(),
  payload text
);
INSERT INTO evt (payload) VALUES ('a'), ('b'), ('c');
SELECT id FROM evt ORDER BY id;     -- 생성 순서대로!
                  id
--------------------------------------
 019eac0e-3cef-75b9-9412-5d0a03f0c78a
 019eac0e-3cef-772c-bd86-5ed97c96eb05
 019eac0e-3cef-773e-8599-ae7d598cefe1
🧩 PG18 이전엔 어떻게 했나

PG18 uuidv7()는 PG18부터 코어에 내장됐습니다. 그 전에는 확장(pgcrypto)이나 애플리케이션 코드에서 UUID를 만들었습니다. 단, 코어의 gen_random_uuid()는 v4(완전 무작위)라 정렬이 안 됩니다 — 정렬 가능한 키가 필요하면 v4가 아니라 uuidv7()를 쓰세요. (이름이 비슷해 헷갈리기 쉬운 함정입니다.)

3timestamptz 함정 ⚠️

시간 컬럼을 만들 때 무심코 timestamp라고 적기 쉽지만, 이건 타임존을 모르는 타입입니다. 거의 모든 앱에서는 timestamptz를 써야 합니다.

⚠️ timestamp(without time zone)는 시간을 어긋나게 만든다

timestamp는 "2026-06-09 11:04"처럼 타임존 정보가 없는 벽시계 값일 뿐입니다. 서버와 클라이언트의 타임존이 다르면 같은 값이 다른 순간을 가리켜 시간이 어긋납니다. 반면 timestamptz는 입력을 내부적으로 UTC로 변환해 저장하고, 조회할 때 세션 타임존으로 표시합니다. (주의: 이름과 달리 타임존을 "저장"하는 게 아니라, UTC로 정규화해 저장하고 표시 시점에 변환합니다.)

아래는 실제 실행 결과입니다. 세션 타임존은 컨테이너 기본값인 Etc/UTC이고, at_right에는 오프셋 +00이 붙어 "이 값은 UTC 기준"임을 알 수 있습니다.

study_db=# — timestamp vs timestamptz
CREATE TABLE log (
  at_wrong timestamp,      -- 타임존 정보 없음 (지뢰)
  at_right timestamptz     -- UTC로 저장, 세션 타임존으로 표시
);
SHOW timezone;
 TimeZone
----------
 Etc/UTC

INSERT INTO log VALUES (now(), now());
SELECT * FROM log;
          at_wrong          |           at_right
----------------------------+-------------------------------
 2026-06-09 11:04:34.802719 | 2026-06-09 11:04:34.802719+00
시간 타입 MySQL

MySQL에는 DATETIME(타임존 변환 없음, 넣은 값 그대로)과 TIMESTAMP(저장 시 UTC로 변환·조회 시 세션 타임존으로)가 있어, 둘 중 무엇을 썼는지가 동작을 가릅니다.

시간 타입 PostgreSQL

Postgres는 timestamp(타임존 모름)와 timestamptz(UTC 저장·세션 타임존 표시)로 나뉩니다. 앱에서는 거의 항상 timestamptz가 정답입니다.

4numeric vs float — 돈은 numeric

마지막 함정은 소수입니다. float8/real은 이진 부동소수점 근사값이라, 우리가 보기엔 딱 떨어지는 값도 정확히 표현하지 못합니다. 그 유명한 0.1 + 0.2를 두 타입으로 더해 봤습니다.

study_db=# — float의 오차 vs numeric의 정확성
SELECT 0.1::float8 + 0.2::float8 AS float_bad,
       0.1::numeric + 0.2::numeric AS numeric_ok;
      float_bad      | numeric_ok
---------------------+------------
 0.30000000000000004 |        0.3

float_bad는 0.30000000000000004 — 근사 오차가 그대로 드러납니다. numeric(임의 정밀도)은 정확히 0.3입니다. 그래서 금액·정밀 계산은 반드시 numeric을 쓰고, 측정값처럼 약간의 오차가 허용되는 과학 계산에만 float8을 씁니다.

6✅ 셀프 체크

Q 새 테이블의 자동증가 PK, SERIAL과 IDENTITY 중 무엇을?

GENERATED ALWAYS AS IDENTITY입니다. SQL 표준이고, 시퀀스를 컬럼이 소유해 소유권·권한이 깔끔합니다. SERIAL은 같은 일을 하는 레거시 축약이라 신규 코드에선 피합니다.

Q UUIDv4 대신 UUIDv7을 PK로 쓰면 좋은 이유는?

v7은 앞부분에 시각이 박혀 시간순으로 정렬되므로, 인덱스에 순차적으로 삽입됩니다 → 페이지 분할↓·캐시 지역성↑. PG18은 uuidv7()가 내장돼 있습니다. (gen_random_uuid()는 v4(무작위)라 정렬이 안 됨에 주의.)

Q timestamp와 timestamptz 중 기본 선택은?

timestamptz입니다. 타임존을 인지해 입력을 UTC로 정규화해 저장하고 조회 시 세션 타임존으로 표시하므로, 다중 타임존 환경에서 안전합니다. timestamp는 타임존을 모르는 벽시계 값이라 어긋날 수 있습니다.
