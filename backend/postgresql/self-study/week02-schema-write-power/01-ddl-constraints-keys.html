무결성을 DB에게 맡기기

테이블 만드는 CREATE TABLE은 누구나 압니다. 차이는 제약(constraint)으로 무결성을 DB가 지키게 하는 것 — 그리고 기본키를 IDENTITY/uuidv7()로 제대로 고르는 것입니다.

“

테이블을 만드는 건 누구나 안다. 차이는 제약으로 데이터 무결성을 DB에 맡기는 것 — 그리고 기본키를 IDENTITY/uuidv7로 제대로 고르는 것.

  • 제약(NOT NULL·UNIQUE·CHECK·FK·DEFAULT)으로 무결성을 DB에 맡긴다
  • IDENTITY vs uuidv7() — 언제 어느 기본키를 고를지 안다
  • 시퀀스를 이해하고, (PG18) 제외 제약·temporal 제약을 맛본다

0무결성(Integrity)이란?

무결성이란 "데이터가 정해진 규칙을 항상 지키고 있는 상태"를 말합니다. 쉽게 말하면 데이터가 올바르고 믿을 수 있는 상태입니다.

📦 비유 — 택배 주소록

택배 주소록에 전화번호 칸이 있다고 해봅시다. 누군가 실수로 전화번호 칸에 "안녕하세요"라고 입력하거나, 같은 사람을 두 번 등록하거나, 이미 탈퇴한 회원의 주문이 남아 있다면 — 이 주소록은 무결성이 깨진 상태입니다. 반대로 모든 데이터가 규칙대로 정확히 들어가 있다면 무결성이 지켜진 상태입니다.

DB에서 무결성을 지키기 위한 구체적인 규칙들을 제약(constraint)이라고 합니다. 예를 들면:

  • 이메일은 중복될 수 없다 → UNIQUE 제약
  • 나이는 반드시 입력해야 하고 0 이상이어야 한다 → NOT NULL + CHECK 제약
  • 주문의 고객 ID는 실제로 존재하는 고객이어야 한다 → FOREIGN KEY 제약

이런 제약들을 DB에 걸어두면, 잘못된 데이터가 들어오려 할 때 DB가 자동으로 거부합니다. 아래에서 각 제약을 하나씩 살펴봅니다.

1CREATE TABLE & 컬럼 제약

테이블을 만들 때 컬럼 이름·타입만 적고 끝내기 쉽지만, 진짜 중요한 건 제약입니다. NOT NULL(비어 있을 수 없음), DEFAULT(기본값), UNIQUE(중복 금지), CHECK(값의 규칙)를 컬럼에 붙이면, "잘못된 데이터가 애초에 들어오지 못하게" DB가 직접 막아 줍니다.

study_db=# — 제약이 붙은 users 테이블
CREATE TABLE users (
  id          int GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  email       text NOT NULL UNIQUE,                 -- 비어 있을 수 없고 중복 불가
  age         int CHECK (age >= 0 AND age < 150),   -- 값의 범위를 DB가 강제
  status      text NOT NULL DEFAULT 'active',       -- 미지정 시 기본값
  created_at  timestamptz NOT NULL DEFAULT now()
);
INSERT INTO users (email, age) VALUES ('a@ex.com', 30), ('b@ex.com', 25);
SELECT id, email, age, status FROM users ORDER BY id;
 id |  email   | age | status
----+----------+-----+--------
  1 | a@ex.com |  30 | active
  2 | b@ex.com |  25 | active

status를 안 넣었는데도 active가 채워졌습니다 — DEFAULT 덕분입니다. 이제 규칙을 어기는 INSERT를 시도하면 DB가 어떻게 막는지 보세요(아래는 의도적으로 실패하는 예라 주석으로 남깁니다).

study_db=# — 제약 위반은 DB가 막는다 (의도적 에러)
INSERT INTO users (email, age) VALUES ('a@ex.com', 40);
ERROR:  duplicate key value violates unique constraint "users_email_key"

INSERT INTO users (email, age) VALUES ('c@ex.com', -5);
ERROR:  new row for relation "users" violates check constraint "users_age_check"
⚠️ 무결성은 앱이 아니라 DB가 지키게 하라

"이메일 중복 체크는 애플리케이션 코드에서 하면 되지" 같은 생각은 위험합니다. 앱에는 버그가 있고, 같은 순간 두 요청이 들어오는 동시성(race condition)도 있습니다. 두 요청이 동시에 "중복 없음"을 확인한 뒤 둘 다 INSERT하면 중복이 들어가 버립니다. 제약을 DB에 두면 이런 경우에도 DB가 마지막 방어선으로 막아 줍니다 — 무결성 규칙은 데이터 옆(DB)에 두는 게 안전합니다.

2외래키(FK)와 참조 무결성

orders.user_id가 실제로 존재하는 users.id만 가리키도록 강제하는 것이 외래키(FOREIGN KEY)입니다. REFERENCES로 선언하면, 없는 부모를 가리키는 주문은 아예 들어가지 못합니다. 그리고 부모가 삭제될 때 자식을 어떻게 할지 ON DELETE로 정합니다.

ON DELETE 옵션부모가 삭제되면…
CASCADE자식 행도 함께 삭제
SET NULL자식의 FK 컬럼을 NULL로
RESTRICT (기본)자식이 있으면 삭제 자체를 거부
study_db=# — FK + ON DELETE CASCADE 시연
CREATE TABLE orders (
  id        int GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  user_id   int NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  amount    numeric(10,2) NOT NULL CHECK (amount > 0)
);
INSERT INTO orders (user_id, amount) VALUES (1, 9.99), (1, 5.00), (2, 12.50);

-- user 1을 지우면 그 주문도 함께 사라진다
DELETE FROM users WHERE id = 1;
SELECT * FROM orders ORDER BY id;
 id | user_id | amount
----+---------+--------
  3 |       2 |  12.50
(1 row)            ↑ user_id=1 주문(1,2)이 CASCADE로 모두 사라짐
외래키 MySQL

MySQL은 InnoDB 엔진에서 FK를 지원합니다(REFERENCES … ON DELETE CASCADE). 단, 과거 MyISAM 엔진은 FK 문법을 받아만 들이고 실제로 강제하지 않았던 함정이 있었습니다.

외래키 PostgreSQL

Postgres는 엔진 구분 없이 항상 FK를 강제합니다. 문법(REFERENCES·ON DELETE CASCADE/SET NULL/RESTRICT)도 InnoDB와 거의 동일해, 둘 다 표준 SQL을 따릅니다.

3기본키 전략 — IDENTITY vs uuidv7()

기본키(Primary Key)는 각 행을 구별하는 고유한 번호표입니다. 예를 들어 회원 테이블에서 "이 사람이 누구인지"를 식별하는 id가 기본키입니다. 선택지는 크게 두 가지입니다.

  • IDENTITY: 행이 추가될 때마다 DB가 자동으로 1, 2, 3… 숫자를 붙여 줍니다.
  • uuidv7(): 019eac43-d675-7f08-… 같은 긴 무작위 문자열을 ID로 씁니다.
상황권장 기본키이유 (쉽게)
대부분의 경우 — 기본 선택IDENTITY1, 2, 3… 숫자라 단순하고 빠름
ID가 외부에 노출될 때
예: URL(/posts/5), API 응답 JSON		uuidv7()숫자면 "다음 글은 /posts/6이겠네" 추측이 가능 — 긴 문자열은 추측 불가
여러 서버가 동시에 ID를 만들 때uuidv7()각 서버가 따로 만들어도 ID가 겹치지 않음
🔗 왜 외부에 노출되는 숫자 ID가 위험한가?

URL이든 API 응답이든 클라이언트에 ID가 전달되는 순간 동일한 문제가 생깁니다. /posts/5를 보면 "다음은 /posts/6이겠네"를 추측할 수 있고, API 응답 { "id": 5 }를 받아도 마찬가지입니다. 악의적인 사용자가 1부터 하나씩 올려가며 데이터를 긁거나, /orders/123으로 "오늘 주문이 123건"이라는 비즈니스 정보가 유출될 수 있습니다. uuidv7()로 바꾸면 { "id": "019eac43-d675-…" }처럼 보여 다음 ID를 예측할 방법이 없습니다.

노출 방식uuidv7() 필요?
내부 DB 조인·집계에만 사용❌ IDENTITY면 충분

🖥️ 왜 여러 서버가 동시에 숫자 ID를 만들면 충돌하는가?

서울 서버와 부산 서버가 직원 번호를 따로 발급하면 둘 다 "1번, 2번, 3번…"을 만들어 번호가 겹쳐버립니다. 숫자 IDENTITY는 반드시 DB 한 곳에 "지금 몇 번까지 발급했어?"를 물어봐야 하므로, 서버가 여러 대면 속도도 느려지고 복잡해집니다. 반면 uuidv7()은 "현재 시각 + 서버별 무작위 값"으로 ID를 만들기 때문에, 각 서버가 DB에 묻지 않고 독립적으로 생성해도 수십억 분의 1 확률로도 겹치지 않습니다.

uuidv7()는 완전히 무작위처럼 보이지만, 앞부분에 생성 시각이 들어 있어 만들어진 순서대로 정렬됩니다. 아래 결과에서 두 행의 id 앞부분 019eac43이 같은 것이 그 증거입니다.

study_db=# — uuidv7() 기본키 (분산/외부노출용)
CREATE TABLE membership (
  id      uuid PRIMARY KEY DEFAULT uuidv7(),
  label   text NOT NULL
);
INSERT INTO membership (label) VALUES ('gold'), ('silver');
SELECT id, label FROM membership ORDER BY id;
                  id                  | label
--------------------------------------+--------
 019eac43-d675-7f08-b52c-2df0e1d35a48 | gold
 019eac43-d676-71fa-92bf-695880d95451 | silver
자동증가 키 MySQL

MySQL은 컬럼에 AUTO_INCREMENT를 붙입니다. 테이블당 하나, 보통 PK에 답니다. 직접 값을 넣어 시퀀스를 어긋나게 할 여지가 있습니다.

자동증가 키 PostgreSQL

Postgres는 SQL 표준 GENERATED ALWAYS AS IDENTITY를 권장합니다. ALWAYS 덕에 직접 삽입을 막아(필요하면 OVERRIDING SYSTEM VALUE 명시) 값이 어긋나지 않습니다.

4시퀀스(SEQUENCE)

IDENTITY 키 뒤에는 사실 시퀀스라는 객체가 숨어 "다음 번호"를 발급하고 있습니다. 시퀀스는 "다음 번호표를 뽑아주는 기계"라고 생각하면 됩니다 — nextval을 호출할 때마다 100, 101, 102… 를 순서대로 내어줍니다. IDENTITY로 테이블을 만들면 PostgreSQL이 이 기계를 자동으로 생성하고, INSERT할 때마다 번호를 하나씩 뽑아 id에 채워줍니다. pg_get_serial_sequence()로 어떤 시퀀스가 컬럼에 묶였는지 확인할 수 있고, 시퀀스를 직접 만들어 nextval/currval로 번호를 뽑아 쓸 수도 있습니다.

study_db=# — IDENTITY 뒤의 시퀀스 + 직접 만들기
-- IDENTITY 컬럼이 소유한 시퀀스 이름 확인
SELECT pg_get_serial_sequence('users', 'id');
    pg_get_serial_sequence
---------------------------
 public.users_id_seq

-- 시퀀스를 직접 만들어 번호 뽑기
CREATE SEQUENCE ticket_seq START 100 INCREMENT 1;
SELECT nextval('ticket_seq');   -- 100 (다음 값 발급)
SELECT nextval('ticket_seq');   -- 101
SELECT currval('ticket_seq');   -- 101 (이 세션의 마지막 값)
🧩 시퀀스는 트랜잭션을 따르지 않는다 — gap이 생기는 이유

gap nextval로 뽑은 번호는 트랜잭션이 롤백돼도 되돌아오지 않습니다.

비유 — 은행 대기표: A·B·C가 동시에 번호표를 뽑아 101·102·103번을 받았습니다. A가 "잘못 왔네" 하고 그냥 나가버렸어도(= 롤백), 기계는 101번을 회수하지 않습니다. 다음 사람은 104번을 받고, 101번은 빈 자리로 남습니다. 만약 "롤백되면 번호를 회수한다"고 설계하면 동시에 100명이 번호표를 뽑을 때 기계가 매번 잠겨 엄청 느려지기 때문에, 일부러 "뽑으면 그냥 끝"으로 만든 겁니다.

gap이 생겨도 괜찮은 이유: 도로 위 번호판이 가 1234, 가 1236으로 1235가 빠져 있어도 각 차를 구별하는 데는 전혀 문제없습니다. ID의 역할은 "각 행을 유일하게 식별하는 것"이지 연속일 필요는 없습니다. 1, 2, 4, 5…여도 충분합니다.

5(PG18) 제외 제약 & temporal 맛보기

지금까지 배운 UNIQUE는 "값이 완전히 같으면 안 됨"을 막습니다. 그런데 현실에서는 "값이 같은 건 괜찮은데, 특정 조건이 둘 다 겹치면 안 되는" 상황이 있습니다. 이걸 처리하는 게 제외 제약(EXCLUDE)입니다.

📅 UNIQUE vs EXCLUDE — 한 줄 비교

UNIQUE: "이메일은 딱 하나만" → 값 하나를 비교
EXCLUDE: "같은 회의실에서 시간이 겹치면 안 됨" → 두 가지 조건을 동시에 비교

예약 시스템에서 UNIQUE로는 이걸 표현할 수 없어요. 같은 방을 다른 시간에 쓰는 건 괜찮으니까요. "방이 같고(=) 그리고 시간대까지 겹치면(&&)" — 이 두 조건이 동시에 참일 때만 거부하는 게 EXCLUDE입니다.

🏢 비유 — 회의실 예약 충돌

팀장 A가 회의실 A를 10시~11시로 예약했습니다. 팀원 B가 동시에 회의실 A를 10시 30분~11시 30분으로 예약 시도합니다. 앱에서만 체크하면 두 요청이 동시에 들어올 때 둘 다 "빈 방이네?"라고 판단하고 저장해버릴 수 있습니다. EXCLUDE를 DB에 걸면 어느 경로로 들어와도 시간대가 겹치는 예약을 DB가 직접 거부합니다.

반면 B가 11시~12시로 예약하면? 시간대가 딱 붙어있지만 겹치지는 않으므로(&&가 거짓) 허용됩니다.

🧩 tsrange — 시작과 끝을 하나로 묶은 타입

tsrange는 timestamp range의 줄임말로, 시작 시각과 끝 시각을 컬럼 두 개 대신 하나의 값으로 묶은 타입입니다.

컬럼 두 개(started_at, ended_at)로 나눠 저장하면 "두 예약이 겹치는지"를 SQL로 검사하기가 복잡합니다. tsrange로 묶으면 && 연산자 하나로 바로 확인할 수 있고, EXCLUDE에서도 그대로 쓸 수 있습니다.

괄호의 의미: [10:00, 11:00)에서 [는 10시를 포함, )는 11시를 미포함합니다(수학의 반열린 구간). 11시 정각에 다음 예약이 시작할 수 있는 이유가 이 때문입니다 — 앞 예약이 11시를 포함하지 않으니 겹치지 않습니다.

비교결과이유
[10:00, 11:00) && [10:30, 11:30)true (겹침)10:30~11:00 구간이 겹침
[10:00, 11:00) && [11:00, 12:00)false (안 겹침)딱 붙어있지만 공유 시각 없음
🧩 EXCLUDE 문법 파헤치기 — USING gist 와 WITH

코드에서 가장 낯선 줄인 EXCLUDE USING gist (room WITH =, during WITH &&)를 단어별로 풀면 이렇습니다.

USING gist — 어떤 인덱스로 검사할지
PostgreSQL 인덱스에는 종류가 있습니다. 기본인 B-tree는 전화번호부처럼 값을 한 줄로 정렬해 =, <, > 비교에 강합니다. 하지만 "두 범위가 겹치는지(&&)"는 시작값과 끝값을 동시에 봐야 판단할 수 있어서 한 줄 정렬로는 안 됩니다. GiST(Generalized Search Tree)는 지도처럼 공간·범위 데이터를 다차원으로 색인해, 겹침 여부를 빠르게 찾을 수 있습니다.

인덱스비유잘하는 연산
B-tree (기본)전화번호부 (가나다 정렬)= < > — 값 크기 비교
GiST지도 (공간 색인)&& — 범위·도형 겹침 검사

WITH — 컬럼마다 "무엇이 충돌인지" 지정
room WITH =은 "room이 같으면 충돌 후보", during WITH &&은 "during이 겹치면 충돌 후보"입니다. 두 조건이 동시에 참일 때만 거부합니다. 방만 같고 시간이 달라도 OK, 시간이 겹쳐도 다른 방이면 OK입니다.

컬럼WITH 연산자의미
room WITH ==방 이름이 같으면 충돌 후보
during WITH &&&&시간대가 겹치면 충돌 후보
🧩 CREATE EXTENSION — PostgreSQL 플러그인 설치

CREATE EXTENSION은 PostgreSQL에 플러그인을 설치하는 명령어입니다. Node.js의 npm install, Python의 pip install과 같은 개념입니다. IF NOT EXISTS는 "이미 설치돼 있으면 에러 내지 말고 넘어가"라는 의미입니다.

btree_gist가 필요한 이유는 우리 EXCLUDE 조건이 하나의 GiST 인덱스에서 두 연산자를 동시에 처리해야 하기 때문입니다. GiST는 범위 겹침(&&)은 원래 잘 처리하지만, 일반 텍스트의 동등 비교(=)는 처리하지 못합니다. btree_gist는 GiST가 =도 처리할 수 있게 가르쳐주는 플러그인입니다.

btree_gist 없을 때btree_gist 있을 때
&& (범위 겹침)GiST 가능 ✅GiST 가능 ✅
= (텍스트 같음)GiST 불가 ❌GiST 가능 ✅
study_db=# — EXCLUDE USING gist (시간대 겹침 방지)
CREATE EXTENSION IF NOT EXISTS btree_gist;  -- 범위 타입과 = 비교를 같이 쓰려면 필요

CREATE TABLE room_booking (
  id      int GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  room    text NOT NULL,
  during  tsrange NOT NULL,                         -- 시작~끝 범위
  EXCLUDE USING gist (room WITH =, during WITH &&)  -- 방이 같고 시간이 겹치면 거부
);
INSERT INTO room_booking (room, during)
  VALUES ('A', '[2026-06-09 10:00, 2026-06-09 11:00)');   -- OK
INSERT INTO room_booking (room, during)
  VALUES ('A', '[2026-06-09 11:00, 2026-06-09 12:00)');   -- 인접(딱 붙음), 겹치지 않음 → OK

-- 10:30~11:30은 기존 10:00~11:00과 겹침 → 거부
INSERT INTO room_booking (room, during)
  VALUES ('A', '[2026-06-09 10:30, 2026-06-09 11:30)');
ERROR:  conflicting key value violates exclusion constraint
🧩 PG18: WITHOUT OVERLAPS — 가격 이력처럼 "기간이 겹치면 안 되는" 데이터

PG18 EXCLUDE가 "회의실 예약 충돌 방지"라면, WITHOUT OVERLAPS는 같은 개념을 기본키에 바로 붙이는 문법입니다(PG18부터 지원).

비유 — 상품 가격 이력: 상품 1번의 가격이 1월~6월엔 10,000원, 6월~12월엔 12,000원이라면 두 행이 모두 product_id = 1이지만 기간이 다르므로 괜찮습니다. 그런데 실수로 "3월~9월에 11,000원"을 추가하면 기존 두 기간과 겹쳐버립니다. WITHOUT OVERLAPS를 PK에 붙이면 이런 겹침을 DB가 자동으로 막습니다.

study_db=# — (PG18) temporal PK · WITHOUT OVERLAPS
CREATE EXTENSION IF NOT EXISTS btree_gist;

CREATE TABLE price_history (
  product_id  int,
  valid_at    daterange,                                        -- 유효 기간
  price       numeric(10,2) NOT NULL,
  PRIMARY KEY (product_id, valid_at WITHOUT OVERLAPS)           -- 같은 상품끼리 기간 겹침 금지
);
INSERT INTO price_history VALUES (1, '[2026-01-01, 2026-06-01)', 10.00);  -- 1~6월 OK
INSERT INTO price_history VALUES (1, '[2026-06-01, 2026-12-01)', 12.00);  -- 6~12월 인접 OK
SELECT * FROM price_history ORDER BY valid_at;
 product_id |        valid_at         | price
------------+-------------------------+-------
          1 | [2026-01-01,2026-06-01) | 10.00
          1 | [2026-06-01,2026-12-01) | 12.00

-- 3~9월은 기존 두 기간과 모두 겹침 → 거부
INSERT INTO price_history VALUES (1, '[2026-03-01, 2026-09-01)', 11.00);
ERROR:  conflicting key value violates exclusion constraint "price_history_pkey"

6✅ 셀프 체크

Q 무결성 제약을 앱이 아니라 DB에 두는 이유는?

앱에는 버그와 동시성(race condition)이 있기 때문입니다. 예를 들어 두 요청이 동시에 "중복 없음"을 확인하고 둘 다 INSERT하면 중복이 들어갑니다. UNIQUE·CHECK·FK를 DB에 두면 그런 경우에도 DB가 마지막 방어선으로 막아 주어, 데이터가 항상 규칙을 지킵니다.

Q IDENTITY와 uuidv7() 중 무엇을 기본키로?

일반적인 단일 DB 테이블이면 작고 빠른 GENERATED ALWAYS AS IDENTITY가 기본입니다. ID를 URL 등 외부에 노출하거나(순번 추측 방지) 분산 환경·클라이언트 생성이 필요하면 uuidv7()를 씁니다. v7은 시간순 정렬이라 인덱스 친화적입니다.

Q ON DELETE CASCADE는 어떻게 동작하나?

부모 행이 삭제되면 그 부모를 참조하던 자식 행도 함께 자동 삭제됩니다. 예에서 users의 id=1을 지우자, 그를 참조하던 orders 행들이 모두 사라졌습니다. 대안으로 자식 FK를 NULL로 만드는 SET NULL, 자식이 있으면 삭제를 거부하는 RESTRICT(기본)가 있습니다.
