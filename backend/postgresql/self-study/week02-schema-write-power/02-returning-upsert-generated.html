RETURNING · UPSERT · MERGE
쓰기 측 강력 기능

쓰기 한 번에 결과까지 받아오고(RETURNING), "있으면 수정, 없으면 삽입"을 한 문장으로(ON CONFLICT) — Postgres의 쓰기 측 무기들입니다.

“

쓰기 한 번에 결과까지 받아오고(RETURNING), '있으면 수정 없으면 삽입'을 한 문장으로(ON CONFLICT) — Postgres의 쓰기 측 무기들.

  • RETURNING으로 왕복을 줄이고, (PG18) OLD/NEW로 변경 전후를 한 번에 받는다
  • UPSERT(ON CONFLICT)와 MERGE로 삽입/수정을 한 문장에 담는다
  • 생성 컬럼 — (PG18) 기본 virtual vs STORED의 차이를 안다

1RETURNING — 쓰기 + 결과를 한 번에

보통 INSERT 후 "방금 만든 행의 id가 뭐지?"를 알려면 다시 SELECT를 날려야 합니다. RETURNING은 그 왕복을 없앱니다 — INSERT/UPDATE/DELETE가 영향받은 행의 컬럼을 그대로 돌려줍니다.

🧾 비유 — 주문하자마자 번호표를 받는 것

음식점에서 주문한 뒤 "제 주문 번호가 뭐예요?"를 따로 물어봐야 한다면 — 그게 보통의 INSERT + SELECT 패턴입니다(네트워크 왕복 2번). RETURNING은 주문하는 순간 직원이 바로 "주문번호 A-123입니다"라고 알려주는 것과 같습니다. 왕복이 1번으로 줄고, 그 사이에 다른 요청이 값을 바꿔버릴 위험(경쟁 조건)도 없어집니다.

study_db=# — INSERT/DELETE ... RETURNING
INSERT INTO products (sku, price) VALUES ('A-1', 100.00)
  RETURNING id, created_at;          -- 재조회 없이 즉시 받는다
 id |          created_at
----+-------------------------------
  1 | 2026-06-09 12:03:07.599352+00

-- DELETE도 무엇을 지웠는지 돌려받는다
DELETE FROM products WHERE sku = 'TMP' RETURNING id, sku;
 id | sku
----+-----
  2 | TMP

PG18에서는 한 걸음 더 나아가, UPDATE의 변경 전 값(OLD)과 변경 후 값(NEW)을 동시에 돌려받을 수 있습니다. 단, RETURNING old.*, new.*처럼 그대로 쓰면 컬럼명이 겹치므로(같은 price가 둘) 별칭을 주는 게 깔끔합니다.

🏦 비유 — 은행 이체 확인서

은행 이체 후 "이전 잔액: 100만원 → 이후 잔액: 90만원"처럼 변경 전후를 한눈에 보여주는 확인서와 같습니다. PG18 이전에는 UPDATE 전에 별도로 SELECT해서 이전 값을 미리 챙겨둬야 했습니다. RETURNING old.price, new.price는 그 수고를 없애고 변경 전후 값을 UPDATE 한 문장으로 받아옵니다.

study_db=# — (PG18) RETURNING OLD / NEW
UPDATE products SET price = price * 1.10 WHERE sku = 'A-1'
  RETURNING old.price AS old_price, new.price AS new_price;
 old_price | new_price
-----------+-----------
    100.00 |    110.00
⚠️ MySQL엔 표준 RETURNING이 없다

MySQL에서는 INSERT 후 LAST_INSERT_ID()를 부르고, 다른 컬럼이 더 필요하면 한 번 더 SELECT를 던져야 합니다(왕복 2회 + 동시성 주의). Postgres의 RETURNING은 이 패턴을 한 문장으로 줄여, 네트워크 왕복과 경쟁 조건을 동시에 없앱니다.

방금 만든 행 받기 MySQL

INSERT 후 SELECT LAST_INSERT_ID();로 id만 받고, 나머지 컬럼은 SELECT … WHERE id = ?로 재조회. UPDATE 전후 값은 별도 SELECT로 직접 챙겨야 합니다.

방금 만든 행 받기 PostgreSQL

INSERT … RETURNING id, created_at처럼 한 문장에 받습니다. PG18은 UPDATE에서 RETURNING old.*, new.*로 변경 전후 값까지 즉시 돌려줍니다.

2UPSERT — INSERT ... ON CONFLICT

"있으면 수정, 없으면 삽입"은 너무 흔한 요구라 별명까지 있습니다 — UPSERT. Postgres는 INSERT … ON CONFLICT로 이를 한 문장·원자적으로 처리합니다. 충돌 시 삽입하려던 행은 특수 테이블 EXCLUDED로 참조합니다.

📱 비유 — 소셜 로그인

카카오 로그인을 처리할 때 "이 이메일로 가입한 회원이 이미 있으면 마지막 로그인 시각만 업데이트, 없으면 새 회원으로 등록"하는 상황을 생각해보세요. 코드로 구현하면 SELECT → 있으면 UPDATE, 없으면 INSERT를 따로 작성해야 하고 동시 요청이 들어오면 경쟁 조건도 생깁니다. ON CONFLICT는 이걸 한 문장·원자적으로 해결합니다.

🎯 ON CONFLICT — 충돌 대상을 명시해야 한다

ON CONFLICT는 "어떤 조건에서 충돌이 발생하는가"를 먼저 지정합니다. 두 가지 방식이 있습니다:

  • ON CONFLICT (sku) — 특정 컬럼의 UNIQUE 제약을 기준으로 충돌을 감지합니다.
  • ON CONFLICT ON CONSTRAINT constraint_name — 제약 이름을 직접 지정합니다. 복합 UNIQUE 제약(컬럼 2개 이상)처럼 컬럼 하나로 특정할 수 없을 때 씁니다.

충돌이 없으면 ON CONFLICT 절 자체가 무시되고 그냥 삽입됩니다. 충돌이 있을 때만 DO UPDATE 또는 DO NOTHING이 실행됩니다.

🔑 비유 — EXCLUDED는 "문 앞에서 거절당한 행"

EXCLUDED는 "삽입하려다 충돌로 거절당한 그 행"을 가리키는 가상 테이블입니다. DO UPDATE SET qty = inventory.qty + EXCLUDED.qty는 "기존 행의 수량(inventory.qty)에, 들어오려던 행의 수량(EXCLUDED.qty)을 더하라"는 뜻 — 즉 기존 값과 새 값을 조합할 수 있습니다. DO NOTHING은 "충돌하면 조용히 무시" — 예를 들어 좋아요 버튼을 두 번 눌러도 한 번만 반영될 때 씁니다.

study_db=# — ON CONFLICT DO UPDATE / DO NOTHING
INSERT INTO inventory (sku, qty) VALUES ('A-1', 10)
  ON CONFLICT (sku) DO UPDATE SET qty = inventory.qty + EXCLUDED.qty;
-- sku='A-1' 이 없었으므로 그냥 삽입 (qty=10)

INSERT INTO inventory (sku, qty) VALUES ('A-1', 5)
  ON CONFLICT (sku) DO UPDATE SET qty = inventory.qty + EXCLUDED.qty;
-- 이번엔 충돌 → 10 + 5 = 15 로 갱신
SELECT * FROM inventory;
 sku | qty
-----+-----
 A-1 |  15

-- DO NOTHING: 충돌하면 조용히 무시(중복 삽입 방지)
INSERT INTO inventory (sku, qty) VALUES ('A-1', 999)
  ON CONFLICT (sku) DO NOTHING;
 sku | qty
-----+-----
 A-1 |  15      ← 999는 무시됨
study_db=# — EXCLUDED 활용 패턴
-- 1. 새 값으로 덮어쓰기
ON CONFLICT (sku) DO UPDATE SET qty = EXCLUDED.qty;

-- 2. 누적하기 (기존 + 새 값)
ON CONFLICT (sku) DO UPDATE SET qty = inventory.qty + EXCLUDED.qty;

-- 3. 더 큰 값만 유지
ON CONFLICT (sku) DO UPDATE SET qty = GREATEST(inventory.qty, EXCLUDED.qty);

-- 4. 조건부 업데이트 (새 값이 더 클 때만)
ON CONFLICT (sku) DO UPDATE SET qty = EXCLUDED.qty
  WHERE EXCLUDED.qty > inventory.qty;

-- 5. 복합 UNIQUE 제약 기준으로 충돌 감지
ON CONFLICT ON CONSTRAINT user_product_unique DO UPDATE SET qty = EXCLUDED.qty;
UPSERT MySQL

MySQL은 INSERT … ON DUPLICATE KEY UPDATE를 씁니다. 새 값은 VALUES(col)(또는 최신 문법의 별칭)로 참조합니다. 어떤 키든 충돌하면 발동하는 점이 다릅니다.

UPSERT PostgreSQL

Postgres는 INSERT … ON CONFLICT (col) DO UPDATE로 충돌 대상 제약/컬럼을 명시합니다. 새 값은 EXCLUDED.col로 참조하고, DO NOTHING으로 무시할 수도 있습니다.

3MERGE — SQL 표준

MERGE는 "두 테이블을 비교해서, 행마다 다르게 처리"하는 명령입니다. ON CONFLICT가 "지금 삽입하려는 한 행이 충돌하면 어떻게 할까"라면, MERGE는 "소스 목록 전체를 대상 테이블과 대조해, 있는 행은 UPDATE·없는 행은 INSERT·필요 없어진 행은 DELETE"를 한 문장으로 처리합니다.

📱 비유 — 새 폰에 연락처 옮기기

구글 연락처에서 내보낸 파일(contacts_export)을 새 폰의 연락처(my_contacts)에 반영한다고 생각해보세요. 사람마다 상황이 다릅니다:

  • 이미 저장된 사람(매칭) → 최신 번호로 덮어씁니다 (WHEN MATCHED → UPDATE)
  • 새로 추가된 사람(미매칭) → 연락처를 새로 등록합니다 (WHEN NOT MATCHED → INSERT)

이 작업을 코드로 짜면 "파일 한 줄씩 읽어서 있으면 UPDATE, 없으면 INSERT" 루프가 필요합니다. MERGE는 이 루프 전체를 SQL 한 문장으로 대체합니다.

💡 MERGE 문장 읽는 법 — 키워드 4개만 기억하면 된다
  • MERGE INTO 대상테이블 — 실제로 바뀔 테이블 (내 연락처)
  • USING 소스테이블 ON 조건 — 비교 기준이 되는 데이터와 매칭 조건 (내보낸 파일, 이름이 같으면 같은 사람)
  • WHEN MATCHED THEN — 조건이 일치하는 행이 있을 때 → 보통 UPDATE
  • WHEN NOT MATCHED THEN — 조건이 일치하는 행이 없을 때 → 보통 INSERT
study_db=# — MERGE (소스 테이블과 병합)
-- stock_feed(소스): 본사에서 받은 재고 데이터 (A-1=100, B-2=7)
-- inventory(대상): 현재 진열대 데이터 (A-1=15만 있음)

MERGE INTO inventory AS tgt         -- 바꿀 대상 테이블
USING stock_feed AS src             -- 비교할 소스 데이터
  ON tgt.sku = src.sku               -- 매칭 조건: sku가 같으면 같은 상품
WHEN MATCHED THEN                    -- 양쪽 모두 있을 때(A-1)
  UPDATE SET qty = src.qty           --   → 본사 수량으로 덮어쓰기
WHEN NOT MATCHED THEN                -- 소스에만 있고 대상엔 없을 때(B-2)
  INSERT (sku, qty) VALUES (src.sku, src.qty);  -- → 새로 삽입

SELECT * FROM inventory ORDER BY sku;
 sku | qty
-----+-----
 A-1 | 100      ← MATCHED → 15에서 100으로 갱신
 B-2 |   7      ← NOT MATCHED → 새로 삽입됨
💡 ON CONFLICT vs MERGE — 한 문장 판단 기준

"지금 INSERT하는 이 한 행"이 충돌 났을 때 처리하는 거라면 → ON CONFLICT가 간결합니다.
"외부 목록 전체"를 대상 테이블과 대조해 매칭/비매칭별로 다르게 처리해야 한다면 → MERGE가 적합합니다.
단순 upsert 한 건에 MERGE는 과합니다 — 코드가 길어지고 읽기 어려워집니다.

4생성 컬럼(Generated Columns)

DB가 자동으로 계산해서 채워주는 컬럼입니다. 예를 들어 "부가세 포함 금액(price_tax)은 항상 price × 1.10"이라면, 앱 코드에서 매번 곱셈을 해서 INSERT할 필요가 없습니다. 컬럼을 만들 때 계산식을 한 번 선언해두면 DB가 알아서 채워줍니다. 직접 값을 넣으려 하면 에러가 납니다 — 항상(ALWAYS) DB가 계산하는 컬럼이기 때문입니다.

📊 비유 — 엑셀의 수식 셀

엑셀에서 B열에 =A1*1.10을 입력해두면, A열 값을 바꿀 때마다 B열이 자동으로 업데이트됩니다. 생성 컬럼이 정확히 이 개념입니다. 앱이 price만 넣어주면 price_tax는 DB가 계산합니다.

계산식을 앱 곳곳에 흩뿌리지 않아도 되고, 실수로 다른 값이 들어갈 걱정도 없습니다.

🗄️ STORED vs virtual — 계산 결과를 저장해두냐, 아니냐

생성 컬럼은 두 가지 방식으로 동작합니다:

  • STORED — INSERT/UPDATE할 때 미리 계산해서 디스크에 저장합니다. 조회할 때는 저장된 값을 그냥 읽으면 되므로 빠릅니다. 대신 공간을 차지하고, 인덱스를 걸 수 있습니다.
  • virtual — 디스크에 저장하지 않고, SELECT할 때마다 그 자리에서 계산합니다. 공간을 아낄 수 있지만, 인덱스를 걸 수 없습니다.

언제 무엇을 쓸까? 검색 조건이나 정렬에 쓰일 컬럼 → STORED(인덱스 필요). 단순 표시용 계산값 → virtual로 충분합니다.

모드언제 계산디스크 저장인덱스
STORED쓸 때 (INSERT/UPDATE)저장함 — 조회 빠름가능 ✅
virtual PG18 기본읽을 때 (SELECT)저장 안 함 — 공간 절약불가 ❌
study_db=# — STORED vs virtual
CREATE TABLE line_item (
  name        text,
  price       numeric(10,2),

  -- STORED: INSERT/UPDATE 시점에 계산해서 디스크에 저장
  price_tax   numeric(10,2) GENERATED ALWAYS AS (price * 1.10) STORED,

  -- virtual(PG18 기본): STORED/VIRTUAL 생략 → SELECT할 때마다 계산
  search_name text GENERATED ALWAYS AS (lower(name))
);

-- name, price만 넣으면 나머지 두 컬럼은 DB가 채워줌
INSERT INTO line_item (name, price) VALUES ('Coffee', 100.00);

SELECT * FROM line_item;
  name  | price  | price_tax | search_name
--------+--------+-----------+-------------
 Coffee | 100.00 |    110.00 | coffee      ← 앱이 넣은 건 name, price뿐

-- 카탈로그로 확인: 's'=stored, 'v'=virtual
SELECT attname, attgenerated FROM pg_attribute
  WHERE attrelid = 'line_item'::regclass AND attgenerated <> '';
   attname   | attgenerated
-------------+--------------
 price_tax   | s             ← STORED
 search_name | v             ← virtual (PG18 기본)
🧩 PG18 이전과의 차이

PG18 PG18 이전에는 생성 컬럼이 STORED만 지원됐습니다. PG18부터 virtual도 가능해졌고, 키워드를 생략하면 기본이 virtual입니다. 즉, 기존 코드에서 STORED를 생략하면 PG18에서는 동작이 달라지므로 주의하세요.

복습

5✅ 셀프 체크

Q RETURNING이 절약하는 왕복은?

INSERT 후 "방금 만든 id·created_at" 같은 값을 알려면 보통 SELECT를 한 번 더 던져야 합니다. RETURNING은 INSERT/UPDATE/DELETE가 영향받은 행의 컬럼을 그 자리에서 돌려줘, 네트워크 왕복과 그 사이의 경쟁 조건을 없앱니다. MySQL은 LAST_INSERT_ID() + 재조회 패턴이 필요합니다.

Q ON CONFLICT의 EXCLUDED란?

충돌 때문에 삽입되지 못한 "들어오려던 행"을 가리키는 가상 테이블입니다. DO UPDATE SET qty = inventory.qty + EXCLUDED.qty처럼 기존 행 값과 새 값을 조합할 때 씁니다. DO NOTHING이면 충돌 시 조용히 무시합니다.

Q ON CONFLICT (sku)와 ON CONFLICT ON CONSTRAINT의 차이는?

ON CONFLICT (sku)는 컬럼 이름으로 충돌 대상을 지정합니다 — 해당 컬럼에 UNIQUE 제약이 있어야 합니다. ON CONFLICT ON CONSTRAINT constraint_name은 제약 이름으로 직접 지정하는 방식으로, 컬럼 2개 이상의 복합 UNIQUE 제약처럼 컬럼 하나로 특정할 수 없을 때 씁니다. 충돌이 없으면 ON CONFLICT 절 자체가 무시되고 그냥 삽입됩니다.

Q 생성 컬럼의 STORED vs virtual 차이는?

STORED는 쓸 때 계산해 디스크에 저장(읽기 빠름·공간 사용), virtual은 읽을 때 계산(공간 절약·쓰기 부담↓)입니다. PG18부터 키워드를 생략하면 기본이 virtual입니다. 인덱스가 필요한 표현식이면 STORED를 명시하세요.
