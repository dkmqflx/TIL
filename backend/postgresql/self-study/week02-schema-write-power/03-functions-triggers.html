1PL/pgSQL 함수 — DB 안에 사는 작은 함수

❓ PL/pgSQL이 뭔가요? — SQL과 무엇이 다른가

SQL은 원래 명령만 됩니다 — "이 데이터 가져와", "이 데이터 바꿔". 변수·조건문·반복문 같은 건 없습니다.

PL/pgSQL은 SQL에 그것을 붙인 겁니다:

할 수 있는 것
SQLSELECT / INSERT / UPDATE / DELETE (단순 명령)
PL/pgSQLSQL + 변수 선언 + IF/ELSE + 반복문 + 예외처리

한 마디로 "Postgres 안에서 돌아가는 미니 프로그래밍 언어"입니다. 이걸로 함수를 만들어 DB 안에 저장해두면, SELECT 함수명(...)으로 어디서든 호출할 수 있습니다.

실무에서 직접 PL/pgSQL 함수를 작성하는 경우는 드물고, 트리거 함수를 만들 때 주로 씁니다 — updated_at 자동 갱신이나 감사 로그처럼 "DB 이벤트가 발생하면 자동으로 뭔가 실행"하는 경우요.

🧮 비유 — 여러 팀이 공유하는 공용 계산기

"할인가 계산 로직"을 앱 코드에서 구현하면, NestJS 서버·관리자 도구·배치 스크립트 등 여러 곳에 같은 계산식이 흩어집니다. 계산식이 바뀌면 다 찾아서 고쳐야 합니다.

PL/pgSQL 함수로 DB에 한 번 정의해두면, 모든 곳에서 같은 함수를 공유합니다. 계산식을 바꿔야 할 때 DB 함수 하나만 수정하면 됩니다.

📐 함수 구조 한눈에 보기
  • CREATE OR REPLACE FUNCTION 함수명(인자) RETURNS 반환타입 — 함수 이름·입력·출력 타입 선언. OR REPLACE는 이미 있으면 덮어씀
  • LANGUAGE plpgsql — PL/pgSQL로 작성된 함수임을 명시
  • AS $$ … $$ — 함수 본문을 감싸는 달러 인용. 본문 안에 작은따옴표가 많아도 충돌하지 않음
  • DECLARE … ; — 지역 변수를 선언하는 블록. 변수가 없으면 생략 가능
  • BEGIN … END; — 실제로 실행할 코드
  • RAISE EXCEPTION '…' — 오류를 발생시키고 트랜잭션 롤백
  • RETURN 값; — 결과를 반환하고 함수 종료
study_db=# — PL/pgSQL 함수 기본 구조
CREATE OR REPLACE FUNCTION discounted_price(price numeric, pct numeric)
RETURNS numeric
LANGUAGE plpgsql
AS $$
DECLARE
  result numeric;            -- 지역 변수 선언
BEGIN
  IF pct < 0 OR pct > 100 THEN
    RAISE EXCEPTION '할인율은 0~100 사이여야 합니다: %', pct;  -- 오류 발생
  END IF;
  result := price * (1 - pct / 100);   -- := 은 PL/pgSQL의 대입 연산자
  RETURN round(result, 2);             -- 소수점 둘째 자리까지 반환
END;
$$;

SELECT discounted_price(10000, 30) AS sale_price;
 sale_price
------------
    7000.00     ← 10000 × (1 - 30/100) = 7000

2트리거 기본 — BEFORE/AFTER · NEW/OLD

트리거(trigger)는 INSERT/UPDATE/DELETE가 일어날 때 DB가 자동으로 함수를 실행시키는 장치입니다. 앱 코드가 아무것도 하지 않아도, 특정 이벤트가 발생하는 순간 미리 정해둔 함수가 실행됩니다.

🚨 비유 — 문에 달린 자동 센서

편의점 문에 달린 센서를 생각해보세요. 손님이 들어오면(이벤트 발생) 자동으로 "딩동" 소리가 납니다(함수 실행). 점원이 직접 버튼을 누르지 않아도 됩니다.

트리거도 마찬가지입니다. "article 테이블에 UPDATE가 생기면" → 자동으로 set_updated_at() 함수 실행. 앱이 일일이 updated_at = now()를 챙기지 않아도 됩니다.

⏱️ BEFORE vs AFTER — 언제 실행되냐의 차이
  • BEFORE — 행이 디스크에 저장되기 직전에 실행됩니다. NEW를 수정하면 수정된 값이 실제로 저장됩니다. updated_at 채우기처럼 "저장될 값을 손보는" 작업에 사용합니다.
  • AFTER — 행이 저장된 이후에 실행됩니다. 이미 저장이 끝났으므로 값을 바꿀 수 없고, 감사 로그 기록 같은 부수 작업에 사용합니다.
🔄 NEW / OLD — 변경 전후의 행

트리거 함수 안에서는 두 가지 특수 변수로 행에 접근할 수 있습니다:

  • NEW — 새로 저장될 행 (INSERT·UPDATE에서 사용 가능). BEFORE 트리거에서 NEW.컬럼 := 값으로 수정하면 실제 저장값이 바뀜
  • OLD — 변경 전 행 (UPDATE·DELETE에서 사용 가능). INSERT에는 OLD 없음
이벤트OLDNEW
INSERT없음삽입될 행 ✅
UPDATE기존 행 ✅수정될 행 ✅
DELETE삭제될 행 ✅없음
트리거 구조 MySQL

MySQL은 CREATE TRIGGER … BEGIN … END 본문에 로직을 직접 적습니다. 같은 로직을 여러 트리거에서 재사용하기 어렵습니다.

트리거 구조 PostgreSQL

Postgres는 함수와 트리거를 분리합니다. RETURNS trigger 함수를 먼저 만들고, CREATE TRIGGER … EXECUTE FUNCTION f()로 연결합니다. 하나의 함수를 여러 테이블이 공유할 수 있어 재사용성이 높습니다.

3⭐ updated_at 자동 갱신 트리거

실무에서 가장 자주 쓰는 트리거입니다. 행이 수정될 때마다 앱이 updated_at = now()를 챙겨야 한다면, UPDATE를 날리는 경로가 10곳이면 10곳 모두 빠뜨리지 않아야 합니다. 하나라도 빠지면 잘못된 시각이 남습니다. BEFORE UPDATE 트리거를 달면 어떤 경로로 수정하든 DB가 항상 자동으로 처리합니다.

🕰️ 비유 — 자동 날짜 도장 기계

updated_at은 "이 글이 마지막으로 언제 바뀌었나"를 적어두는 칸입니다. 이 칸을 앱이 매번 손으로 챙기는 대신, DB 안에 자동 도장 기계를 심어두는 게 이 트리거입니다 — article이 수정될 때마다 "지금 시각"이 updated_at에 자동으로 쾅 찍힙니다.

만드는 건 두 부분입니다. ① 도장 기계 자체 = 함수 set_updated_at()("시각을 찍는다"는 동작), ② 기계를 설치할 위치·타이밍 = 트리거 trg_set_updated_at("article이 수정되기 직전에 돌려라"). 함수만 만들면 아무 일도 안 일어나고, 트리거로 연결해야 비로소 작동합니다.

↩️ BEFORE 트리거는 반드시 RETURN NEW 해야 한다

BEFORE 트리거에서 RETURN NEW를 반환하면 → 수정된 NEW 행이 실제로 저장됩니다.
만약 실수로 RETURN NULL을 반환하면 → 그 행의 INSERT/UPDATE가 통째로 취소됩니다. 데이터가 저장되지 않으므로 주의하세요.

study_db=# — set_updated_at 함수 + 트리거
-- 1단계: 트리거가 호출할 함수를 만든다 (RETURNS trigger 필수)
CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS trigger
LANGUAGE plpgsql
AS $$
BEGIN
  NEW.updated_at := now();   -- 저장 직전에 NEW의 updated_at을 현재 시각으로 교체
  RETURN NEW;               -- 수정된 NEW를 반환해야 이 값이 실제로 저장됨
END;
$$;

-- 2단계: 함수를 어떤 테이블의 어떤 이벤트에 연결할지 지정한다
CREATE TRIGGER trg_set_updated_at
  BEFORE UPDATE ON article   -- article 테이블에 UPDATE 직전
  FOR EACH ROW               -- 영향받은 행마다 한 번씩 실행
  EXECUTE FUNCTION set_updated_at();

이제 행을 수정하면, updated_at을 전혀 건드리지 않았는데도 값이 자동으로 바뀝니다. 아래는 INSERT 후 2초 뒤 UPDATE한 실제 결과 — 12:03:22에서 12:03:24로 갱신됐습니다.

study_db=# — updated_at이 자동으로 갱신됨
INSERT INTO article (title, body) VALUES ('첫 글', '초안');
SELECT id, updated_at AS before_update FROM article;
 id |         before_update
----+-------------------------------
  1 | 2026-06-09 12:03:22.597622+00

UPDATE article SET body = '수정본' WHERE id = 1;   -- updated_at은 안 건드림
SELECT id, updated_at AS after_update FROM article;
 id |         after_update
----+-------------------------------
  1 | 2026-06-09 12:03:24.604974+00   ← 트리거가 자동 갱신

4감사 로그(audit) 트리거

"누가 무엇을 어떻게 바꿨나"를 자동으로 기록하는 것이 감사 로그입니다. AFTER INSERT/UPDATE/DELETE 트리거를 달아두면, 어느 경로로 데이터가 바뀌든 변경 전후 값이 audit_log 테이블에 자동으로 쌓입니다. 앱 코드에는 로그 코드가 한 줄도 없어도 됩니다.

📷 비유 — 블랙박스 카메라

사고가 나기 전·후를 자동으로 촬영해두는 블랙박스처럼, 감사 로그 트리거는 데이터 변경 전·후를 자동으로 스냅샷해서 별도 테이블에 보관합니다. 나중에 "이 데이터가 언제, 어떻게 바뀌었나?"를 추적할 때 꺼내봅니다.

⏱️ 왜 BEFORE가 아니라 AFTER인가?

섹션 3의 updated_at 트리거는 저장되기 전에 값을 바꾸는 게 목적이었으니 BEFORE였습니다. 감사 로그는 반대입니다 — "데이터가 실제로 저장됐다"는 사실을 기록하는 게 목적이므로, 저장이 완료된 뒤에 실행되는 AFTER를 씁니다.

그래서 함수 마지막의 RETURN NULL도 괜찮습니다. AFTER 트리거는 저장이 이미 끝난 뒤라 반환값이 무시됩니다 — NULL을 반환해도 이미 저장된 데이터에 영향이 없어요. BEFORE 트리거에서 RETURN NULL을 하면 저장이 취소됐던 것과 정반대입니다.

🔧 DB가 자동으로 알려주는 이벤트 정보 — TG_OP · TG_TABLE_NAME · to_jsonb

트리거 함수가 실행될 때 DB는 "지금 어떤 상황인지"를 자동으로 알려줍니다. 개발자가 직접 넘기지 않아도 함수 안에서 바로 쓸 수 있습니다.

  • TG_OP — 지금 어떤 작업이 일어났는지: 'INSERT' / 'UPDATE' / 'DELETE' 중 하나
  • TG_TABLE_NAME — 어떤 테이블에서 발생했는지: 예) 'article'. 함수 하나를 여러 테이블에 연결해도 각자 테이블 이름이 자동으로 찍힘
  • to_jsonb(OLD) / to_jsonb(NEW) — 행 전체를 JSON으로 변환. 나중에 컬럼이 추가돼도 자동으로 포함되어 편리함

CASE WHEN이 필요한 이유 — 이벤트마다 존재하는 값이 다릅니다. INSERT는 새로 생기는 것이라 OLD가 없고, DELETE는 없애는 것이라 NEW가 없습니다. 없는 값을 그냥 참조하면 에러가 나므로 CASE WHEN TG_OP <> 'INSERT'처럼 분기해서 안전하게 처리합니다.

이벤트OLD (변경 전)NEW (변경 후)
INSERT없음 → NULL삽입된 행 ✅
UPDATE수정 전 행 ✅수정 후 행 ✅
DELETE삭제된 행 ✅없음 → NULL
study_db=# — audit_changes 함수 (jsonb로 old/new 기록)
CREATE OR REPLACE FUNCTION audit_changes()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  INSERT INTO audit_log (table_name, action, old_row, new_row)
  VALUES (
    TG_TABLE_NAME,           -- 자동 제공: 이 트리거가 발생한 테이블 이름
    TG_OP,                   -- 자동 제공: 'INSERT' / 'UPDATE' / 'DELETE'
    -- INSERT에는 OLD가 없으므로 CASE로 분기
    CASE WHEN TG_OP <> 'INSERT' THEN to_jsonb(OLD) END,
    -- DELETE에는 NEW가 없으므로 CASE로 분기
    CASE WHEN TG_OP <> 'DELETE' THEN to_jsonb(NEW) END
  );
  RETURN NULL;   -- AFTER 트리거는 이미 저장이 끝난 뒤라 반환값이 무시됨
END;
$$;

CREATE TRIGGER trg_audit
  AFTER INSERT OR UPDATE OR DELETE ON article
  FOR EACH ROW EXECUTE FUNCTION audit_changes();
📖 함수 코드 한 줄씩 읽기

함수 본문은 audit_log 테이블에 딱 한 행을 INSERT하는 것이 전부입니다. 네 개의 값을 채웁니다:

  1. TG_TABLE_NAME → 어느 테이블에서 발생했는지 (예: 'article')
  2. TG_OP → 어떤 작업이었는지 (예: 'UPDATE')
  3. CASE WHEN TG_OP <> 'INSERT' THEN to_jsonb(OLD) END → INSERT가 아닐 때만 OLD를 JSON으로 저장, INSERT면 NULL
  4. CASE WHEN TG_OP <> 'DELETE' THEN to_jsonb(NEW) END → DELETE가 아닐 때만 NEW를 JSON으로 저장, DELETE면 NULL

마지막 RETURN NULL은 AFTER 트리거이므로 반환값이 무시됩니다 — 관례상 NULL을 씁니다.

이제 article에 INSERT·UPDATE·DELETE를 차례로 실행하면, audit_log에 변경 이력이 자동으로 쌓입니다. 아래는 실제 실행 결과입니다.

study_db=# — audit_log에 자동으로 쌓인 이력
INSERT INTO article (title, body) VALUES ('둘째 글', 'hello');
UPDATE article SET title = '둘째 글(수정)' WHERE title = '둘째 글';
DELETE FROM article WHERE title = '둘째 글(수정)';

SELECT action, old_row->>'title' AS old_title,
       new_row->>'title' AS new_title
  FROM audit_log ORDER BY id;
 action |   old_title   |   new_title
--------+---------------+---------------
 INSERT |               | 둘째 글
 UPDATE | 둘째 글       | 둘째 글(수정)
 DELETE | 둘째 글(수정) |
⚠️ 트리거 남용 주의

트리거는 강력하지만 숨은 로직입니다 — 코드 어디에도 안 보이는데 행이 바뀌니, 디버깅이 어렵고 "왜 이 값이 바뀌었지?"를 추적하기 힘듭니다. updated_at·감사 로그처럼 가볍고 정형화된 부수 작업에는 훌륭하지만, 무겁거나 비즈니스 핵심 로직(결제·주문 상태 전이 등)은 트리거보다 앱/서비스 계층에 두는 편이 추적·테스트에 유리합니다.

복습

5✅ 셀프 체크

Q updated_at을 트리거로 처리하면 좋은 점은?

앱의 모든 UPDATE 경로마다 updated_at = now()를 챙길 필요가 없어집니다. 어느 코드·도구로 수정하든 DB가 항상 자동으로 갱신하므로, 한 곳이라도 빠뜨릴 걱정이 없습니다. BEFORE UPDATE 트리거에서 NEW.updated_at := now()로 값을 교체하고 RETURN NEW로 반환합니다.

Q BEFORE와 AFTER 트리거의 차이는?

BEFORE는 행이 저장되기 직전에 실행됩니다. NEW를 수정하고 RETURN NEW를 반환하면 수정된 값이 실제로 저장됩니다(updated_at 채우기에 사용). AFTER는 행이 저장된 이후에 실행되므로 값을 바꿀 수 없고, 감사 로그 기록 같은 부수 작업에 사용합니다. AFTER 트리거의 반환값은 이미 저장이 끝난 뒤라 무시됩니다.

Q BEFORE 트리거에서 RETURN NULL을 반환하면?

BEFORE 트리거에서 RETURN NULL을 반환하면 그 행의 INSERT/UPDATE가 취소됩니다. 데이터가 저장되지 않으므로, updated_at 같은 값만 수정하는 BEFORE 트리거는 반드시 RETURN NEW를 반환해야 합니다.

Q 트리거 남용의 위험은?

트리거는 코드에 드러나지 않는 숨은 로직이라, 행이 왜 바뀌었는지 추적이 어렵고 디버깅·테스트가 까다로워집니다. updated_at·감사 로그 같은 가볍고 정형화된 부수 작업엔 적합하지만, 결제·주문 상태 전이 같은 비즈니스 핵심 로직은 앱/서비스 계층에 두는 편이 추적과 테스트에 유리합니다.
