1PL/pgSQL 함수 — DB 안에 사는 작은 함수

Postgres는 SQL만이 아니라 절차적 언어 PL/pgSQL로 함수를 정의할 수 있습니다. 변수·조건문·반복문·예외를 쓸 수 있어, 간단한 계산 로직을 DB 안에 둘 수 있습니다.

🍔 비유 — DB 안에 사는 작은 함수

PL/pgSQL 함수는 주방 안에 붙어 있는 레시피 카드 같은 것입니다. "할인가 계산"처럼 자주 쓰는 절차를 DB 안에 한 번 정의해 두면, 앱마다 똑같은 계산을 복붙하지 않고 SELECT discounted_price(...)로 불러 쓸 수 있습니다. 본문은 $$ ... $$(달러 인용) 사이에 넣습니다.

study_db=# — PL/pgSQL 함수 기본 구조
CREATE OR REPLACE FUNCTION discounted_price(price numeric, pct numeric)
RETURNS numeric
LANGUAGE plpgsql
AS $$
DECLARE
  result numeric;
BEGIN
  IF pct < 0 OR pct > 100 THEN
    RAISE EXCEPTION '할인율은 0~100 사이여야 합니다: %', pct;
  END IF;
  result := price * (1 - pct / 100);
  RETURN round(result, 2);
END;
$$;

SELECT discounted_price(10000, 30) AS sale_price;
 sale_price
------------
    7000.00

2트리거 기본 — BEFORE/AFTER · NEW/OLD

트리거(trigger)는 INSERT/UPDATE/DELETE가 일어날 때 자동으로 함수를 실행시키는 장치입니다. 두 가지 축을 기억하세요.

구분의미
BEFORE행이 저장되기 전 — NEW를 수정해 저장될 값을 바꿀 수 있음
AFTER행이 저장된 후 — 로그 기록 등 부수 작업에 적합
FOR EACH ROW영향받은 행마다 실행
NEW / OLD변경 후 행 / 변경 전 행
트리거 구조 MySQL

MySQL은 CREATE TRIGGER … BEGIN … END 본문에 로직을 직접 적습니다. 트리거가 곧 코드 블록이라, 같은 로직을 여러 트리거에서 재사용하기 어렵습니다.

트리거 구조 PostgreSQL

Postgres는 함수와 트리거가 분리됩니다. RETURNS trigger 함수를 먼저 만들고, CREATE TRIGGER … EXECUTE FUNCTION f()로 연결합니다. 하나의 함수를 여러 테이블·트리거가 공유할 수 있습니다.

3⭐ updated_at 자동 갱신 트리거

실무에서 가장 단골인 트리거입니다. 행이 수정될 때마다 앱이 updated_at = now()를 일일이 챙기는 건 빠뜨리기 쉽습니다. BEFORE UPDATE 트리거로 DB가 자동으로 갱신하게 맡기세요.

study_db=# — set_updated_at 함수 + 트리거
CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS trigger
LANGUAGE plpgsql
AS $$
BEGIN
  NEW.updated_at := now();   -- 저장되기 전(BEFORE)에 값을 손본다
  RETURN NEW;
END;
$$;

CREATE TRIGGER trg_set_updated_at
  BEFORE UPDATE ON article
  FOR EACH ROW
  EXECUTE FUNCTION set_updated_at();

이제 행을 수정하면, 우리가 updated_at을 건드리지 않았는데도 값이 자동으로 바뀝니다. 아래는 INSERT 후 2초 뒤 UPDATE한 실제 결과 — 12:03:22에서 12:03:24로 갱신됐습니다.

study_db=# — updated_at이 자동으로 갱신됨
INSERT INTO article (title, body) VALUES ('첫 글', '초안');
SELECT id, updated_at AS before_update FROM article;
 id |         before_update
----+-------------------------------
  1 | 2026-06-09 12:03:22.597622+00

UPDATE article SET body = '수정본' WHERE id = 1;   -- updated_at은 안 건드림
SELECT id, updated_at AS after_update FROM article;
 id |         after_update
----+-------------------------------
  1 | 2026-06-09 12:03:24.604974+00   ← 트리거가 자동 갱신

4감사 로그(audit) 트리거

"누가 무엇을 어떻게 바꿨나"를 남기는 감사 로그도 트리거의 단골입니다. AFTER INSERT/UPDATE/DELETE 트리거가 변경 전후 행을 to_jsonb()로 직렬화해 별도 audit_log 테이블에 자동 기록합니다. TG_OP(연산 종류)·TG_TABLE_NAME(테이블 이름)은 트리거 함수 안에서 쓸 수 있는 특수 변수입니다.

study_db=# — audit_changes 함수 (jsonb로 old/new 기록)
CREATE OR REPLACE FUNCTION audit_changes()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  INSERT INTO audit_log (table_name, action, old_row, new_row)
  VALUES (
    TG_TABLE_NAME,                                       -- 테이블 이름
    TG_OP,                                               -- INSERT/UPDATE/DELETE
    CASE WHEN TG_OP <> 'INSERT' THEN to_jsonb(OLD) END,  -- 변경 전
    CASE WHEN TG_OP <> 'DELETE' THEN to_jsonb(NEW) END   -- 변경 후
  );
  RETURN NULL;   -- AFTER 트리거의 반환값은 무시됨
END;
$$;

CREATE TRIGGER trg_audit
  AFTER INSERT OR UPDATE OR DELETE ON article
  FOR EACH ROW EXECUTE FUNCTION audit_changes();

이제 article에 INSERT·UPDATE·DELETE를 차례로 실행하면, audit_log에 변경 이력이 자동으로 쌓입니다. 아래는 실제 실행 결과입니다.

study_db=# — audit_log에 자동으로 쌓인 이력
INSERT INTO article (title, body) VALUES ('둘째 글', 'hello');
UPDATE article SET title = '둘째 글(수정)' WHERE title = '둘째 글';
DELETE FROM article WHERE title = '둘째 글(수정)';

SELECT action, old_row->>'title' AS old_title,
       new_row->>'title' AS new_title
  FROM audit_log ORDER BY id;
 action |   old_title   |   new_title
--------+---------------+---------------
 INSERT |               | 둘째 글
 UPDATE | 둘째 글       | 둘째 글(수정)
 DELETE | 둘째 글(수정) |
⚠️ 트리거 남용 주의

트리거는 강력하지만 숨은 로직입니다 — 코드 어디에도 안 보이는데 행이 바뀌니, 디버깅이 어렵고 "왜 이 값이 바뀌었지?"를 추적하기 힘듭니다. updated_at·감사 로그처럼 가볍고 정형화된 부수 작업에는 훌륭하지만, 무겁거나 비즈니스 핵심 로직(결제·주문 상태 전이 등)은 트리거보다 앱/서비스 계층에 두는 편이 추적·테스트에 유리합니다.

복습

5✅ 셀프 체크

Q updated_at을 트리거로 처리하면 좋은 점은?

앱의 모든 UPDATE 경로마다 updated_at = now()를 챙길 필요가 없어집니다. 어느 코드·도구로 수정하든 DB가 항상 자동으로 갱신하므로, 빠뜨림이 사라지고 값이 신뢰할 수 있게 됩니다. BEFORE UPDATE 트리거에서 NEW.updated_at := now()로 처리합니다.

Q BEFORE와 AFTER 트리거의 차이는?

BEFORE는 행이 저장되기 전에 실행돼 NEW를 수정해 저장될 값을 바꿀 수 있습니다(예: updated_at 채우기). AFTER는 행이 저장된 후에 실행돼, 값을 바꾸기보다 로그 기록 같은 부수 작업(예: 감사 로그)에 적합합니다.

Q 트리거 남용의 위험은?

트리거는 코드에 드러나지 않는 숨은 로직이라, 행이 왜 바뀌었는지 추적이 어렵고 디버깅·테스트가 까다로워집니다. updated_at·감사 로그 같은 가벼운 부수 작업엔 좋지만, 비즈니스 핵심 로직은 앱/서비스 계층에 두는 편이 좋습니다.
