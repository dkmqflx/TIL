JSON vs JSONB
연산자 · GIN 인덱스

Postgres는 반정형(semi-structured) 데이터를 일급 타입으로 다룹니다. 앱에서는 거의 항상 json이 아니라 jsonb를 쓰고, @>로 검색하고 GIN으로 인덱싱합니다.

“

Postgres는 반정형 데이터를 일급으로 다룬다. 거의 항상 json이 아니라 jsonb를 쓰고, @>로 검색하고 GIN으로 인덱싱한다.

  • json과 jsonb의 차이를 알고, 왜 앱은 jsonb를 쓰는지 안다
  • 핵심 연산자(->·->>·#>·#>>·@>·?)를 구분해 쓴다
  • jsonb_set·||로 갱신하고, GIN 인덱스로 검색을 가속한다

1json vs jsonb — 무엇을 쓸까

Postgres에는 JSON을 담는 타입이 둘 있습니다. json은 입력한 원문 텍스트를 그대로 보존합니다 — 키 순서, 중복 키, 공백까지 유지하지만 매 연산마다 다시 파싱해야 해서 느립니다. 반면 jsonb는 입력을 한 번 파싱해 바이너리로 저장합니다 — 키를 정렬하고 중복을 제거하며, 그 대신 연산·인덱스가 훨씬 빠릅니다.

jsonjsonb
저장 형태원문 텍스트파싱된 바이너리
키 순서·공백그대로 보존정규화(정렬·압축)
중복 키유지마지막 값만 남김
연산·인덱스느림, GIN 불가빠름, GIN 가능
결론원문 보존이 진짜 필요할 때만앱의 기본 선택

같은 값을 두 타입에 넣어 보면 차이가 한눈에 보입니다. jsonb는 키를 정렬하고 중복 a를 하나로 합쳤습니다.

study_db=# — 같은 값, 다른 타입
SELECT
  '{"b":1, "a":2, "a":3}'::json  AS as_json,    -- 원문 그대로(중복 a 유지)
  '{"b":1, "a":2, "a":3}'::jsonb AS as_jsonb;   -- 정규화(키 정렬, 중복 제거)
        as_json        |     as_jsonb
-----------------------+------------------
 {"b":1, "a":2, "a":3} | {"a": 3, "b": 1}

2핵심 연산자 — -> · ->> · #> · @> · ?

가장 자주 헷갈리는 한 쌍은 ->와 ->>입니다. 둘 다 키로 값을 꺼내지만 반환 타입이 다릅니다 — ->는 jsonb를(따옴표 포함), ->>는 text를(따옴표 없는 순수 문자열) 돌려줍니다.

study_db=# — products 테이블 + -> (jsonb) vs ->> (text)
CREATE TABLE products (
  id    int GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  name  text NOT NULL,
  data  jsonb NOT NULL
);
INSERT INTO products (name, data) VALUES
  ('키보드', '{"brand":"Keychron","price":129,"tags":["mech","wireless"],"spec":{"switch":"brown"}}'),
  ('마우스', '{"brand":"Logitech","price":59,"tags":["wireless"],"spec":{"dpi":8000}}'),
  ('모니터', '{"brand":"LG","price":299,"tags":["4k","ips"],"spec":{"size":27}}');

SELECT
  name,
  data -> 'brand'         AS brand_jsonb,   -- "Keychron" (따옴표 포함 jsonb)
  data ->> 'brand'        AS brand_text,    -- Keychron   (순수 text)
  (data ->> 'price')::int AS price_int     -- text를 int로 캐스팅
FROM products ORDER BY id;
  name  | brand_jsonb | brand_text | price_int
--------+-------------+------------+-----------
 키보드 | "Keychron"  | Keychron   |       129
 마우스 | "Logitech"  | Logitech   |        59
 모니터 | "LG"        | LG         |       299
⚠️ ->와 ->>를 혼동하지 말 것

비교·조인·캐스팅에는 text를 주는 ->>를 쓰세요. WHERE data -> 'brand' = 'LG'는 jsonb("LG")와 text(LG)를 비교하려다 타입이 어긋납니다. data ->> 'brand' = 'LG'가 올바른 비교입니다. 반대로 결과를 다시 키로 파고들 때는 jsonb를 주는 ->를 이어 써야 합니다(예: data -> 'spec' ->> 'switch').

깊은 경로는 #>(jsonb 반환)·#>>(text 반환)에 경로 배열을 줍니다. 그리고 검색의 주인공은 포함 연산자 @>입니다 — "왼쪽 jsonb가 오른쪽을 통째로 포함하는가"를 묻습니다. 키 존재만 확인하려면 ?(단일)·?|(하나라도)·?&(모두)를 씁니다.

study_db=# — #> 경로, @> 포함, ? 키 존재
-- #>> : 경로로 깊이 들어가 text 반환
SELECT name, data #>> '{spec,switch}' AS switch_text FROM products ORDER BY id;
  name  | switch_text
--------+-------------
 키보드 | brown
 마우스 |
 모니터 |

-- @> : 포함(containment). 배열 원소 포함도 된다
SELECT name FROM products WHERE data @> '{"tags":["wireless"]}';
  name
--------
 키보드
 마우스

-- ? : 최상위 키 존재 / ?& : 모든 키 존재
SELECT name FROM products WHERE data ?& array['brand','price'];   -- 세 행 모두
JSON 다루기 MySQL

MySQL은 단일 JSON 타입 하나뿐입니다(내부적으로 바이너리). 값 추출은 주로 함수로 합니다 — JSON_EXTRACT(data, '$.brand'), 또는 단축 연산자 data->'$.brand'(JSON 반환)·data->>'$.brand'(text 반환). 경로는 $.a.b 같은 JSONPath 문자열입니다.

JSON 다루기 PostgreSQL

Postgres는 json/jsonb 두 타입에, 풍부한 전용 연산자(->·->>·#>·@>·? …)를 제공합니다. 특히 @> 포함 검색을 GIN 인덱스로 가속할 수 있는 점이 결정적입니다(MySQL의 일반 JSON 컬럼에는 없는 능력).

3jsonb 갱신 — jsonb_set · || · - · #-

jsonb는 읽기만 하는 게 아니라 부분 갱신도 자연스럽습니다. jsonb_set은 특정 경로의 값을 교체하고, ||는 두 jsonb를 병합(같은 키는 덮어쓰고 없는 키는 추가)합니다. 키 삭제는 최상위는 -, 깊은 경로는 #-로 합니다.

study_db=# — jsonb 부분 갱신
UPDATE products SET data = jsonb_set(data, '{price}', '139') WHERE name = '키보드';
UPDATE products SET data = data || '{"in_stock": true}'::jsonb WHERE name = '마우스';
UPDATE products SET data = data - 'tags' WHERE name = '모니터';        -- 최상위 키 삭제
UPDATE products SET data = data #- '{spec,dpi}' WHERE name = '마우스';  -- 경로 삭제

SELECT name, data FROM products ORDER BY id;
  name  |                                              data
--------+------------------------------------------------------------------------------------------------
 키보드 | {"spec": {"switch": "brown"}, "tags": ["mech", "wireless"], "brand": "Keychron", "price": 139}
 마우스 | {"spec": {}, "tags": ["wireless"], "brand": "Logitech", "price": 59, "in_stock": true}
 모니터 | {"spec": {"size": 27}, "brand": "LG", "price": 299}
⚠️ 무엇이든 jsonb에 쑤셔 넣지 말 것

jsonb가 편하다고 정형(structured) 데이터까지 한 컬럼에 몰아넣으면, 컬럼 타입·제약·외래키·조인이라는 관계형의 강점을 모두 잃습니다. 자주 조회·조인·검증하는 값(가격, 상태, 사용자 id…)은 일반 컬럼으로 빼세요. jsonb는 진짜로 가변적이거나 희소한(sparse) 속성 — 제품마다 다른 스펙, 외부 API 원본 페이로드, 자유로운 설정값 — 에 어울립니다.

4GIN 인덱스 — @> 검색을 가속

@> 같은 포함 검색은 인덱스가 없으면 모든 행을 풀스캔합니다. GIN(Generalized Inverted Index)은 jsonb 안의 키·값을 역색인으로 만들어 이런 검색을 빠르게 합니다. 두 가지 선택지가 있습니다.

인덱스 종류지원 연산자특징
gin (data) (기본)@> · ? · ?| · ?&키 존재 검색까지 지원, 더 큼
gin (data jsonb_path_ops)@> 만더 작고 빠름(키 존재 ? 불가)

@> 포함 검색만 쓴다면 jsonb_path_ops가 인덱스가 작고 빠릅니다. 충분한 행(아래는 1,500행)에서 EXPLAIN을 보면 풀스캔이 아니라 인덱스를 타는 것을 확인할 수 있습니다.

study_db=# — jsonb_path_ops GIN이 @> 검색에 쓰인다
-- 인덱스가 실제로 쓰이려면 행이 충분해야 한다 → events 1,500행 생성
CREATE TABLE events (id int GENERATED ALWAYS AS IDENTITY PRIMARY KEY, payload jsonb NOT NULL);
INSERT INTO events (payload)
  SELECT jsonb_build_object('type', t, 'level', (i % 3))
  FROM generate_series(1, 500) i, (VALUES ('login'),('logout'),('error')) AS v(t);

CREATE INDEX idx_events_payload ON events USING gin (payload jsonb_path_ops);

EXPLAIN (COSTS OFF)
SELECT * FROM events WHERE payload @> '{"type":"error"}';
                         QUERY PLAN
-------------------------------------------------------------
 Bitmap Heap Scan on events
   Recheck Cond: (payload @> '{"type": "error"}'::jsonb)
   ->  Bitmap Index Scan on idx_events_payload
         Index Cond: (payload @> '{"type": "error"}'::jsonb)
💡 SQL/JSON 경로 질의 맛보기

복잡한 조건 추출에는 jsonb_path_query가 강력합니다. 예: jsonb_path_query(data, '$.items[*].q ? (@ > 4)')는 items 배열에서 q > 4인 값만 골라냅니다(5, 9 …). PG17+의 JSON_TABLE(...)은 한발 더 나아가 jsonb를 관계형 행·열로 펼쳐 일반 테이블처럼 다루게 해 줍니다.

복습

5✅ 셀프 체크

Q json과 jsonb 중 무엇을 골라야 하나?

앱 개발에서는 거의 항상 jsonb입니다. 파싱된 바이너리로 저장돼 연산이 빠르고, GIN 인덱스로 @> 검색을 가속할 수 있기 때문입니다. json은 키 순서·중복 키·공백 등 입력 원문을 글자 그대로 보존해야 하는 특수한 경우에만 씁니다.

Q ->와 ->>의 차이는?

둘 다 키로 값을 꺼내지만 반환 타입이 다릅니다. ->는 jsonb(예: "LG", 따옴표 포함), ->>는 text(예: LG)를 돌려줍니다. 비교·조인·캐스팅엔 ->>, 결과를 다시 키로 파고들 땐 ->를 씁니다.

Q @>와 GIN 인덱스는 어떤 관계인가?

@>는 "포함" 검색 연산자이고, GIN 인덱스가 이 검색을 가속합니다. 인덱스가 없으면 풀스캔이지만, gin (data) 또는 @> 전용의 더 작은 gin (data jsonb_path_ops)를 만들면 EXPLAIN에서 Bitmap Index Scan으로 인덱스를 타는 것을 볼 수 있습니다.
