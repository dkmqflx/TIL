배열 · enum · range
표현력 높은 타입을 알맞게

관계형 안에서도 배열·열거형(enum)·범위(range) 같은 표현력 높은 타입을 쓸 수 있습니다. 남용은 금물이지만, 알맞게 쓰면 스키마가 훨씬 또렷해집니다.

“

관계형 안에서도 배열·열거형·범위 같은 표현력 높은 타입을 쓸 수 있다 — 남용은 금물이지만 알맞게 쓰면 강력하다.

  • 배열(text[]·int[]) 타입과 연산(ANY·@>·&&·unnest)을 안다
  • enum 타입을 만들고, 정렬·값 추가의 주의점을 안다
  • range 타입으로 포함(@>)·겹침(&&)을 표현한다

1배열(array) — text[] · int[]

Postgres는 거의 모든 타입을 배열로 쓸 수 있습니다. 컬럼 타입 뒤에 []를 붙이면 됩니다(예: text[]). 리터럴은 '{a,b}' 형태나 ARRAY['a','b'] 두 방식 모두 가능합니다. 태그처럼 단순한 다중 값에 잘 맞습니다.

⚠️ 배열 인덱싱은 1부터 시작한다

대부분의 언어와 MySQL의 사고방식과 달리, Postgres 배열은 1-based입니다 — 첫 원소는 tags[1]입니다(tags[0]이 아님). 자주 실수하는 지점이니 기억하세요.

study_db=# — 배열 컬럼과 기본 함수
CREATE TABLE articles (
  id    int GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  title text NOT NULL,
  tags  text[] NOT NULL DEFAULT '{}'     -- 빈 배열 기본값
);
INSERT INTO articles (title, tags) VALUES
  ('Postgres 입문', '{db,sql,postgres}'),
  ('JSONB 검색',    ARRAY['db','postgres','jsonb']),
  ('CSS 그리드',    '{css,frontend}');

SELECT title, tags[1] AS first_tag, array_length(tags, 1) AS n_tags FROM articles ORDER BY id;
     title     | first_tag | n_tags
---------------+-----------+--------
 Postgres 입문 | db        |      3
 JSONB 검색    | db        |      3
 CSS 그리드    | css       |      2

배열 검색의 핵심 연산은 셋입니다. = ANY(tags)는 "값이 들어 있는가", @>는 "이 부분집합을 포함하는가", &&는 "겹치는 원소가 하나라도 있는가"입니다. 그리고 unnest로 배열을 행으로 펼쳐 집계할 수 있습니다.

study_db=# — ANY · @> · && · unnest
-- ANY: 배열 안에 특정 값이 있는가
SELECT title FROM articles WHERE 'postgres' = ANY(tags);

-- @>: 포함(부분집합) / &&: 겹침(하나라도)
SELECT title FROM articles WHERE tags @> ARRAY['db','sql'];     -- 둘 다 포함
SELECT title FROM articles WHERE tags && ARRAY['css','jsonb'];  -- 하나라도 겹침

-- unnest: 배열을 행으로 펼쳐 집계
SELECT tag, count(*) AS n FROM articles, unnest(tags) AS tag GROUP BY tag ORDER BY n DESC, tag;
   tag    | n
----------+---
 db       | 2
 postgres | 2
 css      | 1
 ...

-- 배열도 GIN 인덱스로 @>, &&, = ANY 검색을 가속
CREATE INDEX idx_articles_tags ON articles USING gin (tags);
⚠️ 배열 vs 별도 테이블 — 트레이드오프

배열은 단순 태그처럼 그 자체로 의미가 닫혀 있을 때 편합니다. 하지만 각 원소가 독립된 엔티티여서 다른 테이블과 조인하거나, 원소별 속성(생성일, 작성자…)·제약이 필요하다면, 배열이 아니라 별도 테이블 + 외래키로 정규화하는 편이 옳습니다. "이 값에 조인이나 추가 속성이 필요한가?"가 갈림길입니다.

2enum — 재사용 가능한 열거형 '타입'

정해진 몇 개의 값만 허용하고 싶을 때 enum을 씁니다. 핵심은 정렬이 알파벳순이 아니라 정의한 순서를 따른다는 점입니다(low < mid < high). 값 추가는 ALTER TYPE ... ADD VALUE로 합니다.

study_db=# — enum 정의 · 정렬 · 값 추가
CREATE TYPE mood AS ENUM ('low', 'mid', 'high');
ALTER TYPE mood ADD VALUE IF NOT EXISTS 'peak';   -- 맨 끝에 값 추가

INSERT INTO mood_log (note, level) VALUES
  ('피곤', 'low'), ('보통', 'mid'), ('최고', 'peak'), ('좋음', 'high');

-- 정렬은 '정의 순서': low < mid < high < peak (알파벳이 아님!)
SELECT note, level FROM mood_log ORDER BY level;
 note | level
------+-------
 피곤 | low
 보통 | mid
 좋음 | high
 최고 | peak
⚠️ ADD VALUE는 트랜잭션 안에서 곧바로 못 쓴다

ALTER TYPE ... ADD VALUE로 추가한 값은, 같은 트랜잭션 안에서는 그 값을 곧장 사용할 수 없습니다(별도 커밋 후에 가능). 또한 enum 값은 중간 위치 삽입·삭제가 번거롭습니다. 값이 자주 바뀌고 추가/삭제가 잦다면, enum보다 참조(lookup) 테이블 + 외래키가 더 유연합니다.

열거형 MySQL

MySQL의 ENUM('low','mid','high')은 컬럼에 종속된 정의입니다. 같은 열거형을 다른 테이블에서도 쓰려면 컬럼마다 다시 적어야 하고, 정의가 흩어지기 쉽습니다.

열거형 PostgreSQL

Postgres는 CREATE TYPE mood AS ENUM (...)로 독립된 타입을 만듭니다. 한 번 정의하면 여러 테이블·함수에서 재사용할 수 있고, 정렬 순서도 정의에 담깁니다.

3range — 범위를 한 값으로

"1부터 10까지", "오전 10시~11시" 같은 범위를 두 컬럼으로 쪼개지 않고 한 값으로 다루는 타입입니다. int4range·numrange·tstzrange 등이 있고, 경계는 [)(시작 포함, 끝 미포함)가 기본이자 권장입니다. 포함은 @>, 겹침은 &&로 묻습니다.

study_db=# — range 포함(@>) · 겹침(&&)
SELECT
  int4range(1, 10)                AS r1,          -- [1,10)
  numrange(0, 100) @> 50::numeric AS contains_50,  -- 포함?
  int4range(1,10) && int4range(5,20) AS overlaps;     -- 겹침?
   r1   | contains_50 | overlaps
--------+-------------+----------
 [1,10) | t           | t

범위 타입의 진가는 겹침 방지입니다. Week 2 · Lesson 01에서 본 EXCLUDE USING gist와 결합하면, "같은 회의실에서 시간대가 겹치는 예약 금지"를 DB가 직접 보장합니다.

📅 비유 — 예약 시간대 겹침 막기

예약 시간을 start/end 두 컬럼으로 두면 "겹침"을 매번 손으로 계산해야 합니다. during tstzrange 한 컬럼으로 두고 EXCLUDE USING gist (room WITH =, during WITH &&)를 걸면, "같은 방(=)이면서 시간대가 겹치면(&&) 거부"를 DB가 강제합니다. 인접(끝=다음 시작)은 [) 경계 덕에 겹치지 않아 통과합니다.

study_db=# — range + EXCLUDE로 예약 겹침 방지
CREATE EXTENSION IF NOT EXISTS btree_gist;

CREATE TABLE room_booking (
  id     int GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  room   text NOT NULL,
  during tstzrange NOT NULL,
  EXCLUDE USING gist (room WITH =, during WITH &&)
);
INSERT INTO room_booking (room, during) VALUES
  ('A', '[2026-06-09 10:00+09, 2026-06-09 11:00+09)'),   -- OK
  ('A', '[2026-06-09 11:00+09, 2026-06-09 12:00+09)');   -- 인접, 안 겹침 → OK

-- 같은 방 A의 겹치는 시간대는 막힌다 (의도적 에러)
INSERT INTO room_booking (room, during)
  VALUES ('A', '[2026-06-09 10:30+09, 2026-06-09 11:30+09)');
ERROR:  conflicting key value violates exclusion constraint

4✅ 셀프 체크

Q 배열을 쓸 때와 별도 테이블로 정규화할 때는?

배열은 단순 태그처럼 값이 그 자체로 닫혀 있고 조인이 필요 없을 때 적합합니다. 각 원소를 다른 테이블과 조인하거나 원소별 속성·제약(작성자, 생성일…)이 필요하면 별도 테이블 + 외래키로 정규화해야 합니다.

Q enum에 값을 추가할 때 주의할 점은?

ALTER TYPE ... ADD VALUE로 추가한 값은 같은 트랜잭션 안에서 곧바로 사용할 수 없습니다(커밋 후 가능). 중간 삽입·삭제도 번거롭습니다. 값이 자주 바뀌면 enum보다 참조 테이블 + 외래키가 유연합니다. 정렬은 알파벳이 아니라 정의 순서를 따른다는 점도 기억하세요.

Q range의 &&는 무엇을 하나?

&&는 두 범위가 겹치는지(overlap)를 판별합니다. EXCLUDE USING gist (room WITH =, during WITH &&)처럼 쓰면 "같은 방의 시간대가 겹치는 예약"을 DB가 거부합니다. 포함 여부는 @>로 판별합니다.
