전문검색
tsvector/tsquery · pg_trgm

LIKE '%...%'는 인덱스를 못 타 느립니다. Postgres는 내장 전문검색(tsvector)과 트라이그램(pg_trgm)으로 진짜 검색을 합니다.

“

LIKE '%...%'는 인덱스를 못 타 느리다. Postgres는 내장 전문검색(tsvector)과 트라이그램(pg_trgm)으로 진짜 검색을 한다.

  • tsvector/tsquery로 전문검색을 하고, 왜 LIKE로는 부족한지 안다
  • ts_rank로 관련도를 매기고, GIN 인덱스로 가속한다
  • pg_trgm으로 유사도·오타 허용·부분일치(ILIKE) 검색을 가속한다

1왜 LIKE '%word%'로는 부족한가

검색을 WHERE body LIKE '%word%'로 처리하면 두 가지 문제가 있습니다. 첫째, 선두 와일드카드(%로 시작)는 일반 B-tree 인덱스를 못 탑니다 — 모든 행을 풀스캔합니다. 둘째, running과 run을 같은 말로 보는 형태소 분석도, 결과를 관련도순으로 정렬하는 랭킹도 없습니다.

🍔 비유 — 색인 없는 책 vs 색인된 책

LIKE '%word%'는 색인(index)이 없는 책에서 특정 단어를 찾으려고 첫 장부터 끝까지 훑는 것과 같습니다. tsvector + GIN은 책 뒤의 찾아보기 색인처럼, "이 단어가 몇 페이지에 있다"를 미리 정리해 두어 곧장 펼칠 수 있게 합니다.

2tsvector / tsquery — 내장 전문검색

전문검색의 두 축은 to_tsvector(문서)와 to_tsquery(질의)이고, 매칭은 @@ 연산자로 합니다. english 사전을 쓰면 어간 분석이 동작해 running 질의가 run이 든 문서에도 매칭됩니다. 질의를 만드는 방법은 셋입니다 — to_tsquery(&·|·! 직접), plainto_tsquery(평문을 AND로), websearch_to_tsquery(웹 검색 문법).

study_db=# — tsvector @@ tsquery 매칭 + 랭킹
-- 'run' 질의가 어간 분석으로 'running'까지 매칭
SELECT title FROM docs WHERE search @@ to_tsquery('english', 'run');
       title
-------------------
 Postgres indexing

-- ts_rank로 관련도 순 정렬
SELECT title, ts_rank(search, q) AS rank
FROM docs, to_tsquery('english', 'index | search') AS q
WHERE search @@ q ORDER BY rank DESC;
       title       |    rank
-------------------+-------------
 JSONB and GIN     |  0.06079271
 Postgres indexing | 0.037995443
 Full text search  | 0.037995443
🧩 한국어는 기본 사전의 한계가 있다

ko Postgres 기본 텍스트 검색 사전은 영어 등 일부 언어의 형태소만 압니다 — 한국어 형태소 분석은 기본 제공되지 않습니다. 그래서 한국어는 'simple' 설정(토큰을 그대로 소문자화, 어간 분석 없음)으로 다루거나, 아래 pg_trgm으로 보완하는 경우가 많습니다. to_tsvector('simple', '포스트그레스 전문검색 예제') → '예제':3 '전문검색':2 '포스트그레스':1처럼 공백 단위로만 끊깁니다.

⚠️ to_tsvector를 매 쿼리 계산하면 인덱스를 못 탄다

WHERE to_tsvector('english', body) @@ ...처럼 매번 to_tsvector를 계산하면, 그 값을 담은 인덱스가 없어 풀스캔이 됩니다. 정답은 생성 컬럼(GENERATED ALWAYS AS (...) STORED)으로 tsvector를 미리 저장하고, 그 컬럼에 GIN 인덱스를 거는 것입니다(또는 동일 식의 expression index). 아래처럼 식을 그대로 쓰면 됩니다.

study_db=# — tsvector 생성 컬럼 + GIN (함정 회피)
CREATE TABLE docs_big (
  id    int GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  title text NOT NULL,
  body  text NOT NULL,
  search tsvector GENERATED ALWAYS AS (              -- 미리 계산해 저장
    to_tsvector('english', title || ' ' || body)
  ) STORED
);
-- generate_series로 5,000행 적재 (.sql 파일 참고)
CREATE INDEX idx_docsbig_search ON docs_big USING gin (search);

-- 충분한 행(5,000)에서 @@ 검색이 실제로 GIN을 탄다
EXPLAIN (COSTS OFF)
SELECT title FROM docs_big WHERE search @@ to_tsquery('english', 'fuzzy');
                      QUERY PLAN
------------------------------------------------------
 Bitmap Heap Scan on docs_big
   Recheck Cond: (search @@ '''fuzzi'''::tsquery)
   ->  Bitmap Index Scan on idx_docsbig_search
         Index Cond: (search @@ '''fuzzi'''::tsquery)

3pg_trgm — 트라이그램 유사도·오타·부분일치

전문검색이 "단어 단위"라면, pg_trgm은 문자열을 세 글자 조각(trigram)으로 쪼개 비교합니다. 덕분에 오타 허용·부분일치에 강합니다. similarity()로 0~1 점수를 내고, % 연산자로 임계값(기본 0.3) 이상이면 매칭으로 봅니다.

study_db=# — 유사도 · 오타 허용
CREATE EXTENSION IF NOT EXISTS pg_trgm;

SELECT similarity('postgres', 'postgrey') AS sim;   -- 오타에도 높은 점수
    sim
-----------
 0.6363636

-- % : 유사도 임계값(기본 0.3) 이상이면 참 → 오타 'Postgrez'도 찾음
SELECT title FROM docs WHERE title % 'Postgrez';
       title
-------------------
 Postgres indexing

핵심은 트라이그램 GIN 인덱스입니다(gin_trgm_ops). 이걸 만들면 일반 B-tree로는 불가능했던 선두 와일드카드 ILIKE '%..%'까지 인덱스를 탑니다. 아래는 5,000행에서 확인한 실제 실행 계획입니다.

study_db=# — trigram GIN이 ILIKE '%..%'를 가속
CREATE INDEX idx_docsbig_body_trgm ON docs_big USING gin (body gin_trgm_ops);

EXPLAIN (COSTS OFF)
SELECT title FROM docs_big WHERE body ILIKE '%fuzzy%';
                    QUERY PLAN
--------------------------------------------------
 Bitmap Heap Scan on docs_big
   Recheck Cond: (body ~~* '%fuzzy%'::text)
   ->  Bitmap Index Scan on idx_docsbig_body_trgm
         Index Cond: (body ~~* '%fuzzy%'::text)
💡 한국어 부분일치엔 pg_trgm이 실용적

한국어처럼 기본 형태소 사전이 없는 경우, pg_trgm 트라이그램 인덱스로 ILIKE '%검색어%'를 가속하면 형태소 분석 없이도 부분일치 검색을 실용적인 속도로 처리할 수 있습니다. 정밀한 자연어 검색이 필요하면 전용 한국어 형태소 확장을 별도로 도입합니다.

전문검색 MySQL

MySQL은 FULLTEXT 인덱스 + MATCH(col) AGAINST('word')로 전문검색을 합니다(InnoDB/ MyISAM). 자연어/불리언 모드를 지원하지만, 오타 허용·트라이그램 유사도 검색은 기본 제공하지 않습니다.

전문검색 PostgreSQL

Postgres는 tsvector + @@ + ts_rank로 어간 분석·랭킹까지 하는 본격 전문검색을 제공하고, 거기에 더해 pg_trgm으로 오타·부분일치를 GIN/GiST 인덱스로 가속합니다. 둘을 상황에 맞게 조합할 수 있는 점이 강점입니다.

복습

4✅ 셀프 체크

Q LIKE '%x%'는 왜 느린가?

선두 와일드카드(%로 시작)는 일반 B-tree 인덱스를 탈 수 없어 전체 행을 풀스캔하기 때문입니다. 게다가 어간 분석·랭킹도 없습니다. 부분일치가 필요하다면 pg_trgm 트라이그램 GIN 인덱스로 ILIKE '%x%'를 가속하는 것이 정답입니다.

Q to_tsvector를 매번 계산하면 안 되는 이유는?

쿼리마다 to_tsvector(...)를 새로 계산하면 그 값을 담은 인덱스가 없어 풀스캔이 됩니다. 생성 컬럼(STORED)에 tsvector를 미리 저장하고 그 컬럼에 GIN 인덱스를 걸면(또는 동일 식의 expression index), @@ 검색이 Bitmap Index Scan으로 인덱스를 탑니다.

Q pg_trgm이 특히 잘하는 것은?

오타 허용과 부분일치입니다. 문자열을 세 글자 trigram으로 쪼개 비교하므로 similarity('postgres','postgrey') 같은 유사도 점수를 내고, % 연산자로 오타가 섞인 질의도 매칭합니다. 트라이그램 GIN 인덱스는 ILIKE '%..%'처럼 형태소와 무관한 부분일치를 가속합니다.
