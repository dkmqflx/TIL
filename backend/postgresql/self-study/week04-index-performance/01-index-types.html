인덱스 종류
B-tree · GIN · GiST · BRIN · partial · expression

인덱스는 책의 색인입니다. Postgres는 B-tree 하나가 아니라 GIN·GiST·BRIN에 partial·expression까지 — 데이터 모양에 맞는 색인을 고릅니다.

“

인덱스는 책의 색인이다. Postgres는 B-tree 하나가 아니라 GIN·GiST·BRIN·partial·expression까지 — 데이터 모양에 맞는 색인을 고른다.

  • B-tree가 왜 기본인지, 인덱스의 쓰기·공간 트레이드오프를 안다
  • GIN·GiST·BRIN·Hash를 각각 언제 쓰는지 구분한다
  • partial·expression·멀티컬럼·covering 인덱스를 만들고 EXPLAIN으로 검증한다

1인덱스 기본 — 왜 빠른가, 무엇을 치르는가

500쪽짜리 햄버거 레시피 책에서 "양파"가 나오는 곳을 찾는다고 합시다. 🍔 처음부터 끝까지 넘기면(Seq Scan) 오래 걸리지만, 책 뒤의 색인에서 "양파 → 12, 88, 301쪽"을 보면 바로 점프합니다. 데이터베이스 인덱스도 똑같습니다 — 컬럼 값을 미리 정렬·정리해 두어, 전체 테이블을 훑지 않고 원하는 행으로 곧장 갑니다.

공짜는 아닙니다. 인덱스는 별도의 디스크 공간을 차지하고, 행을 INSERT·UPDATE·DELETE할 때마다 인덱스도 함께 갱신해야 하므로 쓰기가 느려집니다. 그래서 인덱스는 "많을수록 좋은" 게 아니라 읽기 이득 vs 쓰기·공간 비용을 저울질해 고릅니다.

기본은 B-tree입니다. 등호(=), 범위(<·>·BETWEEN), 정렬(ORDER BY), 그리고 접두 LIKE('서울%')까지 모두 처리합니다. 충분히 큰 테이블(아래는 10만 행)에서 등호 검색이 인덱스를 타는 것을 봅시다.

study_db=# — B-tree, 10만 행에서 등호 검색
CREATE INDEX idx_users_age ON users (age);
ANALYZE users;                          -- 통계 갱신(필수)

EXPLAIN (ANALYZE, BUFFERS, COSTS OFF)
SELECT * FROM users WHERE age = 25;
                                        QUERY PLAN
------------------------------------------------------------------------------------------
 Bitmap Heap Scan on users (actual time=0.120..0.547 rows=1667.00 loops=1)
   Recheck Cond: (age = 25)
   ->  Bitmap Index Scan on idx_users_age (actual time=0.068..0.068 rows=1667.00 loops=1)
         Index Cond: (age = 25)

풀스캔이 아니라 idx_users_age를 통해 1,667행만 정확히 집었습니다. (이렇게 다수 행을 한꺼번에 모으는 경우 Postgres는 Index Scan 대신 Bitmap Index Scan을 자주 고릅니다 — Lesson 02에서 자세히 다룹니다.)

2종류별 용도 — 데이터 모양에 맞춰 고른다

Postgres의 인덱스는 여러 액세스 메서드(access method)로 나뉩니다. 핵심은 "이 컬럼·이 검색에 어떤 색인이 맞나"입니다.

종류언제예
B-tree (기본)등호·범위·정렬·접두 LIKE — 대부분의 경우WHERE age > 30 ORDER BY created_at
GIN한 행에 값이 여러 개 — jsonb·배열·전문검색WHERE data @> '{...}', tags @> ARRAY[...]
GiST범위·기하·근접(nearest) 검색WHERE box && ..., range 겹침, ORDER BY pt <-> ...
BRIN거대한 자연정렬 테이블(시계열 등) — 매우 작음로그 WHERE ts BETWEEN ...
Hash등호 전용(=) — 드묾, 보통 B-tree로 충분WHERE token = '...'

GIN·GiST는 Week 3에서 본 jsonb·전문검색의 엔진이었습니다. 여기서는 앱 개발에서 잘 안 보이지만 강력한 BRIN을 봅시다. 시계열처럼 물리적으로 정렬되어 쌓이는 50만 행 테이블에서, 같은 컬럼에 BRIN과 B-tree를 만들어 크기를 비교하면 차이가 압도적입니다.

study_db=# — BRIN vs B-tree 크기 (50만 행 시계열)
CREATE INDEX idx_sensors_ts_brin  ON sensors USING brin (ts);
CREATE INDEX idx_sensors_ts_btree ON sensors (ts);
SELECT
  pg_size_pretty(pg_relation_size('idx_sensors_ts_brin'))  AS brin_size,
  pg_size_pretty(pg_relation_size('idx_sensors_ts_btree')) AS btree_size;
 brin_size | btree_size
-----------+------------
 24 kB     | 11 MB

-- BRIN으로 범위 조회 (B-tree 제거 후)
EXPLAIN (ANALYZE, BUFFERS, COSTS OFF)
SELECT count(*) FROM sensors WHERE ts BETWEEN '2024-01-02' AND '2024-01-03';
                          QUERY PLAN
--------------------------------------------------------------------
 Aggregate (actual time=8.230..8.231 rows=1.00 loops=1)
   ->  Bitmap Heap Scan on sensors (... rows=86401.00 loops=1)
         Recheck Cond: ((ts >= ...) AND (ts <= ...))
         ->  Bitmap Index Scan on idx_sensors_ts_brin (... rows=6400.00 loops=1)
📝 BRIN은 "블록 범위" 색인

BRIN(Block Range INdex)은 각 행이 아니라 물리 블록 묶음마다 최소/최대값만 저장합니다. 그래서 24 kB로 11 MB B-tree와 같은 범위 조회를 해냅니다. 단, 컬럼이 디스크에 정렬된 순서로 쌓여 있어야 위력을 냅니다 — 시계열 ts, 자동증가 id처럼요. 무작위로 흩어진 값에는 효과가 없습니다.

3partial · expression 인덱스

partial index는 테이블의 일부 행만 인덱싱합니다. "비활성 계정만 조회"가 잦다면 WHERE status = 'inactive' 조건을 붙여 그 5%만 색인합니다 — 인덱스가 작아 빠르고 공간도 아낍니다. 쿼리의 WHERE가 partial 조건과 맞아야 쓰입니다.

study_db=# — partial index (inactive 5%만) + 크기 비교
CREATE INDEX idx_users_inactive ON users (id) WHERE status = 'inactive';

EXPLAIN (ANALYZE, BUFFERS, COSTS OFF)
SELECT id FROM users WHERE status = 'inactive';
                                            QUERY PLAN
---------------------------------------------------------------------------------------------------
 Index Only Scan using idx_users_inactive on users (actual time=0.019..1.057 rows=5000.00 loops=1)

-- 부분 인덱스(inactive 5%)는 전체 status 인덱스보다 훨씬 작다
 partial_size | full_size
--------------+-----------
 128 kB       | 696 kB

expression index는 컬럼이 아니라 함수의 결과를 인덱싱합니다. 대소문자 무시 로그인처럼 WHERE lower(email) = ...를 자주 쓴다면, 그 표현식을 그대로 색인해야 인덱스를 탑니다(Lesson 02의 "인덱스 함정" 참고).

study_db=# — expression index: lower(email)
CREATE INDEX idx_users_lower_email ON users (lower(email));

EXPLAIN (ANALYZE, BUFFERS, COSTS OFF)
SELECT id FROM users WHERE lower(email) = 'user500@example.com';
                                          QUERY PLAN
-----------------------------------------------------------------------------------------------
 Bitmap Heap Scan on users (actual time=0.592..0.593 rows=1.00 loops=1)
   Recheck Cond: (lower(email) = 'user500@example.com'::text)
   ->  Bitmap Index Scan on idx_users_lower_email (... rows=1.00 loops=1)
         Index Cond: (lower(email) = 'user500@example.com'::text)

4멀티컬럼 인덱스 · 컬럼 순서 · covering

여러 컬럼을 묶은 (a, b) 인덱스는 왼쪽 접두(left-prefix) 규칙을 따릅니다 — a 단독, 또는 a, b 조합 쿼리에는 쓰이지만, b만으로는 쓰지 못합니다. 색인이 a 먼저, 그 안에서 b 순으로 정렬돼 있어, a가 없으면 어디서 시작할지 모르기 때문입니다. (전화번호부가 "성→이름" 순이면 성 없이 이름만으로는 못 찾는 것과 같습니다.)

여기에 INCLUDE를 붙이면 covering index가 됩니다 — 자주 함께 조회하는 컬럼을 인덱스에 얹어 두면, 테이블 본체를 안 보고 인덱스만으로(Index Only Scan) 결과를 돌려줍니다.

study_db=# — (email, age) 복합 인덱스의 왼쪽 접두 규칙
CREATE INDEX idx_users_email_age ON users (email, age) INCLUDE (status);

-- 선두 컬럼(email) + age → 복합 인덱스를 탄다. INCLUDE(status)로 Index Only Scan
EXPLAIN (ANALYZE, BUFFERS, COSTS OFF)
SELECT status FROM users WHERE email = 'User500@Example.com' AND age = 38;
                                           QUERY PLAN
-------------------------------------------------------------------------------------------------
 Index Only Scan using idx_users_email_age on users (actual time=0.018..0.019 rows=1.00 loops=1)
   Index Cond: ((email = 'User500@Example.com'::text) AND (age = 38))
   Heap Fetches: 1

-- 선두 컬럼(email) 없이 age 만으로는 (email, age) 인덱스를 못 쓴다 → Seq Scan
EXPLAIN (ANALYZE, BUFFERS, COSTS OFF)
SELECT * FROM users WHERE age = 30;
                            QUERY PLAN
-------------------------------------------------------------------
 Seq Scan on users (actual time=0.007..2.268 rows=1667.00 loops=1)
   Filter: (age = 30)
   Rows Removed by Filter: 98333
💡 PG18 skip scan — 왼쪽 접두 규칙의 예외

PostgreSQL 18은 B-tree skip scan을 추가했습니다. 선두 컬럼의 distinct 값이 매우 적으면(예: status의 active/inactive 둘뿐), 플래너가 그 몇 개 값을 "건너뛰며" 훑어 선두 컬럼 없이도 복합 인덱스를 쓸 수 있습니다(EXPLAIN에 Index Searches: N로 보임). 위 예처럼 선두 컬럼이 email(고유값 다수)이면 skip scan이 무의미해 평소처럼 Seq Scan이 됩니다. 규칙의 큰 줄기는 여전히 "선두 컬럼을 빼지 마라"입니다.

인덱스 종류 MySQL

InnoDB는 사실상 B-tree 하나가 주력입니다. 전문검색은 FULLTEXT 인덱스, 공간 데이터는 SPATIAL 인덱스로 별도 취급합니다. partial index는 없고(생성 컬럼 + 인덱스로 우회), 함수 인덱스·INCLUDE는 비교적 최근 버전에서 제한적으로 지원합니다.

인덱스 종류 PostgreSQL

여러 액세스 메서드(B-tree·GIN·GiST·BRIN·Hash·SP-GiST)를 같은 문법(USING)으로 다룹니다. jsonb·배열·전문검색은 모두 GIN 하나로, 거대한 시계열은 BRIN으로. partial·expression·covering(INCLUDE)이 모두 1급 기능이라, 데이터 모양에 맞춰 색인을 정밀하게 고를 수 있습니다.

⚠️ 인덱스 남발은 부채다

인덱스가 많을수록 읽기는 빨라질 수 있어도 쓰기는 느려지고 공간은 늘어납니다 — 모든 INSERT/UPDATE가 관련 인덱스를 전부 갱신해야 하니까요. 게다가 아무도 안 쓰는 인덱스는 순수한 비용입니다. 실제 쿼리(WHERE/JOIN/ORDER BY)에 근거해 만들고, pg_stat_user_indexes로 사용량을 점검해 안 쓰는 것은 지우세요.

복습

5✅ 셀프 체크

Q GIN 인덱스는 언제 쓰나?

한 행에 값이 여러 개 들어 있고, 그 안의 원소·키를 검색할 때입니다 — jsonb(@> 포함 검색), 배열(원소 포함), 전문검색(tsvector). GIN은 값을 잘게 쪼개 역색인(inverted index)으로 만들어 "이 값을 포함하는 행"을 빠르게 찾습니다.

Q partial index의 이점은?

테이블의 일부 행만 색인하므로 인덱스가 작고 빠르며 공간을 아낍니다. 예에서 inactive 5%만 담은 부분 인덱스(128 kB)는 전체 status 인덱스(696 kB)보다 훨씬 작았습니다. 자주 조회하는 부분집합(활성 행, 미처리 주문 등)이 명확할 때 특히 유용합니다.

Q 멀티컬럼 (a, b) 인덱스를 b만으로 검색에 쓸 수 있나?

원칙적으로 못 씁니다 — 왼쪽 접두 규칙 때문입니다. 색인이 a 먼저 정렬돼 있어 선두 컬럼 a 조건이 없으면 시작점을 모릅니다(예에서 WHERE age=30은 Seq Scan). 단, PG18의 skip scan은 선두 컬럼의 distinct 값이 매우 적을 때 예외적으로 인덱스를 쓸 수 있습니다.
