EXPLAIN ANALYZE
플래너 · 통계 · 인덱스 함정

추측하지 말고 측정하세요. EXPLAIN ANALYZE는 Postgres가 쿼리를 '어떻게' 실행하는지 보여주는 X-레이입니다.

“

추측하지 말고 측정하라. EXPLAIN ANALYZE는 Postgres가 쿼리를 '어떻게' 실행하는지 보여주는 X-레이다. 통계가 틀리면 플랜도 틀린다.

  • EXPLAIN(추정) vs EXPLAIN ANALYZE(실측)를 읽는다
  • Seq Scan vs Index/Bitmap Scan을 플래너가 언제 고르는지 안다
  • 통계·ANALYZE가 플랜을 좌우함을 이해하고 인덱스 함정을 피한다

1EXPLAIN 읽는 법 — 추정 vs 실측

EXPLAIN은 쿼리를 실행하지 않고 플래너의 비용 추정만 보여줍니다. cost=시작..전체는 임의 단위의 추정 비용이고, rows는 예상 행 수입니다. 여기에 ANALYZE를 붙이면 쿼리를 실제로 실행해 actual time(실측 시간)과 실제 rows를 함께 보여줍니다. BUFFERS는 읽은 페이지 수(I/O)를 더합니다.

study_db=# — EXPLAIN(추정만) vs EXPLAIN ANALYZE(실측)
-- 추정만: 실행하지 않고 비용 추정만 (cost, 예상 rows)
EXPLAIN
SELECT * FROM orders WHERE user_id = 42;
                                  QUERY PLAN
-------------------------------------------------------------------------------
 Bitmap Heap Scan on orders  (cost=4.45..74.81 rows=20 width=30)
   Recheck Cond: (user_id = 42)
   ->  Bitmap Index Scan on idx_orders_user  (cost=0.00..4.44 rows=20 width=0)
         Index Cond: (user_id = 42)

-- 실측: 실제 실행 + actual time/rows + 버퍼(I/O)
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM orders WHERE user_id = 42;
                                             QUERY PLAN
-----------------------------------------------------------------------------------------------------
 Bitmap Heap Scan on orders  (cost=4.45..74.81 rows=20 width=30) (actual time=0.050..0.072 rows=20.00 loops=1)
   Recheck Cond: (user_id = 42)
   Buffers: shared hit=20 read=2
   ->  Bitmap Index Scan on idx_orders_user  (cost=0.00..4.44 rows=20 ...) (actual time=0.042..0.042 rows=20.00 loops=1)
         Index Cond: (user_id = 42)
 Execution Time: 0.085 ms

플랜은 아래에서 위로, 안쪽 노드(->)부터 읽습니다. 여기서는 인덱스로 행 위치를 모은 뒤(Bitmap Index Scan) 테이블에서 실제 행을 가져옵니다(Bitmap Heap Scan). 추정 rows=20과 실제 rows=20.00이 일치 — 통계가 정확하다는 신호입니다.

2Seq Scan vs Index Scan vs Bitmap Heap Scan

"인덱스가 있으면 항상 인덱스를 탄다"는 흔한 오해입니다. 플래너는 선택도(selectivity)를 보고 더 싼 쪽을 고릅니다. 찾는 행이 적으면 인덱스로 점프하는 게 이득이지만, 테이블의 대부분을 가져올 거라면 인덱스를 한 행씩 들춰 본체로 점프하는 비용이 더 커서 — 그냥 통째로 순차 스캔(Seq Scan)하는 게 빠릅니다.

같은 컬럼, 같은 테이블에서 선택도만 바꿔 둘을 모두 봅시다. status는 shipped 5%, pending 95%입니다.

study_db=# — 낮은 선택도 → Index Scan / 높은 선택도 → Seq Scan
-- (a) shipped 5% : 적게 골라내므로 인덱스 경로
EXPLAIN (ANALYZE, BUFFERS, COSTS OFF)
SELECT * FROM orders WHERE status = 'shipped';
                                          QUERY PLAN
----------------------------------------------------------------------------------------------
 Index Scan using idx_orders_status on orders (actual time=0.025..1.020 rows=5000.00 loops=1)
   Index Cond: (status = 'shipped'::text)
   Buffers: shared hit=736 read=6

-- (b) pending 95% : 거의 다 가져오므로 인덱스가 손해 → Seq Scan
EXPLAIN (ANALYZE, BUFFERS, COSTS OFF)
SELECT * FROM orders WHERE status = 'pending';
                             QUERY PLAN
---------------------------------------------------------------------
 Seq Scan on orders (actual time=0.006..4.725 rows=95000.00 loops=1)
   Filter: (status = 'pending'::text)
   Rows Removed by Filter: 5000
📝 세 가지 스캔, 한 줄 요약

Seq Scan: 테이블을 통째로 읽는다(작은 테이블·낮은 선택도엔 이게 더 빠름). Index Scan: 인덱스로 행을 하나씩 찾아 본체로 점프(소수 행에 유리). Bitmap Heap Scan: 인덱스로 행 위치들을 비트맵으로 모은 뒤 본체를 블록 순서로 한 번에 읽는다(중간 규모 결과에 유리, 무작위 I/O를 줄임).

3통계 & ANALYZE — 추정이 틀리면 나쁜 플랜

플래너의 모든 판단은 통계에 근거합니다. Postgres는 pg_statistic에 각 컬럼의 값 분포(히스토그램·최빈값 등)를 담아 두고, 이를 보고 "이 조건이면 약 몇 행"인지 추정합니다. 통계는 ANALYZE 명령이나 autovacuum이 갱신합니다. 대량 변경 후 통계가 낡으면 추정이 어긋나 잘못된 플랜을 고릅니다.

일부러 망가뜨려 봅시다. shipped를 5% → 대부분으로 늘린 직후, ANALYZE 전에는 플래너가 아직 "5%"라 믿고 인덱스 경로를 고르지만(실제로는 9만 행이라 비효율), ANALYZE 후에는 통계가 갱신돼 Seq Scan으로 교정됩니다.

study_db=# — 통계가 낡으면 플랜이 틀린다
UPDATE orders SET status = 'shipped' WHERE id <= 90000;  -- 이제 대부분 shipped

-- ANALYZE 전: 여전히 'shipped'=5%로 착각 → 인덱스 경로(잘못된 선택)
EXPLAIN (ANALYZE, COSTS OFF)
SELECT * FROM orders WHERE status = 'shipped';
                                          QUERY PLAN
-----------------------------------------------------------------------------------------------
 Index Scan using idx_orders_status on orders (actual time=3.343..9.765 rows=90500.00 loops=1)
   Index Cond: (status = 'shipped'::text)

-- ANALYZE 후: 통계 갱신 → Seq Scan으로 교정
ANALYZE orders;
EXPLAIN (ANALYZE, COSTS OFF)
SELECT * FROM orders WHERE status = 'shipped';
                             QUERY PLAN
---------------------------------------------------------------------
 Seq Scan on orders (actual time=0.442..5.830 rows=90500.00 loops=1)
   Filter: (status = 'shipped'::text)
   Rows Removed by Filter: 9500
💡 추정이 자꾸 틀린다면

대량 적재·삭제 직후에는 ANALYZE 테이블;을 직접 돌려 통계를 새로 고치세요. 특정 컬럼의 분포가 심하게 치우쳐 추정이 빗나간다면 default_statistics_target(기본 100)을 올려 표본을 키우거나, 컬럼 단위로 ALTER TABLE ... ALTER COLUMN ... SET STATISTICS를 조정합니다. 플랜의 추정 rows와 실제 rows가 크게 다르면 통계를 의심하세요.

4인덱스 함정 — 인덱스를 못 타게 만드는 실수

인덱스를 만들어 두고도 안 타는 경우가 잦습니다. 가장 흔한 함정은 인덱스 컬럼에 함수·연산을 씌우는 것입니다. WHERE upper(status) = 'SHIPPED'는 status 인덱스를 못 씁니다 — 인덱스에는 status의 원래 값이 있지 upper(status)가 없으니까요. 해법은 그 표현식을 그대로 색인하는 expression index(Lesson 01)입니다.

study_db=# — 컬럼에 함수 → Seq Scan, expression index로 교정
-- (함정) upper(status)는 status 인덱스를 못 탄다 → Seq Scan
EXPLAIN (ANALYZE, BUFFERS, COSTS OFF)
SELECT * FROM orders WHERE upper(status) = 'SHIPPED';
                             QUERY PLAN
---------------------------------------------------------------------
 Seq Scan on orders (actual time=0.872..14.114 rows=5000.00 loops=1)
   Filter: (upper(status) = 'SHIPPED'::text)
   Rows Removed by Filter: 95000

-- (교정) 표현식을 그대로 색인하면 다시 인덱스를 탄다
CREATE INDEX idx_orders_upper_status ON orders (upper(status));
EXPLAIN (ANALYZE, BUFFERS, COSTS OFF)
SELECT * FROM orders WHERE upper(status) = 'SHIPPED';
                                             QUERY PLAN
----------------------------------------------------------------------------------------------------
 Bitmap Heap Scan on orders (actual time=0.197..1.111 rows=5000.00 loops=1)
   Recheck Cond: (upper(status) = 'SHIPPED'::text)
   ->  Bitmap Index Scan on idx_orders_upper_status (... rows=5000.00 loops=1)
⚠️ 인덱스를 죽이는 패턴들

① 컬럼에 함수·연산: lower(email)=…, amount*2>…, date(created_at)=… → expression index가 없으면 Seq Scan. ② 앞 와일드카드 LIKE: LIKE '%ipped'는 접두가 없어 B-tree를 못 씀(부분일치는 pg_trgm GIN으로). ③ 암묵적 타입 캐스팅: varchar 컬럼을 int와 비교하면 캐스팅이 끼어 인덱스를 놓칠 수 있음 — 타입을 맞추세요. ④ OR 조건: a=1 OR b=2는 각 컬럼 인덱스를 합치기 어려워 Seq Scan이 되기도 함(UNION으로 분리하면 각자 인덱스를 탐).

study_db=# — 앞 와일드카드 LIKE '%ipped' → Seq Scan
EXPLAIN (ANALYZE, COSTS OFF)
SELECT * FROM orders WHERE status LIKE '%ipped';
                             QUERY PLAN
--------------------------------------------------------------------
 Seq Scan on orders (actual time=0.331..4.821 rows=5000.00 loops=1)
   Filter: (status ~~ '%ipped'::text)
   Rows Removed by Filter: 95000
실행 계획 보기 MySQL

MySQL도 EXPLAIN으로 계획을 봅니다(전통적으로 한 행 표 형태, 8.0+는 FORMAT=TREE·FORMAT=JSON 지원). EXPLAIN ANALYZE(8.0.18+)는 실제 실행 후 실측을 보여줍니다. type(ALL=풀스캔, ref·range=인덱스), rows, key 열을 읽는 식입니다.

실행 계획 보기 PostgreSQL

Postgres는 노드 트리로 더 상세히 보여줍니다. EXPLAIN (ANALYZE, BUFFERS)는 각 노드의 추정 vs 실측 rows, 시간, 버퍼 I/O까지 노출합니다. cost 추정과 actual을 나란히 비교할 수 있어 "통계가 틀렸는지"를 바로 진단할 수 있습니다.

복습

5✅ 셀프 체크

Q cost와 actual time의 차이는?

cost는 플래너가 실행 전에 매긴 추정 비용(임의 단위)이고, actual time은 ANALYZE로 실제 실행한 뒤 측정한 밀리초입니다. EXPLAIN만 쓰면 cost(추정)만, EXPLAIN ANALYZE를 쓰면 둘 다 나옵니다. 추정 rows와 실제 rows가 크게 다르면 통계 문제를 의심합니다.

Q 인덱스가 있는데도 Seq Scan이 더 나을 때는?

찾는 행이 테이블의 큰 비중을 차지할 때입니다(예: 95% pending). 인덱스로 한 행씩 찾아 본체로 점프하는 무작위 I/O보다, 그냥 통째로 순차 읽는 게 더 쌉니다. 작은 테이블에서도 마찬가지입니다. 플래너는 선택도를 보고 이 판단을 내립니다.

Q WHERE 컬럼에 함수를 쓰면 무슨 문제가 생기나?

일반 인덱스를 못 탑니다. 인덱스에는 컬럼의 원래 값이 들어 있는데 upper(status) 같은 함수 결과는 거기에 없기 때문입니다 → Seq Scan. 해법은 그 표현식을 그대로 색인하는 expression index(CREATE INDEX ... (upper(status)))입니다.
