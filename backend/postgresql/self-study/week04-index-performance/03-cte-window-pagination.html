CTE · 윈도우 함수
keyset pagination · N+1

분석·리포팅 SQL의 핵심 도구: WITH(CTE)로 단계를 쪼개고, 윈도우 함수로 순위·누계를 내고, keyset으로 빠르게 페이지를 넘깁니다.

“

분석 SQL의 세 무기: WITH(CTE)로 단계를 쪼개고, 윈도우 함수로 순위·누계를 내고, keyset으로 깊은 페이지를 빠르게 넘긴다.

  • CTE(WITH·재귀)로 복잡한 쿼리를 단계로 분해한다
  • 윈도우 함수로 순위·누계·전후 비교를 내고 GROUP BY와 구분한다
  • keyset pagination으로 OFFSET의 깊은 페이지 문제를 해결하고 N+1을 줄인다

1CTE(WITH) — 단계를 쪼개고, 재귀로 계층을 순회

CTE(Common Table Expression)는 쿼리 안에서 중간 결과에 임시 이름을 붙이는 도구입니다. 🍔 햄버거 레시피를 "① 패티 굽기 → ② 야채 손질 → ③ 조립"처럼 단계로 나눠 적으면 읽기 쉽듯, 거대한 서브쿼리 중첩을 WITH 이름 AS (...)로 펼쳐 단계별로 읽게 합니다.

특히 WITH RECURSIVE는 계층 구조(댓글 트리, 조직도, 카테고리 트리)를 순회합니다. 루트에서 시작해(앵커) 자식으로 내려가며(재귀) 깊이를 누적할 수 있습니다.

study_db=# — 재귀 CTE로 댓글 트리 순회 (depth 계산)
WITH RECURSIVE tree AS (
  -- 앵커: 루트(부모 없음)에서 시작
  SELECT id, parent_id, body, 0 AS depth
  FROM comments WHERE parent_id IS NULL
  UNION ALL
  -- 재귀: 자식으로 내려가며 depth + 1
  SELECT c.id, c.parent_id, c.body, t.depth + 1
  FROM comments c JOIN tree t ON c.parent_id = t.id
)
SELECT id, depth, repeat('  ', depth) || body AS thread FROM tree ORDER BY id;
 id | depth |      thread
----+-------+------------------
  1 |     0 | 원댓글
  2 |     1 |   답글 A
  3 |     1 |   답글 B
  4 |     2 |     답글 A-1
  5 |     3 |       답글 A-1-1
📝 CTE는 기본적으로 인라인 최적화된다

예전 Postgres에서 CTE는 항상 독립 실행(materialize)되어 최적화 경계로 작동했지만, PG12+부터는 (한 번만 참조되고 부작용이 없으면) 인라인해 본 쿼리와 함께 최적화합니다. 일부러 한 번만 계산해 재사용하고 싶으면 WITH x AS MATERIALIZED (...)로 명시할 수 있습니다.

2윈도우 함수 — 행을 접지 않고 옆에 계산

GROUP BY는 행을 접어서 그룹당 한 줄로 줄입니다. 반면 윈도우 함수는 행을 그대로 두고 옆 칸에 계산값을 덧붙입니다 — 각 행을 보면서 "이 행은 지역 내 몇 위인가", "여기까지 누계는 얼마인가", "전날 대비 얼마 늘었나"를 답합니다. 핵심 문법은 함수() OVER (PARTITION BY ... ORDER BY ...)입니다.

study_db=# — ROW_NUMBER/RANK/DENSE_RANK + 누계 + LAG
SELECT
  region, ymd, amount,
  ROW_NUMBER() OVER (PARTITION BY region ORDER BY amount DESC) AS rn,
  RANK()       OVER (PARTITION BY region ORDER BY amount DESC) AS rnk,
  SUM(amount)  OVER (PARTITION BY region ORDER BY ymd)         AS running_total,
  amount - LAG(amount) OVER (PARTITION BY region ORDER BY ymd) AS vs_prev
FROM sales ORDER BY region, ymd;
 region |    ymd     | amount | rn | rnk | running_total | vs_prev
--------+------------+--------+----+-----+---------------+---------
 Busan  | 2024-01-01 |    200 |  2 |   2 |           200 |
 Busan  | 2024-01-02 |    180 |  3 |   3 |           380 |     -20
 Busan  | 2024-01-03 |    220 |  1 |   1 |           600 |      40
 Seoul  | 2024-01-01 |    100 |  3 |   3 |           100 |
 Seoul  | 2024-01-02 |    150 |  1 |   1 |           250 |      50
 Seoul  | 2024-01-03 |    120 |  2 |   2 |           370 |     -30

같은 데이터에 GROUP BY를 쓰면 지역당 한 행(총합)만 남고 개별 일자는 사라집니다. 윈도우 함수는 일자별 원본을 유지하면서 running_total·vs_prev를 옆에 붙인다는 점이 결정적 차이입니다. RANK는 동점 뒤 순위를 건너뛰고(1,1,3), DENSE_RANK는 건너뛰지 않으며(1,1,2), ROW_NUMBER는 무조건 고유 번호를 매깁니다.

3keyset pagination — OFFSET의 깊은 페이지 문제

LIMIT 10 OFFSET 50000으로 깊은 페이지를 넘기면 Postgres는 앞 50,000행을 모두 읽고 버린 뒤 10행을 줍니다 — 페이지가 깊어질수록 점점 느려집니다. keyset(seek) pagination은 "마지막으로 본 행의 키보다 큰 것부터"를 WHERE로 직접 지정해, 인덱스로 그 지점으로 곧장 점프합니다.

10만 행 피드에서 둘을 비교하면 차이가 극명합니다. 정렬 키 (created_at, id)에 복합 인덱스가 있습니다.

study_db=# — OFFSET 50000 (앞 행 다 읽고 버림) vs keyset (즉시 점프)
-- (a) OFFSET 깊은 페이지: 앞 50,010행을 읽고 버린다(느림)
EXPLAIN (ANALYZE, BUFFERS, COSTS OFF)
SELECT id, created_at FROM feed ORDER BY created_at, id OFFSET 50000 LIMIT 10;
                                              QUERY PLAN
-------------------------------------------------------------------------------------------------------
 Limit (actual time=6.475..6.477 rows=10.00 loops=1)
   Buffers: shared hit=368 read=194
   ->  Index Only Scan using idx_feed_created on feed (actual time=0.059..5.471 rows=50010.00 loops=1)
         Heap Fetches: 50010

-- (b) keyset: 마지막으로 본 (created_at,id)보다 큰 행만 → 인덱스로 즉시 점프
EXPLAIN (ANALYZE, BUFFERS, COSTS OFF)
SELECT id, created_at FROM feed
WHERE (created_at, id) > (timestamptz '2024-01-01' + interval '50000 seconds', 50000)
ORDER BY created_at, id LIMIT 10;
                                              QUERY PLAN
-------------------------------------------------------------------------------------------------------
 Limit (actual time=0.006..0.007 rows=10.00 loops=1)
   Buffers: shared hit=4
   ->  Index Only Scan using idx_feed_created on feed (actual time=0.005..0.006 rows=10.00 loops=1)
         Index Cond: (ROW(created_at, id) > ROW(..., 50000))
         Heap Fetches: 10

같은 인덱스를 쓰지만 결과가 다릅니다 — OFFSET은 rows=50010을 읽고(버퍼 562, 6.5ms), keyset은 rows=10만 읽습니다(버퍼 4, 0.02ms). 행 비교 (a, b) > (x, y)가 복합 인덱스의 Index Cond로 그대로 들어가 곧장 점프하는 게 비결입니다.

⚠️ 큰 목록에 OFFSET 페이지네이션을 쓰지 마라

OFFSET은 건너뛸 행을 실제로 읽고 버리므로, 페이지가 깊어질수록 선형으로 느려집니다(무한 스크롤·딥링크에서 치명적). 또 페이지 사이에 행이 삽입·삭제되면 중복·누락이 생깁니다. 정렬 키에 유일성(예: (created_at, id))을 보장하고 그 키 기준 keyset으로 넘기면, 깊이와 무관하게 일정하게 빠르고 안정적입니다.

4N+1 — 여러 번 쿼리 대신 한 방 JOIN

ORM의 흔한 함정인 N+1은 "목록 1번 + 각 항목의 연관 데이터 N번"을 따로 조회하는 패턴입니다. 글 20개를 가져온 뒤 각 글의 작성자를 20번 더 조회하면 쿼리가 21번 — 네트워크 왕복이 쌓여 느려집니다. 해법은 JOIN으로 한 방에 가져오거나, 집계가 필요하면 윈도우 함수·서브쿼리로 합치는 것입니다.

study_db=# — N+1 대신 JOIN 한 방 (각 행에 지역 합 붙이기)
-- 글마다 작성자/집계를 따로 N번 조회하지 말고 한 쿼리로 합친다
SELECT s.region, s.ymd, s.amount, t.region_total
FROM sales s
JOIN (
  SELECT region, SUM(amount) AS region_total FROM sales GROUP BY region
) t ON t.region = s.region
ORDER BY s.region, s.ymd;
 region |    ymd     | amount | region_total
--------+------------+--------+--------------
 Busan  | 2024-01-01 |    200 |          600
 Busan  | 2024-01-02 |    180 |          600
 Busan  | 2024-01-03 |    220 |          600
 Seoul  | 2024-01-01 |    100 |          370
 Seoul  | 2024-01-02 |    150 |          370
 Seoul  | 2024-01-03 |    120 |          370
💡 집계를 옆에 붙일 땐 윈도우가 더 간결

위 "각 행에 지역 합"은 사실 윈도우 함수 한 줄로도 됩니다 — SUM(amount) OVER (PARTITION BY region). JOIN/서브쿼리 없이 같은 결과를 내고 읽기도 쉽습니다. 연관 엔티티를 묶을 땐 JOIN, 같은 테이블의 집계를 행에 붙일 땐 윈도우 — 상황에 맞춰 고르세요.

분석 SQL MySQL

윈도우 함수·CTE·재귀 CTE는 MySQL 8.0부터 지원됩니다(이전 버전엔 없어 사용자 변수·서브쿼리로 우회했습니다). keyset pagination 개념·문법은 동일합니다. 다만 행 비교((a,b) > (x,y)) 같은 튜플 비교 활용은 Postgres 쪽 예제가 더 흔합니다.

분석 SQL PostgreSQL

윈도우 함수·CTE·재귀 CTE를 오래전부터 풍부하게 지원합니다. 행 비교를 복합 인덱스의 Index Cond로 그대로 밀어 넣어 keyset을 깔끔하게 구현하고, MATERIALIZED로 CTE 최적화를 세밀하게 제어할 수 있습니다.

복습

5✅ 셀프 체크

Q 윈도우 함수와 GROUP BY의 차이는?

GROUP BY는 행을 접어서 그룹당 한 줄로 줄입니다. 윈도우 함수는 원본 행을 그대로 두고 각 행 옆에 계산값(순위·누계·전후 비교)을 덧붙입니다. "지역 총합만" 필요하면 GROUP BY, "각 일자 행을 보면서 지역 내 순위·누계도" 필요하면 윈도우 함수입니다.

Q OFFSET 페이지네이션이 깊어질수록 느린 이유는?

OFFSET 50000은 건너뛸 5만 행을 실제로 읽은 뒤 버리고 그다음 10행만 돌려주기 때문입니다(예에서 rows=50010을 읽음). 페이지가 깊어질수록 버리는 양이 선형으로 늘어 느려집니다. keyset은 WHERE (key) > (last)로 인덱스를 타고 곧장 점프해 깊이와 무관하게 빠릅니다.

Q 재귀 CTE는 어디에 쓰나?

계층·그래프 구조를 순회할 때입니다 — 댓글 트리, 조직도(상사→부하), 카테고리 트리, 자재명세(BOM) 등. 앵커(루트)에서 시작해 UNION ALL로 자식을 따라 내려가며 깊이·경로를 누적합니다. 일반 SQL로는 깊이를 모르는 트리를 한 번에 펼치기 어렵습니다.
