트랜잭션
격리 수준

트랜잭션은 '전부 아니면 전무'. 격리 수준은 동시에 도는 트랜잭션이 서로를 얼마나 보는가를 정합니다 — Postgres 기본은 READ COMMITTED.

“

트랜잭션은 '전부 아니면 전무'. 격리 수준은 '동시에 도는 트랜잭션이 서로를 얼마나 보는가'를 정한다 — Postgres 기본은 READ COMMITTED.

  • BEGIN/COMMIT/ROLLBACK과 SAVEPOINT로 트랜잭션을 다룬다
  • 격리 수준 4단계와 막아 주는 이상현상을 구분한다
  • Postgres의 기본 동작(READ COMMITTED)과 MySQL과의 차이를 안다

1트랜잭션 기본 — ACID · BEGIN/COMMIT/ROLLBACK · SAVEPOINT

🍔 송금을 떠올려 보세요. "Alice 계좌에서 200을 빼고, Bob 계좌에 200을 더한다" — 이 둘은 같이 성공하거나, 같이 취소돼야 합니다. 인출만 되고 입금이 실패하면 돈이 증발합니다. 트랜잭션은 여러 문장을 하나의 '전부 아니면 전무' 단위로 묶는 장치입니다.

트랜잭션의 보증을 네 글자로 ACID라 부릅니다. Atomicity(원자성): 전부 적용되거나 전부 취소된다. Consistency(일관성): 제약·규칙을 깨지 않는다. Isolation(격리성): 동시 트랜잭션이 서로를 방해하지 않는다(이 레슨의 주제). Durability(영속성): COMMIT한 결과는 장애가 나도 살아남는다.

study_db=# — BEGIN/COMMIT(송금) 과 ROLLBACK(취소)
-- 송금: 인출과 입금을 한 트랜잭션으로 묶는다(둘 다 성공 또는 둘 다 취소)
BEGIN;
  UPDATE accounts SET balance = balance - 200 WHERE owner = 'Alice';
  UPDATE accounts SET balance = balance + 200 WHERE owner = 'Bob';
COMMIT;
 owner | balance
-------+---------
 Alice |  800.00
 Bob   |  700.00

-- ROLLBACK: 중간에 취소하면 변경이 없던 일이 된다
BEGIN;
  UPDATE accounts SET balance = balance - 300 WHERE owner = 'Alice';
ROLLBACK;
 owner | balance
-------+---------
 Alice |  800.00     -- 그대로 (롤백됨)
 Bob   |  700.00

긴 트랜잭션 안에서 일부만 되돌리고 싶을 때는 SAVEPOINT를 찍고 ROLLBACK TO로 그 지점까지만 취소합니다. 책갈피를 꽂아 두고 거기로만 되감는 셈입니다.

study_db=# — SAVEPOINT / ROLLBACK TO (부분 취소)
BEGIN;
  UPDATE accounts SET balance = balance + 50 WHERE owner = 'Alice';
  SAVEPOINT sp1;
  UPDATE accounts SET balance = balance + 9999 WHERE owner = 'Bob';  -- 실수
  ROLLBACK TO sp1;   -- Bob 변경만 취소, Alice +50 은 살아 있음
  UPDATE accounts SET balance = balance + 10 WHERE owner = 'Bob';    -- 다시 정상 처리
COMMIT;
 owner | balance
-------+---------
 Alice |  850.00     -- +50 유지
 Bob   |  710.00     -- +10 만 (잘못된 +9999 는 취소됨)
💡 앱에서의 트랜잭션

대부분의 드라이버·ORM은 BEGIN/COMMIT을 직접 쓰기보다 트랜잭션 블록(예: with conn.transaction():, @Transactional)으로 감쌉니다. 블록을 정상 통과하면 커밋, 예외가 던져지면 자동 롤백입니다. psql에서 BEGIN 없이 실행한 단일 문장은 각각이 곧 하나의 트랜잭션(autocommit)입니다.

2이상현상 — 동시 트랜잭션이 만드는 네 가지 사고

격리 수준이 낮으면 동시에 도는 트랜잭션이 서로의 미완성 작업을 들여다보며 이상현상(anomaly)을 일으킵니다. SQL 표준은 네 가지를 정의합니다.

  • Dirty read: 아직 커밋되지 않은 다른 트랜잭션의 변경을 읽는다(곧 롤백될 수도 있는 값).
  • Non-repeatable read: 같은 행을 두 번 읽었는데, 그 사이 다른 트랜잭션이 커밋해 값이 바뀐다.
  • Phantom read: 같은 조건으로 두 번 조회했는데, 그 사이 행이 추가/삭제돼 결과 행 수가 바뀐다.
  • Serialization anomaly: 각각은 성공했지만 동시에 돈 결과가 어떤 순차 실행으로도 설명되지 않는다(예: write skew).

아래 표는 PostgreSQL의 실제 동작입니다(공식 문서 Table 13.1 기준). 표준이 "허용될 수 있다"고 둔 칸도 PG에서는 더 강하게 막는 경우가 있습니다 — 다음 절에서 설명합니다.

격리 수준Dirty readNon-repeatablePhantomSerialization anomaly
READ UNCOMMITTED막힘 PG가능가능가능
READ COMMITTED (기본)막힘가능가능가능
REPEATABLE READ막힘막힘막힘 PG가능
SERIALIZABLE막힘막힘막힘막힘

PG 표시 칸이 표준과 다른 부분입니다. PG는 READ UNCOMMITTED에서도 dirty read를 절대 허용하지 않고, REPEATABLE READ가 phantom까지 막습니다.

3격리 수준 4단계 — 그리고 PostgreSQL의 강화

격리 수준은 "동시성 안전"과 "성능·재시도 부담" 사이의 다이얼입니다. 위로 갈수록 안전하지만 충돌·재시도가 늘어납니다. SET TRANSACTION ISOLATION LEVEL ... 또는 BEGIN ISOLATION LEVEL ...로 트랜잭션 단위로 지정합니다.

  • READ UNCOMMITTED — 표준에선 dirty read 허용. PG에선 사실상 READ COMMITTED와 동일(dirty read를 구현하지 않음).
  • READ COMMITTED (PG 기본) — 각 문장이 시작될 때의 스냅샷을 본다. 커밋된 값만 읽지만, 한 트랜잭션 안에서도 문장마다 스냅샷이 갱신돼 non-repeatable·phantom이 생길 수 있다.
  • REPEATABLE READ — 트랜잭션 시작 시점의 스냅샷을 끝까지 본다(스냅샷 격리). PG에서는 이 한 가지로 non-repeatable과 phantom까지 함께 막습니다. 단, 충돌 쓰기를 만나면 could not serialize access로 실패할 수 있다.
  • SERIALIZABLE — REPEATABLE READ에 SSI(Serializable Snapshot Isolation)를 더해 serialization anomaly까지 막는다. 가장 안전하지만 직렬화 실패 시 재시도가 필요하다.
study_db=# — 기본값 확인 & 트랜잭션 단위 설정
-- Postgres 기본 격리 수준
SHOW default_transaction_isolation;
 default_transaction_isolation
-------------------------------
 read committed

-- 이번 트랜잭션만 REPEATABLE READ 로 (스냅샷 격리)
BEGIN;
  SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
  SELECT current_setting('transaction_isolation') AS lvl;
COMMIT;
       lvl
-----------------
 repeatable read

-- SERIALIZABLE 도 같은 방식 (직렬화 실패 시 앱이 재시도해야 함)
BEGIN ISOLATION LEVEL SERIALIZABLE;
  SELECT current_setting('transaction_isolation') AS lvl;
COMMIT;
     lvl
--------------
 serializable
⚠️ PostgreSQL의 격리 수준은 '표준 이름'과 다르게 동작한다

① REPEATABLE READ가 표준보다 강하다. PostgreSQL의 REPEATABLE READ는 스냅샷 격리라 표준과 달리 phantom read까지 막습니다(단, 직렬화 이상은 남아 SERIALIZABLE이 필요할 수 있음). 다만 RR/SERIALIZABLE에서는 충돌하는 쓰기를 만나면 트랜잭션이 could not serialize access due to concurrent update로 실패할 수 있습니다. ② SERIALIZABLE은 재시도 전제. SSI는 직렬화 불가를 감지하면 한쪽을 40001 에러로 abort합니다 — 앱은 이 에러를 잡아 같은 트랜잭션을 다시 실행하도록 설계해야 합니다(락이 아니라 낙관적 방식이라 평소엔 빠르지만 충돌 시 재시도 책임이 앱에 있음).

4두 세션 타임라인 — non-repeatable read를 눈으로

이상현상은 두 트랜잭션이 동시에 돌 때만 보입니다(단일 psql 세션으로는 재현 불가 → 그래서 코드가 아니라 타임라인으로 봅니다). Session A가 같은 행을 두 번 읽는 사이, Session B가 값을 바꾸고 커밋하는 시나리오입니다.

시각Session A READ COMMITTEDSession B
t1BEGIN;
t2SELECT balance ... WHERE owner='Alice'; → 800
t3UPDATE accounts SET balance=900 WHERE owner='Alice'; COMMIT;
t4SELECT balance ... WHERE owner='Alice'; → 900 😱
t5COMMIT; — 같은 트랜잭션인데 값이 바뀜(non-repeatable read)

같은 시나리오를 Session A가 REPEATABLE READ였다면, t4의 두 번째 SELECT도 800을 돌려줍니다 — A는 시작 시점 스냅샷을 끝까지 보기 때문에 B의 커밋이 보이지 않습니다.

시각Session A REPEATABLE READSession B
t1BEGIN ISOLATION LEVEL REPEATABLE READ;
t2SELECT balance ... WHERE owner='Alice'; → 800
t3UPDATE ... =900; COMMIT;
t4SELECT balance ...; → 800 ✅ (스냅샷 고정)
t5COMMIT; — 트랜잭션 내내 일관된 값
기본 격리 수준 MySQL

MySQL InnoDB의 기본은 REPEATABLE READ입니다. 일반 SELECT는 트랜잭션 시작 스냅샷(consistent read)을 보고, phantom은 갭 락(gap lock)/넥스트키 락으로 막습니다. 그래서 MySQL에서 익숙해진 "한 트랜잭션 안에서 읽기가 안 변한다"는 감각이 기본값에서 성립합니다.

기본 격리 수준 PostgreSQL

PostgreSQL의 기본은 READ COMMITTED입니다. 각 문장이 시작될 때의 스냅샷을 보므로, 한 트랜잭션 안에서도 문장 사이에 다른 커밋이 보입니다(non-repeatable·phantom 가능). MySQL에서 넘어온 팀이 가장 자주 헛디디는 실무 차이입니다 — "읽기 일관성"이 필요하면 명시적으로 REPEATABLE READ를 켜세요.

복습

5✅ 셀프 체크

Q READ COMMITTED와 REPEATABLE READ의 차이는?

READ COMMITTED는 문장마다 새 스냅샷을 떠서, 한 트랜잭션 안에서도 다른 트랜잭션의 커밋이 다음 문장에 보입니다(non-repeatable read 가능). REPEATABLE READ는 트랜잭션 시작 시점의 스냅샷을 끝까지 유지해, 그 트랜잭션이 보는 데이터가 변하지 않습니다(PG에선 phantom까지 방지). 대신 충돌 쓰기 시 직렬화 실패로 끝날 수 있습니다.

Q PostgreSQL의 기본 격리 수준은? MySQL과 무엇이 다른가?

PostgreSQL 기본은 READ COMMITTED, MySQL InnoDB 기본은 REPEATABLE READ입니다. 그래서 MySQL에서 당연하던 "한 트랜잭션 내 읽기 불변"이 PG 기본값에서는 성립하지 않습니다. PG에서 같은 보장을 원하면 트랜잭션을 명시적으로 REPEATABLE READ로 시작해야 합니다.

Q SERIALIZABLE을 쓸 때 앱이 반드시 대비할 것은?

재시도(retry)입니다. SERIALIZABLE은 SSI로 직렬화 불가를 감지하면 한쪽 트랜잭션을 40001(serialization failure) 에러로 abort합니다. 앱은 이 에러를 잡아 같은 트랜잭션을 처음부터 다시 실행하는 루프를 갖춰야 합니다. 락이 아니라 낙관적 방식이라 평소엔 빠르지만, 충돌 처리 책임이 애플리케이션에 있습니다.
