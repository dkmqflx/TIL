MVCC · VACUUM
autovacuum · bloat

Postgres는 행을 덮어쓰지 않습니다 — 새 버전을 쓰고 옛 버전을 죽은 튜플로 남깁니다(MVCC). 그 시체를 치우는 청소부가 VACUUM입니다.

“

Postgres는 행을 덮어쓰지 않는다 — 새 버전을 쓰고 옛 버전을 죽은 튜플로 남긴다(MVCC). 그 시체를 치우는 청소부가 VACUUM이다.

  • MVCC 원리를 xmin·xmax·스냅샷으로 이해한다
  • 죽은 튜플이 왜 생기는지, VACUUM이 무엇을 회수하는지 안다
  • autovacuum과 bloat의 관계, 그리고 긴 트랜잭션의 위험을 안다

1MVCC 원리 — UPDATE는 덮어쓰기가 아니다

🍔 레시피 문서를 고칠 때, 옛 버전을 지우지 않고 새 버전을 따로 저장하는 버전 관리를 떠올려 보세요. "현재 읽는 사람"은 자기가 시작할 때의 버전을 계속 보고, 동시에 누가 새 버전을 써도 충돌하지 않습니다. PostgreSQL의 MVCC(Multi-Version Concurrency Control)가 정확히 이렇게 동작합니다 — UPDATE는 기존 행을 제자리에서 고치지 않고, 새 행 버전을 쓰고 옛 버전을 만료 표시합니다.

각 행에는 숨은 시스템 컬럼이 붙어 있습니다. xmin은 이 버전을 만든 트랜잭션 ID, xmax는 이 버전을 만료시킨 트랜잭션 ID(살아 있으면 0), ctid는 물리 위치(블록,오프셋)입니다. "내 스냅샷에서 이 버전이 보이는가"는 xmin/xmax와 내 스냅샷을 비교해 결정됩니다.

study_db=# — UPDATE 3번 → ctid 가 바뀐다(새 물리 버전)
-- 처음 상태: xmin 은 만든 트랜잭션, xmax=0(살아 있음), ctid=(블록,오프셋)
SELECT xmin, xmax, ctid, * FROM items;
 xmin | xmax | ctid  | id |  name  | price
------+------+-------+----+--------+-------
  804 |    0 | (0,1) |  1 | coffee |  4500

-- 같은 논리적 행을 세 번 UPDATE → 매번 새 물리 버전 생성
UPDATE items SET price = 4600 WHERE name = 'coffee';
UPDATE items SET price = 4700 WHERE name = 'coffee';
UPDATE items SET price = 4800 WHERE name = 'coffee';

-- 살아 있는 버전은 하나지만 ctid 가 (0,1) → (0,4) 로 옮겨졌다(=새 버전)
SELECT xmin, xmax, ctid, * FROM items;
 xmin | xmax | ctid  | id |  name  | price
------+------+-------+----+--------+-------
  807 |    0 | (0,4) |  1 | coffee |  4800
📝 xmin/xmax는 매번 다른 값

xmin/xmax는 트랜잭션 ID라 실행할 때마다 값이 달라집니다(위 804·807은 이 실행에서 나온 예시). 반면 ctid는 물리 위치라 같은 시나리오면 재현됩니다 — (0,1)에서 시작해 UPDATE 세 번 뒤 (0,4)로 옮겨진 것이, "옛 버전 3개를 남기고 새 위치에 현재 버전을 썼다"는 MVCC의 증거입니다.

2죽은 튜플 & VACUUM — 시체 치우기

UPDATE·DELETE가 남긴 만료된 옛 버전을 죽은 튜플(dead tuple)이라 부릅니다. 어떤 트랜잭션도 더는 볼 수 없게 된 이 시체들은 디스크 공간만 차지합니다. VACUUM은 이들을 회수해 그 공간을 같은 테이블이 재사용할 수 있게 만듭니다.

위에서 UPDATE를 3번 했으니 죽은 튜플이 3개 쌓였을 것입니다. pg_stat_user_tables의 n_dead_tup으로 확인하고, VACUUM 전후를 비교합니다.

study_db=# — n_dead_tup: VACUUM 전 3, 후 0
-- 통계는 비동기로 모이므로 잠깐 기다린 뒤 갱신
SELECT pg_sleep(1);  ANALYZE items;
SELECT relname, n_live_tup, n_dead_tup
FROM pg_stat_user_tables WHERE relname = 'items';
 relname | n_live_tup | n_dead_tup
---------+------------+------------
 items   |          1 |          3     -- 살아있는 1개 + 죽은 3개

-- VACUUM: 죽은 튜플을 회수해 그 공간을 "재사용 가능"하게 만든다
VACUUM (VERBOSE) items;
INFO:  finished vacuuming "study_db.public.items": index scans: 0
  pages: 0 removed, 1 remain, 1 scanned (100.00% of total) ...
  tuples: 3 removed, 1 remain, 0 are dead but not yet removable ...

SELECT pg_sleep(1);  ANALYZE items;
SELECT relname, n_live_tup, n_dead_tup
FROM pg_stat_user_tables WHERE relname = 'items';
 relname | n_live_tup | n_dead_tup
---------+------------+------------
 items   |          1 |          0     -- 죽은 튜플 회수됨
📝 VACUUM은 디스크를 OS에 돌려주지 않는다

일반 VACUUM은 죽은 튜플 공간을 같은 테이블이 다시 쓰도록 표시만 합니다 — 파일 크기는 거의 그대로입니다(곧 새 행이 그 자리를 채웁니다). 디스크를 정말로 OS에 반환하려면 VACUUM FULL이 테이블을 통째로 재작성하지만, 그동안 테이블에 ACCESS EXCLUSIVE 락이 걸려 읽기·쓰기가 모두 막힙니다. 운영 중에는 함부로 쓰지 마세요(필요하면 pg_repack 같은 무중단 도구를 검토).

3autovacuum & bloat — 청소를 못 따라가면 부푼다

죽은 튜플을 매번 수동으로 치울 수는 없습니다. 그래서 백그라운드 데몬 autovacuum이 임계치를 넘으면 자동으로 VACUUM(과 ANALYZE)을 돌립니다. 임계치는 대략 autovacuum_vacuum_threshold + autovacuum_vacuum_scale_factor × 행수입니다.

study_db=# — autovacuum 임계치 기본값 & 모니터링
SHOW autovacuum_vacuum_threshold;       -- 50
SHOW autovacuum_vacuum_scale_factor;    -- 0.2 (= 행의 20%)
-- → 죽은 튜플이 (50 + 0.2 × 행수) 를 넘으면 autovacuum 발동

-- 마지막 (auto)vacuum 시각 모니터링
SELECT relname, last_vacuum, last_autovacuum
FROM pg_stat_user_tables WHERE relname = 'items';
 relname |         last_vacuum          | last_autovacuum
---------+------------------------------+-----------------
 items   | 2026-06-09 13:02:18.98198+00 |   -- 수동 VACUUM 만 (autovacuum 은 이 데모에서 꺼 둠)

autovacuum이 변경 속도를 못 따라가면 죽은 튜플이 쌓여 테이블·인덱스가 bloat(팽창)됩니다 — 살아 있는 행은 그대로인데 파일은 계속 커지고, 스캔할 페이지가 늘어 느려집니다. 한 행을 1,000번 UPDATE해 일부러 bloat을 만들어 봅시다(autovacuum을 꺼 둔 테이블).

study_db=# — 1행을 1000번 UPDATE → 죽은 튜플 폭증 = bloat
-- 한 행을 반복 UPDATE (autovacuum 꺼 둔 상태)
DO $$ BEGIN FOR k IN 1..999 LOOP
  UPDATE items SET price = price + 1 WHERE name = 'coffee';
END LOOP; END $$;

SELECT pg_sleep(1);  ANALYZE items;
SELECT relname, n_live_tup, n_dead_tup,
       pg_size_pretty(pg_table_size('items')) AS table_size
FROM pg_stat_user_tables WHERE relname = 'items';
 relname | n_live_tup | n_dead_tup | table_size
---------+------------+------------+------------
 items   |          1 |        999 | 88 kB   -- 살아있는 행은 1개인데 88kB(처음엔 16kB)

-- VACUUM 으로 회수: 죽은 튜플은 0 이 되지만 파일 크기(88kB)는 줄지 않는다
VACUUM items;
SELECT pg_sleep(1);  ANALYZE items;
SELECT relname, n_live_tup, n_dead_tup,
       pg_size_pretty(pg_table_size('items')) AS table_size
FROM pg_stat_user_tables WHERE relname = 'items';
 relname | n_live_tup | n_dead_tup | table_size
---------+------------+------------+------------
 items   |          1 |          0 | 88 kB   -- 죽은 튜플은 0, 그러나 크기는 그대로(재사용용 빈 공간)
⚠️ bloat은 모니터링하고, 긴 트랜잭션을 피하라

잦은 UPDATE/DELETE 테이블은 n_dead_tup과 last_autovacuum을 꾸준히 지켜보세요. 죽은 튜플이 계속 늘거나 autovacuum이 안 돌면 테이블·인덱스가 팽창합니다. 핫한 테이블은 autovacuum_vacuum_scale_factor를 테이블 단위로 낮춰(예: 0.02) 더 자주 돌게 할 수 있습니다.

4긴 트랜잭션이 VACUUM을 막는다

VACUUM은 "어떤 살아 있는 트랜잭션도 더는 못 보는" 죽은 튜플만 회수할 수 있습니다. 그런데 오래 열려 있는 트랜잭션(또는 오래된 스냅샷)이 하나라도 있으면, 그 트랜잭션이 볼 수도 있는 옛 버전들을 VACUUM이 치우지 못하고 보존해야 합니다. 결과적으로 죽은 튜플이 회수되지 못해 전체 DB에 bloat이 누적됩니다.

⚠️ 트랜잭션은 짧게 — 화면을 보며 트랜잭션을 열어 두지 마라

"BEGIN 후 사용자 입력을 기다린다", "분석 쿼리를 한 시간째 돌린다", "커넥션을 잡고 잊어버린 좀비 트랜잭션" — 이런 오래된 스냅샷이 VACUUM의 회수 지점을 과거에 묶어 둡니다. SELECT * FROM pg_stat_activity WHERE state <> 'idle' ORDER BY xact_start;로 오래된 트랜잭션을 찾고, idle_in_transaction_session_timeout으로 방치된 트랜잭션을 자동 종료시키세요. 트랜잭션은 짧게 열고 빨리 닫는 것이 MVCC 시스템의 위생 수칙입니다.

옛 버전 관리 MySQL

MySQL InnoDB도 MVCC를 쓰지만, 옛 버전을 테이블 밖의 undo log(롤백 세그먼트)에 둡니다. 현재 행은 제자리에서 갱신되고, 옛 버전이 필요한 읽기는 undo를 따라 되돌려 재구성합니다. 더는 필요 없어진 undo는 purge 스레드가 정리합니다. 긴 트랜잭션이 있으면 undo가 쌓여 "history list length"가 길어집니다(PG의 bloat와 비슷한 증상).

옛 버전 관리 PostgreSQL

PostgreSQL은 옛 버전을 테이블 안(같은 페이지/힙)에 그대로 두고, xmin/xmax로 가시성을 판단합니다. 그래서 죽은 튜플이 테이블에 직접 쌓이고, 이를 VACUUM/autovacuum이 회수합니다. 별도 undo 로그가 없는 대신 VACUUM 운영이 PG 성능 관리의 핵심이 됩니다.

복습

5✅ 셀프 체크

Q 하나의 UPDATE는 무엇을 만드나?

새 행 버전 하나와 죽은 튜플 하나(만료된 옛 버전)를 만듭니다. PG는 행을 제자리에서 덮어쓰지 않고, 새 버전을 쓴 뒤 옛 버전의 xmax를 채워 만료시킵니다. 그래서 같은 행을 N번 UPDATE하면 죽은 튜플이 약 N개 쌓이고, ctid(물리 위치)가 옮겨집니다.

Q VACUUM의 역할은? VACUUM FULL과 무엇이 다른가?

일반 VACUUM은 죽은 튜플을 회수해 그 공간을 같은 테이블이 재사용하도록 표시합니다 — 파일 크기는 거의 그대로이고, 테이블을 잠그지 않습니다. VACUUM FULL은 테이블을 통째로 재작성해 디스크를 OS에 반환하지만, ACCESS EXCLUSIVE 락으로 읽기·쓰기를 모두 막으므로 운영 중에는 위험합니다.

Q 긴 트랜잭션이 bloat을 부르는 이유는?

VACUUM은 "어떤 살아 있는 트랜잭션도 더는 못 보는" 죽은 튜플만 회수할 수 있습니다. 오래 열린 트랜잭션(오래된 스냅샷)이 있으면, 그 스냅샷이 볼 수도 있는 옛 버전들을 VACUUM이 치우지 못하고 보존해야 해서 죽은 튜플이 회수되지 못합니다. 그래서 트랜잭션을 짧게 유지하고 방치된 트랜잭션을 자동 종료시키는 것이 중요합니다.
