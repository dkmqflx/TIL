락 · FOR UPDATE
데드락 · advisory lock

동시에 같은 행을 고치려는 둘을 어떻게 조율할까요. 행 잠금(FOR UPDATE), 데드락, 그리고 큐 패턴을 만드는 SKIP LOCKED·advisory lock.

“

동시에 같은 행을 고치려는 둘을 어떻게 조율하나. 행 잠금(FOR UPDATE), 데드락, 그리고 큐 패턴을 만드는 SKIP LOCKED·advisory lock.

  • 행 수준 락(FOR UPDATE/FOR SHARE)과 NOWAIT·SKIP LOCKED를 안다
  • 데드락을 이해하고 회피하는 법(같은 순서로 잠그기)을 안다
  • advisory lock과 SKIP LOCKED 큐 패턴으로 동시성 문제를 푼다

1행 잠금 — FOR UPDATE · FOR SHARE · NOWAIT · SKIP LOCKED

🍔 탈의실 문을 안에서 잠그는 것을 떠올려 보세요 — 한 사람이 쓰는 동안 다른 사람은 그 칸을 못 씁니다. SELECT ... FOR UPDATE는 고른 행에 "내가 곧 고칠 거다"라는 잠금을 걸어, 다른 트랜잭션이 같은 행을 UPDATE/FOR UPDATE하려 하면 이 트랜잭션이 끝날 때까지 대기시킵니다. FOR SHARE는 더 약해서 "남이 읽는 건 괜찮지만 고치진 마라"입니다.

study_db=# — FOR UPDATE / NOWAIT (단일 세션이라 충돌 없이 통과)
-- 이 행을 "고칠 거다"라고 잠금. 다른 세션의 대기 장면은 아래 타임라인 참조.
BEGIN;
  SELECT id, payload FROM jobs WHERE id = 1 FOR UPDATE;
COMMIT;
 id |   payload
----+--------------
  1 | send-email-1

-- NOWAIT: 잠겨 있으면 기다리지 말고 즉시 에러(단일 세션이라 충돌 없이 통과)
BEGIN;
  SELECT id FROM jobs WHERE id = 2 FOR UPDATE NOWAIT;
COMMIT;

두 세션이 실제로 같은 행을 두고 충돌하면 어떻게 보이는지는 단일 psql로 재현할 수 없습니다(한 세션은 자기 락과 충돌하지 않으니까요). 타임라인으로 봅시다 — Session A가 행을 잠그고 커밋하기 전까지 Session B는 대기합니다.

시각Session ASession B
t1BEGIN; SELECT ... WHERE id=1 FOR UPDATE; → 행 1 잠금 획득
t2(작업 처리 중…)BEGIN; SELECT ... WHERE id=1 FOR UPDATE; → 대기(blocked) ⏳
t3UPDATE jobs SET status='done' WHERE id=1; COMMIT; → 락 해제(여전히 대기 중)
t4대기 풀림 → 갱신된 행 1을 잠그고 진행 ✅
⚠️ 락은 짧게 — 사용자 입력을 기다리며 락을 잡지 마라

행 락은 트랜잭션이 끝날 때 풀립니다. FOR UPDATE로 행을 잠근 채 사용자 확인·외부 API 응답을 기다리면, 그동안 다른 모든 요청이 그 행에서 줄을 섭니다. 잠그기 → 빠르게 갱신 → 즉시 커밋이 원칙입니다. 오래 기다려야 한다면 락을 잡지 말고, 낙관적 잠금(버전 컬럼 비교)이나 큐를 쓰세요.

2데드락 — 서로의 락을 기다리다 멈춤

데드락(deadlock)은 두 트랜잭션이 서로가 가진 락을 기다리며 영원히 멈추는 상황입니다. A는 row1을 잡고 row2를 원하고, B는 row2를 잡고 row1을 원하면 — 둘 다 영영 진행 못 합니다. PostgreSQL은 이를 자동으로 감지해 한쪽 트랜잭션을 ERROR: deadlock detected로 abort하고, 나머지 한쪽은 진행시킵니다(앱은 abort된 트랜잭션을 재시도).

시각Session ASession B
t1BEGIN; UPDATE jobs SET ... WHERE id=1; → 행 1 잠금BEGIN; UPDATE jobs SET ... WHERE id=2; → 행 2 잠금
t2UPDATE ... WHERE id=2; → B가 쥔 행 2 대기 ⏳UPDATE ... WHERE id=1; → A가 쥔 행 1 대기 ⏳
t3PG가 순환 대기 감지 → 한쪽을 ERROR: deadlock detected로 abort나머지 한쪽은 락을 얻어 진행
💡 데드락 회피의 황금률: 항상 같은 순서로 잠그기

여러 행·테이블을 잠가야 한다면 모든 트랜잭션이 동일한 순서(예: 항상 id 오름차순)로 잠그세요. 그러면 순환 대기가 생길 수 없어 데드락이 원천 차단됩니다. 그 밖에: 트랜잭션을 짧게 유지하고, 한 번에 잠그는 행 수를 줄이며, 그래도 드물게 나는 데드락은 재시도 가능한 에러로 다루세요(40P01).

3advisory lock — 행이 아니라 '앱이 정한 키'로 잠그기

때로는 특정 행이 아니라 추상적인 자원을 잠그고 싶습니다 — "이 크론 작업은 한 번에 한 인스턴스만 돌게", "이 사용자에 대한 정산을 동시에 두 번 하지 않게". advisory lock은 행과 무관하게 임의의 정수 키로 잠금을 겁니다. DB는 키의 의미를 모르고, 잠금 규칙은 전적으로 앱의 약속입니다.

study_db=# — pg_try_advisory_lock: 기다리지 않고 성공 여부 반환
-- try_ 계열은 기다리지 않고 t/f 를 즉시 반환(스크립트가 멈추지 않는다)
SELECT pg_try_advisory_lock(42) AS got_lock;    -- 이 세션이 키 42 를 잡음
 got_lock
----------
 t

-- 같은 세션은 재진입 가능(카운트 증가). 다른 세션이었다면 f 를 받는다.
SELECT pg_try_advisory_lock(42) AS got_again;    -- t

-- 잡은 만큼 풀어 준다(두 번 잡았으니 두 번 unlock)
SELECT pg_advisory_unlock(42) AS released_1;   -- t
SELECT pg_advisory_unlock(42) AS released_2;   -- t

-- 현재 들고 있는 advisory lock 확인(모두 풀었으니 0)
SELECT count(*) AS held_advisory_locks
FROM pg_locks WHERE locktype = 'advisory';
 held_advisory_locks
---------------------
                   0
📝 세션 범위 vs 트랜잭션 범위

pg_advisory_lock(key)(블로킹)·pg_try_advisory_lock(key)(즉시 반환)는 세션 범위 — 명시적으로 pg_advisory_unlock하거나 세션이 끊겨야 풀립니다. 반면 pg_advisory_xact_lock(key) 계열은 트랜잭션 범위로, 커밋/롤백 시 자동 해제됩니다(풀기를 잊을 위험이 없어 더 안전한 기본 선택). 분산 락처럼 쓰지만, 잠금이 이 DB 인스턴스 한정임을 기억하세요.

4SKIP LOCKED 큐 패턴 — 워커가 안 겹치게 작업을 집어간다

작업 큐(jobs 테이블)에서 여러 워커가 동시에 다음 작업을 집어갈 때, 같은 행을 두 워커가 잡으면 안 됩니다. 그렇다고 FOR UPDATE만 쓰면 워커들이 같은 맨 앞 행에서 줄을 서 직렬화됩니다. 해법이 FOR UPDATE SKIP LOCKED입니다 — 이미 잠긴 행은 건너뛰고 잠기지 않은 다음 행을 가져오므로, 각 워커가 서로 다른 작업을 충돌 없이 집어갑니다.

study_db=# — FOR UPDATE SKIP LOCKED 로 큐에서 작업 하나 집기
-- 잠긴 행은 건너뛰고 안 잠긴 다음 행을 가져온다(단일 세션이라 맨 앞 행)
BEGIN;
  SELECT id, payload
  FROM jobs
  WHERE status = 'queued'
  ORDER BY id
  FOR UPDATE SKIP LOCKED
  LIMIT 1;
  UPDATE jobs SET status = 'done' WHERE id = 1;  -- 처리 후 상태 변경
COMMIT;
 id |   payload
----+--------------
  1 | send-email-1

SELECT id, payload, status FROM jobs ORDER BY id;
 id |    payload     | status
----+----------------+--------
  1 | send-email-1   | done
  2 | send-email-2   | queued
  3 | resize-image-3 | queued
  4 | build-report-4 | queued
  5 | charge-card-5  | queued

실제 운영에서 워커 A와 워커 B가 동시에 위 쿼리를 실행하면, A는 행 1을 잠그고 B는 그 행을 건너뛰어 행 2를 집어갑니다 — 락 대기 없이 병렬로 큐를 비웁니다. 락을 무한정 기다리고 싶지 않을 때는 lock_timeout으로 상한을 둘 수도 있습니다.

study_db=# — lock_timeout: 락을 너무 오래 기다리면 포기
-- 락을 1초 넘게 기다리면 포기하고 에러(충돌이 없으면 즉시 성공)
SET lock_timeout = '1s';
SHOW lock_timeout;
 lock_timeout
--------------
 1s

BEGIN;
  SELECT id FROM jobs WHERE id = 3 FOR UPDATE;  -- 충돌 없음 → 즉시 성공
COMMIT;
-- 다른 세션이 행 3 을 잡고 있었다면 1초 뒤:
-- ERROR:  canceling statement due to lock timeout  (이 장면은 두 세션에서만 재현)
행 잠금 · 큐 MySQL

MySQL InnoDB도 SELECT ... FOR UPDATE·FOR SHARE를 지원하고, 8.0부터 NOWAIT·SKIP LOCKED가 추가돼 같은 큐 패턴을 쓸 수 있습니다. 데드락도 자동 감지해 한쪽을 롤백합니다. 다만 잠금 단위·갭 락 동작 등 세부는 InnoDB 특유라, PG와 동일하게 가정하지 말고 확인하세요. advisory lock에 해당하는 것은 GET_LOCK(name, timeout)·RELEASE_LOCK(name)입니다.

행 잠금 · 큐 PostgreSQL

PostgreSQL은 FOR UPDATE/FOR SHARE에 NOWAIT·SKIP LOCKED를 제공하고, 데드락을 자동 감지해 한쪽을 40P01로 abort합니다. SKIP LOCKED 큐 패턴과 정수 키 기반 advisory lock(pg_advisory_lock 계열)이 잘 갖춰져 있어, 외부 큐 시스템 없이도 가벼운 작업 큐·분산 락을 DB 안에서 구현하기 좋습니다.

복습

5✅ 셀프 체크

Q SELECT ... FOR UPDATE는 무엇에 쓰나?

고른 행에 쓰기 의도 잠금을 걸어, 다른 트랜잭션이 같은 행을 UPDATE/FOR UPDATE하지 못하게 막습니다(이 트랜잭션이 끝날 때까지 대기). "읽은 값을 보고 갱신"하는 동안 그 사이 다른 트랜잭션이 끼어들지 못하게 하는 읽고-고치기(read-modify-write)의 안전장치입니다. 락은 트랜잭션 종료 시 풀리므로 짧게 잡고 빨리 커밋해야 합니다.

Q 데드락을 어떻게 회피하나?

가장 확실한 방법은 모든 트랜잭션이 자원을 같은 순서로 잠그는 것입니다(예: 항상 id 오름차순). 그러면 순환 대기가 생길 수 없습니다. 더해서 트랜잭션을 짧게 유지하고, 잠그는 행 수를 줄이며, 그래도 드물게 나는 데드락(40P01)은 재시도 가능한 에러로 처리합니다. PG는 데드락을 자동 감지해 한쪽만 abort합니다.

Q SKIP LOCKED 큐 패턴은 어떤 문제를 푸나?

여러 워커가 같은 작업 큐에서 다음 작업을 집어갈 때, 같은 행을 둘이 잡거나 서로 줄을 서서 직렬화되는 문제를 풉니다. SELECT ... FOR UPDATE SKIP LOCKED LIMIT 1은 이미 잠긴 행을 건너뛰어 각 워커가 서로 다른 작업을 락 대기 없이 병렬로 가져가게 합니다 — 외부 큐 없이 DB만으로 작업 큐를 만드는 실무 단골 패턴입니다.
