RLS
Row Level Security

한 테이블에 여러 테넌트의 데이터가 섞여 있어도, RLS를 켜면 DB가 행 단위로 '네가 볼 수 있는 행'만 돌려줍니다 — 앱 코드의 WHERE tenant_id = ?를 DB가 강제합니다.

“

한 테이블에 여러 테넌트의 데이터가 섞여 있어도, RLS를 켜면 DB가 행 단위로 '네가 볼 수 있는 행'만 돌려준다 — 앱 코드의 WHERE tenant_id = ?를 DB가 강제한다.

  • RLS 정책의 USING(가시성)과 WITH CHECK(쓰기 제약)를 구분한다
  • 세션 변수 + 정책으로 멀티테넌시를 구현하고 행이 걸러지는 것을 확인한다
  • 흔한 함정 — superuser·owner의 RLS 우회와 FORCE를 안다

1RLS란 & 왜 — 앱의 WHERE를 DB가 강제한다

🍔 도서관 대출 창구를 떠올려 보세요. 창구 직원은 모든 사람의 대출 기록이 든 같은 장부를 보지만, 당신에게는 당신의 기록만 보여 줍니다. 다른 사람 카드 번호를 댄다고 남의 기록이 나오지 않습니다 — 창구가 "네 것만"이라는 규칙을 강제하기 때문입니다. RLS가 바로 이 창구입니다.

멀티테넌트 SaaS는 보통 한 테이블에 여러 고객사(테넌트)의 행을 함께 담고 tenant_id로 구분합니다. 문제는 앱이 매 쿼리마다 WHERE tenant_id = ?를 빠뜨리지 않아야 한다는 것 — 단 한 군데라도 빠지면 다른 테넌트의 데이터가 유출됩니다. RLS(Row Level Security)는 그 필터를 애플리케이션이 아니라 DB가 행 단위로 강제하게 만듭니다. 개발자가 깜빡해도 DB가 막아 주는, '깊이 있는 방어(defense in depth)'입니다.

2켜기 & 정책 — USING vs WITH CHECK

RLS는 두 단계입니다. 먼저 테이블에 스위치를 켜고(ENABLE ROW LEVEL SECURITY), 그다음 정책(policy)으로 "어떤 행을 허용할지" 규칙을 적습니다. 중요한 기본값: RLS를 켰는데 정책이 하나도 없으면 기본은 deny-all — 아무 행도 안 보입니다.

정책의 두 조건을 꼭 구분하세요. USING은 읽기·기존 행의 가시성(SELECT/UPDATE/DELETE 대상이 되는 행을 거름)이고, WITH CHECK는 새로 삽입·수정되는 행이 만족해야 할 제약(INSERT/UPDATE의 결과 행을 검사)입니다.

study_db=# — RLS 켜고 테넌트 격리 정책 만들기
-- 1) 테이블에 RLS 스위치 켜기 (정책이 없으면 기본 deny-all)
ALTER TABLE documents ENABLE ROW LEVEL SECURITY;

-- 2) 정책: 세션 변수 app.current_tenant 와 같은 tenant_id 행만 허용
--    current_setting(..., true): 변수가 없으면 에러 대신 NULL 반환
CREATE POLICY tenant_isolation ON documents
  USING      (tenant_id = current_setting('app.current_tenant', true)::int)
  WITH CHECK (tenant_id = current_setting('app.current_tenant', true)::int);
CREATE POLICY
📝 USING과 WITH CHECK를 따로 두는 이유

USING만 있으면 "내 테넌트 행은 보고 고칠 수 있다"까지입니다. 하지만 WITH CHECK 없이는 INSERT로 남의 tenant_id를 적어 넣는 것을 막지 못합니다. 둘을 같은 조건으로 주면 "내 테넌트 행만 보이고, 새로 넣는 행도 반드시 내 테넌트여야 한다"가 완성됩니다. 두 조건을 다르게 줄 수도 있습니다(예: 읽기는 넓게, 쓰기는 좁게).

3멀티테넌시 실전 — 테넌트별로 다른 행이 나온다

앱은 커넥션마다 "지금 이 요청은 어느 테넌트인가"를 세션 변수로 설정합니다(SET app.current_tenant = '1' 또는 set_config('app.current_tenant','1',false)). 정책이 그 변수를 참조하므로, 같은 쿼리라도 변수 값에 따라 다른 행이 돌아옵니다.

그런데 여기엔 함정이 있습니다(다음 절). 우리가 psql로 접속한 study 유저는 SUPERUSER라 RLS를 통째로 우회합니다 — 정책을 켜도 5개 행이 다 보입니다. 그래서 RLS가 실제로 행을 거르는 장면을 보려면 NOSUPERUSER 앱 롤을 만들어 SET ROLE로 그 롤이 된 뒤 조회해야 합니다.

study_db=# — SUPERUSER(study)는 RLS를 우회 → 전체가 보인다
-- 접속 유저 study 는 SUPERUSER 라 정책을 무시한다(GOTCHA, 4절 참고)
SELECT id, tenant_id, title FROM documents ORDER BY id;
 id | tenant_id |        title
----+-----------+----------------------
  1 |         1 | Acme 분기 보고서
  2 |         1 | Acme 채용 계획
  3 |         2 | Globex 신제품 로드맵
  4 |         2 | Globex 가격표
  5 |         3 | Initech 회의록
(5 rows)
study_db=# — app_user(NOSUPERUSER)로 바꾸면 RLS가 행을 거른다 ⭐
-- 앱 롤이 된다(이 롤은 SUPERUSER 도 owner 도 아니므로 정책 적용을 받는다)
SET ROLE app_user;

-- 테넌트 1 로 설정 → 테넌트 1 행만 보인다
SET app.current_tenant = '1';
SELECT id, tenant_id, title FROM documents ORDER BY id;
 id | tenant_id |      title
----+-----------+------------------
  1 |         1 | Acme 분기 보고서
  2 |         1 | Acme 채용 계획
(2 rows)

-- 똑같은 쿼리, 테넌트 2 로 바꾸면 → 테넌트 2 행만! (DB가 거른 것)
SET app.current_tenant = '2';
SELECT id, tenant_id, title FROM documents ORDER BY id;
 id | tenant_id |        title
----+-----------+----------------------
  3 |         2 | Globex 신제품 로드맵
  4 |         2 | Globex 가격표
(2 rows)

같은 SELECT * FROM documents인데 app.current_tenant가 1이면 2개 행, 2면 완전히 다른 2개 행이 나옵니다 — 앱이 WHERE를 한 글자도 쓰지 않았는데 DB가 알아서 걸러 줬습니다. 이것이 RLS의 핵심입니다. 쓰기 쪽도 WITH CHECK가 막아 줍니다.

study_db=# — WITH CHECK: 남의 테넌트 행은 못 넣는다
-- 현재 테넌트는 2. 테넌트 1 행을 몰래 넣으려 하면 거부된다.
INSERT INTO documents (tenant_id, title) VALUES (1, '몰래 넣기');
NOTICE:  WITH CHECK 위반으로 INSERT 거부됨:
  new row violates row-level security policy for table "documents"

-- 같은 테넌트(2) 행은 정상 INSERT
INSERT INTO documents (tenant_id, title) VALUES (2, 'Globex 추가 메모');
INSERT 0 1

RESET ROLE;  -- 다시 study(superuser) 로 복귀

4⚠️ 함정 — superuser·owner는 RLS를 우회한다

⚠️ RLS가 적용되지 않는 경우 — 이걸 모르면 "RLS가 안 먹는다"고 착각한다

① SUPERUSER는 RLS를 통째로 우회합니다 — 정책을 켜도 모든 행이 보입니다. ② 테이블 OWNER도 기본적으로 우회합니다. 그래서 앱이 superuser나 owner로 붙으면 RLS는 사실상 꺼진 것과 같습니다(그래서 Week 6·02에서 앱 전용 최소 권한 유저로 붙으라고 합니다). owner에게도 정책을 강제하려면 ALTER TABLE t FORCE ROW LEVEL SECURITY;를 켜세요. ③ 여러 정책이 있으면 같은 명령에 대해 OR로 결합(PERMISSIVE, 기본)되거나, RESTRICTIVE 정책은 AND로 결합됩니다. ④ 정책 조건도 결국 WHERE에 더해지므로 정책이 참조하는 컬럼(예: tenant_id)에 인덱스가 있어야 성능이 납니다.

study_db=# — FORCE RLS 설정 & 카탈로그 확인
-- owner 에게도 RLS 강제(기본은 owner 우회)
ALTER TABLE documents FORCE ROW LEVEL SECURITY;

-- relrowsecurity=RLS 켜짐, relforcerowsecurity=owner 에도 강제
SELECT relname, relrowsecurity, relforcerowsecurity
FROM pg_class WHERE relname = 'documents';
  relname  | relrowsecurity | relforcerowsecurity
-----------+----------------+---------------------
 documents | t              | t
(1 row)

-- 정의된 정책 목록 (polcmd '*' = ALL 명령에 적용)
SELECT polname, polcmd FROM pg_policy
WHERE polrelid = 'documents'::regclass;
     polname      | polcmd
------------------+--------
 tenant_isolation | *
(1 row)
💡 Supabase가 RLS를 핵심으로 쓰는 이유

Supabase(Week 1에서 본 Postgres 기반 BaaS)는 RLS를 보안 모델의 중심으로 씁니다. 브라우저가 DB에 직접 쿼리해도, 로그인한 사용자의 JWT가 세션 변수로 들어오고 RLS 정책(auth.uid() = user_id 같은)이 행을 걸러 주기 때문에 안전합니다. "앱 코드가 아니라 DB가 권한을 강제한다"는 RLS의 발상이 그대로 제품 아키텍처가 된 사례입니다.

행 수준 보안 MySQL

MySQL에는 행 수준 보안 정책이 기본으로 없습니다. 보통 뷰(VIEW)로 테넌트별 필터를 감싸거나, 애플리케이션 로직에서 매 쿼리 WHERE tenant_id = ?를 직접 붙여 처리합니다. 즉 필터를 빠뜨리지 않을 책임이 전적으로 앱에 있어, 한 곳만 실수해도 데이터가 새어 나갈 수 있습니다.

행 수준 보안 PostgreSQL

PostgreSQL은 네이티브 RLS 정책을 제공합니다. ENABLE ROW LEVEL SECURITY + CREATE POLICY로 "누가 어떤 행을 보고/쓸 수 있는지"를 DB 차원에서 선언하면, 앱이 WHERE를 깜빡해도 DB가 강제합니다. 단, 앱은 superuser/owner가 아닌 최소 권한 롤로 붙어야 정책이 실제로 적용됩니다.

5✅ 셀프 체크

Q RLS 정책의 USING과 WITH CHECK는 무엇이 다른가?

USING은 기존 행의 가시성을 거릅니다 — SELECT·UPDATE·DELETE가 대상으로 삼을 수 있는 행을 결정합니다. WITH CHECK는 새로 삽입·수정되는 행이 만족해야 할 제약입니다 — INSERT나 UPDATE의 결과 행을 검사해, 어기면 거부합니다. WITH CHECK 없이 USING만 두면 남의 tenant_id를 적어 넣는 INSERT를 막지 못하므로, 멀티테넌시에선 보통 둘을 같은 조건으로 줍니다.

Q 정책을 켰는데도 모든 행이 다 보였다. 무엇을 의심해야 하나?

접속한 롤이 SUPERUSER이거나 테이블 OWNER일 가능성이 큽니다. 둘 다 기본적으로 RLS를 우회합니다. owner에게도 적용하려면 ALTER TABLE t FORCE ROW LEVEL SECURITY;를 켜고, 앱은 superuser·owner가 아닌 최소 권한 롤로 접속해야 합니다. 또 ENABLE ROW LEVEL SECURITY 자체를 안 켰을 수도 있으니 함께 확인하세요.

Q 멀티테넌시에서 세션 변수(app.current_tenant)는 어떤 역할인가?

정책이 참조하는 "지금 이 커넥션이 누구인가"를 담는 값입니다. 앱은 요청을 처리할 때 커넥션에 SET app.current_tenant = '...'(또는 set_config)로 현재 테넌트를 설정하고, 정책은 tenant_id = current_setting('app.current_tenant')::int로 그 값을 비교합니다. 그래서 같은 쿼리라도 변수 값에 따라 돌아오는 행이 달라집니다. 커넥션 풀을 쓸 때는 요청마다 이 변수를 반드시 설정·초기화해야 다른 테넌트의 값이 새지 않습니다.
