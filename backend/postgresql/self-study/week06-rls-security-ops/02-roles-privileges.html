ROLE · GRANT
앱 전용 최소 권한 유저

Postgres에서 유저와 그룹은 모두 ROLE입니다. 앱은 superuser로 붙지 말고, 꼭 필요한 권한만 가진 '앱 전용 최소 권한 유저'로 붙어야 합니다.

“

Postgres에서 유저와 그룹은 모두 ROLE이다. 앱은 superuser로 붙지 말고, 꼭 필요한 권한만 가진 '앱 전용 최소 권한 유저'로 붙어야 한다.

  • ROLE 모델(유저=그룹=ROLE)과 LOGIN/멤버십을 안다
  • GRANT·REVOKE·스키마 USAGE로 권한을 부여한다
  • 앱 전용 최소 권한 유저 + ALTER DEFAULT PRIVILEGES를 구성한다

1ROLE = 유저 + 그룹 — LOGIN 여부가 가른다

🍔 회사 출입카드를 떠올려 보세요. 어떤 카드는 사람에게(로그인하는 유저), 어떤 카드는 '개발팀' 같은 그룹에 발급됩니다. 사람을 그룹에 넣으면 그룹 권한을 그대로 물려받습니다. Postgres에서는 이 둘이 구분 없이 모두 ROLE입니다 — 옛 CREATE USER는 그저 CREATE ROLE ... LOGIN의 별칭일 뿐입니다.

정리하면: USER란 LOGIN 속성을 가진 ROLE입니다. NOLOGIN 롤은 직접 접속은 못 하지만 권한 묶음(그룹)으로 쓰입니다. 멤버십은 GRANT 그룹롤 TO 유저롤로 만들고, 멤버는 그룹의 권한을 상속합니다.

study_db=# — 앱 유저 롤 두 개 만들기 (둘 다 NOSUPERUSER)
-- app_rw: 읽기/쓰기, app_ro: 읽기 전용. LOGIN 이라 곧 '유저'.
CREATE ROLE app_rw NOSUPERUSER LOGIN PASSWORD 'rw_pw';
CREATE ROLE app_ro NOSUPERUSER LOGIN PASSWORD 'ro_pw';

-- rolcanlogin = t 면 로그인 가능 = 통상적 의미의 '유저'
SELECT rolname, rolsuper, rolcanlogin
FROM pg_roles WHERE rolname IN ('app_rw','app_ro') ORDER BY rolname;
 rolname | rolsuper | rolcanlogin
---------+----------+-------------
 app_ro  | f        | t
 app_rw  | f        | t
(2 rows)

2권한 — 스키마 USAGE부터, 그다음 테이블 권한

PostgreSQL에서 자주 헛디디는 지점: 테이블에 SELECT 권한을 줬는데도 "permission denied for schema"가 난다는 것. Postgres는 스키마에 대한 USAGE 권한이 별도이기 때문입니다. 스키마를 "들여다볼 수 있게" USAGE를 먼저 주고, 그 안의 테이블에 GRANT해야 합니다.

study_db=# — 스키마 USAGE + 테이블 권한 부여
-- 1) 스키마를 쓸 수 있게 USAGE 부터
GRANT USAGE ON SCHEMA public TO app_rw, app_ro;

-- 2) app_rw 는 읽기+쓰기, app_ro 는 읽기만
GRANT SELECT, INSERT, UPDATE, DELETE ON orders TO app_rw;
GRANT SELECT ON orders TO app_ro;

-- orders.id 는 GENERATED ALWAYS AS IDENTITY. IDENTITY 컬럼은 뒤의 시퀀스
-- USAGE 가 필요 없다 — 테이블 INSERT 권한이면 충분(3절에서 증명).
-- 시퀀스 USAGE 는 레거시 serial 이나 앱이 직접 nextval() 호출 시에만 필요:
--   GRANT USAGE ON ALL SEQUENCES IN SCHEMA public TO app_rw;
📝 PUBLIC 의사 롤 — 모두에게 자동 부여되는 함정

PUBLIC은 모든 롤이 자동으로 속하는 가상의 롤입니다. 과거 버전에서는 public 스키마에 PUBLIC이 기본 CREATE 권한을 가져, 누구나 테이블을 만들 수 있었습니다. PostgreSQL 15부터는 이 기본 권한이 제거되어 더 안전해졌습니다. REVOKE로 PUBLIC 권한을 명시적으로 회수해 "기본은 막고 필요한 롤에만 허용"하는 것이 안전한 습관입니다.

3앱 전용 최소 권한 유저 — superuser로 붙지 마라

운영에서 가장 흔한 실수는 앱을 superuser(또는 DB 소유자)로 접속시키는 것입니다. 그러면 SQL 인젝션 한 방에 DROP TABLE·DROP DATABASE까지 갈 수 있고, 무엇보다 RLS(Lesson 01)가 우회됩니다. 앱은 딱 필요한 권한만 가진 롤로 붙어야 합니다 — 읽기/쓰기는 app_rw, 보고서·대시보드는 app_ro 식으로.

study_db=# — 각 롤의 권한을 SET ROLE로 검증
-- (A) app_ro: SELECT 는 되고, INSERT 는 거부된다(의도된 동작)
SET ROLE app_ro;
SELECT count(*) AS visible_rows FROM orders;   -- 3 (읽기 OK)
 visible_rows
--------------
            3

INSERT INTO orders (amount) VALUES (999.00);
NOTICE:  app_ro INSERT 거부됨(정상):
  permission denied for table orders
RESET ROLE;

-- (B) app_rw: SELECT 와 INSERT 모두 성공
SET ROLE app_rw;
INSERT INTO orders (amount) VALUES (500.00);
INSERT 0 1
SELECT count(*) AS rows_after_rw_insert FROM orders;   -- 4
 rows_after_rw_insert
----------------------
                    4
RESET ROLE;

그런데 GRANT ... ON orders는 이미 존재하는 테이블에만 적용됩니다. 나중에 만드는 테이블은 앱 유저가 다시 권한을 못 받죠. 이를 해결하는 것이 ALTER DEFAULT PRIVILEGES — "앞으로 이 스키마에 만들 테이블에는 자동으로 이 권한을 줘라"는 예약 설정입니다.

study_db=# — ALTER DEFAULT PRIVILEGES: 미래 테이블에 자동 부여
-- study 가 앞으로 public 에 만들 테이블은 app_ro 가 자동으로 읽을 수 있게
ALTER DEFAULT PRIVILEGES IN SCHEMA public
  GRANT SELECT ON TABLES TO app_ro;

-- 검증: 설정 후 새 테이블을 만들고 app_ro 가 바로 읽히는지 확인
CREATE TABLE invoices (id int GENERATED ALWAYS AS IDENTITY PRIMARY KEY, total numeric);
INSERT INTO invoices (total) VALUES (10), (20);

SET ROLE app_ro;
SELECT count(*) AS invoices_readable_by_ro FROM invoices;  -- 2 (자동 부여!)
 invoices_readable_by_ro
-------------------------
                       2
RESET ROLE;

-- 등록된 default privilege 확인 (app_ro 에게 read(r) 가 예약됨)
SELECT defaclobjtype, defaclacl FROM pg_default_acl;
 defaclobjtype |    defaclacl
---------------+------------------
 r             | {app_ro=r/study}
(1 row)
⚠️ 앱을 superuser·DB 소유자로 붙이지 마라

앱 커넥션을 superuser나 테이블 owner로 두면 ① RLS가 우회되고(Lesson 01), ② 사고 시 피해가 무제한입니다 — 인젝션·버그 한 번에 스키마 전체가 날아갈 수 있습니다. 앱은 app_rw/app_ro 같은 최소 권한 롤로 붙이고, 스키마 변경(마이그레이션)은 별도의 owner 롤로 분리하세요. ALTER DEFAULT PRIVILEGES는 그 owner가 만든 새 객체에도 앱 권한이 자동으로 따라가게 해 줍니다.

4(PG18) 인증 — scram-sha-256과 OAuth 2.0

롤에 권한을 줬다면, 그다음은 "누가 이 롤로 접속할 수 있는가"를 검증하는 인증입니다. PostgreSQL의 기본 패스워드 인증은 scram-sha-256(평문·MD5보다 안전한 챌린지-응답 방식)입니다. 어떤 클라이언트가 어떤 방식으로 접속할지는 pg_hba.conf가 한 줄씩 규칙으로 정합니다(호스트·DB·유저·인증방식).

PostgreSQL 18의 새 기능: OAuth 2.0 인증 지원. 외부 IdP(Identity Provider, 예: Okta·Entra ID 등)가 발급한 토큰으로 DB에 접속할 수 있게 되어, 사내 SSO·중앙 집중 인증과의 연동이 한결 쉬워졌습니다. 패스워드를 DB마다 따로 관리하지 않고 조직의 인증 체계에 통합하려는 운영 환경에 유용합니다.

유저·권한 모델 MySQL

MySQL은 유저를 'user'@'host' 형태로 정의해 접속 출처(호스트)까지 식별합니다. 권한은 GRANT ... ON db.* TO 'u'@'%'처럼 부여하고, 그룹 개념 대신 롤(8.0+)을 별도로 도입했습니다. 스키마(=데이터베이스)에 대한 별도 USAGE 단계는 PG만큼 두드러지지 않습니다.

유저·권한 모델 PostgreSQL

PostgreSQL은 유저와 그룹을 ROLE 하나로 통합하고, 접속 출처는 pg_hba.conf에서 따로 다룹니다. 테이블 권한 전에 스키마 USAGE가 필요한 점이 MySQL에서 넘어온 팀이 자주 막히는 차이입니다. PG18부터는 scram-sha-256에 더해 OAuth 2.0 인증도 쓸 수 있습니다.

복습

5✅ 셀프 체크

Q ROLE에서 "USER"란 무엇인가?

Postgres에서 유저와 그룹은 구분 없이 모두 ROLE이며, USER란 LOGIN 속성을 가진 ROLE입니다. CREATE USER는 CREATE ROLE ... LOGIN의 별칭일 뿐입니다. NOLOGIN 롤은 직접 접속은 못 하지만 권한을 묶는 그룹으로 쓰이고, GRANT 그룹롤 TO 유저롤로 멤버십을 만들면 권한을 상속합니다.

Q 앱을 최소 권한 유저로 붙여야 하는 이유는?

두 가지입니다. ① 보안 — RLS 우회 방지: superuser·테이블 owner는 RLS를 우회하므로, 앱이 그런 롤로 붙으면 행 수준 격리가 무력화됩니다. ② 사고 시 피해 최소화: SQL 인젝션이나 버그가 있어도 app_rw가 가진 권한 범위(특정 테이블의 CRUD)로 피해가 제한됩니다. 스키마 변경 권한은 별도 owner 롤로 분리하는 것이 좋습니다.

Q ALTER DEFAULT PRIVILEGES는 언제 쓰나?

일반 GRANT는 이미 존재하는 객체에만 적용되므로, 마이그레이션으로 새 테이블을 만들 때마다 앱 유저에게 권한을 다시 줘야 하는 번거로움이 생깁니다. ALTER DEFAULT PRIVILEGES IN SCHEMA ... GRANT ... TO 롤은 "앞으로 (특정 롤이) 이 스키마에 만들 객체에는 자동으로 이 권한을 부여하라"는 예약 설정이라, 새 테이블에도 권한이 자동으로 따라갑니다.
