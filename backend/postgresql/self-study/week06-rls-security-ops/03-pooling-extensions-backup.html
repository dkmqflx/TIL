연결 풀링 · 확장
백업 · 회고

운영의 마지막 조각 — 연결을 풀로 아끼고(PgBouncer), 확장으로 기능을 더하고(pgcrypto·pg_trgm·pgvector), 정기적으로 백업합니다. 그리고 6주를 돌아봅니다.

“

운영의 마지막 조각: 연결을 풀로 아끼고(PgBouncer), 확장으로 기능을 더하고(pgcrypto·pg_trgm·pgvector), 정기적으로 백업한다. 그리고 6주를 돌아본다.

  • 연결 풀링(PgBouncer 풀 모드·풀 사이징·서버리스)을 이해한다
  • 확장 생태계(pgcrypto·pg_trgm·pgvector)를 안다
  • 백업(pg_dump·pg_basebackup·증분)을 알고 6주를 회고한다

1연결 풀링 — '연결은 비싸다'의 해법

🍔 택시 콜센터를 떠올려 보세요. 손님마다 차를 새로 사지 않고, 대기 중인 택시 풀에서 한 대를 배차하고 끝나면 돌려받습니다. DB 연결도 마찬가지입니다. Week 1에서 본 것처럼 PostgreSQL은 연결마다 OS 프로세스를 하나 띄워 연결 생성·유지 비용이 큽니다. 매 요청마다 새로 연결하면 금방 한계(max_connections)에 부딪힙니다.

PgBouncer는 앱과 DB 사이에 끼어 소수의 실제 연결을 재사용해 주는 경량 풀러입니다. 핵심은 풀 모드 선택입니다.

풀 모드연결을 언제 반납특징 · 주의
session클라이언트 연결이 끊길 때가장 안전(세션 상태 보존). 재사용 효율은 낮음
transaction트랜잭션이 끝날 때마다가장 널리 쓰임. 재사용 효율 높음. 단 세션 상태(prepared statement·SET 등)에 주의
statement문장 하나가 끝날 때마다가장 공격적. 멀티문 트랜잭션 불가

풀 사이징의 직관: 연결 수는 무작정 늘리는 게 아닙니다. 실제로 일하는 연결은 CPU 코어 수에 가깝게 두는 것이 좋고(보통 코어 수의 몇 배 이내), 그 위에서 PgBouncer가 수천 개의 클라이언트 연결을 소수의 서버 연결로 접어 줍니다. PgBouncer의 풀 크기 합은 DB의 max_connections보다 작아야 합니다.

⚠️ 서버리스의 connection storm

서버리스(Lambda·Edge 등)는 요청마다 인스턴스가 떴다 사라지며 저마다 새 DB 연결을 열어, 트래픽이 몰리면 순식간에 수천 연결이 폭증(connection storm)해 DB가 too many connections로 무너질 수 있습니다. 해법은 앞단에 외부 풀러(PgBouncer 등)를 두거나, 서버리스용 HTTP/풀링 드라이버(예: 매니지드 Postgres가 제공하는 연결 풀러)를 쓰는 것입니다. transaction 풀 모드가 이 시나리오의 단골 선택입니다.

2확장(extension) — Postgres에 기능을 끼워 넣기

PostgreSQL의 큰 강점은 확장 생태계입니다. CREATE EXTENSION 한 줄로 새 함수·타입·인덱스를 추가할 수 있습니다. 여기서는 공식 이미지의 contrib에 들어 있는 pgcrypto와 pg_trgm을 실제로 실행해 봅니다.

study_db=# — 확장 설치 & pgcrypto 비밀번호 해싱
CREATE EXTENSION IF NOT EXISTS pgcrypto;
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- 주의: gen_random_uuid() 는 PG13+ '코어' 함수다(pgcrypto 아님).
--       pgcrypto 고유의 가치는 crypt()/gen_salt()/digest()/hmac().

-- crypt + gen_salt(bcrypt): 같은 비밀번호도 salt 때문에 매번 다른 해시
SELECT crypt('s3cret', gen_salt('bf')) <> crypt('s3cret', gen_salt('bf'))
       AS two_hashes_differ;
 two_hashes_differ
-------------------
 t

-- 저장된 해시로 검증: 맞는 비번이면 같은 값으로 재계산된다
WITH h AS (SELECT crypt('s3cret', gen_salt('bf')) AS stored)
SELECT (stored = crypt('s3cret', stored)) AS right_password_ok,
       (stored = crypt('wrong',  stored)) AS wrong_password_ok
FROM h;
 right_password_ok | wrong_password_ok
-------------------+-------------------
 t                 | f

-- digest: SHA-256 은 원문 길이와 무관하게 32바이트 고정
SELECT length(digest('hello', 'sha256')) AS sha256_bytes;
 sha256_bytes
--------------
           32
study_db=# — pg_trgm 유사도(오타·유사 검색의 기반)
-- 트라이그램 유사도: 1 에 가까울수록 비슷(Week 3 검색에서 봤던 그 확장)
SELECT word, similarity(word, 'postgre') AS sim
FROM (VALUES ('postgresql'),('postgrey'),('mysql')) AS t(word)
ORDER BY sim DESC;
    word    |    sim
------------+-----------
 postgrey   |       0.7
 postgresql | 0.5833333
 mysql      |         0
(3 rows)

2026년 실무에서 점점 중요해진 확장이 pgvector입니다 — AI 임베딩을 저장하고 의미 기반 유사도 검색(RAG)을 합니다. Postgres가 별도 벡터 DB 없이 그 자체로 벡터 검색 엔진이 되는 셈입니다. 다만 한 가지 중요한 주의가 있습니다.

⚠️ pgvector는 공식 postgres 이미지에 없다 — 그래서 아래는 '실행하지 않는' 예제

vector 확장은 공식 postgres:18 이미지에 들어 있지 않습니다. 이 코드를 그대로 실행하면 CREATE EXTENSION vector에서 실패합니다. 쓰려면 pgvector/pgvector:pg18 이미지를 쓰거나, 확장을 따로 설치해야 합니다. 아래는 "이렇게 쓴다"는 개념 예시로, 이번 주차 검증 스크립트에서는 실행하지 않았습니다.

개념 예시 (pgvector 이미지 필요) — 실행 안 함
CREATE EXTENSION vector;                  -- 공식 postgres:18 엔 없음!

CREATE TABLE items (
  id        bigserial PRIMARY KEY,
  content   text,
  embedding vector(1536)                   -- 임베딩 차원(예: OpenAI)
);

-- 코사인 거리(<=>)로 질의 벡터와 가장 가까운 5개 (RAG 검색)
SELECT id, content FROM items
ORDER BY embedding <=> $1 LIMIT 5;

-- 근사 최근접(ANN) 인덱스: HNSW 또는 IVFFlat
CREATE INDEX ON items USING hnsw (embedding vector_cosine_ops);

3백업 — 복구 테스트까지 해야 진짜 백업

백업은 크게 둘입니다. 논리 백업(pg_dump: SQL/데이터를 추출)과 물리 백업(pg_basebackup: 데이터 파일 자체를 복사). 이 도구들은 SQL이 아니라 셸 명령이므로, 검증 스크립트에서는 개념으로 정리합니다(클러스터 버전만 SQL로 확인).

shell — 백업/복원 명령 정리
# 논리 백업: 커스텀 포맷(-Fc) → 압축 + pg_restore 로 선택 복원 가능
pg_dump -Fc study_db > study.dump
pg_restore -d study_db study.dump

# 롤(권한)·테이블스페이스 등 클러스터 전역 객체까지
pg_dumpall --roles-only > roles.sql

# 물리 백업: 클러스터 파일 전체(복제 구성·PITR 의 토대)
pg_basebackup -D /backup -Ft -z

# (PG17+) 증분 백업: 직전 백업 이후 바뀐 블록만
pg_basebackup --incremental=backup_manifest -D /backup2
study_db=# — 클러스터 버전 확인(증분 백업은 PG17+)
SHOW server_version;
        server_version
-------------------------------
 18.4 (Debian 18.4-1.pgdg13+1)
(1 row)

PITR(Point-In-Time Recovery)는 베이스 백업 + WAL 아카이브를 합쳐 "장애 직전 특정 시각"으로 복원하는 기법입니다. PG17부터는 pg_basebackup --incremental로 직전 백업 이후 바뀐 블록만 떠서 백업 비용을 크게 줄일 수 있습니다.

⚠️ 복구 테스트 없는 백업은 백업이 아니다

가장 흔한 운영 사고는 "백업은 돌고 있었는데 막상 복구해 보니 안 됐다"입니다. 백업 파일 존재 ≠ 복구 가능. 정기적으로 별도 환경에 실제로 복원해 보는 리허설까지 해야 진짜 백업입니다. 복구에 걸리는 시간(RTO)과 잃어도 되는 데이터량(RPO)을 미리 정하고, 그 목표를 만족하는지 검증하세요.

연결 · 기능 MySQL

MySQL은 연결을 스레드로 다뤄(프로세스가 아니라) 연결 비용이 상대적으로 낮고, 풀링 없이도 비교적 많은 연결을 버팁니다. 기능은 상당수가 엔진에 내장되는 경향이고, 백업은 mysqldump·xtrabackup 등을 씁니다.

연결 · 기능 PostgreSQL

PostgreSQL은 연결마다 프로세스를 쓰므로 다수 연결 환경에서 외부 풀러(PgBouncer 등)가 사실상 필수입니다. 대신 확장 생태계(pgcrypto·pg_trgm·pgvector·PostGIS …)로 기능을 자유롭게 더할 수 있고, 백업은 pg_dump(논리)와 pg_basebackup(물리·증분)으로 다룹니다.

5✅ 셀프 체크

Q PgBouncer는 어떤 문제를 푸나?

PostgreSQL은 연결마다 프로세스를 띄워 연결 생성·유지 비용이 크고, max_connections 한계도 있습니다. PgBouncer는 앱과 DB 사이에서 소수의 실제 연결을 재사용(특히 transaction 풀 모드)해, 수천 개의 클라이언트 연결을 소수의 서버 연결로 접어 줍니다. 서버리스의 connection storm처럼 연결이 폭증하는 환경에서 특히 중요합니다.

Q pgvector를 공식 postgres 이미지에서 바로 못 쓰는 이유는?

vector 확장이 공식 postgres:18 이미지에 포함돼 있지 않기 때문입니다(pgcrypto·pg_trgm은 contrib에 들어 있어 바로 됨). pgvector를 쓰려면 pgvector/pgvector:pg18 이미지를 쓰거나 확장을 직접 설치해야 하며, 그래서 CREATE EXTENSION vector는 이번 검증 스크립트에서 실행하지 않고 개념 예시로만 다뤘습니다.

Q pg_dump와 pg_basebackup의 차이는?

pg_dump는 논리 백업으로, 특정 DB의 스키마·데이터를 SQL/아카이브로 추출합니다(선택적 복원·이식에 유리). pg_basebackup은 물리 백업으로, 클러스터의 데이터 파일 자체를 복사합니다(복제 구성·PITR의 토대, PG17+는 증분 백업 가능). 즉 "내용을 다시 만들어 내는" 쪽이 pg_dump, "파일을 통째로 뜨는" 쪽이 pg_basebackup입니다.
